# 功能交付检查清单

## 适用范围

适用于 `SimpleWrite` 仓库中的日常功能开发、缺陷修复与文档整理任务，尤其是同时涉及 Avalonia 视图、ViewModel、交互链路和知识文档时。

## 交付前检查顺序

1. 先确认需求边界，只改和当前任务直接相关的代码与文档。
2. 优先做最小实现，避免顺手引入额外重构。
3. 改完立即做针对性验证，再做一次整体构建或测试收口。
4. 如果实现影响已有约定，同步更新 `Docs/README.md`、`Docs/Knowledge/README.md` 或相关知识文档。

## 代码检查

- 是否遵循当前仓库分层：View 负责界面和事件转发，业务逻辑留在 ViewModel 或业务层。
- 是否复用了现有入口，例如 `SimpleWriteMainViewModel`、`ShortcutManagerHelper`、`PluginCommandPatternProvider`、`RightSlideBar` 等，而不是新建平行链路。
- 是否保持 AOT 与跨平台约束，避免无边界反射、动态代码生成和只适用于单一平台的实现。
- 新增异步 I/O 是否为真正的 `async Task` 链路，而不是同步包装异步。
- 新增空值、路径、文件和平台判断是否做了明确防护。

## Avalonia / XAML 检查

- 新增或调整 `x:Name`、模板、样式、事件处理后，是否立即进行构建验证。
- 共享颜色和样式是否继续放在 `Brushes.axaml` 与 `MainStyles.axaml` 聚合入口中。
- 视图层是否只做 UI 行为，不把业务判断塞回 code-behind。
- 需要访问当前编辑器时，是否继续走 `TextEditorInfo` 或 `MainEditorView` 现有链路。

## Copilot / 业务设计检查

- 新增命令或聊天入口时，是否先确认应该归类到 `Knowledge/Design/` 里的既有设计约定。
- 是否区分“本地工具型命令”和“需要模型参与的命令”，避免职责倒灌。
- 调整日志、会话、工作区工具或消息片段结构时，是否同步评估历史兼容性与文档描述。

## 验证检查

- 有对应测试时，先运行受影响的测试，再根据需要执行更大范围测试。
- 没有现成测试时，至少执行一次相关项目构建，确认没有新增编译错误。
- 涉及第三方库 API 时，先确认当前仓库实际解析到的包版本，再按真实版本实现。

## 文档收口

- 当前任务沉淀出的可复用经验，是否已经写入 `Docs/Knowledge/`。
- 如果知识文档分类、入口或链接发生变化，是否同步更新 `Docs/README.md` 与 `Docs/Knowledge/README.md`。
- 如果原有文档和当前代码实现不一致，是否已按代码现状修正文档，而不是继续保留过时描述。

## 常见经验

- 任何 XAML 结构性改动后，都应立即运行一次构建。
- 事件处理与绑定上下文应显式做模式匹配和空值防护。
- 涉及第三方库 API 差异时，先确认本地实际还原版本，再决定实现方式。
- 文档是代码现状的索引，不是历史实现的备份；发现漂移后应尽快收敛更新。
