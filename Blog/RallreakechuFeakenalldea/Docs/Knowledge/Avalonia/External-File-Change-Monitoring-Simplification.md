# 编辑器外部文件变更处理简化方案

## 背景

`EditorViewModel` 当前使用 `ActiveFileChangeMonitor` 监听正在编辑的本地文件。当其他程序修改文件时，编辑器会根据当前保存状态自动重新加载，或询问用户是否使用磁盘内容。

当前实现为了处理监控期间再次到达的通知，引入了：

- `_isHandlingActiveFileChange`：阻止多个异步处理流程并发执行；
- `_hasPendingActiveFileChange`：记录处理期间是否又收到通知；
- `HandleActiveFileChangeSafelyAsync` 中的循环：存在待处理通知时再次检查文件。

这套机制能够尽量避免遗漏连续变更，但对当前业务场景而言复杂度偏高。

## 优化目标

本次优化以简单、可读和满足主要业务为优先：

- 每次只处理一次有效的外部文件变更；
- 已有处理流程尚未结束时，忽略后续重复通知；
- 保留短暂延迟和磁盘状态复查，过滤文件保存产生的密集通知；
- 不再为了极端连续修改场景维护补偿循环；
- 减少 `EditorViewModel` 中的状态字段和异步控制分支。

## 建议方案

### 1. 保留单次重入保护

继续保留 `_isHandlingActiveFileChange`。当一次外部变更正在处理时，新通知直接返回，不再记录待处理状态。

该保护仍有必要，因为 `FileSystemWatcher` 的一次保存操作可能产生多个通知。若完全取消保护，可能并发读取同一文件或重复显示确认面板。

### 2. 删除待处理循环

删除以下内容：

- `_hasPendingActiveFileChange` 字段；
- 收到重入通知时对该字段的赋值；
- `HandleActiveFileChangeSafelyAsync` 中的 `do/while` 循环；
- 如果不再承担独立职责，可将 `HandleActiveFileChangeAsync` 的单次处理逻辑合并到 `HandleActiveFileChangeSafelyAsync`。

优化后的处理模型为：

1. 已在处理时直接返回；
2. 标记正在处理；
3. 短暂等待，让外部写入尽量完成；
4. 再次确认标签、监控对象和磁盘状态仍然有效；
5. 自动重新加载，或显示一次确认面板；
6. 在 `finally` 中解除处理标记。

### 3. 保留延迟后的状态复查

建议保留当前 100 毫秒延迟，以及延迟后的以下检查：

- 目标仍是当前监控的 `EditorModel`；
- 目标仍是当前标签；
- `HasFileChangedOnDisk` 仍返回 `true`。

这部分不是连续事件补偿，而是对文件监控通知的基本过滤。它可以避免标签已经切换、通知已经过期或文件状态已经同步时继续执行 UI 操作。

### 4. 保持现有业务决策

单次处理后的业务规则保持不变：

- `SaveStatus == SaveStatus.Saved` 时，直接从磁盘重新加载；
- 存在未保存内容时，只显示一次外部变更确认面板；
- 用户选择忽略后，停止当前文件监控，避免持续提示。

## MVVM 职责建议

从长期设计看，文件监控、防抖和通知过滤更适合放在 `Business/FileHandlers` 下的服务中，ViewModel 只负责：

- 当前标签状态判断；
- 调用文件加载逻辑；
- 发起用户确认交互。

但本次建议不同时引入新的服务或抽象。先删除循环和多余状态即可，避免为了整理分层扩大改动范围。后续只有在多个 ViewModel 需要复用文件监控，或监控策略继续复杂化时，再考虑提取服务。

## 可接受的边界

采用单次处理后，存在一个明确且可接受的边界：

> 如果第一次外部变更仍在处理，外部程序又完成了一次新的真实修改，第二次通知可能被忽略。

当前业务中，用户通常只进行一次外部保存；重复通知更多来自底层监控机制，而不是连续业务操作。自动重新加载时通常也会读到磁盘上的最新内容。因此，不为这一低概率场景保留循环补偿。

如果未来确认存在高频外部写入、自动生成文件或多人同步编辑等需求，再重新引入防抖、事件合并或版本队列，并优先放入独立文件监控服务。

## 预期收益

- 外部变更处理从“循环补偿”变为“一次通知、一次检查”；
- 减少一个状态字段和一层异步方法职责；
- 降低阅读和维护 `EditorViewModel` 的成本；
- 保留并发保护，避免重复弹窗和并发重载；
- 设计取舍与当前实际业务复杂度保持一致。

## 后续实施范围

后续执行代码优化时，预计只涉及：

- `SimpleWrite/ViewModels/EditorViewModel.cs`

不修改：

- `ActiveFileChangeMonitor` 的文件监听方式；
- 外部文件变更确认面板；
- 文件读取和磁盘状态比较逻辑；
- 保存、自动保存及标签切换流程。

## 验证建议

代码优化完成后应验证：

1. 外部修改已保存文档时，编辑器能够自动重新加载；
2. 当前文档存在草稿修改时，只显示一次确认面板；
3. 一次外部保存产生多个监控通知时，不会重复加载或重复弹窗；
4. 用户切换标签后，旧标签的延迟通知不会影响当前标签；
5. 用户选择忽略后，当前文件不再持续提示；
6. 项目构建成功，现有文件打开、保存和标签切换行为不受影响。
