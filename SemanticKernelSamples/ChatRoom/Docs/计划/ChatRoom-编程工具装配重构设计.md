# ChatRoom 编程工具装配重构设计

## 背景

`AgentLib.ChatRoom` 中的角色数据与运行时工具应当分开：

- 角色定义和会话存档保存名称、提示词、模型配置等用户数据
- Roslyn、.NET CLI 和文件操作工具属于当前代码提供的运行时能力

当前实现通过 `ChatRoomRoleDefinition.Tools` 保存 `coding_workspace` 声明，再由角色工厂解析声明并创建工具。这使工具进入角色序列化链路，也让历史存档能够决定当前版本角色拥有哪些工具。

这不是需要的行为。编程角色的工具由代码定义：创建或恢复编程角色时直接装配当前版本的工具；保存角色时不保存工具；历史数据中的旧工具字段无条件忽略。

工作区变更也应保持简单。唯一通知链路为：

```text
ChatRoomManager
  → ChatRoomRole
      → 实现工作区感知接口的工具
```

普通工具不接收工作区通知。UI、角色工厂、持久化代码和其他服务不直接更新工具工作区。

## 设计目标

1. 创建编程角色时由代码直接装配工具
2. 角色定义、模板和会话存档不保存工具声明、工具实例或工具配置
3. 历史恢复按照当前代码重新装配工具，不受旧工具字段影响
4. 新建、动态添加和历史恢复使用同一角色创建入口
5. `ChatRoomManager` 是工作区变更的唯一入口和状态所有者
6. `ChatRoomRole` 只向实现可选工作区感知接口的工具转发通知
7. 普通工具完全不感知工作区变化
8. 工作区变更代码只存在于管理器、角色转发和具体感知工具三个小范围
9. 工作区切换失败时尽量保留原有可用工具资源
10. 聊天室关闭、切换取消和初始化失败时不泄漏外部进程

## 非目标

1. 不保存或迁移 `ToolDefinition`、工具 ID、工具参数和工具版本
2. 不根据角色存档中的工具声明决定运行时能力
3. 不引入工具 Resolver、按工具 ID 注册的 Provider 或配置驱动装配体系
4. 不引入第二个工作区状态中心
5. 不让能力快照、角色工厂、UI 或持久化代码参与工作区传播
6. 不要求所有工具实现工作区更新方法
7. 不新增任意 Shell 命令执行能力
8. 不重新设计 Roslyn 查询、文件编辑或 .NET CLI 命令的具体功能

## 核心设计

### 工具是运行时能力，不是角色数据

以下内容不得进入角色定义、模板或会话 JSON：

- `AITool` 实例
- `coding_workspace` 等工具声明
- Roslyn Language Server 配置和进程信息
- .NET CLI 工具实例及命令日志
- 当前工作区路径
- 工具等待、可用、失败或降级状态

稳定的角色种类标识属于角色身份数据，不属于工具声明。它只用于在恢复角色时选择对应的角色工厂。

若旧 JSON 中存在工具字段，反序列化过程只需容忍并无条件忽略：不映射到运行时模型、不校验、不迁移，也不参与角色种类或能力判断。新数据停止写出该字段。

### 创建角色时直接装配工具

`CodingAssistantRoleFactory` 或等价的编程角色创建入口负责表达“这是编程角色”，并直接创建带有编程工具的普通 `ChatRoomRole`。

```text
CodingAssistantRoleFactory
  → 创建 ChatRoomRole
  → 创建 Roslyn、.NET CLI 等运行时工具
  → 将工具加入角色
  → 返回运行时角色
```

工具名称、类型和装配关系由代码固定，不读取持久化工具列表，也不通过工具 ID 动态发现。

通用角色创建逻辑不需要识别 `coding_workspace`。若编程角色和普通角色存在共同创建代码，可以抽取基础方法，但不应为此增加工具声明解析层。

### 恢复角色时重新执行创建逻辑

```text
读取角色数据
  → 根据稳定的角色种类选择角色工厂
  → 工厂按当前代码创建角色并装配工具
  → 恢复消息和 AgentSession
```

若旧存档没有稳定角色种类，只能使用工具字段之外的模板或角色身份信息。若没有任何可用信息，则按普通角色恢复或报告无法识别，不能使用旧工具字段推断角色类型。

## 工作区变更设计

### 唯一通知链路

工作区只能通过 `ChatRoomManager` 修改：

```text
UI / ChatRoomService
  → ChatRoomManager.SetWorkspacePathAsync
      → ChatRoomRole.SetWorkspacePathAsync
          → IChatRoomWorkspaceAwareTool.SetWorkspacePathAsync
```

不允许以下调用：

- UI 直接通知角色或工具
- `ChatRoomService` 逐个通知角色或工具
- 角色工厂在工作区变化时重新创建角色
- 持久化恢复代码直接设置工具工作区
- 普通工具订阅全局工作区事件
- 额外服务维护另一份权威工作区路径并再次广播

### 可选工作区感知接口

工作区感知是一种可选能力，不属于所有角色工具的基础契约。建议增加独立接口：

```text
IChatRoomWorkspaceAwareTool
  - SetWorkspacePathAsync(workspacePath, cancellationToken)
```

需要感知工作区的工具同时实现角色工具接口和该接口：

```text
CodingWorkspaceRoleTool
  : IChatRoomRoleTool
  , IChatRoomWorkspaceAwareTool
```

与工作区无关的工具不实现该接口，也不增加空方法。这样新增普通工具时无需考虑工作区逻辑。

接口名称应表达“可感知工作区变化”，而不是把工作区更新强加到所有工具上。具体参数类型和可空语义结合现有 `WorkspacePath` 约定确定；清空工作区时应使用现有统一表达，不再增加另一套状态协议。

### ChatRoomManager 的职责

`ChatRoomManager` 是唯一工作区状态所有者，负责：

- 接受外部工作区变更请求
- 规范化并校验路径
- 串行化工作区变更
- 比较新旧路径，忽略相同规范化路径
- 通知当前所有角色
- 为动态加入的角色补发当前工作区
- 在通知成功后更新管理器保存的工作区状态
- 在通知失败时保留或恢复原工作区状态

管理器只认识 `ChatRoomRole`，不识别 `CodingWorkspaceRoleTool`、Roslyn 或 CLI 等具体工具。

### ChatRoomRole 的职责

`ChatRoomRole.SetWorkspacePathAsync` 只做薄转发：

1. 从角色已有工具中筛选实现 `IChatRoomWorkspaceAwareTool` 的实例
2. 将工作区路径和取消令牌传给这些工具
3. 普通工具不参与调用
4. 不识别任何具体工具类型
5. 不创建 Roslyn、CLI 或其他工作区资源
6. 不保存第二份权威工作区状态

角色内部若存在多种工具集合，工作区感知工具的枚举应集中在一个私有方法中，避免每次增加工作区相关功能都复制筛选逻辑。

### 工作区感知工具的职责

具体工具自行处理与自身相关的工作区变化，例如：

- 创建绑定新工作区的候选会话
- 启动 Roslyn Language Server
- 创建 .NET CLI 工具
- 更新文件工具的根目录
- 成功后替换旧资源
- 清空工作区时释放资源
- 初始化失败或取消时释放候选资源并保留旧资源

候选会话、先建后换、进程取消和资源释放属于具体工具内部实现，不进入 `ChatRoomManager` 和 `ChatRoomRole`。

如果多个编程工具需要共享同一 Roslyn 或 CLI 会话，可以在这些工具内部依赖一个范围很小的共享资源对象。该对象只服务于具体工具实现：

- 不作为工作区状态所有者
- 不被 UI、角色工厂或持久化代码调用
- 不额外广播工作区变化
- 只在收到工作区感知工具调用后更新内部资源

## 生命周期

### 创建和恢复角色

```text
ChatRoomManager 选择角色工厂
  → 工厂创建角色并直接装配工具
  → 管理器把当前工作区通知给新角色
  → 工具按需初始化工作区资源
  → 角色加入聊天室
```

新建、动态添加和历史恢复都使用该流程。没有工作区时，管理器按现有约定通知空工作区，工作区感知工具保持未初始化状态。

为避免角色加入后才发现工具初始化失败，建议先完成当前工作区通知，再把角色提交到角色集合。失败时释放候选角色和已创建工具，不留下半初始化角色。

### 设置或切换工作区

```text
ChatRoomManager.SetWorkspacePathAsync
  → 规范化并校验新路径
  → 获取工作区变更串行锁
  → 保存原工作区路径
  → 依次通知 ChatRoomRole
      → Role 只通知工作区感知工具
  → 全部成功后提交新路径
```

工具内部采用先建后换，因此单个工具初始化失败时应继续保留自己的旧资源。

如果部分角色已经更新、后续角色失败，管理器使用相同通知链路把原路径通知给已经更新的角色，执行回滚。回滚失败应记录完整错误并将聊天室标记为需要重新同步，不能静默保留角色间不一致状态。

首版可以按角色和工具顺序串行通知，以保持行为简单、顺序明确。除非有测量结果证明这里是性能瓶颈，否则不引入并行广播、版本竞争或复杂快照协议。

### 动态添加角色

工作区已经设置时，管理器在角色加入集合前调用角色的工作区更新方法。角色仍只通知自身实现工作区感知接口的工具。

普通角色即使包含大量工具，只要这些工具未实现工作区感知接口，就不会收到任何工作区调用。

### 移除角色

移除角色时释放角色及其工具。若多个工具共享底层会话，共享资源对象自行处理引用和最终释放，管理器不参与具体资源计数。

### 关闭聊天室

关闭顺序建议为：

1. 阻止新的角色发言和工作区变更
2. 捕获并提交需要持久化的角色、消息和会话状态
3. 释放角色及其工具
4. 工具取消进行中的 Roslyn 与 CLI 调用并释放外部进程
5. 等待持久化完成

工作区清理仍通过角色和工具自身的释放逻辑完成，不需要管理器绕过角色直接释放具体编程工具资源。

## 并发与资源安全

### 管理器串行化工作区变更

同一时间只允许一次 `ChatRoomManager.SetWorkspacePathAsync`。较晚请求等待较早请求完成，不允许多个调用并发覆盖角色工具状态。

工作区变更锁只由管理器持有。UI、角色和工具不各自实现一套全局切换版本控制。

### 工具内部串行化资源替换

工作区感知工具应协调以下操作：

- `SetWorkspacePathAsync`
- 工具调用
- `DisposeAsync`

具体工具必须保证：

- 较旧初始化结果不会覆盖较新状态
- 初始化失败时释放候选资源
- 切换取消时保留旧资源
- 工具调用不会访问已释放会话
- 重复释放安全

若需要调用租约，只在具体工具或其内部共享资源对象中实现，不把租约概念传播到管理器、角色或普通工具。

### Roslyn 与 CLI

若多个编程角色共享 Roslyn，需要验证：

- `RoslynLspClient` 请求 ID 和响应分发是否线程安全
- 多角色并发查询是否串线
- 单次请求取消是否影响其他请求
- 切换和关闭时如何排空进行中请求

如果客户端不能安全并发，应在编程工具内部串行化请求，而不是把并发协调放入 `ChatRoomManager`。

共享 `DotNetCliTools` 时，build/test 首版按工作区串行执行。每次执行返回独立调用 ID、退出码和日志快照，避免多个角色覆盖“最后日志”。这些行为同样封装在编程工具内部。

## 错误与可观测性

需要明确区分：

- 工作区路径无效
- 某个工作区感知工具初始化失败
- Roslyn 初始化失败且禁止降级
- Roslyn 初始化失败但允许仅使用 CLI
- 工作区切换被取消
- 工作区回滚失败

`ChatRoomManager` 只报告“哪个角色的工作区通知失败”和最终切换结果。具体 Roslyn、CLI 或文件工具错误由对应工具提供诊断，不让管理器依赖具体实现。

降级策略属于应用级运行时资源策略，不属于角色、模板或会话存档。日志不得记录 API Key、用户文件内容或不必要的本地绝对路径。

## 分阶段实施

### 阶段 1：移除工具持久化依赖

1. 停止向 `ChatRoomRoleDefinition.Tools` 写入 `coding_workspace`
2. 保存角色和模板时不再写出工具字段
3. 读取历史数据时无条件忽略旧工具字段
4. 删除根据工具声明决定角色能力的逻辑
5. 明确不依赖工具字段的稳定角色种类标识

### 阶段 2：创建时直接装配工具

1. 让 `CodingAssistantRoleFactory` 创建角色时直接加入编程工具
2. 让新建、动态添加和历史恢复使用同一工厂链路
3. 让普通角色创建逻辑不引用具体编程工具
4. 删除不再需要的工具 Resolver、Provider 和配置迁移代码

### 阶段 3：收敛工作区通知链路

1. 增加可选的 `IChatRoomWorkspaceAwareTool`
2. 让 `ChatRoomRole` 只通知实现该接口的工具
3. 让 `ChatRoomManager` 成为唯一工作区变更入口
4. 让动态添加角色由管理器补发当前工作区
5. 删除 UI、服务、工厂及其他位置直接更新工具工作区的代码

### 阶段 4：强化工具内部资源切换

1. 让编程工作区工具实现工作区感知接口
2. 在工具内部实现候选会话先建后换
3. 在工具内部协调调用、切换和释放
4. 按需要封装小范围共享 Roslyn 与 CLI 资源
5. 验证 Roslyn 并发和 CLI 命令串行化

### 阶段 5：补齐失败与回滚

1. 串行化管理器工作区变更
2. 工具通知失败时保留管理器旧路径
3. 对已成功更新的角色使用相同链路回滚
4. 暴露回滚失败和工具降级诊断
5. 增加切换、取消、回滚和关闭测试

## 测试策略

### 持久化边界

1. 保存编程角色时不写出工具声明或运行时工具
2. 历史旧工具字段不映射、不校验、不迁移
3. 修改代码中的工具集合后，恢复角色获得当前工具集合
4. 普通角色不会因旧数据包含 `coding_workspace` 而获得编程工具
5. 编程角色不会因旧数据缺少工具字段而丢失编程工具

### 角色创建

1. 新建编程角色时直接获得编程工具
2. 动态添加和历史恢复使用相同装配逻辑
3. 普通角色不获得 Roslyn 或 CLI 工具
4. 名称校验不会创建运行时工具

### 工作区通知

1. 管理器是唯一工作区变更入口
2. 角色只通知实现工作区感知接口的工具
3. 普通工具不会收到工作区通知
4. 一个角色中的多个感知工具都收到同一路径
5. 动态加入角色在提交前收到当前工作区
6. 相同规范化路径不会重复通知
7. 清空工作区只通知工作区感知工具

### 切换与资源安全

1. 工具初始化新工作区失败时保留旧资源
2. 某个角色通知失败时管理器保留旧路径
3. 部分角色成功后失败时，已更新角色收到旧路径回滚通知
4. 两次并发切换按管理器串行顺序执行
5. 工具调用进行中切换时不访问已释放资源
6. 聊天室关闭会释放角色工具和外部进程
7. Roslyn 多角色并发查询不会串线
8. CLI build/test 按工作区串行执行

所有新增异步 MSTest 测试使用 `[TestMethod(DisplayName = "...")]`，并配置硬超时，避免 Language Server、子进程或并发测试挂起测试进程。

## 影响文件建议

| 文件或区域 | 预期变化 |
|------------|----------|
| `CodingAssistantRoleFactory.cs` | 不再写入工具声明；创建角色时直接加入编程工具 |
| `ChatRoomRoleDefinition.cs` | 移除工具持久化字段；旧 JSON 字段无条件忽略 |
| 角色/模板序列化代码 | 不保存工具声明、工具状态或工作区信息 |
| `IChatRoomWorkspaceAwareTool.cs` | 新增可选工作区感知接口 |
| `ChatRoomRole.cs` | 集中筛选感知接口并转发工作区通知 |
| `ChatRoomManager.cs` | 唯一工作区入口、串行通知、动态角色补发和失败回滚 |
| `ChatRoomService.cs` | 只委托管理器，不直接通知角色或工具 |
| `RoleLobbyViewModel.cs` | 通过服务/管理器添加角色，不直接创建或更新工具 |
| `MentionParser.cs` | 使用纯名称校验，不创建临时角色和工具 |
| `CodingWorkspaceRoleTool.cs` | 实现工作区感知接口并在内部安全替换会话 |
| `CodingWorkspaceToolSession.cs` | 由具体工具或其内部共享资源对象管理 |
| `DotNetCliTools.cs` | 串行化命令并提供独立调用结果 |

## 验收标准

1. 角色定义、模板和会话 JSON 不保存工具声明或运行时工具
2. 历史工具字段不决定角色能力
3. 编程角色每次创建或恢复时由当前代码直接装配工具
4. 工作区变更只走 `ChatRoomManager → ChatRoomRole → 工作区感知工具`
5. 只有实现可选感知接口的工具收到工作区通知
6. UI、服务、角色工厂和持久化代码不直接更新工具工作区
7. 管理器不引用 Roslyn、CLI 或具体工作区工具类型
8. 角色不引用具体工作区工具类型，也不保存第二份权威路径
9. 工具在内部完成候选资源创建、替换、取消和释放
10. 工作区切换失败后管理器和角色工具不会静默处于不一致状态
11. 动态添加角色使用同一通知链路获得当前工作区
12. 聊天室关闭、切换取消和初始化失败不会泄漏外部进程
13. 相关构建和测试全部通过
