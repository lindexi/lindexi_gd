# ChatRoom 编码角色设计思考

## 背景

`AgentLib.ChatRoom` 已具备多角色会话、独立模型、按角色授权工具、工作区路径和人工审批能力。本次又将 `CodingwhejeedenochaDeelemjuher` 中基于 Roslyn Language Server 的只读代码理解能力收进 `AgentLib.ChatRoom`，可以查询解决方案项目、项目文件、符号定义、实现和引用。

这些能力为“专门用于写代码的角色”提供了代码理解基础，但距离可靠地修改代码仍有明显差距。编码角色不能只依靠提示词和文件写入工具；它需要形成“理解需求 → 探索代码 → 制定方案 → 修改文件 → 编译测试 → 复查结果”的闭环，并且必须把只读分析、工作区写入和命令执行划分为不同风险等级。

本文只讨论编码角色的设计方向，不实现角色、工具注入或 UI 代码。

## 设计目标

1. 让编码角色先理解工作区和现有约定，再修改代码
2. 将 Roslyn 符号查询与现有文件系统工具组合，减少纯文本猜测
3. 对写文件、删除文件、执行命令和修改项目配置建立明确审批边界
4. 每次修改后必须通过编译、测试或静态诊断验证，而不是只输出“已完成”
5. 将编码角色保持为普通角色，通过 `ChatRoomRoleDefinition`、独立模型、系统提示词和工具授权组合出编码能力
6. 允许管理者、测试角色或审查角色参与协作，但避免多个角色同时写同一工作区
7. Roslyn 等代码工具默认不向普通角色开放，只按角色和任务需要显式授权
8. 控制上下文规模，只把与当前任务有关的文件、符号和诊断提供给模型

## 非目标

1. 本阶段不实现编码角色预设或专用角色类型
2. 本阶段不实现文件编辑、Git、终端、构建或测试工具
3. 本阶段不让 Roslyn Language Server 直接修改代码
4. 本阶段不设计无人监督的自动提交、自动推送或自动发布
5. 本阶段不承诺支持所有编程语言；迁入的 Roslyn 能力主要面向 C#、Visual Basic 和部分 Razor 文件

## 当前能力分析

### 已具备能力

| 能力 | 当前基础 | 适合用途 |
|------|----------|----------|
| 多角色对话 | `ChatRoomManager`、`ChatRoomRole` | 编码、测试、审查角色协作 |
| 工作区路径 | `ChatRoomManager.WorkspacePath` | 限定文件工具访问范围 |
| 设置路径审批 | `WorkspacePathTools` | 用户确认目标代码仓库 |
| 文件系统工具 | `WorkspaceToolProvider` | 带行号读取、文本搜索、创建文件和基于快照的字符串替换 |
| Roslyn 项目目录 | `WorkspaceProjectCatalog` | 获取解决方案、项目和项目文件 |
| Roslyn 符号查询 | `RoslynAgentTools` | 查找定义、实现和引用 |
| .NET 构建与测试 | `DotNetCliTools` | 通过 `dotnet build` 和 `dotnet test` 验证代码修改 |
| 角色管理工具 | `ChatRoomRoleManagementTools` | 动态创建测试或审查角色 |
| 独立模型 | 每个角色独立模型配置 | 为编码角色选择更强模型 |
| 人工审批 | `HumanApprovalTool` | 包装高风险工具 |

### 仍缺少能力

| 能力 | 缺少内容 | 风险 |
|------|----------|------|
| 原子批量编辑 | 多处替换全部成功或全部回滚 | 当前批量替换可能部分成功 |
| 文件删除 | 路径约束、项目引用处理和审批 | 越权写入或误删文件 |
| 结构化编译诊断 | 从构建输出中提取项目、文件、行列和错误代码 | 模型需要自行阅读命令输出 |
| 测试发现与筛选 | 按项目、类型或方法发现和运行测试 | `run_tests` 当前只能执行整个目标 |
| Git 变更审查 | diff、状态和未提交改动保护 | 覆盖用户已有修改 |
| 指令发现 | 仓库规则、README、编辑器配置 | 生成代码不符合项目规范 |
| 任务状态 | 可观察的计划与步骤进度 | 长任务容易偏离目标 |

## 核心设计决策

### 决策 1：编码角色是具有特定配置和工具授权的普通 ChatRoomRole

编码角色在角色模型上与其他角色没有本质区别。它仍然是普通的 `ChatRoomRole`，通过配置、提示词和工具授权获得编码能力，不需要立即引入 `CodingChatRoomRole` 之类的子类。建议继续使用 `ChatRoomRoleDefinition`：

- 用 `SystemPrompt` 定义编码工作流和行为约束
- 用 `ModelProviderId`、`ModelId` 选择适合代码任务的模型
- 用 `ParticipationMode = MentionOnly` 避免无关任务中自动参与
- 按角色显式授权代码理解、编辑和验证工具
- 用 `MemoryContent` 保存稳定的团队规范，而不是保存当前任务细节

只有当编码角色需要独立生命周期、专用状态机或与普通角色不同的持久化数据时，才考虑新增类型。

### 决策 2：工具默认不可见，按角色显式授权

工具不应因为已经在聊天室或依赖注入容器中注册，就自动提供给所有角色。默认策略应是最小权限：角色未被显式授权的工具，对该角色不可见，也不能调用。

编码角色模板默认配置完成编码闭环所需的 Roslyn 查询、文件读取、受控写入和验证工具，但创建出的角色实例仍通过统一的显式授权机制获得这些工具。普通聊天角色模板默认不配置代码工具。测试角色和审查角色可按模板或当前任务需要配置 Roslyn 查询、文件读取、构建结果和测试结果等只读能力，但不因此获得工作区写入权。

| 角色 | Roslyn 查询 | 文件读取 | 工作区写入 | 构建与测试 |
|------|-------------|----------|------------|------------|
| 普通聊天角色 | 默认不授权 | 默认不授权 | 不授权 | 不授权 |
| 管理者角色 | 按需授权 | 按需授权 | 不授权 | 查看结果或按需运行 |
| 编码角色 | 模板默认配置 | 模板默认配置 | 持有写入租约时授权 | 模板默认配置 |
| 测试角色 | 按需授权 | 按需授权 | 不授权 | 按需授权 |
| 审查角色 | 按需授权 | 按需授权 | 不授权 | 查看结果或按需运行 |

这里的“默认不授权”不是在架构上限定 Roslyn 工具只能由编码角色使用。Roslyn 能力本质上是只读代码理解能力，可以安全地共享给多个经过授权的分析角色；需要严格保持单一持有者的是工作区写入能力。

### 决策 3：只读工具与写入工具分层

工具应按风险分为三层：

| 层级 | 工具示例 | 默认策略 |
|------|----------|----------|
| 只读分析 | 项目列表、文件列表、符号定义、引用、读取文件、文本搜索 | 可直接调用 |
| 工作区写入 | 字符串替换、整体写入、创建文件、删除文件 | 展示变更摘要后审批 |
| 命令执行 | 构建、测试、格式化、Git 操作 | 命令白名单、工作目录限制、超时与审批 |

`RoslynAgentTools` 属于只读分析层。它不应因为编码角色需要写代码而扩展为写入工具，否则会混淆生命周期和安全边界。

### 决策 4：一个工作区同时只允许一个写入者

聊天室可以有多个分析角色，但同一时间只能有一个角色持有工作区写入权。否则可能出现：

- 编码角色与修复角色同时修改同一文件
- 两个补丁互相覆盖
- 文件状态在补丁检查与应用之间发生竞争
- 构建和测试基于尚未完成的一组修改运行

建议引入“工作区写入租约”概念：

1. 角色准备修改前申请写入租约
2. 租约记录角色 ID、工作区、任务 ID 和开始时间
3. 其他角色只能读取或等待
4. 修改、构建和测试完成后释放租约
5. 取消、异常或超时也必须释放租约

首版可以不实现复杂锁服务，但编码流程和 UI 必须遵守单写入者规则。

### 决策 5：修改已有文件优先使用精确字符串替换

现有 `WorkspaceToolProvider` 已提供以下能力：

- `ReadFileLines`：按行读取并记录文件快照
- `ReplaceStringInFile`：要求旧文本唯一匹配后执行单处替换
- `MultiReplaceStringInFile`：顺序执行多处替换，但当前不保证原子性
- `WriteFileContent`：文件不存在时创建文件，文件已存在时要求先读取且快照未失效

修改已有文件时，编码角色应先调用 `ReadFileLines` 读取相关区域，再优先使用 `ReplaceStringInFile` 或 `MultiReplaceStringInFile`。这些工具通过文件快照和旧文本唯一匹配避免基于过期上下文误改文件。

`WriteFileContent` 已覆盖创建文件场景，因此当前不需要再增加独立的 `create_file` 工具。整体覆盖已有文件虽然受快照保护，但仍应作为无法使用精确替换时的后备方式。`MultiReplaceStringInFile` 允许部分成功，不能将其描述为原子补丁；调用后必须检查每项结果。

### 决策 6：先探索再计划，计划后才写入

编码角色应有明确阶段：

```text
收到任务
  → 确认工作区与需求
  → 读取仓库指令和项目配置
  → 获取解决方案与项目结构
  → 搜索相关概念和符号
  → 读取最少必要文件
  → 形成修改计划
  → 获取写入审批
  → 执行精确字符串替换或创建文件
  → 编译与测试
  → 审查 Git diff
  → 汇报结果和未解决风险
```

对于单文件、目标明确的小修改，可以缩短计划，但不能跳过工作区确认、相关代码读取和修改后验证。

### 决策 7：验证是完成条件，不是可选步骤

编码角色只有在满足以下条件之一时才能宣称完成：

1. 相关项目成功编译，并且相关测试通过
2. 无可运行测试，但文件诊断和项目构建通过
3. 环境阻止验证，角色明确报告阻塞原因、已执行命令和未验证范围

不能使用“代码看起来正确”代替验证。若构建存在与本次修改无关的既有失败，应区分：

- 本次修改引入的错误：必须修复
- 修改前已存在的错误：记录证据，不扩大任务范围
- 环境或依赖错误：给出可复现命令和输出摘要

### 决策 8：Git diff 是最终审查输入

完成修改和验证后，编码角色应读取当前 Git 变更，并检查：

- 是否只修改了任务需要的文件
- 是否意外包含生成物、密钥、本地绝对路径或临时文件
- 是否覆盖用户原有未提交修改
- 是否存在仅格式化导致的大范围无关 diff
- 新公共 API 是否有文档和测试
- 项目文件、包版本和目标框架是否被无意改动

编码角色默认不执行提交和推送。若未来支持，应将提交与推送设为独立、显式审批动作。

## 推荐工具集

### 1. 工作区与指令工具

- `set_workspace_path`：用户审批后设置工作区
- `get_workspace_status`：返回根目录、解决方案、Git 状态和当前写入者
- `find_instruction_files`：查找仓库级和目录级编码规则
- `read_instruction_file`：读取适用于目标文件的规则

### 2. 代码理解工具

使用本次迁入的：

- `get_projects_in_solution`
- `get_files_in_project`
- `code_search`
- `find_symbol`
- `find_all_references`

其中 `code_search` 实际基于 LSP `workspace/symbol`，更准确的描述是“符号名称搜索”。如果后续增加纯文本或向量搜索，应拆分为不同工具，避免模型误以为它可以按任意自然语言找到实现。

配合现有 `WorkspaceToolProvider`：

- `ReadFileLines`：按行范围读取，并可返回行号
- `FindFilesMatchingPattern`：正则或精确文本搜索

### 3. 修改工具

- `ReplaceStringInFile`：单处唯一文本替换
- `MultiReplaceStringInFile`：顺序执行多处唯一文本替换
- `WriteFileContent`：创建文件；覆盖已有文件时必须先读取且快照未失效

写入工具必须验证路径位于工作区内，并拒绝 `bin`、`obj`、`.git`、`.vs` 等目录。

### 4. 验证工具

- `run_build`：固定调用 `dotnet build`，可指定工作区内的解决方案或项目
- `run_tests`：固定调用 `dotnet test`，可指定工作区内的解决方案或项目

这两个工具由 `AgentLib.ChatRoom.Tools.Coding.DotNetCliTools` 提供，与 Roslyn 相关工具一起归类为编程角色工具。它们不放入通用 `AgentLib` 的 `WorkspaceToolProvider`，也不加入所有聊天室角色的默认工具集合，而是由角色授权机制按需注入。

命令执行工具不接受任意 Shell 字符串，只允许选择工作区内的解决方案或项目路径。当前阶段先返回 `dotnet` 原始输出、退出码和成功状态；测试发现、筛选运行、结构化诊断和 Git diff 工具留待后续实现。

## RoslynAgentTools 的生命周期

`RoslynAgentTools` 会启动外部 `roslyn-language-server` 进程，因此不能在每次角色发言时临时创建。建议生命周期为“工作区会话级”：

1. 用户确认工作区路径
2. 聊天室或工作区能力服务创建一个 `RoslynAgentTools`
3. 同一工作区中经过授权的编码、测试或审查角色共享其只读能力
4. 工作区切换时释放旧会话并创建新会话
5. 聊天室关闭时调用 `DisposeAsync`

工具实例的生命周期与工具对角色的可见性是两个独立问题。底层可以复用同一个 `RoslynAgentTools` 和 Language Server 进程，但在为角色构造工具集合时，只向经过授权的角色公开对应函数。实例共享不表示聊天室中的所有角色都能调用这些工具。

不建议将其加入 `ChatRoomAutoLoopRunner` 的默认工具列表，因为：

- 每个聊天室角色都会看到与任务无关的代码工具
- 默认注入无法自然管理外部进程生命周期
- 工作区尚未设置时无法启动
- 非编码聊天会承担额外启动和上下文成本

更合适的接入点是后续的“工作区能力服务”，由宿主负责创建、缓存和释放工具实例，再由角色模板或角色能力配置决定哪些工具对当前角色可见。

## 编码角色系统提示词结构

系统提示词不应堆叠具体项目知识，而应描述稳定工作方式。建议包含以下部分：

### 身份与职责

- 你是当前聊天室中唯一拥有代码写入权限的编码角色
- 你的职责是实现用户明确要求的代码变更
- 不主动扩大范围，不顺手重构无关代码

### 探索规则

- 修改前必须读取相关文件
- 不猜测路径；先查询解决方案和项目结构
- 查符号定义与引用时优先使用 Roslyn 工具
- 需要自然语言或文本定位时使用文本搜索，不把符号搜索当语义搜索
- 读取并遵守目标目录适用的指令文件

### 修改规则

- 使用最小补丁
- 不编辑自动生成文件
- 不覆盖用户已有修改
- 不添加未使用抽象、依赖或配置
- 新公共 API 必须有文档和测试

### 验证规则

- 修改后先检查文件诊断
- 再运行最小相关构建和测试
- 最后审查 Git diff
- 验证失败时修复根因，不能隐藏错误

### 协作规则

- 需求不明确时向用户提问
- 架构选择有明显分歧时让管理者或架构角色参与
- 修改完成后可 @测试角色 或 @审查角色，但写入权仍保留在编码角色
- 不让其他角色直接编辑同一工作区

## 与其他角色的协作方式

### 管理者角色

管理者负责：

- 澄清最终目标和优先级
- 判断是否需要拆分任务
- 在编码角色完成后汇总结论
- 决定是否让测试或审查角色继续检查

管理者默认不需要代码工具；确有分析需要时可以获得只读工具，但不应获得工作区写入工具，否则会破坏单写入者规则。

### 测试角色

测试角色使用只读代码工具、构建结果和测试结果，提出：

- 缺少的边界场景
- 测试命名和结构问题
- 可复现失败步骤
- 回归风险

测试角色可以建议补丁，但由编码角色执行。

### 审查角色

审查角色以 Git diff 和相关上下文为主要输入，关注：

- 正确性
- 安全性
- 可维护性
- 并发与资源释放
- API 兼容性
- 测试充分性

审查意见应回到编码角色处理，避免审查角色直接修改。

## 审批与安全模型

### 工作区边界

所有文件路径必须：

1. 转换为绝对路径
2. 解析符号链接或重解析点风险
3. 验证位于已批准工作区内
4. 对输出优先展示相对路径，避免泄露本地绝对路径

### 高风险操作

以下操作必须单独审批：

- 删除或移动文件
- 修改解决方案、项目文件和包版本
- 执行包安装、数据库迁移或发布命令
- 执行 Git reset、clean、checkout、commit、push
- 访问工作区外路径
- 运行来源不明的仓库脚本

### 命令限制

命令执行服务应提供：

- 固定工作目录
- 允许命令或结构化操作列表
- 超时和取消
- 标准输出长度限制
- 环境变量脱敏
- 后台进程回收
- 明确退出码

不应默认允许模型拼接任意 shell 命令。

## 上下文管理

编码任务容易产生大量文件内容和诊断。建议采用分层上下文：

1. **稳定上下文**：角色规则、仓库指令、目标框架
2. **任务上下文**：用户需求、当前计划、已完成步骤
3. **代码上下文**：相关符号、文件片段、调用关系
4. **验证上下文**：编译错误、测试失败、Git diff

每次发言只注入当前阶段需要的内容。例如修改阶段不应重复注入完整解决方案文件列表；验证阶段优先注入错误附近代码和本次 diff。

角色私有历史可以保留决策过程，但公开消息只应报告计划、变更摘要、验证结果和阻塞项，不公开冗长工具输出。

## 错误恢复策略

### Roslyn Language Server 启动失败

- 报告命令不可用或初始化失败
- 保留文件读取和文本搜索能力
- 不伪装为已完成语义分析
- 提示用户安装或配置语言服务器命令

### 补丁上下文不匹配

- 重新读取目标区域
- 检查是否存在用户并发修改
- 重新生成最小补丁
- 不退化为整文件覆盖

### 构建失败

- 先定位本次修改相关诊断
- 一次只修复一个明确根因
- 连续失败达到限制后停止，并向用户报告证据
- 不通过禁用警告、跳过项目或删除测试隐藏问题

### 测试失败

- 区分测试本身错误、实现错误和环境错误
- 记录具体测试名和断言差异
- 只修复与当前任务有关的问题
- 对已存在失败提供基线证据

## 推荐实施顺序

### 第一阶段：只读编码助手

1. 将 `RoslynAgentTools` 接入工作区会话生命周期
2. 建立按角色显式授权工具的机制，普通角色默认看不到 Roslyn 工具
3. 复用 `WorkspaceToolProvider` 已有的带行号文件读取和文本搜索工具
4. 提供编码角色模板，但不授予写入能力
5. 验证编码角色能正确定位项目、符号和相关文件，并验证其他角色只能在授权后调用这些工具

### 第二阶段：受控写入

1. 复用 `ReadFileLines`、`ReplaceStringInFile`、`MultiReplaceStringInFile` 和 `WriteFileContent`
2. 引入写入租约
3. 对每次写入展示变更摘要并审批
4. 明确批量替换允许部分成功，并要求逐项检查结果

### 第三阶段：基础验证闭环

1. 使用 `DotNetCliTools.run_build` 调用 `dotnet build`
2. 使用 `DotNetCliTools.run_tests` 调用 `dotnet test`
3. 将退出码和受限长度的原始输出反馈给编码角色
4. 建立最大修复轮次和失败退出规则

后续再增加测试发现与筛选、结构化编译诊断和 Git diff 审查，不作为当前阶段实现 `run_build`、`run_tests` 的前置条件。

### 第四阶段：多角色协作

1. 增加测试角色和审查角色模板
2. 由管理者协调角色发言
3. 保持编码角色为唯一写入者
4. 对最终 diff 形成独立审查结论

## 验收标准建议

后续实现编码角色时，至少应通过以下场景：

1. 用户未设置工作区时，角色先请求设置并等待审批
2. 角色能从 `.slnx` 找到目标项目和相关文件
3. 角色能通过符号查询找到定义和引用，而不是猜测文件路径
4. 角色修改已有文件时先读取，再使用唯一文本替换，不覆盖无关内容
5. 工作区外路径、`bin`、`obj` 和自动生成文件写入被拒绝
6. 两个角色同时请求写入时，仅一个获得写入权
7. 修改引入编译错误时，角色不会宣称完成
8. 相关测试失败时，角色报告具体失败并尝试有限次数修复
9. 最终 Git diff 不包含临时文件、本地绝对路径或用户敏感信息
10. 聊天室关闭或切换工作区后，Roslyn Language Server 进程被正确释放
11. 普通角色默认看不到 Roslyn 工具，测试角色或审查角色获得显式授权后可以复用同一工作区会话的只读能力
12. `run_build` 和 `run_tests` 只执行固定的 `dotnet` 子命令，并拒绝工作区外的目标路径

## 结论

专门用于写代码的角色在角色模型上仍然是普通的 `ChatRoomRole`，不需要新增继承体系。它与其他角色的差异来自系统提示词、模型配置和显式授予的工具集合；同时，它还需要受控的工作区生命周期、精确的代码理解工具、单写入者机制、补丁式修改、人工审批以及编译测试验证闭环。

`RoslynAgentTools` 和 `DotNetCliTools` 统一放在 `AgentLib.ChatRoom.Tools.Coding` 下，分别提供只读代码理解能力以及固定的 `dotnet build`、`dotnet test` 验证能力。它们默认不向普通角色开放，但可以按角色显式授权；底层工具实例可以按工作区共享，角色可见性必须保持隔离。文件读取、搜索、创建和精确替换继续复用通用 `WorkspaceToolProvider`，避免在 ChatRoom 层重复实现。后续再补充结构化诊断、测试筛选和 Git diff 审查。
