# ChatRoom 共享工作区工具架构设计

## 文档定位

本文对聊天室编程工具的归属和装配方式进行进一步收敛。

此前的 `ChatRoom-CodingAssistantRoleFactory-Plan.md` 和 `ChatRoom-编程工具装配重构设计.md` 已经明确：运行时工具不应进入角色定义、角色模板或会话存档。但此前方案仍通过 `ChatRoomRoleKind` 选择角色工厂，再为特定角色装配编程工具。

这种做法虽然移除了工具声明的持久化，却仍然把工具能力绑定到了角色类型。本文进一步确认：

> Roslyn、.NET CLI 和工作区文件操作属于当前聊天室所连接工作区的运行时能力，不属于某种角色的身份数据。

若本文与前述文档中“按角色种类选择工厂并装配编程工具”的内容冲突，以本文为准。前述文档中关于工具不持久化、工作区资源安全切换、并发控制和异步释放的结论仍然有效。

## 背景

当前实现为了让编程助手在会话恢复后重新获得编程工具，引入了以下链路：

```text
ChatRoomRoleDefinition.Kind
  → ChatRoomRoleFactory 判断角色种类
  → CodingAssistantRoleFactory 创建角色
  → 为角色装配 CodingWorkspaceRoleTool
  → CodingWorkspaceRoleTool 创建 CodingWorkspaceToolSession
```

这条链路要求角色定义持久化一个稳定种类。否则，模板转换为会话角色并生成新的 `RoleId` 后，系统无法判断恢复出的角色是否应该重新获得编程工具。

`ChatRoomRoleKind` 因此不是孤立问题。只要产品要求“部分角色获得一套工具，其他角色不获得”，系统就必然需要：

1. 一个稳定的角色分类或授权判据
2. 一段根据判据选择工具装配逻辑的代码
3. 模板复制和会话持久化对该判据的传播
4. 角色编辑时对该判据的保留
5. 恢复历史会话时对该判据的解释

把枚举替换为字符串、能力列表或工厂 ID，只会改变判据的表示形式，不会消除选择逻辑和工具耦合。

## 当前实现揭示的真实归属

### 工作区属于聊天室

`ChatRoomManager` 已经是工作区路径的唯一权威状态所有者。工作区由当前聊天室选择，而不是由单个角色分别选择。

工作区发生变化时，所有参与同一聊天室的 AI 角色面对的是同一个代码目录、解决方案和构建环境。因此，工作区能力天然具有聊天室作用域。

### 公共工具已经按发言上下文注入

`ChatRoomAutoLoopRunner.StepAsync` 在每次角色发言时统一创建并追加：

- 角色管理工具
- 设置工作区路径工具

`ChatRoomRole.SpeakAsync` 已支持通过 `additionalTools` 接收本次发言可使用的公共工具。这说明聊天室已经具备在不修改角色定义的情况下向所有 AI 角色提供运行时能力的通道。

编程工具可以复用同一通道，不需要先进入角色工厂，再保存到角色实例。

### 编程助手仍是普通角色

当前编程助手没有独立的运行时类型。它仍然使用 `ChatRoomRole`，主要差异是：

- 显示名称
- 系统提示词
- 默认参与模式
- 角色大厅中的描述和标签

这些都属于角色模板数据。编程助手可以继续作为一份面向软件开发任务优化的模板存在，但不需要成为特殊的运行时角色种类。

### 当前装配方式会复制重型资源

当编程工具归角色所有时，每个编程角色都可能分别创建：

- `CodingWorkspaceRoleTool`
- `CodingWorkspaceToolSession`
- `RoslynAgentTools`
- Roslyn Language Server 进程
- `DotNetCliTools`

多个角色连接同一聊天室工作区时，这会造成重复进程、重复索引、重复生命周期管理和更复杂的并发切换。

这些资源描述的是工作区，而不是角色，因此应尽量在聊天室作用域内创建和释放一次。

## 核心结论

### 工具能力按聊天室统一提供

只要宿主为当前聊天室启用了编程工作区能力，所有会调用模型的角色都获得同一类工作区工具。

人类角色不经过 LLM 发言链路，因此不会接收或调用这些工具，不需要额外的角色类型判断。

目标模型为：

```text
聊天室连接工作区
  → 工作区运行时创建当前代码定义的工具
  → 每次 AI 角色发言时统一注入工具
  → 角色根据自身提示词和任务决定是否调用
```

系统不再判断“这个角色是不是编程助手”，只判断“当前聊天室是否存在可用的工作区运行时”。

### 角色模板描述行为，不授予工具

角色模板继续保存：

- 角色名称
- 系统提示词
- 模型偏好
- 参与模式
- 记忆
- 管理者配置
- 技能文件夹

角色模板不保存：

- 角色种类
- 运行时配置 ID
- 工厂 ID
- 工具 ID
- 能力列表
- Roslyn 或 CLI 配置
- 工作区路径
- 工具实例和工具状态

“编程助手”模板通过提示词引导模型主动探索代码、修改文件、构建和测试，但模板本身不决定工具是否存在。

### 具体工具耦合只能存在于运行时组合边界

具体编程工具不可能在整个程序中完全没有引用，因为系统总要在某个位置创建它们。

合理的耦合边界是应用组合根和具体工具提供者：

```text
App 组合根
  → 配置聊天室工作区工具提供者
  → ChatRoomService 为当前聊天室创建运行时环境
  → ChatRoomManager 使用抽象的工具集合和生命周期契约
```

以下区域不应引用 `RoslynAgentTools`、`DotNetCliTools` 或“编程助手”概念：

- `ChatRoomRoleDefinition`
- `RoleTemplate`
- 角色模板复制逻辑
- 会话持久化逻辑
- `ChatRoomRole`
- 角色编辑 ViewModel
- 角色类型选择逻辑

具体工具提供者可以知道如何创建和释放 Roslyn、CLI 等资源，但它不根据角色类型选择工具。

### 不提前引入角色级授权模型

当前设计不以“不同 AI 角色拥有不同工具权限”为目标。现有工作区文件工具本身也不是按编程助手种类授权。

因此本次不新增：

- `[Flags]` 能力枚举
- `RuntimeProfileId`
- `RoleFactoryId`
- 按角色注册的工具 Provider
- 工具白名单或黑名单
- 多态角色定义

如果未来出现明确的安全需求，例如只允许部分角色写文件或运行构建，应单独设计通用的工具授权策略。授权策略应围绕工具、操作和会话安全边界建模，而不是重新引入 `ChatRoomRoleKind`。

## 目标架构

### 总体结构

```text
应用组合根
  → 创建聊天室运行时工具提供者

ChatRoomService
  → 为当前聊天室创建 ChatRoomManager
  → 为当前聊天室提供独立的运行时工具生命周期

ChatRoomManager
  → 持有唯一 WorkspacePath
  → 持有聊天室级运行时工具提供者
  → 切换工作区时更新提供者一次
  → 发言时提供当前工具集合
  → 关闭聊天室时释放提供者一次

ChatRoomAutoLoopRunner
  → 获取聊天室级工具
  → 追加角色管理和工作区控制工具
  → 作为 additionalTools 传给当前发言角色

ChatRoomRole
  → 保存角色定义
  → 管理模型、提示词、记忆和 AgentSession
  → 让角色局部默认文件工具使用聊天室工作区路径
  → 不识别编程角色种类
  → 不创建或拥有 Roslyn/CLI 运行时
```

### 聊天室运行时工具提供者

建议用一个职责很小的聊天室级对象表达以下语义：

- 根据当前工作区创建运行时工具
- 暴露当前可用的 `AITool` 集合
- 安全切换或清空工作区
- 管理重型资源和外部进程
- 在聊天室关闭时异步释放

`ChatRoomManager` 只依赖这些通用语义，不依赖具体工具类型。

首版不需要工具发现系统或按 ID 注册的插件体系。组合根可以直接配置一个具体实现，也可以通过简单工厂或委托为每个聊天室创建实例。

### 工具集合的来源

每次 AI 角色发言时，最终工具集合可以由以下来源合并：

1. `CopilotChatManager` 已有默认工具
2. 聊天室共享工作区工具
3. 聊天室角色管理工具
4. 设置工作区路径等本轮协调工具

按工具名称去重，并保留明确的优先级规则。聊天室共享工具不写入角色定义，也不缓存到角色模板。

### 共享能力不等于所有对象必须单例

“工具属于聊天室”描述的是能力归属和装配规则，不要求所有工具对象无条件共享。

需要区分两类状态：

#### 适合聊天室共享的重型资源

- Roslyn Language Server 进程
- 工作区索引和 Roslyn 客户端
- 同一工作区的构建与测试调度
- 工作区切换和外部进程生命周期

这些资源创建成本高，且描述同一个工作区，应由聊天室级运行时统一管理。

#### 可能需要角色隔离的调用状态

- 角色独立的 `AgentSession`
- 文件工具“先读取后写入”的快照
- 单个角色的模型和对话历史
- 需要区分调用者的临时日志视图

如果某项状态需要隔离，可以为每个角色创建轻量适配器，或在调用时传递调用上下文。但所有 AI 角色仍获得同一类工具，不通过角色种类选择是否装配。

现有 `CopilotChatManager` 的默认文件工具可以在首轮重构中保持角色局部实例，以保留读取快照隔离。本文首先迁移新引入的 Roslyn 和 .NET CLI 运行时，不要求同时重写全部默认文件工具。

## 工作区生命周期

### 创建聊天室

创建聊天室时：

1. 为该聊天室创建一个运行时工具提供者
2. 初始工作区为空，工具集合可以为空或处于未初始化状态
3. 创建普通角色不触发 Roslyn、CLI 或其他工作区资源初始化
4. 添加多个角色不会重复创建工作区运行时

### 设置工作区

设置工作区时：

```text
UI / ChatRoomService
  → ChatRoomManager.SetWorkspacePathAsync
      → 规范化并校验路径
      → 聊天室运行时工具提供者创建候选工作区会话
      → 成功后替换旧会话
      → 提交新的 WorkspacePath
```

不再按角色遍历并分别创建编程工具。

若角色内部仍保留现有默认文件工具，管理器可以继续用通用方式同步路径，但该操作不再负责选择或创建 Roslyn、CLI 等专用工具。

### 添加和恢复角色

添加新角色或恢复历史角色时：

1. 直接创建普通 `ChatRoomRole`
2. 注册模型提供商并恢复角色的 `AgentSession`
3. 不读取角色种类
4. 不创建角色私有编程工具
5. 角色下一次发言时自然获得当前聊天室的共享工具

因此，模板创建、动态添加和历史恢复不再需要专用角色工厂链路。

### 移除角色

移除角色时只释放角色自身的模型会话和角色局部状态。

移除任意角色都不应关闭聊天室共享的 Roslyn Language Server，也不应重建工作区工具集合。

### 切换工作区

工作区工具提供者继续采用候选资源先建后换：

1. 保留旧工作区运行时
2. 创建新工作区候选会话
3. 候选会话完全初始化后原子替换
4. 释放旧会话
5. 创建失败或取消时保留旧会话

由于共享资源只存在一份，不再需要对多个角色逐一通知、记录已更新角色并执行逆序回滚。

### 关闭聊天室

关闭顺序建议为：

1. 阻止新的发言和工作区切换
2. 停止并等待自动循环退出
3. 持久化角色、消息和 AgentSession 状态
4. 释放全部角色
5. 释放聊天室运行时工具提供者
6. 等待 Roslyn Language Server 等外部进程退出

共享运行时只释放一次。

## 数据模型和持久化

### 删除角色种类

从 `ChatRoomRoleDefinition` 删除：

- `ChatRoomRoleKind` 枚举
- `Kind` 属性

不使用其他字段替代它。特别是不增加 `RuntimeProfileId`、模板来源 ID 或工具能力字段。

### 历史 JSON

`System.Text.Json` 默认容忍未映射属性。删除 `Kind` 后，当前版本读取包含旧 `Kind` 字段的会话或模板时会忽略该字段。

后续保存会自然停止写出 `Kind`。不需要迁移枚举数值，也不需要根据旧值恢复工具。

旧数据中的工具声明同样继续无条件忽略。恢复后，所有 AI 角色根据当前聊天室运行时获得当前代码版本提供的工具。

### 模板

编程助手可以继续作为角色大厅模板存在。它可以是普通预置模板，也可以保持为当前进程注册的运行时模板，但两种方式都只影响模板展示和角色定义内容。

模板 ID 不参与工具装配，模板转换为会话角色后也不需要保留来源信息。

## 工厂和依赖收敛

### ChatRoomRoleFactory

删除角色种类后，不再存在多套角色创建策略。

如果 `ChatRoomRoleFactory` 和 `IChatRoomRoleFactory` 仅用于：

- 判断 `ChatRoomRoleKind`
- 选择 `CodingAssistantRoleFactory`
- 设置主线程调度器

则应优先删除这层抽象，由现有创建入口直接构造普通 `ChatRoomRole`。不要为了保留现有结构而留下只有一个实现、没有外部依赖边界的接口。

如果后续仍有独立且真实的角色创建依赖，再根据实际需求保留最小工厂，而不是预留按角色类型扩展点。

### CodingAssistantRoleFactory

`CodingAssistantRoleFactory` 不再创建带工具的运行时角色。

其中仍有价值的内容是：

- 编程助手默认系统提示词
- 默认名称和参与模式
- 角色大厅模板描述与标签

这部分可以收敛为普通模板提供者，或直接并入现有预置模板定义。名称应表达模板职责，不再使用 `RoleFactory` 暗示特殊运行时创建逻辑。

### CodingWorkspaceRoleTool

`CodingWorkspaceRoleTool` 不再属于某个角色。应将其职责调整为聊天室级工作区工具提供者，名称也应去除 `Role`。

它继续负责：

- 工作区切换
- `CodingWorkspaceToolSession` 生命周期
- Roslyn 降级策略
- 工具集合暴露
- 异步释放

但不再由角色工厂创建，也不再保存到 `ChatRoomRole`。

## 并发和资源安全

### Roslyn

共享 Roslyn 运行时前需要验证：

- 请求 ID 生成是否线程安全
- 响应分发是否能支持多个角色并发查询
- 单个调用取消是否会影响其他调用
- 切换工作区时如何等待或取消进行中的请求
- 关闭聊天室时是否能可靠终止外部进程

若客户端不能安全并发，应在具体工作区运行时内部串行化请求，而不是重新退回每角色一套 Language Server。

### .NET CLI

同一工作区的构建和测试首版继续串行执行，避免多个角色同时修改构建输出和占用相同中间目录。

当前“最后一次日志”若由多个角色共享，可能出现后一次调用覆盖前一次日志的问题。可以选择：

- 明确把最后日志定义为聊天室共享状态
- 为每次调用返回调用 ID，并按调用 ID 查询日志
- 为角色提供轻量日志视图

该问题属于调用结果建模，不应通过复制整套 CLI 工具规避。

### 文件工具

文件工具的读取快照可能承担“先读后写”的安全约束。若直接共享同一实例，角色 A 的读取可能允许角色 B 写入。

因此首版可以保留角色局部的默认文件工具实例，同时确保所有角色使用相同工作区路径并获得相同工具类型。后续若要共享文件工具底层服务，需要显式携带调用者上下文，不能丢失读取快照隔离。

## 安全边界

启用聊天室编程工作区能力意味着：

- 所有 AI 角色都可以看到工作区工具
- 所有 AI 角色都可能发起代码查询、文件操作、构建或测试
- 工具自身必须继续限制访问范围和命令范围
- 写入、构建、测试等敏感操作应沿用审批和路径校验机制

如果产品不能接受所有 AI 角色获得这些能力，就必须明确设计工具授权策略。该策略应直接回答：

- 哪个角色可以调用哪个工具
- 哪类操作需要审批
- 授权由用户、会话还是宿主决定
- 授权是否需要持久化

在这些需求尚未出现前，不通过“编程助手种类”隐式承担安全职责。

## 分阶段实施建议

### 阶段 1：移除角色分类

1. 删除 `ChatRoomRoleKind`
2. 删除 `ChatRoomRoleDefinition.Kind`
3. 删除模板复制中的 `Kind` 传播
4. 删除角色种类的序列化和往返测试
5. 让旧 JSON 中的 `Kind` 被自然忽略

### 阶段 2：建立聊天室级工具生命周期

1. 将 `CodingWorkspaceRoleTool` 调整为聊天室级工具提供者
2. 让每个聊天室只持有一个 `CodingWorkspaceToolSession`
3. 让 `ChatRoomManager.SetWorkspacePathAsync` 更新共享运行时一次
4. 让聊天室关闭时释放共享运行时一次
5. 保持候选会话先建后换和失败保留旧资源

### 阶段 3：统一发言工具注入

1. 由 `ChatRoomAutoLoopRunner.StepAsync` 获取聊天室共享工具
2. 与角色管理工具、工作区控制工具一起作为 `additionalTools` 传入
3. 简化 `ChatRoomRole.MergeTools` 的工具来源
4. 删除 `ChatRoomRole` 中编程运行时工具集合
5. 删除角色对 Roslyn/CLI 专用运行时的工作区通知和释放逻辑
6. 保留角色局部默认文件工具所需的工作区路径同步

### 阶段 4：收敛角色创建

1. 删除 `ChatRoomRoleFactory` 中按种类分派的逻辑
2. 评估并删除不再需要的 `IChatRoomRoleFactory`
3. 将编程助手创建逻辑收敛为普通模板提供逻辑
4. 让模板创建、动态添加和历史恢复都直接创建普通角色
5. 修正角色编辑时手工重建定义导致的其他字段丢失问题

### 阶段 5：验证共享资源并发

1. 验证 Roslyn 客户端多调用并发
2. 验证 CLI 构建和测试串行化
3. 明确共享日志语义
4. 验证工作区切换期间的调用取消和资源替换
5. 验证关闭聊天室不会遗留外部进程

## 测试策略

### 数据模型和持久化

1. 角色定义不再包含角色种类或运行时工具字段
2. 会话和模板 JSON 不写出 `Kind`
3. 包含历史 `Kind` 字段的 JSON 可以正常读取
4. 包含历史工具字段的 JSON 可以正常读取且字段被忽略
5. 模板转换不传播任何运行时装配判据

### 工具可用性

1. 设置工作区后，任意 AI 角色发言都获得当前编程工具
2. 普通助手、架构师、审查员和编程助手使用相同工具集合
3. 人类角色不会进入工具调用链路
4. 未设置工作区时不暴露依赖工作区的可执行工具，或工具明确返回未设置状态
5. 修改模板名称或提示词不会改变工具集合

### 生命周期

1. 一个聊天室最多创建一套 Roslyn 工作区运行时
2. 添加多个角色不会启动额外 Language Server
3. 移除角色不会关闭共享 Language Server
4. 切换工作区只创建一套候选运行时
5. 切换失败保留旧运行时
6. 关闭聊天室释放共享运行时和外部进程
7. 切换会话不会复用已释放的旧工作区资源

### 并发

1. 多个角色并发调用 Roslyn 查询不会串线
2. 一个调用取消不会错误取消其他角色的调用
3. 构建和测试按工作区串行执行
4. 工作区切换期间不会访问已释放会话
5. 重复关闭和重复释放保持幂等

所有新增异步 MSTest 测试应使用 `[TestMethod(DisplayName = "...")]` 并配置硬超时，避免 Language Server、子进程或并发测试挂起测试进程。

## 验收标准

1. `ChatRoomRoleDefinition` 不包含角色种类、工具声明或运行时配置标识
2. 编程助手只是普通角色模板，不存在专用运行时角色类型
3. 系统不根据 `RoleId`、角色名称、系统提示词或模板 ID 推断工具能力
4. 所有 AI 角色通过统一发言上下文获得当前聊天室工作区工具
5. `ChatRoomRole` 不创建、不持有、不释放 Roslyn 或 CLI 运行时
6. 每个聊天室最多持有一套重型工作区运行时资源
7. 添加、编辑、移除或恢复角色不会触发编程工具重新装配
8. 工作区切换只更新聊天室级运行时一次
9. 角色、模板和会话 JSON 不保存 `Kind`、工具 ID、工作区路径或工具状态
10. 旧 `Kind` 和旧工具字段可以被安全忽略
11. 具体编程工具耦合只存在于组合根和具体运行时提供者
12. 若没有明确授权需求，不引入角色级工具权限模型
13. Roslyn、CLI 和文件工具的并发与状态隔离边界经过测试确认
14. 聊天室关闭、切换取消和初始化失败不会泄漏外部进程
15. 相关构建和测试全部通过
