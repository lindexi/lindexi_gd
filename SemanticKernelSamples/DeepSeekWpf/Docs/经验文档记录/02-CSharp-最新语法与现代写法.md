# C# 最新语法与现代写法总结

## 1. 类型设计优先级

- 表达不可变配置、值对象、请求参数时，优先考虑 `record`。
- 需要属性变更通知、可编辑 UI 状态时，优先使用普通 `class` 并结合 `INotifyPropertyChanged`。
- 当对象语义强调“数据内容相等”而不是“引用相等”时，`record` 更合适。

## 2. `record` 的推荐用法

- 配置对象适合使用 `record` + `init` 属性，便于构造后保持稳定。
- 更新配置对象时优先使用 `with` 表达式，而不是手写 `Clone()`。
- 对外暴露配置快照时，使用不可变 `record` 可以减少共享可变状态带来的副作用。

## 3. 集合与对象初始化

- 优先使用集合表达式 `[]` 初始化空集合，提高可读性。
- 对象初始化器适合表达配置装配与 DTO 构造。
- 当默认值固定且语义清晰时，可在属性声明处直接赋默认值。

## 4. 模式匹配

- 使用 `is`, `switch`, 属性模式、关系模式和逻辑模式简化条件判断。
- 对空值、类型分支、状态分支的判断优先使用模式匹配，而不是层层 `if` 嵌套。

## 5. 空值处理

- 开启可空引用类型后，使用 `?`、`??`、`??=`、空条件访问等语法表达真实空值语义。
- 避免为了消除警告而滥用 `!`，应通过更清晰的流程控制来证明对象非空。

## 6. 异步与流式写法

- I/O 密集操作优先使用 `async`/`await`。
- 对逐步产生结果的数据流，优先考虑 `IAsyncEnumerable<T>`，适合聊天流式输出、日志流和增量处理场景。
- 取消控制优先通过 `CancellationToken` 贯穿接口边界。

## 7. 简化语法

- 文件作用域命名空间适合减少缩进层级。
- 目标类型 `new()` 适合上下文明确的构造场景。
- 表达式体成员适合单一表达式逻辑，但不要为了简短牺牲可读性。
- 原始字符串字面量适合多行模板、提示词和 JSON 片段。

## 8. 面向桌面应用的实践建议

- 配置对象用 `record`，界面状态对象用可通知的 `class`。
- 文件读取、序列化、网络请求尽量异步化，避免阻塞 UI 线程。
- 对流式 AI 输出，使用增量消息模型，而不是等待完整字符串后一次性回填。
- 把临时编辑状态标记为运行时状态，避免误持久化到业务数据文件中。

## 9. 在 MVVM 项目中的落地建议

- `ViewModel` 中保留命令和状态，配置模型使用不可变 `record`。
- 用 `with` 生成新配置快照并交给设置服务保存。
- 对聊天消息流采用“领域数据 + UI 临时状态”并存的设计，但要通过序列化忽略临时状态字段。
