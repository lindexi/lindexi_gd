# WPF MVVM 最佳实践总结

## 1. 分层原则

- `View` 只负责界面结构、样式、交互绑定，不承载业务逻辑。
- `ViewModel` 负责状态组织、命令编排、输入校验、页面级流程控制。
- `Model` 负责业务数据结构和持久化对象定义。
- `Service` 负责文件、网络、模型调用、日志、配置等基础设施能力。

## 2. 绑定与通知

- 所有界面状态统一通过 `INotifyPropertyChanged` 暴露。
- 可编辑集合优先使用 `ObservableCollection<T>`。
- 对命令可用性相关的属性，在变更后同步刷新命令状态。
- 输入框尽量使用 `UpdateSourceTrigger=PropertyChanged`，让交互状态及时反馈到 `ViewModel`。

## 3. 命令设计

- 按钮、菜单、快捷操作统一走 `ICommand`，避免把事件处理逻辑塞到代码隐藏。
- 长耗时操作使用异步命令，并在执行期间控制按钮禁用、显示状态和取消能力。
- 需要停止生成、撤销等交互时，优先通过 `CancellationTokenSource` 建模。

## 4. 可维护结构

- 为 `ViewModels/`、`Models/`、`Services/`、`Infrastructure/`、`Views/` 建立清晰目录。
- 把通用能力（如属性通知、命令实现）下沉到 `Infrastructure/`。
- 通过接口抽象 `IAiChatService`、`IChatRepository`、`ISettingsService` 等边界，便于后续替换真实模型调用。

## 5. WPF 界面实践

- 使用 `DataTemplate` 将 `ViewModel` 映射到具体 `View`，减少窗口级分支判断。
- 对列表、表单、消息流等区域采用组合式布局，避免单个 XAML 过度膨胀。
- 保持 `Window` 作为壳层，主业务 UI 尽量下沉到 `UserControl`。
- 通过资源字典或应用资源集中管理基础样式，保持统一视觉。

## 6. 状态与持久化

- 聊天记录、设置项等用户数据应落盘保存，并在启动时恢复。
- 持久化逻辑不要直接写在 `ViewModel` 内部，应通过仓储或服务完成。
- 目录、文件路径等运行时配置统一由设置服务维护。

## 7. 面向 AI 聊天场景的补充建议

- 聊天会话、消息、设置三类状态拆开建模，避免页面对象膨胀。
- 把“发送中”“可停止”“编辑消息”“继续历史对话”等状态显式建模成属性和命令。
- 先用可替换的模拟 AI 服务打通界面与流程，再替换真实大模型接入。
- 日志记录要覆盖发送、停止、保存设置、持久化等关键行为，便于后续排障。

## 8. 本项目落地建议

- 主窗口仅负责导航与承载页面内容。
- 聊天页负责会话列表、消息列表、输入区、消息编辑区。
- 设置页负责缓存路径、日志路径、模型名称等配置维护。
- 所有页面共享统一服务实例，由 `App` 作为简单组合根进行装配。
