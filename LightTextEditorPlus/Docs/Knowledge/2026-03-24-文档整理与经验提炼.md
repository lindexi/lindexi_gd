# 文档整理与经验提炼

## 背景与目标

当前 `Docs` 目录已经覆盖使用说明、维护约束、行为定义、实现定义、术语表、参考资料、问题列表和待办路线图，信息量很完整，但首次进入仓库时阅读成本偏高。此次整理的目标不是删减原始文档，而是把高频信息提炼成一份经验文档，帮助后续协作者快速找到正确入口，并理解各文档分别回答什么问题。

## 关键文件

- `README.md`
- `Docs/使用说明文档.md`
- `Docs/维护文档.md`
- `Docs/行为定义.md`
- `Docs/实现定义.md`
- `Docs/术语表.md`
- `Docs/参考文档.md`
- `Docs/待办.md`
- `Docs/Knowledge/README.md`

## 主要决策

### 按“问题类型”而不是“文件来源”整理文档

现有文档最适合按用途分成七类：

1. 产品总览：`README.md`
2. 对外使用：`Docs/使用说明文档.md`
3. 行为选择：`Docs/行为定义.md`
4. 实现约束：`Docs/实现定义.md`
5. 维护规则：`Docs/维护文档.md`
6. 概念统一：`Docs/术语表.md`
7. 补充资料：`Docs/参考文档.md`、`Docs/待办.md`、`Docs/问题列表/*`

这样整理之后，后续查阅者可以先判断自己面对的是“怎么用”“为什么这样设计”“代码里必须保持什么约束”还是“还有哪些未完成事项”，再进入对应文档，而不是从头到尾通读全部资料。

### 提炼一条稳定的阅读顺序

建议后续协作者按以下顺序阅读：

1. `README.md`：先了解项目能力边界、平台分层和核心入口。
2. `Docs/使用说明文档.md`：确认对外 API 的主路径和示例写法。
3. `Docs/行为定义.md`：处理选区、换行、行高、标点等设计分歧时优先查这里。
4. `Docs/实现定义.md`：落代码时核对换段字符、空文本、删除越界、缓存等底层约束。
5. `Docs/术语表.md`：避免“行/段”“光标偏移/文档偏移”“Bullet/Number Marker”这类概念混用。
6. `Docs/维护文档.md`：做跨平台 API、共享文件、分析器约束时再深入阅读。
7. `Docs/待办.md` 与 `Docs/问题列表/*`：用于判断当前能力缺口和历史问题背景。

### 统一经验入口，减少重复阅读

知识库文档适合沉淀“如何更快理解现有体系”，而不是复制原始文档全文。因此这份经验文档只提炼规律，不替代原始定义；当需要精确规则时，仍应回到对应原文。

## 修改摘要

### 文档结构归纳

从现有资料中可以归纳出以下高价值经验：

#### 1. 这个仓库的核心不是“控件”，而是“排版内核 + 多平台适配”

- `LightTextEditorPlus.Core` 负责平台无关的排版与编辑能力。
- `LightTextEditorPlus.Wpf`、`LightTextEditorPlus.Avalonia`、`LightTextEditorPlus.Skia` 分别负责平台接入。
- 平台层共享 API 通过 `Build/Shared/API` 与分析器约束保持一致。

后续阅读代码时，优先把“核心规则”和“平台实现”分开看，否则容易把平台差异误认为核心行为。

#### 2. 文档已经形成清晰分工，后续新增内容也应遵循分工

- 新增 API 示例，放到 `Docs/使用说明文档.md`。
- 新增设计取舍，放到 `Docs/行为定义.md`。
- 新增底层实现规则，放到 `Docs/实现定义.md`。
- 新增维护方式、目录规则、共享代码约束，放到 `Docs/维护文档.md`。
- 新增概念定义或命名统一，放到 `Docs/术语表.md`。
- 新增任务复盘与协作心得，放到 `Docs/Knowledge/*`。
- 新增缺陷细节，放到 `Docs/问题列表/*`；未完成规划继续放 `Docs/待办.md`。

这条分工规则本身就是最重要的整理结果，后续继续写文档时只要遵守它，文档量增加也不会失控。

#### 3. 文本库存在几条必须优先记住的底层约束

- 换段统一使用 `\n`，不要把 `\r\n` 当作内部标准。
- 空文本对外至少表现为一段一行，但内部按需创建，目的是性能优化。
- 编辑后文本会变脏，布局相关信息只有在完成布局后才可信。
- 删除超范围字符遵循 `string` 一样的严格越界语义，优先抛异常而不是静默截断。
- 字符属性大量依赖不可变对象与缓存，新增能力时要先考虑是否会破坏共享对象假设。

这些规则散落在 `README.md`、`Docs/实现定义.md` 和 `Docs/术语表.md` 中，整理后可视为实现层的共通知识。

#### 4. 行为层以 WPF 观感为基线，但目标是向 Word/PPT 靠拢

- 光标、换行、行距、标点分行等行为不是纯数学问题，而是排版产品取舍。
- 文档里已经明确“设计上以 WPF 为准，其他平台行为向 Word 和 PPT 靠拢”。
- 处理跨平台差异时，不应只问“某个平台怎么做”，还要问“当前库的统一行为基线是什么”。

这意味着行为变更前应优先查 `Docs/行为定义.md`，避免只凭某个平台现状直接改代码。

#### 5. 维护层的关键经验是“入口集中、实现分布、约束外置”

- `TextEditor` 作为大类保留统一入口，减少外部使用复杂度。
- 具体实现通过分布文件拆开，降低单文件维护成本。
- 共享 API 定义放在 `Build/Shared/API`。
- 平台 API 一致性不通过接口强绑，而通过分析器和约束文本维持命名与数量一致。

这套方式很适合当前项目：既保留轻量入口，又避免多平台代码完全复制失控。

#### 6. `Docs/术语表.md` 不是附录，而是避免误解的前置文档

文本排版领域里很多词天然容易混淆，例如：

- 行 与 段
- Char 与 Run
- 光标偏移量 与 文档偏移量
- 字族、字体、字型、字形
- BulletMarker 与 NumberMarker
- 合写字 与 连写字

后续写代码、写测试、写知识库文档时，应尽量回到 `Docs/术语表.md` 对齐表达，避免不同文档说的是同一件事却用了不同名字。

#### 7. `Docs/待办.md` 更像路线图和问题账本，不适合作为规则来源

`Docs/待办.md` 很适合了解项目当前边界、已完成能力、历史问题与修复线索，但它混合了功能规划、问题记录和优化事项。阅读时应把它当成“现状与方向”材料，而不是最终行为规范。真正的规则仍应以下列文档为准：

- 使用方式：`Docs/使用说明文档.md`
- 行为规则：`Docs/行为定义.md`
- 实现规则：`Docs/实现定义.md`
- 维护规则：`Docs/维护文档.md`

### 后续维护建议

1. 保持“原始定义文档 + 知识库经验文档”双层结构。
2. 新的知识库总结优先沉淀查阅路径、决策原因和避坑经验，不重复粘贴已有原文。
3. 如果后续 `Docs/待办.md` 继续膨胀，建议再拆分为“路线图”和“问题回顾”两类文档。
4. 如果 `Docs/问题列表` 数量继续增长，建议补一个索引页，按排版、命中测试、平台渲染、输入交互分类。

## 验证记录

- 手工阅读并交叉核对 `README.md`、`Docs/*.md` 与 `Docs/Knowledge/README.md` 的职责边界。
- 新增知识库文档并在知识库入口增加索引，确认文档可被后续协作者直接发现。
- 本次仅涉及 Markdown 文档整理，未修改运行时代码。

## 风险与后续建议

- 当前经验文档属于高层归纳，若原始规则后续发生变动，需要同步更新这里的阅读建议。
- 若后续希望进一步“瘦身”文档，优先做目录索引和分类导航，不建议直接删除原始设计说明。
