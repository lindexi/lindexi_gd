# Markdown 高亮跨平台构建修复

## 背景与目标

`LightTextEditorPlus.Highlighters` 中的 Markdown 高亮实现最初直接引入了 Avalonia 专有的绘制上下文、画刷和 Skia 相关类型，导致同一份共享代码在 `LightTextEditorPlus.Highlighters.Wpf` 项目下无法编译。

本次目标是尽量通过命名空间 `using` 别名让 Avalonia 与 WPF 共用同一份高亮逻辑，只在确实存在平台差异的绘制和颜色对象上使用条件编译。

## 关键文件与模块

- `LightTextEditorPlus.Highlighters/MarkdownDocumentHighlighter.cs`
- `LightTextEditorPlus.Highlighters/IDocumentHighlighter.cs`
- `LightTextEditorPlus.Highlighters/PlainTextDocumentHighlighter.cs`
- `LightTextEditorPlus.Highlighters/CSharpDocumentHighlighter.cs`
- `LightTextEditorPlus.Highlighters/OtherCodeDocumentHighlighter.cs`
- `LightTextEditorPlus.Highlighters/TextRunPropertySetter.cs`
- `LightTextEditorPlus.Highlighters/TextEditorColorCode.cs`
- `LightTextEditorPlus.Highlighters/WpfTextEditorDrawingContext.wpf.cs`

## 主要决策与原因

1. 优先用类型别名统一共享逻辑
   - 对 `RunProperty` 和绘制上下文统一使用条件 `using` 别名
   - 这样可以保留同一套高亮流程，减少 `.ava.cs` / `.wpf.cs` 分裂范围

2. 仅在平台真正有差异的地方使用条件编译
   - 文本前景色对象在 Avalonia 使用 `SolidColorSkiaTextBrush`
   - WPF 使用 `ImmutableBrush`
   - 背景矩形绘制分别调用各自平台的 `DrawingContext.DrawRectangle`

3. 为 WPF 增加最小绘制上下文类型
   - 新增 `WpfTextEditorDrawingContext.wpf.cs`
   - 结构与 Avalonia 版本保持一致，便于 `IDocumentHighlighter` 和各实现继续共用方法签名

4. 顺手修正共享代码中的 Skia 假设
   - `TextEditorColorCode` 原本也直接依赖 `SkiaTextRunProperty` 和 `SolidColorSkiaTextBrush`
   - 这里同样改为基于 `RunProperty` 别名和平台化前景色创建，避免 WPF 再次编译失败

5. 保持对较低目标框架兼容
   - `LightTextEditorPlus.Highlighters.Wpf` 目标框架是 .NET 6
   - 因此不能使用 `ArgumentException.ThrowIfNullOrWhiteSpace`
   - 改为显式 `string.IsNullOrWhiteSpace` 检查加 `ArgumentException`

## 修改点摘要

- 将 `IDocumentHighlighter` 的渲染上下文参数改为条件别名统一后的类型
- 将多个高亮器中的 `AvaloniaTextEditorDrawingContext` 改为共享别名
- 将 `MarkdownDocumentHighlighter` 中的 `SkiaTextRunProperty` 改为 `RunProperty` 别名
- 将 Markdown 标题粗细、URL 前景色、代码语言前景色、代码块背景绘制改为平台分支实现
- 新增 `WpfTextEditorDrawingContext.wpf.cs`，补齐 WPF 编译所需上下文类型
- 将 `TextRunPropertySetter` 改为使用 `RunProperty` 别名
- 将 `TextEditorColorCode` 改为使用 `RunProperty` 别名和平台化颜色画刷创建
- 修复 .NET 6 下不可用 API 与未使用变量警告

## 验证方式（构建/测试/手工验证）

- 运行构建
  - `dotnet build LightTextEditorPlus.Highlighters/LightTextEditorPlus.Highlighters.Avalonia.csproj -c Release`
  - 结果：成功
- 运行构建
  - `dotnet build LightTextEditorPlus.Highlighters/LightTextEditorPlus.Highlighters.Wpf.csproj -c Release`
  - 结果：成功

  - `dotnet test Tests/LightTextEditorPlus.Avalonia.Tests/LightTextEditorPlus.Avalonia.Tests.csproj -c Release`

## 后续建议

- 如果后续还要在共享高亮项目中加入平台相关渲染能力，优先先抽象出别名层，再决定是否需要拆分文件
- `IDocumentHighlighter` 目前的公共成员 XML 注释缺失较多，后续若继续维护该项目，可统一补齐文档注释以减少构建警告
- 若后续 WPF 也需要更复杂的高亮背景或前景叠加效果，可考虑继续沿用当前“共享逻辑 + 平台最小渲染适配”模式，而不是直接复制整份高亮实现
