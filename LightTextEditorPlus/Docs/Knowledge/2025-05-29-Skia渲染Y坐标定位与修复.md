# Skia 渲染 Y 坐标定位与修复总结

**日期**: 2025-05-29  
**任务**: 定位并修复 Skia 渲染链路中的 Y 坐标异常问题

## 背景与目标

在 `LightTextEditorPlus.Skia` 的实际渲染中，发现最终字符绘制位置、行范围以及选择范围的 Y 坐标存在异常，怀疑点一开始集中在基线计算。

本次任务的目标包括：
- 在 `LightTextEditorPlus.SkiaDemo` 中直接使用 SkiaSharp 对“微软雅黑”固定字号进行测量
- 对比直接 SkiaSharp 测量值与文本库内部 `SkiaCharInfoMeasurer`、`HorizontalSkiaTextRenderer` 的计算结果
- 检查 `LineLayoutData` 的行高与 Y 坐标是否符合预期
- 检查 `RenderInfoProvider.GetSelectionBoundsList` 返回的选择范围 Y 坐标是否正确
- 在明确定位问题后完成修复，并保留可复用的诊断输出

## 关键文件与模块

- Demo 入口: `Demos/LightTextEditorPlus.SkiaDemo/Program.cs`
- Skia 字符测量: `LightTextEditorPlus.Skia/Platform/SkiaCharInfoMeasurer.cs`
- Skia 横排渲染: `LightTextEditorPlus.Skia/Rendering/Core/HorizontalSkiaTextRenderer.cs`
- 水平布局计算: `LightTextEditorPlus.Core/Layout/HorizontalArrangingLayoutProvider.cs`
- 行布局数据: `LightTextEditorPlus.Core/Document/DocumentManagers_/ParagraphManagers_/Paragraphs_/Lines_/LineLayoutData.cs`

- 行高公式参考: `LightTextEditorPlus.Core/Layout/LineSpacing_/LineSpacingCalculator.cs`

## 主要决策与原因

### 1. 先验证 SkiaSharp 原始字体度量

没有直接假设是基线计算错误，而是先在 Demo 中单独对 `SKFontMetrics`、`MeasureText`、`GetGlyphWidths` 做探针输出。

在本机 “微软雅黑” 对应的 `Microsoft YaHei`、16 号字下，核心数据如下：
- `Ascent = -16.93`
- `Descent = 4.188`
- `Top = -16.578`
- `Bottom = 4.086`
- `LineHeight = 21.117`

这些值与 SkiaSharp 绘制结果一致，说明基线偏移本身没有明显错误。

### 2. 在整条渲染链路中增加控制台诊断输出

为了确认问题发生在哪一层，而不是只看最终结果，在以下类型中加入了大量调试输出：
- `SkiaCharInfoMeasurer`：输出字符的 FrameSize、FaceSize、Baseline、GlyphIndex
- `HorizontalSkiaTextRenderer`：输出实际绘制点、基线偏移、字体度量
- `LineLayoutData`：输出 `CharStartPoint`、`LineContentStartPoint`、`LineCharTextSize`、`LineContentSize`、`LineSpacingThickness`
- `RenderInfoProvider`：输出每次选择范围的首尾字符范围、结果范围、所在行的内容范围

这样可以将“字体度量错误”和“布局几何错误”分离开。

### 3. 将问题定位为行高小于字符高度

通过诊断输出发现，异常时不是基线值不对，而是行布局中出现了：
- `LineContentSize.Height = 19.283`
- 实际字符高度约为 `21.117`
- `LineSpacingThickness.Top = -1.834`

这意味着最终布局得到的行高比字符实际高度还小，导致字符和选择范围都被向上挤出行顶，形成负的或偏小的 Y 坐标。

因此修复策略不是调整基线，而是在水平布局阶段保证：**行高不能小于当前行字符实际高度**。

### 4. 选择范围需要与行内容范围求交

`RenderInfoProvider.GetSelectionBoundsList` 原先只对高度进行截断：
- `Height = Math.Min(textRect.Height, currentLineLayoutData.LineContentSize.Height)`

这只能限制高度，不能校正已经偏移的顶部坐标，因此在字符本身已经被上移时，选择框仍然会带着错误的 Y 值。

新的做法是：
- 先取首尾字符范围并求并集
- 再与 `currentLineLayoutData.GetLineContentBounds()` 求交

这样可以同时修正 Y、Height 以及越界部分，更符合“选择范围应落在当前行内容区域内”的语义。

## 修改点摘要

1. 完全重写 `LightTextEditorPlus.SkiaDemo`：
   - 增加 SkiaSharp 原始字体探针
   - 增加真实 `SkiaTextEditor` 渲染探针
   - 输出文档、段落、行、字符、光标、选择范围信息
   - 生成 `skia-font-probe.png` 与 `skia-editor-render.png`
   - 增加 `DiagnosisSummary`，直接输出首行、首字符、首个选择范围之间的 Y 差值

2. 在 `SkiaCharInfoMeasurer` 中增加字符测量调试输出。

3. 在 `HorizontalSkiaTextRenderer` 中增加最终绘制点与基线调试输出。

4. 在 `LineLayoutData` 中增加行布局调试输出，用于观察行起点、内容范围和行距厚度。

5. 在 `HorizontalArrangingLayoutProvider` 中增加保护逻辑：
   - 当计算得到的行高小于字符实际高度时，将行高提升到字符高度

6. 在 `RenderInfoProvider.GetSelectionBoundsList` 中：
   - 保留调试输出
   - 将选择范围改为与行内容范围求交
   - 对空选择返回空列表，并在调试模式下写日志而不是抛异常，以对齐现有测试约定

## 后续建议

1. `LightTextEditorPlus.SkiaDemo` 可以继续保留为 Skia 字体与布局探针，不建议回退为最小示例。
2. 若后续继续调查其他字体，可在 Demo 中加入不同字体族、不同字号、不同语言脚本的批量测量。
3. 若未来需要减少调试噪音，可以考虑为当前控制台输出增加统一的调试开关，而不是直接删除。
4. 竖排渲染链路也建议复用本次方法，补充同级别的行高、基线、选择范围诊断输出。
5. 如果继续扩展选择范围逻辑，优先保持“最终结果与行内容范围求交”的策略，避免仅裁剪高度导致的局部修复。

## 技术要点

1. 本次问题的根因不是 SkiaSharp 的基线计算，而是布局层允许行高小于字符高度。
2. 行布局一旦出现负的顶部行距，字符绘制、光标、选择范围都会同步出现 Y 偏移。
3. 选择范围的正确修复方式是几何求交，而不是单独限制高度。
4. 对复杂渲染问题，按“原始字体度量 → 字符测量 → 行布局 → 最终绘制 → 选择范围”的顺序逐层输出数据，定位效率最高。
