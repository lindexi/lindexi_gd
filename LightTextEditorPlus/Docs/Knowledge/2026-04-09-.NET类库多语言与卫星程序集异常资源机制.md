# .NET 类库多语言与卫星程序集异常资源机制

## 背景与目标

当前 `LightTextEditorPlus` 的异常消息主要直接写死为中文字符串。这对中文开发者友好，但不利于其他语言文化的开发者接入，也不利于后续按语言独立交付资源。

本次文档先总结 .NET 类库多语言的通用最佳实践，再给出本仓库在异常消息场景下的落地建议。目标是：

- 让中文成为默认中性资源；
- 后续可以按文化生成卫星程序集；
- 业务代码不再直接硬编码用户可见异常消息；
- 资源切换遵循 .NET 现有 `ResourceManager` 回退机制。

## .NET 类库多语言最佳实践

以下结论参考 Microsoft Learn 关于 `ResourceManager`、`NeutralResourcesLanguageAttribute`、卫星程序集的官方说明。

### 1. 使用 `resx` + `ResourceManager` 作为基础方案

对于 .NET 类库，最稳妥的字符串本地化方式仍然是：

- 使用 `.resx` 保存资源；
- 使用 `ResourceManager` 在运行时按 `CurrentUICulture` 取值；
- 让默认语言资源随主程序集发布；
- 让其他语言资源按文化生成卫星程序集。

这样可以直接复用 .NET 自带的文化回退、探测路径和程序集加载机制。

### 2. 显式声明中性资源语言

应在主程序集上声明：

- `[assembly: NeutralResourcesLanguage("zh-Hans")]`

这样做有两个价值：

- 告诉运行时默认回退文化是什么；
- 当默认资源放在主程序集时，减少 `ResourceManager` 的额外探测，提高查找效率。

如果未来默认资源也要独立打成卫星程序集，则可以改用带 `UltimateResourceFallbackLocation.Satellite` 的重载；但在常规类库里，默认资源放主程序集通常更简单。

### 3. 默认资源放主程序集，附加语言放卫星程序集

推荐采用 .NET 的 hub-and-spoke 模型：

- 主程序集：代码 + 默认中性资源；
- 卫星程序集：仅包含特定文化资源。

例如：

- `ExceptionMessages.resx`：中性中文资源；
- `ExceptionMessages.en-US.resx`：英语资源；
- 未来还可扩展 `ExceptionMessages.ja-JP.resx`、`ExceptionMessages.fr-FR.resx`。

SDK 风格项目在构建时会自动把文化资源生成到对应目录下的 `.resources.dll`，这正是后续支持卫星程序集的基础。

### 4. 不要在业务代码里直接写死可见消息

异常消息、提示文本、诊断说明等只要可能被调用方看到，就不应散落在业务代码中直接硬编码。

推荐做法：

- 资源文件中维护稳定的 key；
- 优先让 `resx` 自动生成强类型属性；
- 业务代码直接引用强类型资源属性；
- 资源 key 一旦公开使用，就尽量保持稳定，避免频繁改名。

这样做可以避免：

- 同一条消息在多个地方复制；
- 翻译时难以集中维护；
- 后续切换文化时出现遗漏。

### 5. 强类型资源优先于手写 provider

不要在各处直接拼 `ResourceManager` 和硬编码 key，也不建议再包一层手写 provider 方法。

更适合当前仓库的方式是：

- 让 `ExceptionMessages.resx` 自动生成 `ExceptionMessages.Designer.cs`；
- 让抛异常的位置直接引用 `ExceptionMessages.TextSpanCharObject_CharCountMustBeSurrogatePair` 这类属性；
- 仅在使用点做 `string.Format(CultureInfo.CurrentCulture, ...)` 这类参数格式化。

这样有几个直接收益：

- 抛异常的位置能直接看到资源成员名；
- 可以从代码直接跳转到生成属性和 `resx`；
- 不需要维护大量没有业务语义的转发方法；
- 对 `AllInOne` 这种源码合并项目也更容易复用同一套调用方式。

### 6. 资源查找使用 `CurrentUICulture`，格式化使用 `CurrentCulture`

这是类库里比较稳妥的约定：

- 选哪种语言：看 `CultureInfo.CurrentUICulture`；
- 数字、日期等格式化：看 `CultureInfo.CurrentCulture`。

对异常消息来说，文本语言属于 UI 语义，应该跟随 `CurrentUICulture`；而插值参数如果含数字、日期，则用 `CurrentCulture` 进行格式化更符合 .NET 习惯。

### 7. 资源粒度按“领域”划分，不按“类数量”失控拆分

不建议每个类一个资源文件，也不建议所有文案都堆进一个超大资源文件。

对类库更合适的方式通常是按领域拆分，例如：

- `ExceptionMessages.resx`
- `DiagnosticsMessages.resx`
- `PublicApiStrings.resx`

本仓库当前先从异常消息开始，优先建立 `ExceptionMessages` 这一层。

### 8. 为本地化回退行为编写测试

至少应覆盖两类行为：

- 当目标文化存在资源时，能拿到对应语言；
- 当目标文化不存在资源时，能回退到中性资源。

对类库来说，这类测试比手工检查更重要，因为它直接验证：

- 资源命名是否正确；
- 卫星程序集是否被正确构建与复制；
- 中性语言声明是否生效。

### 9. 面向 NuGet/复用类库时，优先考虑卫星程序集兼容性

如果后续需要把语言资源按文化独立部署，推荐继续沿用标准卫星程序集约定：

- 文化目录名必须与文化名一致；
- 卫星程序集命名遵循 `<主程序集名>.resources.dll`；
- 若未来进入强命名/GAC 场景，卫星程序集也需要对应签名策略。

这可以保证宿主应用、测试工程和未来 NuGet 分发都复用同一套加载行为。

### 10. 源码合并项目必须同步嵌入 `resx`

如果项目使用了类似 `AllInOne.Wpf`、`AllInOne.Avalonia` 这种“直接把别的项目源码编进当前程序集”的方式，仅仅 `Compile Include="..\LightTextEditorPlus.Core\**\*.cs"` 是不够的。

还必须同步嵌入对应的 `EmbeddedResource`：

- 否则强类型资源类虽然能编译通过；
- 但运行时会因为程序集内没有对应资源而找不到消息；
- 文化资源也不会生成对应的卫星程序集。

因此，源码合并项目要把被合并源码目录中的 `.resx` 一并纳入构建，并保持资源逻辑名与强类型资源类使用的基名一致。

## 本仓库建议的落地约束

### 第一阶段

先完成机制搭建，不大规模替换全部异常：

- 在 `LightTextEditorPlus.Core` 中声明中性资源语言为 `zh-Hans`；
- 新增 `ExceptionMessages.resx` 作为默认中文资源；
- 新增 `ExceptionMessages.en-US.resx` 作为试点英语资源；
- 让 `ExceptionMessages.resx` 自动生成强类型资源类；
- 先替换 `TextSpanCharObject` 中的异常消息；
- 补充测试验证中性回退与英语命中；
- 为 `AllInOne` 项目补齐资源嵌入。

### 第二阶段

在机制稳定后，再批量迁移其他异常类型：

- 按模块逐步替换硬编码消息；
- 补充资源 key 命名规范；
- 视需要增加更多文化资源；
- 在打包流程中检查卫星程序集输出。

## 本次试点的 key 设计建议

建议采用“类型 + 语义”的稳定命名方式，例如：

- `TextSpanCharObject_CharCountMustBeSurrogatePair`
- `TextSpanCharObject_CharCountMustBeOneOrTwo`

优点是：

- 能直接看出来源；
- 后续全仓迁移时不易冲突；
- 即使消息文案调整，key 也不必跟着变化。

## 验证建议

本次机制搭建完成后，至少验证：

- `CurrentUICulture = en-US` 时试点异常输出英文；
- `CurrentUICulture = fr-FR` 且无法语资源时回退到中文；
- 构建输出中存在对应文化目录和资源程序集；
- 主程序集在无卫星程序集时仍能返回中文中性资源；
- `AllInOne.Wpf` 与 `AllInOne.Avalonia` 的输出目录中也存在对应语言的 `.resources.dll`。
