# 键盘范围选择与按词选择

## 背景与目标

本次任务是为文本库补充键盘范围选择能力，支持 `Shift + 方向键` 扩展选区，并支持 `Ctrl + Shift + 左右键` 按单词扩展选区，同时保持 WPF、Avalonia 与 Core 的选择语义一致。

## 主要决策与原因

1. 将选择扩展语义放在 Core 的 `TextEditorCore.Select(SelectionType)` 中实现。
   - 原因：平台层只负责解析输入，选择锚点与终点计算应保持跨平台一致。
2. 扩展选择时始终保留 `CurrentSelection.StartOffset` 作为锚点；无选择时以当前光标作为锚点。
   - 原因：这与现有 `Selection` 的方向性设计一致，也和此前 `Shift + 鼠标点击` 的实现策略一致。
3. 复用现有 `KeyboardCaretNavigationHelper` 光标导航结果计算选择终点，并增加“忽略当前选择”的导航重载。
   - 原因：避免为选择逻辑复制一套按字符、按单词、按行的导航代码。
4. WPF 继续通过命令绑定处理键盘手势，Avalonia 继续在按键分支中解析修饰键。
   - 原因：遵循各平台现有输入接线方式，减少变更面。
5. 测试中对 `ControlShiftLeft` 场景优先断言范围边界偏移，而不是直接比较完整 `Selection`。
   - 原因：`Selection` 比较包含方向与 `CaretOffset.IsAtLineStart` 等附加状态，而本场景关注的是选择边界是否正确。

## 修改点摘要

- 为 Core 新增 `TextEditorCore.Select(SelectionType)` API，用于按键盘语义扩展选区。
- 扩展 `KeyboardCaretNavigationHelper`，允许在指定当前光标并忽略现有选择的条件下计算新的光标终点。
- 为共享 `TextEditor` API 暴露 `Select(SelectionType)` 入口。
- 为共享 `TextEditorHandler` 增加 `Select(SelectionType)` 部分方法。
- WPF 接入：
  - `Shift + Left/Right/Up/Down`
  - `Ctrl + Shift + Left/Right`
- Avalonia 接入：
  - `Shift + Left/Right/Up/Down`
  - `Ctrl + Shift + Left/Right`
- 补充 Core 测试，覆盖：
  - Shift 向右扩展
  - 已有反向选择继续扩展
  - Ctrl+Shift 向右按词选择
  - Ctrl+Shift 向左按词选择
- 补充 WPF/Avalonia 处理器测试，验证平台交互层可触发选择扩展。
- 顺带修正测试辅助类的隐藏成员警告，保持 Release 测试输出干净。

## 后续建议

1. 如果后续还要支持 `Shift + Home/End`、`Ctrl + Shift + Home/End`，建议继续复用 `SelectionType -> CaretMoveType` 的统一映射方式。
2. 如果 WPF 后续存在旋转文本区域的更多组合键行为，继续沿用 `CaretTransformDirectionHelper` 统一转换 `SelectionType` 与 `CaretMoveType`。
3. 若需要更贴近编辑器产品行为，可继续补充“先有选择再反向收缩”“跨段按词扩展”等更细粒度测试。
