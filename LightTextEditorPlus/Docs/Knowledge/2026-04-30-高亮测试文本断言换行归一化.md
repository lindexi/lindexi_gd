# 高亮测试文本断言换行归一化

## 背景与目标

`LightTextEditorPlus.Highlighters.Avalonia.UnitTests` 中部分测试会比较代码字符串或文档字符串。服务端拉取代码后，多行原始字符串可能因为工作区换行差异表现为 `\r\n`，而文本编辑器内部文本按维护约束统一为 `\n`，导致 `Assert.Equal` 直接比较失败。

本次目标是在不依赖 Git 换行配置的情况下，让测试中的文本内容断言忽略 `\r\n` 与 `\n` 差异，同时将该单元测试项目下的 `.cs` 文件统一保存为 CRLF。

## 关键文件与模块

- `LightTextEditorPlus.Highlighters.Avalonia.UnitTests/DocumentHighlighterTestHelper.cs`
  - 增加统一文本断言入口 `AssertTextEqual`。
  - `AssertTextPreserved` 复用该入口。
- `LightTextEditorPlus.Highlighters.Avalonia.UnitTests/*.cs`
  - 将直接比较编辑器全文内容的断言改为统一文本断言。
  - 单行、无换行符的普通 token 断言保持原样。

## 主要决策与原因

- 文本比较时把 `\r\n` 与单独 `\r` 都归一为 `\n`。
  - 符合维护文档中“换段统一使用 `\n`，不要把 `\r\n` 当作内部标准”的约束。
- 只替换编辑器文本内容相关的 `Assert.Equal(expected, actual)`。
  - URL、token、枚举、数值等单行或非文档内容断言不做放宽，避免降低测试精度。
- 不修改生产代码。
  - 问题来自测试源文件换行与内部文本标准差异，测试断言层处理即可。

## 修改点摘要

- 新增 `DocumentHighlighterTestHelper.AssertTextEqual(string expectedText, string actualText)`。
- 新增私有 `NormalizeLineEndings` 方法用于断言前归一化换行。
- 更新 `AssertTextPreserved` 使用归一化文本断言。
- 批量替换高亮 Avalonia 单元测试项目中针对编辑器全文内容的直接 `Assert.Equal`。
- 将 `LightTextEditorPlus.Highlighters.Avalonia.UnitTests` 下 `.cs` 文件统一保存为 CRLF。

## 后续建议

- 后续新增“编辑器文本内容保持不变”类测试时，优先使用 `DocumentHighlighterTestHelper.AssertTextPreserved` 或 `AssertTextEqual`。
- 若其它测试项目也存在类似多行代码字符串与编辑器内部文本比较，应在对应测试项目建立同样的测试辅助断言，而不是修改 Git 换行配置。
