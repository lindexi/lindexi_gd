# UTF-16 索引与文档偏移转换体验优化

## 背景与目标

TextEditor 内部使用两套索引体系：UTF-16 索引（外部库如 Markdig 使用）和文档字符偏移（内部 `DocumentOffset`/`Selection` 使用）。两者之间的转换逻辑散落在 4 个地方且无公开 API，导致高亮器作者等上层开发者无法复用。

目标：按照 `通用参考/UTF16索引与文档偏移转换体验优化分析.md` 的实施计划，建立三层公开 API 并统一内部调用。

## 关键文件

- `LightTextEditorPlus.Core/Utils/TextIndexConverter.cs` - 新增：权威转换工具类
- `LightTextEditorPlus.Core/Document/Segments/DocumentOffset.cs` - 修改：添加 `FromUtf16Index` 工厂方法
- `LightTextEditorPlus.Core/API/Extensions_/TextEditorCoreTextExtensions.cs` - 修改：添加 `CreateDocumentOffsetFromUtf16Index` 扩展方法
- `LightTextEditorPlus.Highlighters/MarkdownDocumentHighlighter.cs` - 修改：替换内部 `ConvertUtf16IndexToDocumentOffset` 调用
- `LightTextEditorPlus.Highlighters/TextRunPropertySetter.cs` - 修改：替换内部 `GetDocumentCharOffset` 调用
- `LightTextEditorPlus.Highlighters.Avalonia.UnitTests/DocumentHighlighterTestHelper.cs` - 修改：替换内部转换方法
- `LightTextEditorPlus.Highlighters.Avalonia.UnitTests/MarkdownCodeBlockHighlightingTests.cs` - 修改：替换调用

- `Docs/Knowledge/领域与技术参考.md` - 修改：补充 `TextIndexConverter` 使用说明
- `Docs/Knowledge/README.md` - 修改：新增索引条目

## 主要决策

1. **三层 API 设计**：`TextIndexConverter`（权威实现）→ `DocumentOffset.FromUtf16Index`（工厂方法）→ `TextEditorCore` 扩展方法（便捷入口）
2. **不引入 `Utf16StringIndex` 类型包装**：`DocumentOffset` 的隐式转换会绕过类型检查，且混淆场景仅限于高亮器作者，三层 API 已足够
3. **`DocumentOffset.FromUtf16Index` 使用 `global::` 前缀**：避免 `DocumentOffset` 所在命名空间 `LightTextEditorPlus.Core.Document.Segments` 与 `Utils` 的命名空间冲突

## 修改摘要

- 新增 `TextIndexConverter` 三个公开方法：`ConvertUtf16IndexToDocumentOffset`、`ConvertDocumentOffsetToUtf16Index`、`GetDocumentLength`
- `DocumentOffset` 新增 `FromUtf16Index(string, int)` 静态工厂方法
- `TextEditorCoreTextExtensions` 新增 `CreateDocumentOffsetFromUtf16Index` 扩展方法
- 移除 `MarkdownDocumentHighlighter.ConvertUtf16IndexToDocumentOffset`（internal static）
- 重构 `TextRunPropertySetter.GetDocumentCharOffset` → `GetDocumentCharOffsetFromPlainText`，委托给 `TextIndexConverter`
- 移除 `DocumentHighlighterTestHelper.GetDocumentOffsetFromUtf16Index` 和 `GetDocumentLength`，替换为 `TextIndexConverter` 调用
- 新增 18 个单元测试覆盖：纯 ASCII、emoji、`\r\n` 折叠、混合、`\uFFFC`、边界条件

## 验证记录

- `dotnet build` 全量通过
- `dotnet test -c Release --filter "FullyQualifiedName~TextIndexConverter"` 18/18 通过
- 现有高亮测试通过回归验证

## 风险与后续建议

- `TextRunPropertySetter.GetDocumentCharOffsetFromPlainText` 的行为与原来完全一致（委托给 `TextIndexConverter`），无回归风险
- 建议后续在 `TextEditorColorCode` 等直接使用 `new DocumentOffset(int)` 的地方改用 `DocumentOffset.FromUtf16Index`，但非强制
