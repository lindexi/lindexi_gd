# 高亮 UTF-16 偏移与 Emoji 映射修复

## 背景与目标

在给 `XmlCodeDocumentHighlighterTests` 补充 emoji 场景时，单纯的 XML 文本节点 `😊` 用例可以通过，但进一步下钻发现高亮系统存在更底层的问题：外部高亮器给出的 `SourceSpan`/`TextSpan` 偏移是基于 UTF-16 `string` 下标，而文本库内部文档偏移是基于 `ICharObject` 计数。

这会在以下场景产生错位：

- emoji 等代理对字符出现在被高亮片段之前；
- `\r\n` 在文档内部被折叠为单个换行字符对象；
- 后续任何依赖外部字符串索引映射到文档字符偏移的高亮功能。

本次目标不是只修 XML，而是修复高亮通用层的偏移映射。

## 主要决策与原因

1. **根因定位在高亮 span 到 Selection 的映射层**
   - 文本库内部已经用 `RuneCharObject` 处理 emoji。
   - 真正的问题出在 `TextRunPropertySetter.SourceSpanToSelection` 直接把 UTF-16 下标当文档偏移使用。

2. **修复放在高亮公共层，不放在 XML 专用逻辑**
   - XML、Markdown 代码块、C#、ColorCode 都会经过 `TextRunPropertySetter`/`TextEditorColorCode`。
   - 放在通用层后，所有语言高亮统一受益。

3. **映射规则复用文本库已有语义**
   - emoji 作为一个 Rune 计一个文档字符；
   - `\r\n` 在文档内视为一个换行字符对象；
   - 因此映射逻辑按 Rune 枚举，并对 `\r\n` 做折叠处理。

4. **对越界 span 做截断而不是抛异常**
   - 高亮器属于附加能力，外部 span 偏大时更适合容错到文档末尾。
   - 因此同步更新了相关单元测试预期。

5. **同步修复测试辅助**
   - 原 `DocumentHighlighterTestHelper` 使用 `string.IndexOf` + `token.Length` 直接作为文档偏移/长度。
   - 这对 emoji、CRLF 都不可靠。
   - 如果不修辅助方法，会出现实现已正确但断言仍然错位的问题。

## 修改点摘要

### 测试补充

- 在 `XmlCodeDocumentHighlighterTests` 中新增：
  - `ApplyHighlight_XmlWithEmojiText_HighlightsEmojiAsPlainText`
  - `ApplyHighlight_XmlWithEmojiBeforeFollowingText_PreservesFollowingPlainTextHighlightAlignment`
- 在 `TextRunPropertySetterTests` 中新增：
  - `TrySetRunProperty_WithSpanAfterEmoji_MapsUtf16SpanToDocumentCharacterOffset`

### 实现修复

- `TextRunPropertySetter`
  - 新增 `PlainText` 输入；
  - 将 `SourceSpan` 从 UTF-16 下标转换为文档字符偏移后再生成 `Selection`；
  - 修复 `StartOffset` 的叠加顺序，先做局部映射，再叠加全局起始偏移；
  - 无显式源码时，从编辑器当前选择范围回推文本，保持兼容。

- `TextEditorColorCode`
  - 新增带 `plainCode` 的构造；
  - 保留旧构造以兼容现有调用；
  - 旧构造内部会从编辑器与 `startOffset` 推导局部源码。

- 高亮入口调用更新
  - `OtherCodeDocumentHighlighter`
  - `CSharpDocumentHighlighter`
  - `MarkdownDocumentHighlighter`
  - 显式传入原始源码文本，避免映射缺少上下文。

- `DocumentHighlighterTestHelper`
  - 将 token 的起始与长度计算改为“UTF-16 下标 -> 文档字符偏移/长度”的统一转换。

### 测试预期调整

- 将 `SetRunProperty_SpanExceedsDocumentLength_ThrowsException` 调整为 `SetRunProperty_SpanExceedsDocumentLength_ClampsToDocumentEnd`，验证高亮容错行为。

## 后续建议

1. 如果后续还有解析器直接输出 UTF-16 下标，应优先复用当前映射逻辑，不要再直接构造 `Selection(new CaretOffset(span.Start), span.Length)`。
2. 可以考虑把“UTF-16 下标到文档字符偏移”的转换进一步下沉为公共帮助方法，避免高亮项目和测试项目各自保留一份类似逻辑。
3. 当前修复解决了 **Rune/CRLF 层级** 的一致性问题，但对更复杂的 grapheme cluster（如 ZWJ 家庭 emoji、组合附加符）仍需结合产品定义确认“一个字符”的最终语义。若未来要支持“用户感知字符”粒度，需要单独设计比 Rune 更高层的 cluster 模型。
