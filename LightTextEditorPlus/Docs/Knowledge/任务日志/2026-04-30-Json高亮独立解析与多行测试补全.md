# Json 高亮独立解析与多行测试补全

## 背景与目标

本次任务聚焦现有代码高亮中的 Json 场景拆分。

目标有两项：

1. 当输入内容可以按宽松规则成功解析为 Json 时，不再完全依赖 ColorCode，而是走独立的 Json 解析高亮路径；
2. 为 Json 高亮补充缺失的多行长文本单元测试，并采用测试驱动方式，先用测试定义预期，再修复实现缺陷。

同时要求保留回退策略：如果 Json 解析失败，依然退回到既有 ColorCode 逻辑。

## 关键文件与模块

- `LightTextEditorPlus.Highlighters/OtherCodeDocumentHighlighter.cs`
- `LightTextEditorPlus.Highlighters/MarkdownDocumentHighlighter.cs`
- `LightTextEditorPlus.Highlighters/CodeHighlighters/JsonCodeHighlighter.cs`
- `LightTextEditorPlus.Highlighters/CodeHighlighters/ColorCodeCodeHighlighter.cs`
- `LightTextEditorPlus.Highlighters/TextEditorColorCode.cs`
- `LightTextEditorPlus.Highlighters/CodeHighlighters/CsharpCodeHighlighter.cs`
- `LightTextEditorPlus.Highlighters.Avalonia.UnitTests/OtherCodeDocumentHighlighterTests.cs`

## 主要决策与原因

1. Json 独立高亮使用 `Utf8JsonReader`
   - 相比在字符串上做正则或二次猜测，`Utf8JsonReader` 能直接提供 token 类型与起始位置；
   - 同时可通过 `AllowTrailingCommas = true` 和 `CommentHandling = Skip` 支持宽松 Json；
   - 对多行长文本和嵌套结构更稳定。

2. 仅在 `languageId == "json"` 时优先尝试独立解析
   - 这样修改面最小；
   - 不影响其他语言仍旧走 ColorCode 的现有行为。

3. Json 解析失败时必须回退到 ColorCode
   - 这是用户明确要求；
   - 因此 `JsonCodeHighlighter` 只暴露 `TryApplyHighlight`，成功则消费，失败则由调用方继续走 ColorCode。

4. Markdown 代码块中的 Json 也复用同一独立解析路径
   - 仓库内已有“Markdown 代码块高亮应与独立高亮一致”的约束；
   - 如果只修改独立高亮，不修改 Markdown 内部 Json，会造成同一段 Json 在两处着色不一致。

5. 旧 Json 测试基线从“等同 ColorCode”改为“符合 Json 语义”
   - 需求已经变为 Json 独立解析优先；
   - 因此旧的“逐字符颜色完全等于 ColorCode”对 Json 不再成立，测试需要同步改成语义断言。

6. 对新增多行测试统一使用 `\n`
   - 仓库维护文档明确内部换段以 `\n` 为标准；
   - 否则测试中的原始字符串在不同环境下可能带入 `\r\n`，导致偏移和断言不稳定。

## 修改点摘要

- `OtherCodeDocumentHighlighter.cs`
  - 增加 `_languageId` 与 `JsonCodeHighlighter` 字段；
  - 在 `ApplyHighlight` 中对 Json 先尝试独立解析高亮，成功则直接返回，失败再回退到 ColorCode。

- `JsonCodeHighlighter.cs`
  - 新增基于 `Utf8JsonReader` 的 Json 高亮实现；
  - 支持宽松解析：尾随逗号、注释跳过；
  - 对 `PropertyName` 映射为 `ClassMember`，字符串映射为 `String`，数字映射为 `Number`，`true/false/null` 映射为 `DeclarationTypeSyntax`；
  - 通过字符级扫描计算字符串 token 的实际长度，避免多行或转义内容下定位错误。

- `MarkdownDocumentHighlighter.cs`
  - 为 fenced code block 内的 `json` 语言复用 `JsonCodeHighlighter`；
  - 保证 Markdown 代码块 Json 与独立 Json 高亮行为一致。

- `ColorCodeCodeHighlighter.cs`
  - 增加最终写入前的边界保护；
  - 避免异常输入下第三方返回越界区间时写入失败。

- `TextEditorColorCode.cs`
  - 为 `ScopeType.ClassMember` 增加颜色映射；
  - 使 Json key、属性名、成员名等不再退回纯文本色。

- `CsharpCodeHighlighter.cs`
  - 为“标识符后接括号”的独立成员声明片段增加最小兜底识别；
  - 修复新增 `ClassMember` 颜色后暴露出来的既有 C# 测试回归。

- `OtherCodeDocumentHighlighterTests.cs`
  - 新增多行长 Json 测试，覆盖 key、字符串、数字、布尔、null；
  - 新增宽松 Json 多行测试，覆盖注释与尾随逗号；
  - 新增非法 Json 回退到 ColorCode 的测试；
  - 将既有单行 Json 测试改为 Json 语义断言，而不是继续以 ColorCode 为基线。

## 后续建议

- 如果后续继续增强 Json 语义高亮，可考虑补充对象/数组分隔符、括号类 token 的独立着色，而不是只覆盖值语义。
- 当前宽松规则已覆盖注释和尾随逗号；若未来还要支持更宽松的 Json 方言，应先明确方言边界，再决定是否继续扩展解析路径。
- `ColorCodeCodeHighlighter` 的越界保护目前是最小防御性修复；如后续持续遇到第三方返回异常区间，建议单独补底层回归测试并再做收敛。
