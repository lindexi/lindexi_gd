# Knowledge 知识库目录整理

## 背景与目标

`Docs/Knowledge/` 目录下积累了 46 个 markdown 文件（1 个 README 模板 + 2 个提炼文档 + 42 个任务日志 + 1 个整理计划），全部平铺在同一层级。存在提炼文档编号缺失、旧任务日志过时、通用技术参考与项目特定知识混放、README 缺少索引等问题。本次建立分层分类体系并补全索引。

## 关键文件与模块

- `Docs/Knowledge/README.md` — 重写为完整索引 + 使用说明
- `Docs/Knowledge/开发约定与模式.md` — 去掉"二、"编号前缀，修复内部链接
- `Docs/Knowledge/领域与技术参考.md` — 去掉"三、"编号前缀，修复内部链接
- `Docs/Docs.csproj` — 增加 Knowledge 子目录引用（通配符 `*.md`）
- `.github/copilot-instructions.md` — 更新 Knowledge 目录结构描述和协作流程
- `Docs/Knowledge/2026-06-01-Knowledge目录整理计划.md` — 整理计划文档，标记已完成

## 主要决策与原因

1. **分层分类体系**：提炼文档在顶层，任务日志、通用参考、归档分别在子目录
   - 原因是提炼文档需要优先被阅读，子目录减少文件数量噪音
2. **去掉提炼文档的"二、""三、"编号**：搜索未找到"一、"原始内容
   - 原因是不完整的编号体系反而造成困惑
3. **`2025-01-XX` 文件直接归档**：不拆分提取
   - 原因是该文件作为整体已过时，有效内容已被提炼文档覆盖
4. **`2026-04-09` 文件整体移入通用参考**：不拆分
   - 原因是该文件既有通用技术也有项目决策，但整体偏通用，且拆分会破坏阅读连贯性
5. **`Docs.csproj` 使用通配符**：`任务日志\*.md` 等
   - 原因是 42 个任务日志逐条列出维护成本过高

## 修改点摘要

### 目录重组
- 创建子目录：`任务日志/`、`通用参考/`、`_archive/`
- 42 个任务日志文件 → `任务日志/`
- 1 个通用技术文件 → `通用参考/`
- 1 个过时文件 → `_archive/`

### 文件修改
- `README.md`：从 41 行模板重写为 125 行完整索引
- `开发约定与模式.md`：去掉"二、"前缀，内部链接从"具体任务日志文件"改为指向 `任务日志/` 目录
- `领域与技术参考.md`：去掉"三、"前缀，同上
- `Docs.csproj`：增加 Knowledge 子目录的 `None` 引用
- `.github/copilot-instructions.md`：
  - Knowledge 描述从单行扩展为完整子目录说明
  - "任务完成后"章节更新为写入 `Docs/Knowledge/任务日志/`
  - 新增通用参考和索引更新要求
- `2026-06-01-Knowledge目录整理计划.md`：状态标记为已完成

## 验证方式

- 运行 `Get-ChildItem -Recurse -Name` 确认文件结构
- 确认 README.md 索引与目录实际文件一致（共 48 条索引已核对）
- `Docs.csproj` 通过通配符覆盖所有子目录

## 后续建议

- 当 `任务日志/` 中某个主题的文件累计较多时，将共性规律提炼到顶层提炼文档
- 定期（每季度）审查 `任务日志/`，将其中实现已被完全替换的文件移入 `_archive/`
- 如果后续 `_archive/` 文件增多，可在归档文件头部加注释说明归档时间和原因
