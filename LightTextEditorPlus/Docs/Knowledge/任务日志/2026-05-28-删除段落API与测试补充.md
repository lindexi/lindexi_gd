# 删除段落 API 与测试补充

## 背景与目标

本轮目标是在文档层补充“按段落删除”的能力，并沿着 `DocumentManager -> TextEditorCore -> TextEditor` 的公开链路提供 API。删除行为需要覆盖非末段、末段和仅剩一段时的差异，同时尽量复用现有文本删除基础设施，保证撤销重做、光标更新与文档变更事件行为一致。

## 关键文件与模块

- `LightTextEditorPlus.Core/Document/DocumentManagers_/DocumentManager_/DocumentManager.cs`
- `LightTextEditorPlus.Core/API/TextEditorCore.Edit.cs`
- `Build/Shared/API/TextEditor.Edit.Shared.cs`
- `Build/Shared/API/Constraints/TextEditor.Edit.Shared.txt`
- `Tests/LightTextEditorPlus.Core.Tests/API/TextEditorEditTest.cs`
- `Tests/LightTextEditorPlus.Core.Tests/API/TextEditorUndoRedoTest.cs`

## 主要决策与原因

1. `DocumentManager.RemoveParagraph(ITextParagraph)` 不直接重写底层删除流程，而是先将“删除段落”转换为精确的 `Selection`，再复用既有 `RemoveInner -> EditAndReplaceRunInner -> ReplaceCore` 链路，减少重复逻辑。
2. 非末段删除时，选择范围包含“当前段内容 + 段尾换行符”；末段删除时，选择范围改为“前置换行符 + 当前段内容”，避免错误保留孤立换行。
3. 当文档只剩一段时，不删除整段容器，只删除段内字符，依赖既有空文本保底一段的约定，避免出现“文档无段落”的不稳定中间状态。
4. 撤销测试采用现有 `TextChangeOperation` 语义校验：撤销后文本恢复，但光标位置仍按现有替换逻辑落到恢复内容之后，而不是强制回到原删除起点。

## 修改点摘要

### 1) DocumentManager 增加删除段落能力

新增 `RemoveParagraph(ITextParagraph)`：

- 校验 `null` 与段落实例类型；
- 使用 `ParagraphManager.GetParagraphIndex` 确认段落属于当前文档；
- 单段时仅删除段内文本；
- 非末段时删除段内容并带上段尾换行；
- 末段时删除前置换行与段内容；
- 最终统一转发到现有删除逻辑处理。

### 2) Core 与共享 API 暴露

新增：

- `TextEditorCore.RemoveParagraph(ITextParagraph)`
- `TextEditor.RemoveParagraph(ITextParagraph)`

并在共享约束文件 `TextEditor.Edit.Shared.txt` 中补充 `RemoveParagraph` 定义，保持多平台 API 一致性。

### 3) 测试补充

在 `TextEditorEditTest` 中新增 `RemoveParagraphTest`，覆盖：

- 空文本删除空段；
- 仅一段文本删除该段；
- 多段删除首段；
- 多段删除中间段；
- 多段删除末段；
- 多段删除空段。

在 `TextEditorUndoRedoTest` 中补充删除段落后的撤销重做场景，验证文本与光标状态恢复符合当前实现语义。

## 后续建议

- 如后续还会补充“段落级编辑”能力，可考虑继续抽取 `DocumentManager` 内部“段落到 Selection”的辅助方法，减少边界分支重复。
- 可以再补一组针对共享 `TextEditor` 外层 API 的集成测试，验证多平台共享入口与 Core 行为保持一致。
- 若产品期望“撤销段落删除后光标回到原段起点”，需要单独评估 `TextChangeOperation` 的统一语义，避免影响现有文本替换撤销行为。
