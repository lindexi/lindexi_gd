# 竖排排版位置错误问题调查

## 背景与目标

用户反馈竖排排版结果"偏了"——字符位置跑偏。本次调查的目标是定位竖排排版位置错误的根因。

## 调查过程

### 1. 布局架构探索

文本编辑器使用**水平排列坐标系（Horizontal Arranging CS）**：
- **X 轴** = inline 方向（文本流动方向）
- **Y 轴** = block 方向（行列堆叠方向）

对于横排：X=左→右，Y=上→下
对于竖排：X=上→下，Y=列排列方向（标准竖排从右到左）

布局分为三个阶段：
1. **预布局阶段**（`PreUpdateDocumentLayout`）— 段落→行→字符的排列计算
2. **回溯最终布局阶段**（`FinalUpdateDocumentLayout`）— 对齐、边距等抛光计算
3. 渲染层将水平排列坐标转换为实际屏幕坐标

### 2. `VerticalArrangingLayoutProvider` 分析

`VerticalArrangingLayoutProvider`（仅 48 行）继承自 `HorizontalArrangingLayoutProvider`（1418 行），**仅重写了 2 个方法**：
- `GetLineMaxWidth()` — 用 `DocumentHeight` 替代 `DocumentWidth`
- `CalculateDocumentOutlineSize()` — 交换宽高

关键发现：`HorizontalArrangingLayoutProvider` 中与竖排行为差异的关键方法全部是 **`private`**，无法被子类重写：
- `GetNextLineStartPoint` — `Offset(0, Height)`
- `UpdateTextLineStartPoint` — 字符沿 X 轴排列
- `UpdateWholeLineLayout` — 行高/列宽计算
- `BuildParagraphTextSize` — 段落尺寸计算
- `FinalUpdateParagraphLineLayout` — 缩进、水平对齐全在 X 轴
- `FinalUpdateParagraphLayout` — 段落宽度计算

### 3. 坐标转换链路追踪

追踪从布局到渲染的坐标转换链路，发现关键问题在 `LineLayoutData.GetLineDrawingArgument()`：

```csharp
// LineLayoutData.cs 核心代码：
return new LineDrawingArgument(..., 
    CharStartPoint.ToCurrentArrangingTypePoint(),  // ✅ StartPoint 已转换到屏幕坐标
    GetCharList())
{
    LineContentSize = LineContentSize,      // ❌ 仍是水平排列CS，未转换！
    LineCharTextSize = LineCharTextSize,    // ❌ 仍是水平排列CS，未转换！
};
```

## 根因分析

### 核心问题

**`LineContentSize` 和 `LineCharTextSize` 没有跟着 `StartPoint` 一起通过 `ToCurrentArrangingTypePoint()` / `SwapWidthAndHeight()` 转换到当前排列类型的屏幕坐标。**

渲染器拿到的 `argument.StartPoint` 是屏幕坐标，而 `argument.LineContentSize` 是水平排列 CS 的尺寸——宽高在竖排下含义完全相反。

### 具体表现

在 `VerticalSkiaTextRenderer.RenderCharList` 中：

```csharp
// charData.GetStartPoint() 返回什么坐标？→ 推断是水平排列CS
(double x, double y) = charData.GetStartPoint();
var contentMargin = argument.StartPoint.X - x;
// ↑ argument.StartPoint.X 是屏幕坐标，x 是水平排列CS坐标 → 混合坐标系！

if (!ArrangingType.IsLeftToRightVertical)
{
    x -= (argument.LineContentSize.Height - contentMargin);
    // ↑ LineContentSize.Height 是水平排列CS的Height（=竖排的列宽），
    //   但在这里被当作屏幕坐标使用
}
```

这导致渲染器在两种坐标系之间来回混合计算，产出的最终渲染位置必然偏了。

## 主要决策与原因

### 1. 水平排列 CS 模型设计

布局系统内部统一使用水平排列 CS，通过 `ToCurrentArrangingTypePoint()` 在最终输出时转换到屏幕坐标。这个设计本身是合理的。

### 2. `LineDrawingArgument` 构建不一致

问题出在 `LineLayoutData.GetLineDrawingArgument()` 中：
- `StartPoint` 调用了 `ToCurrentArrangingTypePoint()` 转换
- `LineContentSize` 和 `LineCharTextSize` 没有转换

这导致渲染器拿到的数据处于两种不同的坐标系，无法正确计算。

2. 渲染器中的 `SwapWidthAndHeight()` 调用是否能正确补偿坐标系差异？
