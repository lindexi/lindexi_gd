# 指导布局设置回流与文档结构修正

## 背景与目标

前一版指导布局实现主要提供了“导出指导布局快照”的能力，但存在三个问题：

1. 在 `OnLayoutCompleted` 中立即构造 `GuidingLayoutInfo`，对大多数业务场景属于额外开销
2. 缺少“把指导布局重新设置回文本库并参与布局渲染”的核心能力
3. 文档中指导布局章节的组织位置不符合现有文档结构习惯

本次目标是把指导布局从“只读结果”升级为“下一次布局的一次性输入 + 可导出结果”的能力，同时修正文档结构。

## 主要决策与原因

### 1. 不在布局完成事件中立即构造指导布局快照

改为：

- `RenderInfoProvider.GetCurrentGuidingLayoutInfo()` 按需导出
- `TextEditorCore.GetCurrentGuidingLayoutInfo()` 按需导出

原因：

- 大多数业务只关心渲染，不关心指导布局
- 在 `OnLayoutCompleted` 内每次都构造快照会增加不必要的遍历与分配

### 2. 指导布局作为下一次布局的一次性输入

新增核心入口：

- `SetGuidingLayoutInfoForNextUpdateLayout`

原因：

- 指导布局本质上是一次布局约束输入，不应长期保留在 `TextEditorCore`
- 应在布局开始时由 `LayoutManager` 校验并消费，避免状态陈旧

### 3. 复用现有字符测量、行高与坐标计算流程

当前实现只用指导布局驱动“每段几行、每行取多少字符”，然后继续复用现有：

- 字符测量
- 行距计算
- 行内字符坐标计算
- 段落与文档范围计算

原因：

- 避免出现第二套独立布局算法
- 保证默认布局与指导布局布局的一致性与可维护性

### 4. 校验失败时只记录日志并回退默认布局

原因：

- 指导布局属于可选增强能力，不应把文本框直接打挂
- 当前库已有日志体系，适合用于报告不匹配原因

## 修改点摘要

- 移除 `LayoutCompletedEventArgs.GuidingLayoutInfo`
- 移除 `OnLayoutCompleted` 中的立即指导布局快照构造
- 新增 `GuidingLayoutInfoValidator`，用于校验：
  - 布局方式
  - 段落数量
  - 每段字符数量
  - 每行字符数量
  - 空段要求
- 将指导布局放入下一次布局配置，由 `LayoutManager` 在布局开始前校验并一次性消费
- 在段落行布局时，将 `GuidingParagraphLayoutInfo` 和 `GuidingLineLayoutInfo` 显式放入布局参数
- 在分行算法阶段按指导布局的每行字符数执行分行，继续复用原有整行测量逻辑
- 公开 `TextEditor` 层共享 API：
  - `GetCurrentGuidingLayoutInfo`
  - `SetGuidingLayoutInfoForNextUpdateLayout`
- 补充测试：
  - Core：获取、设置、无效指导布局回退日志
  - WPF：设置指导布局后按指定行字符数渲染
  - Avalonia：设置指导布局后按指定行字符数渲染
- 修正文档结构位置：
  - 使用说明：移动到高级用法
  - 实现定义：移动到渲染章节前
  - 术语表：移动到字体回滚后
  - 行为定义：移动到中文标点前

## 验证方式（构建/测试/手工验证）

建议执行：

- `dotnet test -c Release Tests/LightTextEditorPlus.Core.Tests/LightTextEditorPlus.Core.Tests.csproj`
- `dotnet test -c Release Tests/LightTextEditorPlus.Tests/LightTextEditorPlus.Tests.csproj`
- `dotnet test -c Release Tests/LightTextEditorPlus.Avalonia.Tests/LightTextEditorPlus.Avalonia.Tests.csproj`

同时关注：

- 设置无效指导布局时是否在本次布局记录警告日志并回退默认分行
- 设置有效指导布局后，目标文本框是否按指导布局的行字符数渲染

## 后续建议

- 后续可继续支持更强的指导布局输入，例如：
  - 精确行坐标
  - 精确段落边界
  - 精确字符点位
- 如未来平台差异较大，可增加更多指导布局失配诊断日志
- 可再补充项目符号、缩进、段前段后、竖排与指导布局组合测试
