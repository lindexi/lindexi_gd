# 空段拖拽选择修复

## 背景与目标

`TextEditorHandler.HandleDragSelect` 在拖拽选择到空段时，会因为命中结果标记为超过文本字符范围而忽略选择更新，导致无法把选择范围扩展到空段行首。本次按 TDD 流程先补充复现用例，再最小化修复共享交互逻辑。

## 关键文件

- `Build/Shared/API/Handlers/TextEditorHandler.Shared.cs`
- `Tests/LightTextEditorPlus.Tests/TextEditorHandlerTest.cs`
- `Tests/LightTextEditorPlus.Avalonia.Tests/TextEditorHandlerTest.cs`

## 主要决策

- 在 WPF 与 Avalonia 交互测试中同时覆盖该共享处理器行为，避免只验证单个平台。
- 保留原有“拖拽超过非空文本范围时忽略”的行为，只对空段命中放行，降低行为变更范围。
- 测试通过包装受保护方法并设置单击后的 `ClickCount` 状态，直接验证 `HandleDragSelect` 的核心行为。

## 修改摘要

- 新增“鼠标拖拽选择到空段时，应扩展选择到空段行首”测试，覆盖文本 `1\n\n2` 中第二段空段。
- 调整 `HandleDragSelect`：当命中结果为 `IsOutOfTextCharacterBounds` 但命中段落是空段时，仍使用 `HitCaretOffset` 更新选择。

## 验证记录

- 先运行新增目标测试，WPF 与 Avalonia 均失败，复现选择停留在 `0-0` 的问题。
- 修复后运行新增目标测试，2 个测试全部通过。
- 运行 `dotnet test LightTextEditorPlus.sln -c Release --filter DragSelectShouldSelectToEmptyParagraph`，新增 Release 目标测试通过。
- 运行 `dotnet test LightTextEditorPlus.sln -c Release`，总计 1878，失败 0，成功 1860，跳过 18。

## 风险与后续建议

- 空段命中当前仍可能带有 `IsOutOfTextCharacterBounds` 标记，后续如重构命中语义，可考虑在核心命中层面更明确地区分“空段行内命中”和“文档外越界”。
