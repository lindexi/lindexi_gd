# WPF 竖排文本渲染接入

## 背景与目标

WPF 平台此前已经补齐了竖排字符测量，但 `LightTextEditorPlus.Wpf/Layers/TextView.cs` 在遇到 `ArrangingType.Vertical` 和 `ArrangingType.Mongolian` 时仍直接抛出 `NotSupportedException`。本次任务的目标是仿照横排渲染器实现 WPF 竖排文本渲染器，并将其接入 `TextView`。

## 关键文件与模块

- `LightTextEditorPlus.Wpf/Rendering/VerticalTextRenderer.cs`
- `LightTextEditorPlus.Wpf/Layers/TextView.cs`
- `LightTextEditorPlus.Wpf/Rendering/HorizontalTextRenderer.cs`
- `LightTextEditorPlus.Skia/Rendering/Core/VerticalSkiaTextRenderer.cs`
- `Tests/LightTextEditorPlus.Tests/TextEditorTest.cs`

## 主要决策与原因

1. 新增独立的 `VerticalTextRenderer`
   - 不直接在 `HorizontalTextRenderer` 中塞入大量分支，避免横排和竖排逻辑互相污染。
2. 复用横排渲染器的字符拆分、字体回退与装饰渲染模式
   - 保持 WPF 平台内部实现风格一致，减少行为差异。
3. 采用逐字 `GlyphRun` 绘制
   - 竖排下每个字符都需要单独计算位置，逐字绘制更容易对齐 Skia 的竖排定位公式。
4. 使用 `isSideways: true` 的 `GlyphRun`
   - 让 WPF 直接参与竖排旋转绘制，避免额外包裹复杂的变换矩阵。
5. 列级缓存沿 `StartPoint.X` 复用
   - 横排是按行 Y 偏移复用，竖排改为按列 X 偏移复用，保持缓存思路一致。

## 修改点摘要

### `LightTextEditorPlus.Wpf/Rendering/VerticalTextRenderer.cs`

- 新增 WPF 竖排文本渲染器。
- 参考 `HorizontalTextRenderer` 组织结构，保留：
  - 段落/行遍历
  - 按 `RunProperty` 拆分字符
  - 字体回退处理
  - 文本装饰渲染
- 参考 Skia 竖排渲染公式：
  - 使用 `charData.Size.SwapWidthAndHeight()` 和 `charData.FaceSize.SwapWidthAndHeight()` 还原实际绘制尺寸
  - 使用 `Baseline` 与 `space` 修正竖排字符的 Y 位置
  - 对非从左到右竖排做列宽补偿

### `LightTextEditorPlus.Wpf/Layers/TextView.cs`

- 将 `ArrangingType.Vertical` 和 `ArrangingType.Mongolian` 的渲染路径接入 `VerticalTextRenderer`
- 保持横排仍使用 `HorizontalTextRenderer`

### `Tests/LightTextEditorPlus.Tests/TextEditorTest.cs`

- 新增 `VerticalRenderTest`
- 验证竖排文本可以完成真实渲染路径，不再因 `TextView` 抛出不支持异常

## 后续建议

1. 后续可补充包含中文、拉丁字母、数字混排的竖排视觉集成测试。
