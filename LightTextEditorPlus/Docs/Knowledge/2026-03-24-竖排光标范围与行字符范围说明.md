# 竖排光标范围与行字符范围说明

## 背景与目标

`LightTextEditorPlus.Core/Rendering/CaretRenderInfo.cs` 的 `GetCaretBounds` 之前只实现了横排光标，竖排分支直接返回 `TextRect.Empty`。

本次任务的目标有两个：

1. 仿照横排语义补齐竖排光标范围计算。
2. 顺带梳理 `CharData`、`CharDataInfo`、`LineLayoutData` 里的各类范围和坐标语义，避免后续在竖排场景下误用范围数据。

## 关键文件与模块

- `LightTextEditorPlus.Core/Rendering/CaretRenderInfo.cs`
- `LightTextEditorPlus.Core/Document/DocumentManagers_/ParagraphManagers_/Paragraphs_/Chars_/CharData.cs`
- `LightTextEditorPlus.Core/Document/DocumentManagers_/ParagraphManagers_/Paragraphs_/Chars_/CharDataInfo.cs`
- `LightTextEditorPlus.Core/Document/DocumentManagers_/ParagraphManagers_/Paragraphs_/Chars_/CharLayoutData.cs`
- `LightTextEditorPlus.Core/Document/DocumentManagers_/ParagraphManagers_/Paragraphs_/Lines_/LineLayoutData.cs`
- `LightTextEditorPlus.Wpf/Layers/Caret_/SelectionAndCaretLayer.cs`
- `LightTextEditorPlus.Wpf/Rendering/VerticalTextRenderer.cs`
- `Tests/LightTextEditorPlus.Core.Tests/Rendering/CaretRenderInfoTest.cs`

## 主要决策与原因

1. 竖排普通光标按“横排插入光标旋转 90 度”处理
   - 横排普通光标是竖线。
   - 竖排普通光标对应为横线，放在字符前方或后方。
2. 竖排覆盖模式按“横排下划线旋转 90 度”处理
   - 右到左竖排显示为后一个字符左侧的竖线。
   - 左到右竖排（蒙古文方向）显示为后一个字符右侧的竖线。
3. 竖排不能直接把 `GetStartPoint + Size` 当作最终可视矩形
   - 右到左竖排下，`GetStartPoint()` 更接近列锚点，实际可视宽高要按竖排渲染语义交换解释。
   - 因此本次光标范围计算显式使用“可视宽 = `Size.Height`、可视高 = `Size.Width`”的规则。
4. 行范围要区分“内容带”和“外接带”
   - `GetLineContentBounds()` 更适合光标、选择、字符相关绘制定位。
   - `OutlineBounds` 更适合空白点击命中、段落/行外接区域判断。

## 修改点摘要

## CharData 与 CharDataInfo 的范围语义

### `CharData.GetStartPoint()`

- 表示字符布局锚点，坐标相对文档左上角。
- 横排下，这个点基本可直接理解为字符框左上角。
- 右到左竖排下，这个点更接近“列锚点”，后续做可视矩形时通常需要结合竖排宽高语义一起解释，而不能简单把它当作最终的左上角。

### `CharDataInfo.FrameSize` / `CharData.Size`

- 表示字符的布局框尺寸，也就是字外框尺寸。
- 这是布局推进最核心的尺寸：排版、光标、选择、换行都更依赖它。
- `CharData.Size` 只是 `FrameSize` 的别名。

### `CharDataInfo.FaceSize` / `CharData.FaceSize`

- 表示字面实际着墨区域尺寸，也就是字墨范围。
- 一般小于等于 `FrameSize`。
- 更适合文本装饰、精细渲染修正，不适合直接拿来做布局推进。

### `CharData.GetBounds()`

- 是 `new TextRect(GetStartPoint(), Size)` 的快捷结果。
- 横排下可直接视为字符布局范围。
- 竖排下要特别注意：它沿用了当前坐标系和字符尺寸的存储语义，做最终视觉矩形时往往还需要结合排版方向做宽高交换或锚点修正。

## LineLayoutData 的范围语义

### `CharStartPoint`

- 行在文档中的起始点，属于“整行起点”。
- 它还没有叠加本行缩进与对齐带来的空白偏移。

### `LineContentStartPoint`

- 行内内容真正开始放置的位置。
- 等于 `CharStartPoint + IndentationThickness + HorizontalTextAlignmentGapThickness` 的左上偏移。
- 更接近字符布局和字符绘制要使用的起始点。

### `LineContentSize`

- 表示行内容带尺寸。
- 注释定义里，这个尺寸会把：
  - 缩进影响
  - 水平对齐带来的 gap
  - 行距厚度
  一起纳入考虑。
- 因此它不是“纯字形并集”，而是“当前行内容带”的尺寸。

### `GetLineContentBounds()`

- 返回当前行的内容带范围。
- 适合：
  - 光标定位
  - 选择高亮
  - 行内字符相关绘制
  - 命中到行内容区后的进一步处理
- 数值特点：
  - 通常比纯字符并集更大，因为会带上行距等留白。
  - 通常比 `OutlineBounds` 更贴近实际内容。
  - 在允许悬挂标点时，宽度还可能超过 `OutlineBounds`。

### `OutlineBounds`

- 返回当前行的外接范围，是更偏“命中与容器判断”的范围。
- 包含：
  - 段落左右边距影响
  - 对齐带来的空白
  - 缩进带来的空白
- 适合：
  - 判断点击是否落在当前行所属区域
  - 处理“点在行空白处但仍应命中本行”的逻辑
  - 做更宽松的行外接范围判断

## `GetLineContentBounds` 和 `OutlineBounds` 的差异

### 概念差异

- `GetLineContentBounds()`：内容带，偏向“画内容”和“贴内容”的范围。
- `OutlineBounds`：外接带，偏向“命中区域”和“容器区域”的范围。

### 数值差异

常见情况下：

- `OutlineBounds.Left` 会小于等于 `GetLineContentBounds().Left`
- `OutlineBounds.Top` 会小于等于 `GetLineContentBounds().Top`
- `OutlineBounds` 往往会把对齐和缩进留下的空白也包进去
- `GetLineContentBounds()` 会更贴近实际内容带
- 若启用悬挂标点，`GetLineContentBounds()` 的宽度可能反而比 `OutlineBounds` 更大

也就是说，两者不是简单的“谁严格包含谁”的永久关系，而是各自服务于不同语义。

### 什么时候用哪个

- 要做光标、选择、字符贴边绘制：优先 `GetLineContentBounds()`。
- 要做“我点在这一行所属区域里了吗”的宽松命中：优先 `OutlineBounds`。
- 要做字符精确位置：优先 `CharData.GetStartPoint()` + `CharData.Size/FaceSize`，不要只看行范围。

## 竖排光标的用法链路

当前 WPF 光标绘制链路是：

1. `SelectionAndCaretLayer` 获取当前 `CaretRenderInfo`
2. 调用 `CaretRenderInfo.GetCaretBounds(...)`
3. 将返回的 `TextRect` 直接画成矩形

因此 `GetCaretBounds` 返回值必须已经是最终可绘制范围，不能再把“还需要业务层自己纠偏”的工作留给 WPF 层。

本次改动就是在 Core 层把这个矩形补齐，让 WPF 和其他平台共用一致的竖排光标范围语义。
