## 1. API 设计模式

### 1.1 三重重载规则

每个修改文档状态或光标位置的公开方法，提供三种重载：

```csharp
// 重载 1：作用于当前光标位置（默认）
public void SetBold() { SetBold(CaretOffset); }

// 重载 2：作用于指定 CaretOffset
public void SetBold(int caretOffset) { ... }

// 重载 3：作用于指定段落的指定偏移（用于内部批量操作）
internal void SetBold(Paragraph paragraph, int offsetInParagraph) { ... }
```

**规则**：
- 重载 1 **必须**转发到重载 2，不能在重载 1 中写业务逻辑。
- 重载 3 为 `internal`，供内部批量操作绕过重复的光标查找。

### 1.2 段落级 API 同理

```csharp
public void DoSomething()              // 当前段落
public void DoSomething(int paraIndex) // 指定段落索引
```

## 2. 功能开关模式

### 2.1 判断位置

- **在具体公开 API 入口处判断**，不在通用内部方法中判断。

✅ 正确：
```csharp
public void SetVerticalLayout() 
{
    if (!FeatureFlags.IsVerticalEnabled)
        return; // 或 throw
    SetRunProperty(RunProperty.Vertical, true);
}
```

❌ 错误：
```csharp
internal void SetRunProperty(...) 
{
    if (property == RunProperty.Vertical && !FeatureFlags.IsVerticalEnabled)
        return; // 不应在这里拦截
    ...
}
```

**原因**：`SetRunProperty` 是直通管道，不应感知业务功能开关。功能开关与公开 API 语义绑定，不是与底层属性绑定。

### 2.2 开关定义

- 集中在 `FeatureFlags` 类中，以 `IsXxxEnabled` 命名。
- 默认值在静态构造函数中设置，可通过配置文件或测试钩子覆盖。

## 3. 跨平台策略

### 3.1 优先级

1. **共享代码文件**（`Build/Shared/`）— 同一份源码被多个项目引用
2. **条件 using 别名** — 统一类型名，避免 `#if` 散落在方法内部
3. **条件编译块** — 仅在真正有行为差异的代码路径使用
4. **平台特定文件**（`.ava.cs` / `.wpf.cs`）— 仅在签名相同但实现完全不同时使用

### 3.2 平台特定文件命名

- `Foo.ava.cs` — Avalonia 实现
- `Foo.wpf.cs` — WPF 实现
- 不含后缀的 `Foo.cs` 是共享文件或抽象定义。

## 4. 测试模式

### 4.1 测试覆盖要求

- 每个新功能：`Core.Tests` 测逻辑 + `Tests` (WPF) 测集成 + `Avalonia.Tests` 测集成。
- WPF 和 Avalonia 测试应**对称对齐**：相同的测试类名、相同的方法名、相同的断言逻辑（仅平台适配部分不同）。

### 4.2 文本断言规范

- **换行归一化**：所有文本断言前需用 `NormalizeNewlines()` 或等效方法将 `\r\n` 转为 `\n`，避免 CI 环境差异导致误报。
- 优先用 `Assert.Equal(expected, actual)` 而非 `Assert.Contains`，前者在失败时给出完整 diff。

### 4.3 测试数据

- 测试用例中的中文、蒙古文等内容的 Unicode 码点需在注释中标明，方便调试。

## 5. 验证流程

每次 AI 协作完成后，执行以下验证：

```bash
dotnet build                          # 全解决方案编译
dotnet test -c Release tests/Core.Tests          # 核心测试
dotnet test -c Release tests/Tests               # WPF 测试
dotnet test -c Release tests/Avalonia.Tests      # Avalonia 测试
```

- **顺序固定**：先 Core 后平台，Core 失败时不继续测平台。
- 所有测试必须零失败。
- 必须采用 Release 状态进行测试

## 6. 内部通信模式

- **使用构造注入的回调接口**，不使用 `event` 或 `EventHandler`。
- 回调接口以 `I` 开头、`Callback` 结尾（如 `ITextChangeCallback`）。
- 目的：减少内部分配，避免事件泄漏，使依赖关系显式化。

## 7. 换行约定

- **内部统一使用 `TextContext.NewLine` **。
- 从外部传入的文本在入口处归一化为 `TextContext.NewLine`。
- 不把 `\r\n` 或 `\n` (LF) 当作"标准"或"默认"。
- 输出给平台渲染时按需转换（如 WPF 的 `TextBox` 需要 `\r\n`）。

## 8. 知识文件命名规范

```
Knowledge/YYYY-MM-DD-任务简述.md
```

- 日期使用任务**创建日期**，非完成日期。
- 任务简述用中文，简洁明确（不超过 15 字），用连字符连接。
- 示例：`2026-03-17-段落便捷API重载.md`

## 9. 可见性约定

- 公开 API：`public`，面向库使用者。
- 内部 API：`internal`，面向内部模块间调用。
- 平台特定 API：`internal` 或 `protected`，不暴露给使用者。
- 测试可见：必要时用 `[InternalsVisibleTo]` 暴露给测试项目。

## 10. 代码风格速记

- 命名空间按文件夹结构对齐。
- 命名空间声明用块语法（`namespace Foo { }`），不用文件范围语法。
- `using` 放在命名空间内部。
- 方法体内尽量减少嵌套层级，提前 return
