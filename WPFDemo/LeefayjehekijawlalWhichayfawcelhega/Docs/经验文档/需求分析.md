# AI 多页课件生成需求分析

本文用于分析当前 WPF 项目的核心需求，目标是根据用户输入的完整 PPT 大纲，先由 Agent 自主拆分页并生成逐页图片提示词，再由用户确认后调用兼容 OpenAI 的图片模型生成每页候选图片，最后导出用户选中的页面图片。

## 一、需求来源

当前项目的模型对接方式，按照现有文档 `Docs/Csharp Microsoft Agent Framework 与 豆包 对接.md` 提供的兼容 OpenAI 接入思路实现。也就是说：

- 使用 `Microsoft Agent Framework` 组织 Agent 能力。
- 通过兼容 OpenAI 的接口方式访问豆包。
- 先创建用于文本生成的模型连接。
- 再创建用于图片生成的模型连接。

## 二、核心业务目标

本项目不是单次生成一张图片的小工具，而是一个“多页课件页面生成器”。用户的目标是：

1. 输入一组完整的 PPT 课件大纲。
2. 让系统结合整份大纲自动决定页面边界。
3. 为每一页生成适合图片模型使用的提示词。
4. 由用户逐页确认提示词是否可用，并允许手工新增或删除页面。
5. 对单页或多页触发图片生成。
6. 从每页 4 张候选图片里选中 1 张。
7. 将所有已选图片按页面顺序导出到指定目录下的时间命名子目录中。

## 三、模型调用链路分析

根据需求，模型调用需要分成两个阶段。

### 1. 提示词生成阶段

这一阶段使用模型 `ep-20260306101224-c8mtg`。

职责是：

- 接收用户输入的完整 PPT 课件大纲。
- 根据全局上下文判断应拆分成多少页。
- 为每一页生成适合图片生成的中文提示词。
- 通过工具一次性提交逐页提示词列表，而不是直接在正文里拼接输出。

这个阶段必须先于图片生成阶段完成。

### 2. 图片生成阶段

这一阶段使用模型 `ep-20260120102721-c4pxb`。

职责是：

- 根据单页提示词生成 4 张候选图片。
- 通过 OpenAI SDK 的图片生成客户端访问兼容 OpenAI 的图片接口。
- 将结果绑定到对应的页面项中。
- 允许单页独立重试。

这一阶段不能在提示词生成完成后自动全部触发，必须保留用户确认步骤。

## 四、关键交互要求

### 1. 提示词需要先展示给用户

即使系统已经成功生成图片提示词，也不能直接开始调用图片模型。正确流程应为：

- 系统先根据完整大纲生成逐页提示词。
- 将提示词展示在页面列表里。
- 用户可查看并手工调整提示词。
- 用户还可以手工新增页面或删除页面。
- 用户决定是否为某一页执行图片生成。

### 2. 单页操作优先

虽然系统可以支持批量生成提示词，但图片生成必须允许单页独立操作。这样可以满足：

- 只重试某一页。
- 只修改某一页提示词后再生成。
- 某些页面暂时不生成，也不影响其他页面导出准备。

### 3. 候选图片必须可选中

每一页默认生成 4 张图片。用户需要：

- 查看 4 张候选图片。
- 通过单选方式选择其中 1 张。
- 默认第一张为选中状态。
- 在重新生成后更新可选项。

### 4. 导出按当前选中结果执行

导出行为依赖每一页当前的选中图片，而不是依赖最后一次生成结果或固定序号。

## 五、界面需求分析

主界面至少需要包含以下区域。

### 1. 输入区域

用于录入完整 PPT 大纲内容。页面拆分不应依赖按行或按空行规则，而应交给 Agent 根据整体语义判断。

### 2. 全局操作区域

用于放置：

- 服务配置输入框（Endpoint、API Key、模型编号）
- 生成提示词按钮
- 选择导出目录按钮
- 新增页面按钮
- 全局状态展示

### 3. 页面列表区域

列表中的每一项代表一页内容，并且需要按顺序排列。每项包含三列：

- 第一列：页码、删除按钮与提示词文本框，允许查看与编辑。
- 第二列：4 张候选图片，支持单选其中一张。
- 第三列：重新生成按钮，用于针对当前页重新生成图片。

如果某页还没有生成图片，第二列应该展示占位提示，而不是空白状态。

### 4. 底部导出区域

页面最下方需要提供导出按钮。点击后将所有已选图片导出到用户预设目录下以时间命名的子目录中。

## 六、数据结构需求分析

围绕页面列表，至少需要以下核心数据。

### 1. 页面项数据

每页应包含：

- 页面顺序号
- 当前提示词
- 当前候选图片列表
- 当前选中图片
- 当前阶段状态
- 当前错误信息

### 2. 应用级数据

应用层至少需要：

- 原始 PPT 大纲文本
- 服务配置（Endpoint、API Key、模型编号）
- 页面项集合
- 当前导出目录
- 全局忙碌状态
- 全局消息

## 七、流程需求分析

推荐主流程如下：

1. 用户输入完整 PPT 大纲。
2. 系统调用提示词生成 Agent，并将整份大纲交给 Agent 判断分页。
3. Agent 通过工具一次性返回逐页提示词列表。
4. 系统将提示词写入页面列表。
5. 用户检查、修改、新增或删除页面。
6. 用户点击某页生成图片。
7. 系统通过 OpenAI SDK 调用图片模型生成 4 张候选图片。
8. 用户选择每页最终图片。
9. 用户点击导出。
10. 系统创建时间命名子目录并按顺序输出图片文件。

## 八、非功能性要求

### 1. 可维护性

项目需要按照 MVVM 组织，不将模型调用逻辑堆积在代码隐藏文件。

### 2. 可恢复性

单页失败时，其他页不受影响，用户可以局部重试。

### 3. 可扩展性

后续可能补充：

- 页码重排
- 模板风格选择
- 批量重新生成
- 生成 PPT 文件

因此当前设计应尽量避免把页面项写死成一次性流程。

## 九、实现边界

在当前阶段，优先完成以下内容：

- 文本大纲输入
- 按页生成提示词
- 单页确认后生成图片
- 候选图片选择
- 导出已选图片
- 对接兼容 OpenAI 的模型服务基础框架

对于真正的联网调用、密钥读取、下载图片格式细节与更复杂的错误重试策略，本轮先把结构搭建完整并预留扩展点。

## 总结

本项目的关键不只是“接入某个模型服务”，而是建立一个可逐页确认、可单页重试、可选择最终图片并可顺序导出的课件生成工作台。因此界面、状态模型和服务边界都需要围绕“完整大纲 + Agent 自主分页 + 多页列表 + 用户确认 + 单页生成 + 最终导出”来设计。
