# 项目框架设计

本文用于描述 AI 多页课件生成项目的基础框架设计，目标是在 WPF 中以 MVVM 方式实现“页面大纲输入、逐页提示词生成、单页确认后生成图片、候选图片选择、顺序导出”的主流程。

## 一、总体设计目标

框架设计需要满足以下几点：

- 让界面层保持轻量。
- 让豆包调用逻辑集中到服务层。
- 让每页状态独立，便于单页重试。
- 让导出逻辑只依赖最终选中结果。
- 让后续继续扩展为真正的课件生成器时不需要重写主结构。

## 二、分层设计

### 1. View 层

`View` 使用 WPF XAML 实现，负责：

- 输入框布局
- 全局按钮与状态栏布局
- 页面列表展示
- 候选图片列表展示
- 提示词编辑区域展示

`View` 不直接处理豆包请求与导出细节。

### 2. ViewModel 层

`ViewModel` 负责组织界面状态和命令，核心包括：

- `MainViewModel`
- `SlidePageViewModel`
- `ImageCandidateViewModel`

职责拆分如下：

- `MainViewModel`：维护全局输入文本、页面集合、导出目录、全局状态、生成提示词与导出命令。
- `SlidePageViewModel`：维护单页内容、提示词、候选图片、所选图片、当前阶段、错误信息，以及单页图片生成命令。
- `ImageCandidateViewModel`：维护候选图片的预览数据、序号与导出二进制内容。

### 3. Service 层

服务层建议包含以下服务：

- `DoubaoPromptAgentService`：负责通过 Microsoft Agent Framework 调用 `ep-20260306101224-c8mtg` 生成逐页提示词。
- `DoubaoImageGenerationService`：负责调用 `ep-20260120102721-c4pxb` 生成图片，并下载候选图片数据。
- `SlideOutlineParser`：负责把输入框里的大纲文本拆分为一页页内容。
- `ImageExportService`：负责将已选图片导出到时间命名目录。

### 4. Model 层

模型层用于承载纯数据，建议包含：

- `DoubaoOptions`
- `PageGenerationStage`
- `GeneratedImageResult`
- 图片生成接口返回 DTO

## 三、关键流程设计

### 1. 提示词生成流程

1. 用户在输入框中录入 PPT 大纲。
2. `MainViewModel` 调用 `SlideOutlineParser` 拆分页面。
3. 为每页创建 `SlidePageViewModel`。
4. `MainViewModel` 调用 `DoubaoPromptAgentService` 逐页生成提示词。
5. 每页进入“待确认生成图片”状态。

### 2. 单页图片生成流程

1. 用户检查或修改该页提示词。
2. 用户点击当前页的生成按钮。
3. `SlidePageViewModel` 调用 `DoubaoImageGenerationService`。
4. 服务返回 4 张候选图片。
5. `SlidePageViewModel` 更新候选图片集合，并默认选中第一张。

这一步就是显式确认节点，避免系统自动调用图片模型。

### 3. 导出流程

1. 用户确认导出目录。
2. 用户点击底部导出按钮。
3. `MainViewModel` 收集每页的已选图片。
4. `ImageExportService` 在基础导出目录下创建时间命名子目录。
5. 按页面顺序输出图片文件。

## 四、状态设计

单页页面项推荐使用枚举表达阶段，至少包含：

- `PendingPromptGeneration`
- `PromptReady`
- `AwaitingImageGeneration`
- `GeneratingImages`
- `ImagesReady`
- `Failed`
- `Exported`

通过显式状态，界面可以更自然地控制按钮文本、禁用逻辑和错误提示。

## 五、配置设计

为了避免把密钥直接写入代码，建议：

- 通过环境变量读取豆包 API Key。
- 在 `DoubaoOptions` 中固定默认 Endpoint。
- 将文本模型与图片模型编号配置化，但提供当前需求的默认值。

推荐默认配置如下：

- Endpoint：`https://ark.cn-beijing.volces.com/api/v3`
- 提示词模型：`ep-20260306101224-c8mtg`
- 图片模型：`ep-20260120102721-c4pxb`

## 六、界面结构设计

主窗口可以分成三块：

### 1. 顶部输入与操作区

包含：

- PPT 大纲输入框
- 导出目录输入框
- 生成提示词按钮
- 全局状态文本

### 2. 中间页面列表区

列表项按三列组织：

- 左列：提示词文本框
- 中列：4 张候选图片列表
- 右列：单页生成或重新生成按钮

### 3. 底部导出区

包含导出按钮，触发最终文件输出。

## 七、异常处理设计

- 提示词生成失败时，仅标记对应页面失败。
- 图片生成失败时，仅标记对应页面失败。
- 全局状态栏显示最近一次操作结果。
- 单页错误信息直接显示在对应页面行中，方便用户定位。

## 八、实现策略

当前阶段采用“先搭骨架、再接能力”的实现方式：

1. 先搭建 MVVM 基础结构。
2. 再接入提示词生成 Agent。
3. 再补充图片生成与图片下载。
4. 最后完成导出与文档整理。

## 总结

本项目推荐使用 `MainWindow -> MainViewModel -> SlidePageViewModel -> Service` 的结构，将页面状态和豆包调用逻辑彻底分离。这样不仅能满足当前的多页课件生成需求，也能为后续加入更多编辑和导出能力保留足够清晰的演进空间。
