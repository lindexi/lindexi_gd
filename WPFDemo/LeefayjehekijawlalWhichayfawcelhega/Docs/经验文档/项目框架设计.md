# 项目框架设计

本文用于描述 AI 多页课件生成项目的基础框架设计，目标是在 WPF 中以 MVVM 方式实现“完整大纲输入、Agent 自主分页生成逐页提示词、单页确认后生成图片、候选图片选择、顺序导出”的主流程。

## 一、总体设计目标

框架设计需要满足以下几点：

- 让界面层保持轻量。
- 让 AI 服务调用逻辑集中到服务层。
- 让每页状态独立，便于单页重试。
- 让导出逻辑只依赖最终选中结果。
- 让后续继续扩展为真正的课件生成器时不需要重写主结构。

## 二、分层设计

### 1. View 层

`View` 使用 WPF XAML 实现，负责：

- 输入框布局
- 全局按钮与状态栏布局
- 页面列表展示
- 候选图片列表展示
- 提示词编辑区域展示

`View` 不直接处理模型请求与导出细节。

### 2. ViewModel 层

`ViewModel` 负责组织界面状态和命令，核心包括：

- `MainViewModel`
- `SlidePageViewModel`
- `ImageCandidateViewModel`

职责拆分如下：

- `MainViewModel`：维护完整大纲文本、服务配置、页面集合、导出目录、全局状态、生成提示词与导出命令。
- `SlidePageViewModel`：维护单页提示词、候选图片、所选图片、当前阶段、错误信息，以及单页图片生成命令。
- `ImageCandidateViewModel`：维护候选图片的预览数据、序号与导出二进制内容。

### 3. Service 层

服务层建议包含以下服务：

- `PromptAgentService`：负责通过 Microsoft Agent Framework 将完整大纲交给 Agent，由 Agent 自主拆分页并通过工具一次性提交逐页提示词。
- `ImageGenerationService`：负责使用 OpenAI SDK 的图片生成客户端调用兼容 OpenAI 的图片接口，生成候选图片数据。
- `ImageExportService`：负责将已选图片导出到时间命名目录。

### 4. Model 层

模型层用于承载纯数据，建议包含：

- `AiProviderOptions`
- `PageGenerationStage`
- `GeneratedImageResult`
- 图片生成接口返回 DTO

## 三、关键流程设计

### 1. 提示词生成流程

1. 用户在输入框中录入完整 PPT 大纲。
2. `MainViewModel` 调用 `PromptAgentService`。
3. Agent 结合完整上下文决定页面边界，并通过工具一次性返回逐页提示词列表。
4. `MainViewModel` 为每页创建 `SlidePageViewModel`。
5. 用户可继续手工新增页面、删除页面或直接编辑提示词。

### 2. 单页图片生成流程

1. 用户检查或修改该页提示词。
2. 用户点击当前页的生成按钮。
3. `SlidePageViewModel` 调用 `ImageGenerationService`。
4. 服务通过 OpenAI SDK 的图片生成客户端返回 4 张候选图片。
5. `SlidePageViewModel` 更新候选图片集合，并默认选中第一张。

这一步就是显式确认节点，避免系统自动调用图片模型。

### 3. 导出流程

1. 用户确认导出目录。
2. 用户点击底部导出按钮。
3. `MainViewModel` 收集每页的已选图片。
4. `ImageExportService` 在基础导出目录下创建时间命名子目录。
5. 按页面顺序输出图片文件。

## 四、状态设计

单页页面项推荐使用枚举表达阶段，至少包含：

- `PendingPromptGeneration`
- `PromptReady`
- `AwaitingImageGeneration`
- `GeneratingImages`
- `ImagesReady`
- `Failed`
- `Exported`

通过显式状态，界面可以更自然地控制按钮文本、禁用逻辑和错误提示。

## 五、配置设计

为了避免把密钥直接写入代码，同时方便切换服务商，建议：

- 通过界面输入 API Key，而不是依赖环境变量。
- 在 `AiProviderOptions` 中保存当前界面输入的 Endpoint 与模型编号。
- 为常用 Endpoint 与模型编号提供默认值，但允许用户随时修改。

推荐默认配置如下：

- Endpoint：`https://ark.cn-beijing.volces.com/api/v3`
- 提示词模型：`ep-20260306101224-c8mtg`
- 图片模型：`ep-20260120102721-c4pxb`

这些值只是默认项，不应在代码层写死为唯一服务商。

## 六、界面结构设计

主窗口可以分成三块：

### 1. 顶部输入与操作区

包含：

- 服务配置输入框（Endpoint、API Key、模型编号）
- 完整 PPT 大纲输入框
- 导出目录输入框
- 生成提示词按钮
- 新增页面按钮
- 全局状态文本

### 2. 中间页面列表区

列表项按三列组织：

- 左列：页码、删除按钮与提示词编辑框
- 中列：4 张候选图片列表
- 右列：单页生成或重新生成按钮

### 3. 底部导出区

包含导出按钮，触发最终文件输出。

## 七、异常处理设计

- 提示词生成失败时，仅标记对应页面失败。
- 图片生成失败时，仅标记对应页面失败。
- 全局状态栏显示最近一次操作结果。
- 单页错误信息直接显示在对应页面行中，方便用户定位。

## 八、实现策略

当前阶段采用“先搭骨架、再接能力”的实现方式：

1. 先搭建 MVVM 基础结构。
2. 再接入提示词生成 Agent。
3. 再补充图片生成与图片下载。
4. 最后完成导出与文档整理。

## 总结

本项目推荐使用 `MainWindow -> MainViewModel -> SlidePageViewModel -> Service` 的结构，将页面状态和 AI 服务调用逻辑彻底分离。这样不仅能满足当前的多页课件生成需求，也能为后续加入更多编辑、换用其他服务商和扩展导出能力保留足够清晰的演进空间。
