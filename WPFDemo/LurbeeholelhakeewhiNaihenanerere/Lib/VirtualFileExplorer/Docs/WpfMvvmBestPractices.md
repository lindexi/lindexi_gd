# WPF MVVM 最佳实践总结

## 目标

本项目中的文件管理器控件需要作为可复用库交付，因此 MVVM 的目标不是只让页面能跑起来，而是让界面结构、命令行为、样式替换、宿主集成和后续扩展都保持稳定。

## 视图职责

- 视图只负责布局、样式、模板、绑定和交互触发。
- 视图不直接承担文件管理业务，不在代码后置中拼装列表数据。
- 视图中的可变交互应优先绑定到命令和依赖属性，避免把状态散落在事件处理器里。
- 作为库控件时，视图应暴露模板和样式入口，允许宿主替换局部 UI。

## 视图模型职责

- 视图模型负责导航状态、选中状态、显示模式、命令状态和列表数据组织。
- 视图模型对外提供稳定的可绑定属性，例如当前文件夹、面包屑、文件集合、文件夹集合、混合项集合、当前显示模式。
- 视图模型统一协调重命名、复制、移动、删除等操作，避免控件直接依赖具体文件系统实现。
- 视图模型应将耗时操作设计为异步入口，避免阻塞 UI 线程。

## 模型职责

- 模型对象表达文件和文件夹的领域信息，例如名称、路径标识、大小、时间戳、是否可编辑、是否可删除。
- 如果操作能力需要按对象暴露，模型可以携带委托或引用管理器，但不应把 UI 状态写入模型。
- 文件系统相关能力应以抽象管理器统一提供，便于切换物理文件系统与虚拟提供器。

## 可绑定设计

- 所有会驱动界面刷新的状态都应实现属性变更通知。
- 集合项使用 `ObservableCollection<T>`，单值状态使用 `INotifyPropertyChanged`。
- 需要在多个属性之间联动时，应在主属性变更后同步通知派生属性。
- 命令的可执行状态变化要有刷新机制，避免按钮状态失真。

## 命令设计

- 用户操作优先建模为命令，例如进入文件夹、返回上级、刷新、删除、复制、移动、切换显示模式。
- 命令中只放协调逻辑，具体文件系统行为委托给 `VirtualFileManager` 或具体项对象。
- 对于库代码，应优先提供可复用命令参数模型，减少对具体控件类型的依赖。

## 模板与样式扩展

- 可替换部件优先通过依赖属性、`DataTemplate`、`Style` 和资源键暴露。
- 平铺模式和详细列表模式应共享同一批数据源，避免为不同视图复制状态。
- 默认样式要开箱即用，但不能阻止宿主替换项模板、工具栏或命令入口。

## 导航与状态管理

- 当前目录、导航历史和面包屑应由单一状态源维护，避免多个集合互相推导导致不一致。
- 面包屑项应直接携带目标文件夹引用，点击时无需再次解析字符串路径。
- 选中项和当前目录变化后，应及时刷新命令可执行状态。

## 库项目约束

- 对外 API 保持最小但完整，优先暴露领域能力和模板扩展点。
- 默认实现面向虚拟文件系统抽象，不把物理文件系统写死在控件层。
- 示例应用只用于验证集成，不承载库内部逻辑。

## 测试与验证建议

- 物理文件管理器优先做真实目录验证，确认枚举、重命名、复制、移动、删除行为符合预期。
- 视图模型至少要验证导航、显示模式切换和命令启用状态。
- 控件层重点验证绑定是否连通、模板替换是否生效、默认样式是否可直接使用。
