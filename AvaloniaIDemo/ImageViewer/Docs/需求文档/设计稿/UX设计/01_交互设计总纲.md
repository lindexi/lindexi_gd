# 交互设计总纲 — 图片查看器（v1.0）

> 产出人：UX设计师  
> 基于：`图片查看器_需求分析文档_v1.0.md`  
> 定位：UX 设计方向总纲，所有后续交互方案的"宪法"

---

## 一、设计哲学：板正之道的 UX 翻译

需求文档核心气质定调为「板正」——规矩、正统、专业、克制。UX 视角下，这 4 个词翻译为：

| 板正关键词 | UX 翻译 | 设计约束 |
|-----------|---------|---------|
| **规矩** | 行为可预测 | 所有交互结果必须符合用户基于同类工具的肌肉记忆（Windows 照片查看器、浏览器图片查看、Photoshop） |
| **正统** | 不搞"创新" | 不自创交互范式。缩放=Ctrl+滚轮，全屏=F11，翻看=←→，旋转=R/L——这些都是"行规" |
| **专业** | 信息透明 | 状态栏永远可见，缩放比例有反馈，位置信息不隐藏。用户随时知道自己"在哪"、"在看什么" |
| **克制** | 不打扰 | 无弹窗教程、无首次引导、无动画炫技、无功能堆砌。工具的存在感让位于内容 |

**UX 设计的核心使命：让用户忘记「图片查看器」这个壳，只看到图片本身。**

## 二、用户心智模型

### 2.1 用户是谁（来自需求 §三）

- 普通大众用户，非专业摄影师/设计师
- 操作水平：基础~中等
- 场景：日常翻看照片、查看截图、浏览下载的图片
- 核心情绪：**"我就想看一眼这张图"** ——需求极其单一、极其短暂

### 2.2 用户期望的心智模型（基于竞品肌肉记忆）

```
打开图片 → 看到了 → （可能放大看看细节）→ （可能翻几张）→ 关掉
               ↓
           （偶尔需要旋转一下拍歪的照片）
           （偶尔需要全屏沉浸看大图）
```

整个交互路径极度扁平，不存在"探索"需求。用户不会在图片查看器里"逛"。

### 2.3 心智模型对应的 UX 策略

| 用户意图 | 脑内期望 | UX 怎么满足 |
|---------|---------|------------|
| "我就看一眼" | 打开即看，无等待 | 适应窗口默认显示，图片优先渲染，加载态要轻 |
| "放大看看细节" | 滚轮就行 | Ctrl+滚轮，以光标为中心缩放，这是全球标准 |
| "这是第几张来着？" | 想知道位置 | 状态栏显示"第 N/M 张"，工具栏翻看按钮有边界反馈 |
| "拍歪了" | 转一下 | 旋转按钮+快捷键，转完就走，别问我保不保存 |
| "想沉浸看" | 全屏 | F11，黑底，别的都消失 |
| "看完了" | 关掉 | ESC / 点 ×，窗口消失，无挽留 |

## 三、关键设计决策（Design Decisions）

以下每条决策都与需求文档的具体条目绑定，便于追溯。

### DD-1：默认「适应窗口」而非 1:1

- **对应需求**：§4.1「打开后图片适应窗口显示」
- **决策**：默认适应窗口
- **论证**：目标用户是"看一眼"型，不是"审片"型。1:1 作为默认会让大量图片溢出窗口，需要用户手动缩小——这与"零认知负担"矛盾。1:1 留给专业人士，适应窗口留给大众用户。
- **参考**：Windows 照片查看器、macOS Preview 均默认适应窗口
- **替代方案被否决**：1:1 默认 → 符合专业工具逻辑但不符合目标用户画像

### DD-2：双击切换「适应窗口 ↔ 1:1」

- **对应需求**：§4.2「双击图片切换至 1:1 实际大小（100%）显示；再次双击恢复适应窗口」
- **决策**：双击是 toggle，在两个状态间切换
- **论证**：这是最高频的缩放操作——"看看这张图的原始细节"。双击比任何按钮+快捷键都快。反复双击在两个最常用缩放比例间切换，符合 Fitts' Law 的"最小操作成本"原则。
- **状态记忆**：如果用户在 1:1 状态下手动缩放到 200%，再次双击 → 回到适应窗口；再双击 → 回到 1:1（而非 200%）

### DD-3：旋转仅视图、不持久

- **对应需求**：§4.3「仅视图旋转，不修改原文件。关闭图片即丢失旋转状态」
- **决策**：旋转后无任何保存提示、无状态残留
- **论证**：核心逻辑——"旋转是临时的观看辅助，不是编辑行为"。如果弹出「是否保存？」对话框，反而增加认知负担。用户若需要永久旋转，那不在本产品范围（后续可加「另存为」）。
- **翻看行为澄清**：旋转→翻到下一张→翻回来→旋转丢失。这是因为"旋转是当前视图的临时属性，不是图片的持久属性"。翻看相当于"离开再回来"，状态不保留。

### DD-4：缩放指示短暂浮现

- **对应需求**：§4.2「缩放比例变化时短暂显示百分比（如右下角浮现「150%」，1.5 秒后消失）」
- **决策**：缩放指示器在右下角浮动 1.5 秒，带 200ms 淡出
- **论证**：用户缩放时需要确认当前比例，但不需要一直看到——长期显示占用视觉空间且无持续价值。1.5 秒 + 淡出是行业惯例（参考 YouTube 音量指示器、VLC 播放器的音量 OSD）。

### DD-5：翻看边界「到头即停」而非循环

- **对应需求**：§4.4「到目录第一张时『上一张』按钮灰掉；到最后一张时『下一张』按钮灰掉。不循环」
- **决策**：到达边界后按钮禁用（灰色），快捷键不响应
- **论证**：循环翻看会让用户迷失——"我到底翻到哪儿了？"。明确的边界感让用户知道自己在目录中的位置。这是「规矩」的典型体现。

### DD-6：全屏工具栏 2 秒自动隐藏

- **对应需求**：§4.5「鼠标移动时短暂显示底部工具栏（2 秒无操作自动隐藏）」
- **决策**：工具栏从底部滑入，鼠标静止 2 秒后滑出；鼠标再次移动立即滑入
- **论证**：全屏的核心价值是"沉浸"，工具栏应当召之即来、挥之即去。2 秒阈值参考了视频播放器的 UI 隐藏策略（YouTube 3 秒、VLC 可配置），对图片查看器场景稍缩短以保持简洁。

### DD-7：工具栏布局——顶部工具栏 + 底部状态栏

- **对应需求**：§4.7「状态栏常驻显示于窗口底部」+ §4 各功能入口
- **决策**：
  - **顶部工具栏**：功能操作区（缩放、旋转、翻看、全屏按钮）
  - **底部状态栏**：信息展示区（分辨率、大小、格式）
  - 工具栏与状态栏颜色保持统一，不抢夺图片注意力
- **论证**：经典的"操作在上、信息在下"布局，符合 F 型视觉扫描习惯。也与 Windows 照片查看器、浏览器布局一致。

## 四、用户旅程概览（User Journey Map）

```
┌─────────┐   ┌─────────┐   ┌──────────────┐   ┌──────────┐   ┌─────────┐
│ 触发打开 │ → │ 加载过渡 │ → │ 适应窗口查看  │ → │  浏览/操作 │ → │  关闭   │
│(双击/右键)│   │(短暂呈现)│   │(默认状态)    │   │(缩放/旋转/ │   │(ESC/×) │
└─────────┘   └─────────┘   └──────────────┘   │  翻看/全屏)│   └─────────┘
                                               └──────────┘
```

### 旅程各阶段 UX 关注点

| 阶段 | UX 关注点 | 设计策略 |
|------|----------|---------|
| 触发打开 | 关联正确、打开快 | 系统集成无感知，冷启动性能兜底 |
| 加载过渡 | 不白屏、不卡 UI 线程 | 轻量加载态（非模态），图片渐进显示 |
| 适应窗口 | 图片完整可见、比例正确 | 等比缩放算法，窗口默认大小合理 |
| 浏览/操作 | 流畅、可预测、有反馈 | 所有操作有视觉/状态反馈，快捷键全量覆盖 |
| 关闭 | 干净利落、不挽留 | 直接关闭，无弹窗、无提示 |

## 五、设计原则清单（Design Principles）

以下原则作为所有后续交互方案的"自查清单"：

| # | 原则 | 检验标准 |
|---|------|---------|
| 1 | **零学习成本** | 不做任何需要教程才能理解的设计。所有交互使用行业标准范式 |
| 2 | **键盘优先** | 鼠标能做的，键盘必须能做。每个功能有且仅有一个主快捷键 |
| 3 | **防误操作** | 旋转不保存、缩放可重置、全屏可退出。所有操作可逆 |
| 4 | **状态可见** | 当前在哪（第几张）、缩放比例、旋转角度——这些信息用户需要时就能看到 |
| 5 | **容错性** | 错误的打开方式（不支持格式、损坏文件）有清晰、友好的反馈 |
| 6 | **跨平台一致** | Windows/Linux 交互逻辑一致，快捷键在平台差异处做适配 |
| 7 | **克制不打扰** | 无弹窗广告、无引导教程、无动画炫技、无功能堆砌 |

## 六、UX 风险识别（基于需求文档的已知模糊点）

在分析需求文档 v1.0 后，UX 视角识别出以下需要明确的问题：

| # | 风险/模糊点 | 影响的功能 | 建议 | 状态 |
|---|-----------|-----------|------|------|
| R1 | GIF 动图旋转后是否继续播放？旋转的是每一帧还是整体容器？ | §4.3 旋转 | 建议：旋转整体视图容器，GIF 继续播放。旋转不暂停动画 | 待需求确认 |
| R2 | GIF 动图在翻看切换时，从 GIF 翻到静态图再翻回，GIF 是否重新开始播放？ | §4.4 翻看 | 建议：重新开始播放，保持行为简单一致 | 待需求确认 |
| R3 | 损坏的图片文件——需求只说"不支持的格式弹提示"，损坏但格式正确的文件如何处理？ | §4.1 打开 | 建议：增加"图片文件已损坏或无法解码"独立提示，区别于格式不支持 | 待需求确认 |
| R4 | 0 字节图片文件如何处理？ | §4.1 打开 | 建议：按损坏文件处理 | 待需求确认 |
| R5 | 翻看时遇到不支持格式"静默跳过"——如果目录中全是不可识别文件，是否显示空状态？ | §4.4 翻看 | 建议：当前图片可显示但翻看无更多图片时，翻看按钮灰掉即可；极端情况（唯一图片不可识别）按格式不支持处理 | 已可决策 |
| R6 | 超长文件名在标题栏如何截断？ | §4.1 标题栏 | 建议：中间截断（`DSC0...001.jpg`），保留扩展名和后缀可见 | 已可决策 |
| R7 | 全屏下鼠标 2 秒不动隐藏工具栏——移动鼠标后工具栏出现，此时鼠标在工具栏上方还是在图片区域？ | §4.5 全屏 | 建议：工具栏出现时不改变鼠标位置，工具栏覆盖在图片底部上方 | 已可决策 |
| R8 | 「适应窗口」按钮/快捷键——已有双击切换，是否还需要独立按钮？ | §4.2 缩放 | 建议：需要。双击是效率入口，按钮是发现性入口（新用户不知道双击）。两者并存 | 已可决策 |
| R9 | 缩放至 3200% 上限后再滚轮放大——是否给反馈？ | §4.2 缩放 | 建议：短暂显示"3200%（已达上限）"，1.5 秒消失。下限同理 | 已可决策 |
| R10 | 拖拽平移时图片小于窗口——是否允许拖拽？ | §4.6 拖拽 | 建议：不允许。图片未超出窗口时拖拽无意义，光标不变成抓手 | 已可决策 |

## 七、与 UI 设计师的协作接口

UX 已确定的交互规范，UI 设计师可直接在此基础上发挥：

| UX 决策（UI 无需重新讨论） | UI 可自由发挥的 |
|--------------------------|---------------|
| 工具栏位置：顶部（功能）+ 底部（信息）| 工具栏颜色、图标风格、间距 |
| 全屏工具栏 2 秒自动隐藏 | 滑入/滑出动画细节 |
| 缩放指示器右下角 1.5 秒浮现 | 指示器样式（气泡/标签/半透明） |
| 状态栏三栏信息（分辨率｜大小｜格式）| 分隔符样式、字体大小 |
| 双击切换适应窗口↔1:1 | 双击的视觉反馈（可选） |
| 按钮功能分组（缩放组、旋转组、翻看组、全屏）| 各组之间的分隔方式 |
