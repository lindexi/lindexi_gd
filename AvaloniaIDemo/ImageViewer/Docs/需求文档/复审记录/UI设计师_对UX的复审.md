## 三、对 UX 7 个需求模糊点的复审（Q1~Q7）

UX 在 06 完成说明中主动提出的 7 个需求模糊点，已经被需求分析师在 RE-01 中全部裁决。我从 UI 实现角度评价这些裁决：

| # | UX 模糊点 | 需求裁决 | UI 实现可行性评价 |
|---|---------|---------|------------------|
| **Q1** | GIF 动图支持旋转？ | 暂搁置，v2.0 评估 | ✅ **同意**。v1.0 不引入 GIF 动图可避免引入新依赖（ImageShape 库需确认）。如未来加入，UI 需新增"播放/暂停"按钮。 |
| **Q2** | 翻看边界循环 vs 停止？ | 采纳"停止" | ✅ **同意**。停止行为 UI 实现简单（if first/last block），循环则需额外状态。 |
| **Q3** | 切换图片缩放继承 vs 重置？ | 采纳"重置为适应窗口" | ✅ **同意**。继承需在内存维护"每张图片的缩放比例"映射，重置更简单且符合用户预期。 |
| **Q4** | 全屏下滚轮=翻看 vs 缩放？ | 采纳"翻看 + Ctrl+滚轮=缩放" | ✅ **同意**。符合 Windows 照片 / macOS Preview 惯例，用户零学习成本。 |
| **Q5** | 单实例跨虚拟桌面行为？ | 采纳"移到当前桌面" | ✅ **同意**。符合 OS 单实例应用惯例，UI 实现需调用 OS API（如 Windows 的 SetWindowPos）。 |
| **Q6** | 是否提供 Delete 删除？ | 采纳"不提供" | ✅ **同意**。删除不可逆，引入风险大，UI 工具栏无入口。 |
| **Q7** | 工具栏是否允许自定义？ | 采纳"不允许" | ✅ **同意**。保持产品一致性，"板正"气质的体现。UI 实现也省去自定义布局引擎。 |

**结论**：✅ 7 个裁决全部合理，UI 均可低成本实现。

## 四、UI 实现可行性评估

### 4.1 控件实现难度

| UX 元素 | UI 实现方式 | 难度 | 备注 |
|---------|------------|------|------|
| 工具栏按钮组 | OS 原生按钮 + Token 样式 | ⭐ 低 | 标准控件 |
| 缩放下拉菜单 | Popup 容器 + ListItem | ⭐⭐ 中 | 需自定义样式 |
| OSD 浮层 | Toast 容器 + opacity 过渡 | ⭐⭐ 中 | 需定时器控制 |
| 全屏切换 | OS API（Windows: SetWindowPos 等）| ⭐⭐ 中 | 需注意多平台差异 |
| 单实例行为 | OS API（Windows: FindWindow + SetForegroundWindow）| ⭐⭐⭐ 中高 | 跨虚拟桌面需额外处理 |
| 错误对话框 | Modal Dialog + Icon + Text + Button | ⭐ 低 | 标准控件 |

**总体**：✅ 实现难度可控，无技术风险。
