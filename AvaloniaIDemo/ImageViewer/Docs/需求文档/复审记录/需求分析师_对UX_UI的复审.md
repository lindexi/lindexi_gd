## 二、3 项冲突的裁决（C1 / C2 / C3）

### 🔴 冲突 C1：旋转后是否弹出"是否保存"对话框？

| 维度 | 现状 |
|------|------|
| UX 设计 | S4 线框包含"是否保存"对话框；02 文档写"关闭时如果有旋转状态，询问是否保存" |
| UI 立场 | 不设计此对话框（理由：§4.3 明确"仅视图旋转，关闭即丢失"） |
| 需求基线 | §4.3：「仅视图旋转，不修改原文件。关闭图片即丢失旋转状态，下次打开恢复原方向」；§2.2 明确不包含编辑/导出 |

#### 我的裁决：**采纳 UI 立场，UX 需更新**

**裁决理由**（按优先级）：

1. **范围裁定第一**：需求文档 §2.2 明文规定 v1.0 不包含编辑/导出/打印/分享等功能。保存旋转后的图片本质上是写入原文件，属于编辑/导出范畴，超出 v1.0 范围。

2. **需求 §4.3 语义明确**：使用了"仅视图旋转"、"不修改原文件"、"关闭即丢失"、"不保留"四个排他性表达，没有保留任何"可选保存"的语义空间。

3. **UX 自身的论证矛盾**：UX 01 总纲中写到"旋转是临时的观看辅助，不是编辑行为"，06 完成说明中也写"无状态残留"。但 02 文档却冒出"关闭时询问是否保存"，与自身设计哲学不一致。

4. **产品定位一致**：本产品是"查看器"不是"编辑器"。若提供"保存旋转结果"功能，会引入写入原文件的潜在风险（覆盖、权限、并发），与"克制"气质冲突。

#### 裁决后动作清单

- [x] **UX**：删除 S4 线框中的"是否保存"对话框
- [x] **UX**：删除 02 文档中"关闭时如果有旋转状态，询问是否保存"段落
- [x] **后续**：如果需求方未来希望加"另存为"，建议放到 v2.0 单独迭代（需更新需求文档）
- [x] **需求文档**：维持 v1.0 §4.3 不变，无需 v1.1 更新

### 🔴 冲突 C2：是否提供 `Ctrl+S` / `Ctrl+Shift+S` 快捷键？

| 维度 | 现状 |
|------|------|
| UX 设计 | UX 04 快捷键表 2.1 节包含 `Ctrl+S`（保存）和 `Ctrl+Shift+S`（另存为） |
| UI 立场 | v1.0 不提供（理由：与 C1 同源，§2.2 排除编辑/导出） |
| 需求基线 | §2.2 不包含编辑/导出；§4 各功能定义也未提及保存 |

#### 我的裁决：**采纳 UI 立场，UX 需移除**

**裁决理由**：

1. **基线一致性**：与 C1 同源。若 C1 已裁决"不设计保存对话框"，则 Ctrl+S/另存为 也没有触发对象。
2. **UX 自洽性问题**：UX 04 已自评"v1.0 快捷键不可自定义"（DD-7），保持"设定即最终"。C1 已裁决不保存，则 C2 必须同步移除，否则 UX 设计自相矛盾。
3. **无独立价值**：单独存在的保存/另存为快捷键，若工具栏没入口，快捷键无效，是死键。
4. **`Ctrl+C` 不冲突**：复制图片到剪贴板是读取操作（不是写入文件），属于查看器合理范围，可保留。

#### 裁决后动作清单

- [x] **UX 04**：删除 2.1 节的 `Ctrl+S` 与 `Ctrl+Shift+S` 两行
- [x] **UX 04**：保留 `Ctrl+C`（复制图片到剪贴板）
- [x] **需求文档**：维持 v1.0 §2.2 不变

### 🔴 冲突 C3：是否提供 `Delete` 删除快捷键？

| 维度 | 现状 |
|------|------|
| UX 设计 | UX 04 第 2.6 节保留 `Delete` 键，但标注"⚠️ 待需求确认是否提供删除功能" |
| UI 立场 | 不设计任何删除入口（理由：§2.2 明确不包含"删除图片功能"） |
| 需求基线 | §2.2 明确不包含"删除图片功能" |

#### 我的裁决：**采纳 UI 立场，UX 需删除**

**裁决理由**：

1. **基线明确**：需求文档 §2.2 用排除法明文写了"删除图片功能"——这是产品边界设定，不是 UX 可越权扩展的。
2. **高风险操作**：删除是不可逆操作。在 v1.0 阶段引入的风险（误删、权限、回收站逻辑）与"克制"气质不符。
3. **职责划分**：用户若有删除需求，应使用 OS 文件管理器（回收站机制成熟）。产品不重复造轮子。
4. **UX 自身不自信**：UX 已主动标注"⚠️ 待确认"，说明对该功能是否纳入判断不确定。应尊重基线裁决。

#### 裁决后动作清单

- [x] **UX 04**：明确删除 `Delete` 快捷键条目（不留灰色态）
- [x] **需求文档**：维持 v1.0 §2.2 不变

## 三、UX 提出的 7 个需求模糊点（Q1~Q7）裁决

| # | UX 提出的模糊点 | 我的裁决 | 依据 |
|---|----------------|---------|------|
| **Q1** | GIF 动图是否支持旋转？ | **暂搁置** | v1.0 不引入 GIF 动图播放能力（需 ImageShape 库支持，研发未确认）。留待 v2.0 评估 |
| **Q2** | 翻看到首尾时循环 vs 停止？ | **采纳"停止"** | 与 §4.4「到头即停」明文一致，符合"规矩"气质 |
| **Q3** | 切换图片时缩放比例继承 vs 重置？ | **采纳"重置为适应窗口"** | 避免用户在多图间缩放状态不一致产生的迷失感，且适应窗口是 §4.1 默认行为 |

| **Q5** | 单实例跨虚拟桌面行为？ | **采纳"移到当前桌面"** | 与 OS 单实例应用惯例一致。移到当前虚拟桌面显示窗口，避免焦点丢失 |
| **Q6** | 是否提供 Delete 删除？ | **采纳"不提供"**（= C3 裁决） | §2.2 排除 |
| **Q7** | 工具栏是否允许自定义？ | **采纳"不允许"** | 板正克制，保持产品一致性。DD-7 已确立 |

## 六、需求文档更新建议（v1.1）

基于本次复审结论，建议需求文档 v1.0 → v1.1 增加 3 项明确化：

### 建议在 §4.1「打开后图片适应窗口显示」后追加：

> AC-01a：切换图片时，缩放比例自动重置为"适应窗口"，不继承上一张的缩放状态。

### 建议在 §4.4「到目录第一张时按钮灰掉」后追加：

> AC-04a：到达边界时不循环翻看，按钮处于禁用状态，快捷键 `←` `→` `Home` `End` 在边界处不响应。

### 建议在 §4.6「鼠标移动时短暂显示底部工具栏」后追加：

> AC-06a：全屏模式下，无修饰键的滚轮 = 翻看（上一张/下一张）；`Ctrl+滚轮` = 缩放。两种行为不混叠。

### 不变的部分（明确"维持原状"）：

- §2.2 范围不变（不含编辑/导出/删除）
- §4.3 旋转语义不变（仅视图，不持久，关闭即丢失）
- 13 条 AC 主体不变
