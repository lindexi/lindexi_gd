## 三、对 UI 微调的逐项复审（UX 立场）

### ⚠️ 微调 #1：缩放指示器淡出 200ms → 300ms

| 维度 | 内容 |
|------|------|
| UX 原设计 | "短暂浮现后淡出"，未明确毫秒数；早期脑暴提过 200ms |
| UI 决定 | 300ms 淡出 + 1.5s 显示 |
| 设计理由 | "让消失更轻柔"，符合克制气质 |

**UX 评价**：✅ **同意**。300ms 在视觉感知上确实是"轻柔而非骤停"，且 OSD 是辅助信息，确实需要柔和淡出。如果未来有用户反馈"消失太快看不清"，可以再回到 200ms。先用 300ms 是合理决策。

---

### ⚠️ 微调 #2：全屏工具栏动效细节补充

| 维度 | 内容 |
|------|------|
| UX 原设计 | "2 秒自动隐藏"，"滑出"二字模糊 |
| UI 决定 | 2 秒 + slide-down + opacity 双属性 200ms 过渡 |

**UX 评价**：✅ **同意**。"滑出"原本是模糊描述，UI 明确为 slide-down + opacity 双属性是设计细化而非偏离，对研发实现更友好。

---

### ⚠️ 微调 #3：S8 文件损坏图标色改为警告橙

| 维度 | 内容 |
|------|------|
| UX 原设计 | 未指定颜色，与 S7 共用线框 |
| UI 决定 | `--warning` #D4A040 橙色，区别于 S7 的 `--error` #E05555 红色 |
| 设计理由 | S7=用户选错格式（确定性问题，红色）；S8=文件可能临时问题（疑似问题，橙色）|

**UX 评价**：✅ **强烈同意**。颜色差异传达"问题性质不同"，降低用户挫败感，是优秀的设计决策。建议后续在 UX 文档的"视觉与色彩"章节补充这条规则——**红色=用户操作导致，橙色=系统/文件导致**。

---

### ⚠️ 微调 #4：S9 加载态不用 spinner 用静态文字

| 维度 | 内容 |
|------|------|
| UX 原设计 | 提到"加载中状态"，未明确形态 |
| UI 决定 | 仅显示"正在加载..."静态文字，200ms 内完成则跳过 |
| 设计理由 | spinner 在短加载场景制造焦虑，静态文字更克制；参考 macOS Preview |

**UX 评价**：✅ **同意**。图片查看器的加载通常很快（<500ms），spinner 反而是噪音。macOS Preview 的处理方式确实是行业最佳实践。UX 文档可以补一句：**S9 加载态用静态文字而非 spinner，加载 <200ms 直接跳过**。

## 四、对 UI 视觉规范的复审

### 4.1 颜色 Token 体系（01_视觉风格定义.md）

| Token | UI 定义 | UX 立场评价 |
|-------|---------|-------------|
| `--surface-chrome` #2C2C2C | 工具栏/状态栏背景 | ✅ 暖暗灰，比纯黑 `#000000` 高级，符合"板正" |
| `--surface-image-area` #1A1A1A | 图片显示区背景 | ✅ 比 chrome 深一阶，让图片"浮起来" |
| `--surface-hover` / `--surface-pressed` | 交互态 | ✅ 反馈清晰，符合 OS 习惯 |
| `--text-primary` #E8E8E8 | 主要文字 | ✅ 非纯白，对比度舒适 |
| `--text-secondary` / `--text-tertiary` #999 / #666 | 次要/三级文字 | ✅ 层级清晰 |
| `--accent` #4A90D9 | 强调色（缩放比例等）| ✅ 蓝色符合"克制"，不是高饱和橙红 |
| `--error` #E05555 | 错误（红）| ✅ S7 用色 |
| `--warning` #D4A040 | 警告（橙）| ✅ S8 用色 |

**结论**：✅ Token 体系完整、克制、专业。可直接进入研发。
