- **技术栈**：Avalonia（跨平台桌面）+ ImageShape（图片解码）
- **目标平台**：Windows 10+ / Linux（Ubuntu/Debian/Fedora）
- **用户群**：普通大众用户
- **气质关键词**：板正、克制、专业、简洁（非极简）

## 三、协作流程（共 5 阶段）

### 阶段 B：UX 先行（当前阶段 ⏳）

**@角色**：UX设计师
**任务**：基于 v1.0 需求文档，输出《交互设计总纲》+ 关键页线框图
**输出位置**：`设计稿\UX设计\01_交互设计总纲.md` 等
**完成判据**：UX文档包含工具栏布局、信息架构、快捷键体系、关键线框图

### 阶段 C：UI 接力

**@角色**：UI设计师
**任务**：基于 v1.0 需求文档 + UX 方案，输出视觉风格定义 + 高保真设计稿
**输出位置**：`设计稿\UI设计\01_视觉风格定义.md` 等
**完成判据**：UI文档包含色板/字体/间距/组件库/关键页高保真稿

### 阶段 D：交叉评审（角色相互沟通环节）

| 评审人 | 评审对象 | 评审重点 | 输出文件 |
|--------|----------|----------|----------|
| 需求分析师 | UX + UI 方案 | 需求一致性、AC可追溯、范围偏差 | `复审记录\需求分析师_对UX_UI的复审.md` |
| UX设计师 | UI 视觉稿 | 交互一致性、键盘/手势/状态可实现性 | `复审记录\UX设计师_对UI的复审.md` |
| UI设计师 | UX 线框图 | 视觉可行性、间距字号合理性 | `复审记录\UI设计师_对UX的复审.md` |
| 测试工程师 | 全部文档 | 可测试性、AC可量化性、边界场景覆盖 | `复审记录\测试工程师_评审意见.md` |

### 阶段 E：测试用例产出

**@角色**：测试工程师
**任务**：基于需求 + UX + UI 输出完整测试用例（功能/异常/兼容/性能/易用）
**输出位置**：`测试用例\` 目录下
**完成判据**：13 条 AC 全部覆盖，异常/边界场景有独立用例

### 阶段 F：闭环迭代

- 测试工程师发现的问题 → 反馈给需求分析师/UX/UI → 更新文档
- 各方反复对齐直到所有文档版本一致且无歧义
- 最终输出 v1.0 设计基线 + v1.0 测试基线

---

## 四、协作规则

1. **角色间沟通** 通过协调者（助手）发起
2. **每个角色产出** 必须写入对应目录的文件，不允许只口述
3. **跨角色评审** 必须明确指出问题点、影响范围、建议方案
4. **任何需求变更** 必须先经需求分析师评估，更新需求基线后其他角色再响应
5. **版本管理**：每次更新需注明版本号（如 v1.0 → v1.1）和变更说明
6. **测试驱动**：测试工程师在任何阶段都可以提出质疑和场景补充
