# 搞一个「让 LLM 直接生成 PPT」的小实验

### 为什么又造一个轮子？

市面上 AI PPT 的方案我大概看了几类：

- **PPTX 直接操作**：让模型写 Python 调 python-pptx，能跑但生成的版面感很差，全是模板填空
- **HTML 转 PPT**：模型写 HTML/CSS 然后截图或转 PPTX，问题是 CSS 太灵活了，模型容易写出各种骚操作，渲染结果不可控
- **JSON Schema 约束**：定义一套 JSON 结构让模型填，结构是稳了，但模型对 JSON 的排版直觉很差，而且嵌套层级多了之后 JSON 可读性爆炸

我想试的是第四条路：**自定义一套极简的 XML 标签语言，只保留 PPT 排版中最核心的几个元素，让模型生成 XML，然后我们这边写一个确定性渲染引擎来画出来。**

### 核心理念：「生成 → 渲染 → 反馈」闭环

这是最让我兴奋的点。

模型的天然短板是什么？**它不知道文本渲染出来到底会换几行、占多高。** 因为它没有字体度量信息。

我们的做法很简单：

1. 模型生成 XML（其中文本可以写 `Width="340"` 约束宽度，但不写高度）
2. 渲染引擎真实地排版这段文本，算出来「哦这段文字在 340px 宽度下折了 3 行，实际高度 63px」
3. 引擎把 `RenderSize="340x63"` `RenderLocation="80x120"` `ActualLineCount="3"` 回填到 XML 里，还给模型
4. 模型看到回填数据，发现「卧槽溢出了」，下一轮把字号从 16 改成 14，或者把容器高度从 48 改成 72

**等于说模型是设计师，渲染引擎是它的「像素级排版助手」。** 模型不需要猜，它可以直接试，然后看实际结果。

如果模型支持多模态（GPT-4o、Claude 3.5 之类），我们还能直接把渲染截图丢给它，连「这个间距看着不太协调」这种主观审美问题都能纠正。

### 有多简单？整个标签体系就 5 个标签

```
Page          — 画布，1280x720
Panel         — 容器，可以嵌套
Rect          — 矩形（圆角卡片什么的）
TextElement   — 文本
Image         — 图片
```

属性也很克制，总共 20 来个。文本对齐、颜色、圆角、透明度……够用但不泛滥。没有 CSS 那种几百个属性的地狱。

图片不是写 URL，而是写一个 `Source="img_hero_001"` 这种资源 ID。图片哪来的？另说——可能是 RAG 检索、可能是 SubAgent 去 Unsplash 搜、也可能就是个内置图库。重点是**不在生成阶段污染 XML**。

### 当前进展和想验证的东西

最想验证的假设：**模型看到 ActualLineCount 反馈后，能不能在下一次生成中有效纠正溢出？** 如果能，这条路就基本跑通了。

### V1 不做的东西（先克制）

- 富文本（`<Span>` 多样式混排）
- 样式复用系统
- 流式布局（V1 纯绝对定位）
- 动画 / 过渡
- measure_text 工具调用

这些都在 V2 规划里，但第一版必须够小够快，跑通闭环再说
