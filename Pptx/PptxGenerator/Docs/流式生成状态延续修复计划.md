# 流式生成状态延续修复计划

## 概述

本文档描述 `StreamingSlideGenerator` 在错误反馈重试和跨轮对话场景下的状态丢失问题及修复方案。

当前 `RunStreamingLoopAsync` 每次调用都会创建全新的 `SlideMlPipelineContext` 和 `SlideStreamingPipeline`，导致上一轮已合并的 XML DOM 树状态无法传递到下一轮。这与 [SlideML 流式输出实现计划](./SlideML%20流式输出实现计划.md) 中"中断后上下文保留"的设计意图相矛盾。

---

## 问题分析

### 问题 A：错误反馈重试丢失合并状态

`GenerateAsync` 的重试循环中，每次调用 `RunStreamingLoopAsync` 都会执行：

```csharp
var context = new SlideMlPipelineContext();
var streamingPipeline = new SlideStreamingPipeline(_promptProvider, _renderTool.RenderPipeline, _dispatcher);
```

这导致：

1. `SlideMlStreamingMerger` 的 `XDocument` DOM 树被重置为空——上一轮已成功合并的片段全部丢失
2. `_idIndex`、`_styleIdIndex`、`_danglingElements` 全部清空——后续片段无法通过 Id 匹配已有元素
3. 虽然错误反馈消息（`BuildErrorFeedback`）将上一轮的 `CurrentMergedXml` 作为文本附在消息中，但 LLM 必须从头重新输出所有片段，无法做增量修正
4. `SlideMlPipelineContext` 的诊断信息也被重置

**影响**：重试效率极低。LLM 需要重新输出完整内容而非仅输出修正部分，浪费 token 和时间。合并器无法利用已有的 DOM 状态做增量 Merge。

### 问题 B：跨轮对话无法延续状态

`SlideGenerationPipeline.SendMessageAsync` 中，每次调用都创建新的 `StreamingSlideGenerator`：

```csharp
var generator = new StreamingSlideGenerator(
    _copilotChatManager, _promptProvider, SlideMlRenderTool, _dispatcher);
await generator.GenerateAsync(userMessage, isFirstMessage, cancellationToken, ...);
```

当 `isFirstMessage` 为 `false`（用户发送第二条消息）时：

1. 新的 `StreamingSlideGenerator` 没有上一轮的 `SlideStreamingPipeline` 实例
2. 新的 `SlideStreamingPipeline` 内部的 `SlideMlStreamingMerger` DOM 树为空
3. LLM 无法在已有幻灯片基础上做增量修改（添加/删除/调整元素），必须从零重新生成

**影响**：用户无法通过后续对话迭代修改幻灯片。例如用户说"把标题改成红色"，LLM 看到的是空状态，无法定位已有标题元素进行属性 Merge。

### 根因总结

| 组件 | 当前行为 | 期望行为 |
|------|----------|----------|
| `RunStreamingLoopAsync` | 每次创建新的 `SlideStreamingPipeline` + `SlideMlPipelineContext` | 重试时复用同一实例，保留 DOM 树和 Id 索引 |
| `SlideMlStreamingMerger` | 仅有 `Reset()` 清空状态，无从 XML 恢复的方法 | 需要提供从已有 XML 初始化 DOM 树和索引的能力 |
| `SlideStreamingPipeline` | 仅有 `Reset()` 清空状态，无从 XML 恢复的方法 | 需要提供从已有 XML 初始化的能力 |
| `StreamingSlideGenerator` | 每次 `GenerateAsync` 调用均为无状态实例 | 需要持有或接收上一轮的合并状态 |
| `SlideGenerationPipeline.SendMessageAsync` | 每次创建新的 `StreamingSlideGenerator` | 需要传递已有合并 XML 供新轮初始化 |
| `BuildErrorFeedback` | 要求 LLM"重新输出完整的 SlideML XML" | 应改为要求 LLM"仅输出修正和后续片段" |

---

## 修复方案

### 核心思路

1. **重试循环内复用 Pipeline**：将 `SlideStreamingPipeline` 和 `SlideMlPipelineContext` 的创建提升到 `GenerateAsync` 的重试循环之外，使其在重试轮次间保持状态
2. **Merger 支持从 XML 恢复**：为 `SlideMlStreamingMerger` 新增 `LoadFromXml` 方法，从完整 XML 字符串重建 DOM 树和 Id/StyleId 索引
3. **Pipeline 暴露初始化入口**：为 `SlideStreamingPipeline` 新增 `LoadFromXml` 方法，委托给内部 merger
4. **跨轮对话传递状态**：`StreamingSlideGenerator.GenerateAsync` 接收可选的初始 XML 参数；`SlideGenerationPipeline.SendMessageAsync` 在 `isFirstMessage` 为 `false` 时传入上一轮的渲染回填 XML
5. **修正错误反馈措辞**：将"重新输出完整的 SlideML XML"改为"仅输出修正和后续片段"，配合增量合并

### 设计决策

| 决策 | 理由 |
|------|------|
| 重试时复用同一 `SlideStreamingPipeline` | 与实现计划文档 §中断后上下文保留的关键 一致——"重试时不创建新的 SlideMlStreamingMerger，保留已合并的 DOM 树状态" |
| 重试时复用同一 `SlideMlPipelineContext` 但调用 `Reset()` 清空诊断信息 | 诊断信息（Errors/Warnings）是每轮独立的，不应跨轮累积；但 `MaterialResourceManager` 等资源管理器应跨轮保留 |
| Merger 的 `LoadFromXml` 解析 XML 后重建 Id/StyleId 索引 | 后续片段需要通过 Id 匹配已有元素，索引必须完整重建 |
| 跨轮对话时使用 `LatestRenderedXml`（渲染回填后的 XML）作为初始状态 | 回填后的 XML 包含 `RenderSize`/`RenderLocation` 等引擎计算值，但这些属性在后续 Merge 中会被片段显式属性覆盖或保留，不影响合并逻辑 |
| `SlideMlFragmentExtractor` 每轮重试仍新建 | 片段提取器的缓冲区不应跨轮保留（上一轮的残留缓冲区内容已无意义），与实现计划文档步骤 7 一致 |

## 实施步骤

### 步骤 1：SlideMlStreamingMerger 新增 LoadFromXml

**文件**：`Code/PptxGenerator.Core/Streaming/SlideMlStreamingMerger.cs`（修改）

**新增方法**：

```csharp
/// <summary>
/// 从完整 XML 字符串加载初始状态，重建 DOM 树和 Id/StyleId 索引。
/// 用于跨轮对话或错误重试时恢复已有幻灯片状态。
/// </summary>
/// <param name="xml">完整的 SlideML XML 字符串（根元素为 Page 或悬空元素）。</param>
/// <param name="context">渲染上下文，用于收集加载过程中的警告和错误。</param>
public void LoadFromXml(string xml, SlideMlPipelineContext context)
```

**实现要点**：

- 调用 `Reset()` 清空现有状态
- 使用 `XDocument.Parse(xml)` 解析 XML
- 如果根元素为 `Page`：直接作为 `_document`，调用 `RegisterIdElements` 和 `RegisterStyleIdElements` 重建索引
- 如果根元素不是 `Page`（悬空元素作为根）：将根元素作为悬空元素加入 `_danglingElements`，同样注册索引
- 解析失败时通过 `context.AddError` 报告，不抛出异常
- 忽略 `RenderSize`/`RenderLocation`/`ActualLineCount` 等引擎回填属性（不剥离，但后续 Merge 中片段显式属性会覆盖它们）

**边界情况**：

| 场景 | 处理 |
|------|------|
| XML 为空或空白 | 不做任何操作，保持空状态 |
| XML 解析失败 | 报 `[Error]`，保持空状态 |
| XML 根元素为 Page | 正常加载，重建全部索引 |
| XML 根元素为非 Page 元素（悬空元素） | 作为悬空元素加载 |
| XML 包含多个顶层元素 | `XDocument.Parse` 只接受单根，报 `[Error]` |

### 步骤 2：SlideStreamingPipeline 新增 LoadFromXml

**文件**：`Code/PptxGenerator.Core/Streaming/SlideStreamingPipeline.cs`（修改）

**新增方法**：

```csharp
/// <summary>
/// 从完整 XML 字符串加载初始合并状态。
/// 用于跨轮对话或错误重试时恢复已有幻灯片状态。
/// </summary>
/// <param name="xml">完整的 SlideML XML 字符串。</param>
/// <param name="context">渲染上下文。</param>
public void LoadFromXml(string xml, SlideMlPipelineContext context)
{
    ArgumentNullException.ThrowIfNull(xml);
    ArgumentNullException.ThrowIfNull(context);
    _merger.LoadFromXml(xml, context);
}
```

### 步骤 3：重构 StreamingSlideGenerator 支持状态延续

**文件**：`Code/PptxGenerator.Core/Pipeline/StreamingSlideGenerator.cs`（修改）

#### 3a. GenerateAsync 签名变更

```csharp
public async Task GenerateAsync(
    string userMessage, bool isFirstMessage, CancellationToken cancellationToken,
    Action? onPropertiesChanged = null,
    string? initialXml = null)
```

新增 `initialXml` 参数：当 `isFirstMessage` 为 `false` 时，由调用方传入上一轮的渲染回填 XML，用于初始化合并器状态。

#### 3b. GenerateAsync 重构

```
GenerateAsync(userMessage, isFirstMessage, cancellationToken, onPropertiesChanged, initialXml)
  │
  ├─ 1. 在重试循环外创建 context 和 streamingPipeline
  │     var context = new SlideMlPipelineContext();
  │     var streamingPipeline = new SlideStreamingPipeline(...);
  │
  ├─ 2. 如果 initialXml 非空，调用 streamingPipeline.LoadFromXml(initialXml, context)
  │     初始化合并器 DOM 树和索引
  │
  └─ 3. for (attempt = 0; attempt <= MaxRetries; attempt++)
        │
        ├─ 重试时 context.Reset()（仅清空诊断信息，不清空合并器状态）
        ├─ 调用 RunStreamingLoopAsync(currentMessage, includeSystemPrompt, ..., streamingPipeline, context)
        └─ 错误反馈逻辑不变，但 streamingPipeline 和 context 跨轮保留
```

#### 3c. RunStreamingLoopAsync 签名变更

```csharp
private async Task<(bool HasErrors, string ErrorFeedback)> RunStreamingLoopAsync(
    string userMessage, bool includeSystemPrompt,
    CancellationTokenSource errorCancellationTokenSource, CancellationToken externalCancellationToken,
    SlideStreamingPipeline streamingPipeline, SlideMlPipelineContext context)
```

接收外部传入的 `streamingPipeline` 和 `context`，不再自行创建。

#### 3d. RunStreamingLoopAsync 内部调整

- 移除 `var context = new SlideMlPipelineContext()` 和 `var streamingPipeline = new SlideStreamingPipeline(...)` 的创建
- `Rendered` 事件订阅保持不变（每轮都订阅，使用后取消订阅或在循环外订阅一次）
- 错误收集逻辑不变
- `context.Reset()` 的调用移到 `GenerateAsync` 的重试循环中（在调用 `RunStreamingLoopAsync` 之前）

#### 3e. BuildErrorFeedback 措辞修正

将"请修正上述错误，重新输出完整的 SlideML XML"改为：

```
请修正上述错误，仅输出修正和后续片段。
不要重复已成功的片段，合并器已保留之前成功合并的内容。
```

### 步骤 4：SlideGenerationPipeline 传递初始 XML

**文件**：`Code/PptxGenerator.Core/Pipeline/SlideGenerationPipeline.cs`（修改）


## 数据流（修复后）

### 错误反馈重试场景

```
GenerateAsync(userMessage, isFirstMessage: true)
  │
  ├─ 创建 context + streamingPipeline（循环外）
  │
  ├─ attempt 0:
  │    RunStreamingLoopAsync(msg, ..., streamingPipeline, context)
  │    → LLM 输出 <Page>...</Page> + <Panel>...</Panel>（部分成功后出错）
  │    → merger DOM 树保留已成功合并的片段
  │    → 返回 (true, errorFeedback)
  │
  ├─ context.Reset()  ← 仅清空诊断信息
  │
  ├─ attempt 1:
  │    RunStreamingLoopAsync(errorFeedback, ..., streamingPipeline, context)
  │    → streamingPipeline 仍是同一个实例，DOM 树保留 attempt 0 的合并结果
  │    → LLM 仅输出修正片段（如 <Panel Id="x" .../> 修正属性）
  │    → merger 通过 Id 匹配已有元素，做增量 Merge
  │    → 成功，返回 (false, "")
  │
  └─ 完成
```

### 跨轮对话场景

```
用户第一轮："生成一个标题页"
  → GenerateAsync(msg, isFirstMessage: true, initialXml: null)
  → streamingPipeline 从空状态开始
  → LLM 输出完整幻灯片片段序列
  → 最终渲染，SlideMlRenderTool.LatestRenderedXml = "<Page>...</Page>"

用户第二轮："把标题改成红色"
  → GenerateAsync(msg, isFirstMessage: false, initialXml: LatestRenderedXml)
  → streamingPipeline.LoadFromXml(LatestRenderedXml, context)
  → merger DOM 树 = 上一轮的完整幻灯片
  → LLM 输出 <TextElement Id="title" Foreground="#FF0000"/>
  → merger 通过 Id="title" 匹配已有元素，属性 Merge
  → 最终渲染，标题变红
```

## 风险与开放问题

1. **回填属性的处理**：`LatestRenderedXml` 包含 `RenderSize`/`RenderLocation`/`ActualLineCount` 等引擎回填属性。`LoadFromXml` 加载时保留这些属性，后续 Merge 中片段的显式属性会覆盖它们。如果片段未显式声明这些属性，旧的回填值会保留——但下次渲染时会重新回填，不会产生错误。可选择在 `LoadFromXml` 中剥离回填属性，但这会增加复杂度，暂不处理。

2. **Rendered 事件订阅时机**：当前 `RunStreamingLoopAsync` 每次调用都订阅 `streamingPipeline.Rendered` 事件。重构后如果 `streamingPipeline` 跨轮复用，需要确保事件订阅不会重复累积。方案：在 `GenerateAsync` 中订阅一次，或使用局部变量捕获确保每次 `RunStreamingLoopAsync` 取消上一次的订阅。

3. **AgentSession 复用**：当前每次 `RunStreamingLoopAsync` 都通过 `CreateManualSendMessageContextAsync` 创建新的会话上下文。`ManualSendMessageContext` 内部复用 `CopilotChatSession.AgentSession`（如果已存在），因此 AgentSession 的历史是跨轮保留的。这一行为无需修改。

4. **initialXml 过大时的 token 消耗**：跨轮对话时将完整渲染 XML 传入合并器，LLM 不需要重新输出完整 XML，只需输出修改片段。但如果 LLM 的上下文窗口中没有上一轮的完整内容（被 ChatReducer 压缩），LLM 可能不知道当前幻灯片的结构。方案：在非首轮消息的提示词中附加上一轮的合并 XML 摘要或 `get_slide_state` 工具供 LLM 查询。这超出本次修复范围，作为后续优化。

5. **错误反馈中包含的 XML 与合并器状态的一致性**：`BuildErrorFeedback` 使用 `streamingPipeline.CurrentMergedXml` 构建反馈消息。修复后合并器跨轮保留，`CurrentMergedXml` 反映的是已成功合并的部分（不包含出错的片段），反馈消息中的 XML 与合并器状态一致。LLM 看到的"已生成的 SlideML"与合并器实际状态吻合，可以正确做增量修正。
