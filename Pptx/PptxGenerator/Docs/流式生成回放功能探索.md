# 流式生成回放功能探索

## 概述

本文档记录对“在某条用户消息上右击选择重新开始，并通过 `FakeChatClient` 回放原本流式生成内容”的探索结果。

目标功能：

1. 用户在聊天列表中的某条用户消息上右击，选择“重新开始”。
2. 系统清空当前 SlideML 流式状态，包括合并器 DOM、Id/StyleId 索引、片段提取器缓冲区、渲染结果等。
3. 系统清空或重建当前聊天 Session，使后续真实生成不继续携带被截断后的对话历史。
4. 系统使用 `FakeChatClient` 回放原本已存在的聊天内容，让 SlideML 流式管道重新走一遍增量解析、合并、渲染流程。
5. 回放结束后，系统从被右击的用户消息开始重新生成，得到新的后续 SlideML 结果。

本次探索只分析现状与设计接入点，不修改业务代码。

## 现有链路梳理

### 1. WPF 聊天界面

当前 WPF Demo 的右侧聊天区域由 `CharUserControl.xaml` 承载：

```text
MainWindow.xaml
  └─ CharUserControl
       └─ ListBox ItemsSource="{Binding ChatMessages}"
            └─ DataTemplate DataType="CopilotChatMessage"
                 └─ Border + StackPanel + ItemsControl MessageItems
```

`MainWindowViewModel.ChatMessages` 直接绑定：

```text
SlideChatManager.Pipeline.ChatManager.ChatMessages
```

当前消息气泡没有 `ContextMenu`，也没有针对单条消息的命令绑定。发送入口为：

```text
MainWindowViewModel.RunSendMessageAsync
  → SlideChatManager.SendMessageAsync(... useStreaming: IsStreamingMode ...)
  → SlideGenerationPipeline.SendMessageAsync(... useStreaming: true ...)
  → StreamingSlideGenerator.GenerateAsync
```

### 2. 流式状态位置

当前流式状态已经提升到 `SlideGenerationPipeline`：

```text
SlideGenerationPipeline
  └─ _streamingState: SlideStreamingState?
       ├─ Pipeline: SlideStreamingPipeline
       │    └─ SlideMlStreamingMerger
       │         ├─ XDocument DOM
       │         ├─ IdIndex
       │         ├─ StyleIdIndex
       │         └─ DanglingElements
       └─ Context: SlideMlPipelineContext
            ├─ Warnings / Errors
            └─ MaterialResourceManager
```

`SlideGenerationPipeline.SendMessageAsync` 在以下场景重置 `_streamingState`：

| 场景 | 当前行为 |
|------|----------|
| `isFirstMessage == true` | `_streamingState = null` |
| `createNewSession == true` | `_streamingState = null` |
| 普通跨轮对话 | 复用 `_streamingState` |

`SlideStreamingPipeline` 提供两类重置：

| API | 行为 | 适用场景 |
|-----|------|----------|
| `Reset()` | 清空片段提取器与合并器 DOM 状态 | 回放前彻底清空 SlideML 状态 |
| `ResetExtractor()` | 只清空片段提取器缓冲区，保留 DOM | 错误重试前清理残留 token |

### 3. 聊天 Session 位置

`CopilotChatManager` 管理聊天 Session：

```text
CopilotChatManager
  ├─ ChatSessions: ObservableCollection<CopilotChatSession>
  ├─ SelectedSession: CopilotChatSession
  │    ├─ ChatMessages: ObservableCollection<CopilotChatMessage>
  │    └─ AgentSession: AgentSession?
  │         └─ InMemoryChatHistoryProvider
  └─ CreateManualSendMessageContextAsync()
```

流式生成时，`StreamingSlideGenerator.RunStreamingLoopAsync` 会调用：

```text
CopilotChatManager.CreateManualSendMessageContextAsync
  → 使用 SelectedSession
  → 使用 AgentApiEndpointManager.PrimaryModel.GetChatClientAsync()
  → ManualSendMessageContext.GetAgentSessionAsync()
       ├─ 若 SelectedSession.AgentSession 已存在，则复用
       └─ 否则创建新 AgentSession 并写回 SelectedSession
```

这意味着“聊天列表可见消息”和“AgentSession 内部历史”是两个需要同时处理的状态：

| 状态 | 作用 | 回放前处理要求 |
|------|------|----------------|
| `CopilotChatSession.ChatMessages` | UI 可见聊天列表 | 截断到回放边界之前或新建会话后重放 |
| `CopilotChatSession.AgentSession` | AgentFramework 的真实历史 | 必须清空或新建，否则会携带旧历史 |

`AgentLib` 已提供 `CreateSessionFromMessage(CopilotChatMessage fromMessage)`，它会从当前会话复制从最早消息到指定消息的 UI 消息，并切换到新会话。但该方法不会复制 `AgentSession`，因此新会话的 AgentSession 为空。这个特性对“重新开始”很有价值。

## FakeChatClient 能力分析

`FakeChatClient` 位于 AgentLib：

```text
AgentLib.Core.AgentApiManagers.LanguageModelProviders.Fakes.FakeChatClient
```

核心能力：

| 成员 | 用途 |
|------|------|
| `OnGetResponseAsync` | 模拟非流式响应 |
| `OnGetStreamingResponseAsync` | 模拟流式响应，返回 `IAsyncEnumerable<ChatResponseUpdate>` |
| `OnGetService` | 模拟服务查询 |
| `OnDispose` | 模拟释放 |

流式回放可使用：

```text
OnGetStreamingResponseAsync = (messages, options, cancellationToken) => replayUpdates
```

现有单元测试已经用 `FakeChatClient` 逐字符模拟 SlideML 输出，完整走通：

```text
CopilotChatManager
  → CreateManualSendMessageContextAsync
  → ChatClientAgent.RunStreamingAsync
  → StreamingSlideGenerator
  → SlideStreamingPipeline.ProcessIncrementalTextAsync
  → SlideMlStreamingMerger
  → RenderPipeline
```

项目引用关系也允许在 PptxGenerator Core 使用 FakeChatClient：

```text
PptxGenerator.Core
  └─ ProjectReference AgentLib
```

因此不需要新增跨项目依赖。

## 回放功能的关键语义

### 1. 右击“重新开始”的建议语义

建议将“重新开始”定义为：

> 以被右击的用户消息为新的最后一条真实用户输入，先回放它之前已经确认的对话结果，恢复 SlideML 状态；然后从该用户消息重新发起真实生成，替换该消息之后的所有内容。

举例：

```text
U1：生成首页
A1：输出 Page + title
U2：加一个卡片
A2：输出 card1
U3：把标题改红色
A3：输出 title Foreground="#FF0000"
```

用户右击 `U2` 选择“重新开始”后，建议流程为：

```text
1. 截断边界：U2
2. 回放 U1/A1，恢复到 U1 完成后的 SlideML 状态
3. 删除或隔离 U2/A2/U3/A3 的旧历史
4. 以 U2 作为新的真实请求重新生成
```

这样“重新开始”表示“从这条用户消息开始重新走”，而不是“把这条用户消息已有的旧回复也回放一遍”。

### 2. 为什么不直接回放到被右击消息之后

如果回放 `U2/A2` 后再重新生成 `U2`，会造成同一用户意图被执行两次。对于增量式 SlideML 来说，这可能带来重复元素、重复工具调用和不一致的合并结果。

因此更清晰的边界是：

| 消息范围 | 处理方式 |
|----------|----------|
| 被右击用户消息之前的完整轮次 | 用 FakeChatClient 回放，恢复状态 |
| 被右击用户消息本身 | 作为新的真实请求发送 |
| 被右击用户消息之后 | 丢弃或保留在旧会话中，不参与新会话 |

## 建议设计

### 方案概览

建议在 `SlideGenerationPipeline` 或其上层新增一个“重新开始”入口，内部按以下阶段执行：

```text
RestartFromUserMessageAsync(targetUserMessage)
  │
  ├─ 1. 捕获当前会话快照
  │    ├─ 找到 targetUserMessage 的索引
  │    ├─ 收集 target 之前的完整 user/assistant 轮次
  │    └─ 提取每条 assistant 消息中的文本内容作为回放输出
  │
  ├─ 2. 重建聊天 Session
  │    ├─ 新建空会话，或调用 CreateSessionFromMessage(target 前一条消息)
  │    └─ 确保新会话 AgentSession 为空
  │
  ├─ 3. 清空 SlideML 状态
  │    ├─ _streamingState = null 或 _streamingState.Pipeline.Reset()
  │    ├─ Context.Reset()
  │    └─ 清空/覆盖 SlideMlRenderTool.Latest* 渲染结果
  │
  ├─ 4. 安装 FakeChatClient
  │    ├─ 保存原 PrimaryModel
  │    ├─ 注册/切换到 FakeLanguageModelProvider
  │    └─ FakeChatClient 按轮次返回旧 assistant 文本流
  │
  ├─ 5. 静默回放历史轮次
  │    ├─ 对每个历史 user 消息调用 SendMessageAsync(useStreaming: true)
  │    ├─ 首条历史消息使用 isFirstMessage: true
  │    └─ 后续历史消息使用 isFirstMessage: false
  │
  ├─ 6. 恢复真实模型
  │    └─ PrimaryModel = 原模型
  │
  └─ 7. 发送 targetUserMessage 进行真实生成
       └─ SendMessageAsync(targetText, isFirstMessage: false, useStreaming: true)
```

### 回放数据来源

回放需要从旧 UI 消息中提取两类数据：

| 数据 | 来源 | 用途 |
|------|------|------|
| 用户消息文本 | `CopilotChatMessage.Content` | 作为回放时的输入消息 |
| 助手流式输出文本 | `CopilotChatMessage.Content` | 作为 FakeChatClient 的流式响应 |

当前 `CopilotChatMessage.Content` 会拼接所有 `CopilotChatTextItem` 的文本。对于 SlideML 流式生成来说，助手的 SlideML XML 主要通过文本 token 进入，因此可以作为第一版回放数据来源。

但需要注意：`CopilotChatMessage` 还可能包含工具项、图片项、推理项等。回放 SlideML 状态时，真正驱动 `SlideStreamingPipeline` 的只有 `AgentResponseUpdate.Text`。因此首版可以只回放文本内容；后续如需完全复刻模型输出，再考虑保存原始 `AgentResponseUpdate` 序列。

### FakeChatClient 的流式输出策略

建议实现一个回放队列：

```text
Queue<string> assistantReplayTexts
```

每次 `OnGetStreamingResponseAsync` 被调用时：

1. 从队列取出下一条 assistant 文本。
2. 按字符或按原始 chunk 切成 `ChatResponseUpdate`。
3. 逐个 yield 给 `ChatClientAgent.RunStreamingAsync`。

按字符回放最简单，现有测试已经验证可行。若未来要更接近原始体验，可在真实生成时额外记录原始 chunk。

## UI 接入点

### 1. CharUserControl.xaml

在消息气泡 `Border` 上增加 `ContextMenu`：

```text
Border（消息气泡）
  └─ ContextMenu
       └─ MenuItem Header="重新开始"
```

仅用户消息应该显示或启用该菜单项。可通过以下方式之一实现：

| 方式 | 说明 |
|------|------|
| Converter 判断 `Role == ChatRole.User` | 改动小，符合当前 XAML 风格 |
| ViewModel 暴露 `CanRestartFromMessage` | 逻辑更集中，便于单元测试 |

### 2. MainWindowViewModel

建议新增命令：

```text
RestartFromMessageCommand: ICommand
```

命令参数为 `CopilotChatMessage`。执行时：

```text
RunRestartFromMessageAsync(CopilotChatMessage message)
  → SlideChatManager.RestartFromMessageAsync(message, useStreaming: IsStreamingMode)
```

ViewModel 负责忙碌状态、状态文本和命令可用性；具体回放逻辑不应放在 ViewModel。

### 3. SlideChatManager / SlideGenerationPipeline

`SlideChatManager` 可新增兼容入口，委托给 `SlideGenerationPipeline`：

```text
SlideChatManager.RestartFromMessageAsync(CopilotChatMessage message, bool useStreaming, CancellationToken cancellationToken)
  → Pipeline.RestartFromMessageAsync(...)
```

核心逻辑建议放在 `SlideGenerationPipeline`，因为它同时拥有：

- `CopilotChatManager`
- `SlideMlRenderTool`
- `_streamingState`
- `_promptProvider`
- `_dispatcher`

这些正是回放所需的状态和能力。

## 状态清理边界

回放前需要清理的状态分三类。

### 1. SlideML 合并状态

应清理：

```text
_streamingState = null
```

或：

```text
_streamingState.Pipeline.Reset()
_streamingState.Context.Reset()
```

推荐使用 `_streamingState = null`，让下一次 `SendMessageAsync(useStreaming: true, isFirstMessage: true)` 自然创建全新的 `SlideStreamingState`，避免遗漏内部新增状态。

### 2. 渲染结果状态

`SlideMlRenderTool` 持有：

```text
LatestPreviewImage
LatestRenderedXml
LatestSlideXml
LatestWarnings
```

回放前如果不清理，UI 左侧可能短暂显示旧结果；回放过程中又会被新渲染覆盖。

建议增加一个明确的清理入口，例如：

```text
SlideMlRenderTool.ResetLatestResult()
```

如果暂时不新增 API，也可以接受回放首个片段渲染后覆盖旧结果，但用户体验会弱一些。

### 3. 聊天 Session 状态

必须确保新的 `SelectedSession.AgentSession` 为空，否则回放时 `ManualSendMessageContext.GetAgentSessionAsync` 会复用旧历史。

可选做法：

| 做法 | 优点 | 风险 |
|------|------|------|
| 使用 `CreateSessionFromMessage` 复制到 target 前一条消息 | 复用 AgentLib 现有能力，新会话 AgentSession 为空 | 需要处理 target 是第一条消息时没有前一条消息 |
| 新建空会话后手动回放 UI 消息 | 语义最清晰 | 需要额外控制 UI 是否重复出现回放消息 |
| 直接清空当前 `SelectedSession.AgentSession` | 改动少 | UI 消息与 Agent 历史不一致，且会修改原会话 |

建议采用“新建会话 + 回放”模型，保留旧会话作为历史快照，不在原会话上原地删除。

## 回放时 UI 消息处理

这里有一个关键设计选择：回放是否要把消息再次追加到聊天列表？

### 方案 A：可见回放

每次回放都调用现有 `SendMessageAsync`，让用户看到消息重新生成一遍。

优点：

- 实现简单。
- UI 与 AgentSession 完全一致。
- 方便观察回放过程。

缺点：

- 用户会看到历史消息快速重刷，可能闪烁。
- 如果历史较长，等待时间增加。

### 方案 B：静默回放

回放只驱动 AgentSession 和 SlideML 状态，不向 UI 追加消息；回放结束后 UI 只展示截断后的历史和新的生成。

优点：

- 用户体验更像“回到过去并重新生成”。
- 不会出现重复消息闪烁。

缺点：

- 需要 AgentLib 或 Pipeline 提供“只写 AgentSession、不写 UI”的能力。
- 当前 `CreateManualSendMessageContextAsync` 默认由调用方主动 `AppendMessagesToSessionAsync`，可以不调用它，但 `StreamingSlideGenerator` 当前内部固定调用了该方法。

首版建议采用方案 A，先保证功能闭环。后续如体验不佳，再重构出静默回放模式。

## 风险与开放问题

### 1. 回放文本是否完整

当前从 `CopilotChatMessage.Content` 取助手文本，只包含文本项，不包含：

- 推理内容
- 工具调用输入输出
- 原始 chunk 边界
- 图片、多模态内容

对恢复 SlideML DOM 来说通常足够，因为流式管道只消费 `update.Text`。但如果模型输出中存在非文本内容影响 Agent 历史，回放会不完全。

### 2. 错误重试消息是否参与回放

流式生成遇到错误时，当前实现会把错误反馈消息也追加到 Session，并让 LLM 继续输出修正片段。UI 中可能出现多轮内部错误反馈。

回放时需要决定：

| 选择 | 说明 |
|------|------|
| 全量回放所有可见消息 | 最忠实，但可能把内部重试也重放出来 |
| 只回放用户可见的真实用户消息与最终助手消息 | 更贴近用户心智，但需要识别内部反馈 |

当前 `CopilotChatMessage.IsPresetInfo` 可用于识别部分预置信息，但错误反馈消息是否有明确标记需要进一步确认。

### 3. Fake 模型切换后的模型列表影响

若通过 `AgentApiEndpointManager.RegisterLanguageModelProvider(new FakeLanguageModelProvider(fakeChatClient))` 临时注册 Fake 模型，需要确认：

- 是否会污染 UI 模型下拉列表。
- 是否能可靠恢复原 `PrimaryModel`。
- 是否需要避免重复注册 Fake provider。

更稳妥的方式是新增一个局部可替换 `IChatClient` 的回放入口，而不是临时改全局 PrimaryModel。但这会涉及 `StreamingSlideGenerator` 构造与 `CreateManualSendMessageContextAsync` 调用方式调整。

### 4. 首条用户消息的处理

如果用户右击第一条用户消息：

```text
U1
A1
U2
A2
```

则回放边界之前没有任何历史。此时应：

1. 新建空会话。
2. 清空 SlideML 状态。
3. 直接以 `isFirstMessage: true` 发送 U1 进行真实生成。

### 5. 非流式模式

用户需求明确是“Streaming 生成 SlideML”的回放功能。首版可以只在 `IsStreamingMode == true` 时启用“重新开始”。非流式模式可禁用菜单项或提示暂不支持。

## 建议实施步骤

### 步骤 1：定义回放边界模型

新增内部数据结构描述回放计划：

```text
ReplayPlan
  ├─ TargetUserMessage
  ├─ TargetUserText
  ├─ PreviousTurns: IReadOnlyList<ReplayTurn>
  └─ IsTargetFirstMessage

ReplayTurn
  ├─ UserText
  └─ AssistantText
```

从当前 `SelectedSession.ChatMessages` 中解析：

1. 找到目标用户消息索引。
2. 只接受 `Role == User` 的目标消息。
3. 收集目标之前完整的 User/Assistant 成对轮次。
4. 丢弃目标之后的消息。

### 步骤 2：新增 Pipeline 回放入口

在 `SlideGenerationPipeline` 中新增：

```text
RestartFromMessageAsync(CopilotChatMessage targetMessage, CancellationToken cancellationToken)
```

职责：

- 创建 `ReplayPlan`。
- 新建或切换 Session。
- 清空 `_streamingState`。
- 清理渲染结果。
- 执行 FakeChatClient 回放。
- 恢复真实模型。
- 发送 target 消息。

### 步骤 3：封装 FakeChatClient 回放器

建议新增内部类：

```text
SlideStreamingReplayChatClientFactory
```

或更简单地在 Pipeline 内部创建 `FakeChatClient`。

注意点：

- `OnGetStreamingResponseAsync` 必须支持取消令牌。
- 响应队列耗尽时应返回空流或抛出清晰异常。
- 非流式 `OnGetResponseAsync` 可返回空响应，避免意外调用时崩溃不清晰。

### 步骤 4：新增 ViewModel 命令

在 `MainWindowViewModel` 中新增：

```text
RestartFromMessageCommand
```

命令可执行条件：

- 当前不忙。
- `IsStreamingMode == true`。
- 参数是用户消息。

### 步骤 5：新增 XAML 右键菜单

在 `CharUserControl.xaml` 的消息气泡 `Border` 上增加 `ContextMenu`，绑定命令到父级 DataContext。

需要注意 `ContextMenu` 不在视觉树中，绑定父级 `DataContext` 时通常要通过 `PlacementTarget` 或代理对象。

### 步骤 6：补充测试

建议新增测试覆盖：

1. 从第二条用户消息重新开始时，只回放第一轮并重新生成第二轮。
2. 右击第一条用户消息时，不执行历史回放，直接全新生成。
3. 回放前 `_streamingState` 被重置，旧 XML 不残留。
4. 新会话 `AgentSession` 为空，并在回放后重新建立。
5. FakeChatClient 队列按调用顺序返回历史助手输出。

## 当前结论

1. 现有流式状态已经集中在 `SlideGenerationPipeline._streamingState`，具备清空和重建的基础。
2. `SlideStreamingPipeline.Reset()` 可以清空 SlideML 合并状态；更推荐直接将 `_streamingState` 置空，让后续首次回放自然创建新状态。
3. 聊天 Session 不能只清 UI 消息，还必须避免复用旧 `AgentSession`。新建会话是更安全的方向。
4. `FakeChatClient.OnGetStreamingResponseAsync` 可以完整模拟流式输出，现有测试已经验证它能驱动当前 SlideML 流式管道。
5. 首版建议采用“可见回放”：用 FakeChatClient 重新调用现有流式发送链路，先恢复历史状态，再恢复真实模型并重新发送目标用户消息。
6. 后续若要优化体验，可再引入“静默回放”模式，避免历史消息在 UI 中重新刷出。
