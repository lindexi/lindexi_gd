# SlideML 流式输出 — 决策记录

本文档记录了 SlideML 流式输出规范制定过程中讨论的关键决策，特别是被废弃的方案及其理由。

---

## 决策 1：整体替换 vs 属性级 Merge

**讨论脉络**：

① 流式场景下，片段中的元素通过 `Id` 匹配已有元素后，如何处理属性？最初倾向整体替换——片段元素完全覆盖已有元素，不写的属性恢复默认值。

② 以 header Panel 为例：
```xml
<!-- 初始 -->
<Panel Id="header" X="0" Y="0" Width="1280" Height="120"/>

<!-- 整体替换：只写 Background → X/Y/Width/Height 丢失 -->
<Panel Id="header" Background="#1A1A2E"/>
```

LLM 只想加个背景色，却必须把位置尺寸重写一遍。

③ **最终方案**：属性级 Merge。片段中显式声明的覆盖，未声明的保留。LLM 只需写变化的属性。

---

## 决策 2：子元素 Merge 的算法演进

**讨论脉络**：

① 最初约定子元素整体替换：片段里的子元素列表就是最终列表。LLM 只想给 `cards` 追加一个 `card3`，却必须把已有的 `card1`、`card2` 全部重写一遍。

② 用户提出关键想法——「直接靠只带子元素内容即可」，不引入新标签或新属性。在 Panel 里写一个已有 Id = 保留该子元素，写新 Id = 追加，不写的 Id = 删除。示例：
```xml
<Panel Id="cards">
  <Rect Id="card1"/>          <!-- 只写 Id，原样保留 -->
  <Rect Id="card3" .../>      <!-- 新元素追加 -->
</Panel>
<!-- card2 没写 → 删除 -->
```

③ 嵌套场景暴露问题。用户问：`outer` 里套 `inner`，想追加 `inner2`，怎么写？
```xml
<Panel Id="outer">
  <Panel Id="inner"/>
  <Panel Id="inner2" .../>
</Panel>
```
按 ② 的想法，`inner` 出现在片段里 → 参与 Merge。但 `inner` 是自闭合的，它的子元素会怎样？这触发了关于自闭合语义的讨论（见决策 4）。

④ 自闭合区分被否决后，需要对「未提及子元素」的语义重新抉择。我提出 `ReplaceChildren` 属性来区分两种模式，用户否决——「那就比你上面给的要添加一个 ReplaceChildren 的方案更加简洁」。

⑤ 用户提出最终算法：以片段中第一个已有 Id 为锚点。经过多轮例子推演：

```
原有 [card1, card2, card3]

片段 [card4, card2]
  → 锚点 card2 在位置 1，移除 card2，位置 1 插入 [card4, card2]
  → 结果 [card1, card4, card2, card3]

片段 [card3]
  → 锚点 card3 在位置 2，移除 card3，位置 2 插入 [card3]
  → 结果 [card1, card2, card3]（等效于没动）

片段 [card5]（全为新 Id，无锚点）
  → P = 末尾，插入 [card5]
  → 结果 [card1, card2, card3, card5]
```

⑥ 至此，规范在所有层级上达成了一致规则：**「未提及 = 保留」**。

| 层级 | 未提及 = 保留的体现 |
|------|---------------------|
| 属性 | 片段中未声明的属性 → 保留原有值 |
| 容器自身（无子元素） | 片段中容器不含子元素 → 现有子元素保持不动 |
| 容器内子元素 | 片段中未出现的子元素 → 保留不动 |
| 整棵树 | 未被任何片段提及的元素 → 保持原样 |

这意味着 LLM 在任何层级都只需要写**变化的部分**。无论是一个属性、一个子元素，还是一个深层嵌套的子树——不碰就是保留。排序算法的设计（以已有 Id 为锚点、移除冲突后插入）正是为了让这条规则在子元素层级也能成立，而不像早期「子元素整体替换」方案那样在此处断裂——当时属性是 Merge 的，子元素却被整体替换，行为不一致。

⑦ 被否决的中间方案：
- `ChildrenMerge` 属性（`"Append"` / `"Replace"`）— 用户认为不如排序算法简洁
- `ReplaceChildren="True"` 属性 — 引入新属性，与增量式心智模型不匹配

## 决策 3：未提及 = 保留 vs 未提及 = 删除

**讨论脉络**：

① 决策 2 早期版本中子元素整体替换，天然是「未提及 = 删除」：片段里没出现的子元素就被删掉。

② 嵌套场景下，用户问「在 outer 里面添加一个 inner2 的情况，这时候你觉得应该如何写？」我给出的回答要求 LLM 写出 `<Panel Id="inner"/>` 来占位保留。但用户追问「那个时候 inner 里面是否就没有了元素了？」——暴露了「提一嘴保留但可能清空子树」的冲突。

③ 两条路的完整对比：

| | 未提及 = 删除（声明式） | 未提及 = 保留（增量式） |
|---|---|---|
| 追加 card2 | 必须重写 card1 | 只写 card2 |
| 更新 card1 属性 | 必须重写 card2 | 只写 card1 |
| 删除 card2 | 不提即可（自然） | 需要 `<Remove>`（额外操作） |
| 嵌套追加 inner2 | 必须提 inner 占位，子树可能出问题 | inner 完全不受影响 |

④ 核心判断来自心智模型：流式输出是 LLM 边想边写，每一步在**追加**而非重做。写 card3 时不需要重新声明 card1 和 card2。增量式更匹配。

⑤ 「未提及 = 保留」带来的删除问题，用最小标签 `<Remove TargetId="x"/>` 解决。删除是罕见操作（流式构建中远少于追加），不值得为它牺牲增量式的自然心智模型。

⑥ 与决策 2 的关联：排序算法的最终设计使「未提及 = 保留」能够在子元素层级贯彻。早期「子元素整体替换」方案在子元素层级断裂——属性是 Merge 的，子元素却被整体替换，两条规则互相矛盾。当前方案下，属性、子元素、嵌套子树，三个层级行为一致。LLM 的心智模型是统一的：「我只说变化的部分」。

⑦ 被否决的方案：
- 方案 A：`ReplaceChildren="True"` 属性，在 Panel 上切全量模式。用户认为不够简洁。
- 方案 B：`<Remove TargetId="parent" ChildId="child"/>` 双属性。后因 Id 全局唯一，简化为单属性（决策 6）。

## 决策 4：自闭合元素区分语义

**讨论脉络**：

① 在决策 2 的嵌套场景中，`<Panel Id="inner"/>` 写法引发讨论：它和 `<Panel Id="inner"></Panel>` 是否该有不同语义？

② 我提出：`<Panel Id="x"/>` 表示「保留子树不动」，`<Panel Id="x"></Panel>` 显式空内容表示「清空子元素」，借用 XML 自闭合 vs 显式闭合的区分。

③ 用户明确反对：「不能靠 XML 专用语义进行区分。这很可能在某些序列化过程中丢失，且带来了奇怪的理解成本。」

④ **最终方案**：不区分。`<Panel Id="x"/>` 和 `<Panel Id="x"></Panel>` 在 XML 语义上完全等价，解析器统一处理：容器不含子元素时，仅执行属性 Merge，现有子元素保持不动。

⑤ 删除子元素的唯一途径是 `<Remove>`。

## 决策 5：`<Add>` 标签

**讨论脉络**：

① 在讨论子元素操作时，我提出三种方案：隐式子元素 Merge（按 Id 匹配）、显式 `<Add>` / `<Remove>` 操作标签、`ChildrenMerge` 属性控制。

② 方案 B 的 `<Add>` 标签设计：
```xml
<Add TargetId="cards">
  <Rect Id="card3" .../>
</Add>
```
`TargetId` 指向父容器，可选的 `BeforeId` 控制插入位置。`<Remove>` 配 `TargetId` + `ChildId` 做删除。

③ 用户提出更简洁的想法——不引入 `<Add>`，直接在 Panel 里带子元素即可。「你提醒我了，完全可以做到非上面三个方式的之外的方式。直接靠只带子元素内容即可。比如 `<Panel><Rect Id="card1" /> <!--不写内容就是保留--> <Rect Id="card3" .../></Panel>`」

④ **最终方案**：不引入 `<Add>`。复用 `<Panel Id="x">...</Panel>`，排序算法自动处理追加和插入。零新标签。

---

## 决策 6：`<Remove>` 标签属性数量

**讨论脉络**：

① 最初 `<Remove>` 设计为双属性：`TargetId`（父容器）+ `ChildId`（子元素）。
```xml
<Remove TargetId="cards" ChildId="card2"/>
```

② 用户认为应该更简单：「只要 `<Remove TargetId="cards" />` 就可以了，只有 TargetId 一个属性就可以了。因为设计为 Id 为不重复。」

③ **最终方案**：`<Remove TargetId="x"/>` — 单属性。Id 全局唯一，不需要额外指定父容器来定位。

---

## 决策 7：「替代删除方式」的引入与删除

**讨论脉络**：

① 在规范初稿中，我写了一段「替代删除方式」：LLM 重新声明父容器，只写想保留的子元素，未提及的会在排序算法步骤②中被移除。

```xml
<!-- 骨架 -->
<Panel Id="cards">
  <Rect Id="card1"/>
  <Rect Id="card2"/>
  <Rect Id="card3"/>
</Panel>

<!-- 片段：只写 card1 和 card3 → card2 被删 -->
<Panel Id="cards">
  <Rect Id="card1"/>
  <Rect Id="card3"/>
</Panel>
```

② 这段逻辑是决策 2 早期「子元素整体替换」思维的残余——片段声明完整子元素列表，不提等于删。

③ 决策 3 确定「未提及 = 保留」后，这段直接与之冲突。「未提及 = 保留」意味着片段中没提 card2 → card2 保留不动，但「替代删除方式」却说 card2 被删除。两条规则互相矛盾。

④ 用户明确指出：「替代删除方式的规则比较冲突，我期望全删掉。」

⑤ **最终方案**：删除整节。删除的唯一途径是 `<Remove>`。排序算法的步骤②只用来处理「片段中出现的已有 Id 需要从原位置移除后重新插入以调整顺序」，不承担隐式删除职责。

## 决策 8：`AfterId` / `BeforeId` 位置标记

**讨论脉络**：

① 讨论子元素插入位置时，曾考虑是否需要显式标记（如 `AfterId="card1"`）来控制新元素插入位置。

② 用户认为排序算法的锚点机制已经覆盖。要插入到 card1 和 card2 之间，写出 card1、新元素、card2 即可。要放到最前面，写一个已有 Id 靠前即可。

③ **最终方案**：不引入位置标记属性。不为罕见场景增加概念负担。

---

## 决策 9：流结束标记

**讨论脉络**：

① 讨论是否需要 LLM 显式输出一个结束标记（如 `<SlideMLDone/>`），让解析器知道流已结束。

② 用户认为不添加过多概念：「直接靠 EOF 好了。」

③ **最终方案**：解析器以 EOF 作为流结束信号，不引入额外标记。

---

## 决策 10：「两阶段输出」命名

**讨论脉络**：

① 初稿将 LLM 输出过程描述为「骨架阶段」和「填充阶段」两个阶段。

② 用户指出：「不应该叫两阶段输出，至少在填充阶段里面，可以源源不断地间歇填充。骨架阶段也不意味着以后骨架部分不可变。」

③ **最终方案**：统称「片段序列」。第一个片段通常是 `<Page>`，但 `Page` 本身也可以作为片段在后续再次出现，更新页面属性或调整顶层结构。不设硬边界。

---

## 决策 11：Id 重复的错误分类

**讨论脉络**：

① 初稿错误处理中写：「`Id` 在全局范围内重复 → 报 `[Error]`，后者覆盖前者」。

② 用户质疑：如果 Id 重复，为什么报 Error？不应该是替换吗？指出这实际上表达的是——如果一个 Id 原本在 Panel1 里，后面又在 Panel2 里出现，此时应该报 Error。

③ 讨论后区分三种情况：

| 场景 | 说明 | 行为 |
|------|------|------|
| 片段中元素 Id 匹配到已有元素 | 正常 Merge，属性覆盖 | ✅ 正常流程 |
| 同一个 Id 出现在文档树的两个不同父容器下 | card1 同时是 PanelA 和 PanelB 的子元素 | ❌ Error |
| 同一个片段内出现两个相同 Id | 片段自身冲突 | ❌ Error |

④ **最终方案**：拆为两条——
- 同一个 Id 出现在文档树的两个不同父容器下 → Error，后者覆盖前者（后者所在的片段生效）
- 同一个片段内出现两个相同 Id → Error，按文档顺序后者覆盖前者
