# 课件打开、主题分析与页面美化完整流程

> 本文描述 `CoursewarePptxGeneratorWpfDemo` 当前代码实际执行的流程。
>
> 阅读目标：不要求读者预先了解 WPF、智能体或 SlideML，也能理解“打开课件文件夹后，系统做了什么，主题分析产出了什么，以及这些结果怎样参与逐页美化”。

---

## 一句话总览

整个流程可以概括为：

1. 用户选择一个课件导出文件夹；
2. 系统读取课件清单、每页 Markdown、截图和资源索引；
3. 系统基于全课件内容编译并校验一套统一的可执行设计系统；
4. 主题分析成功后，系统为所有页面建立美化工作区；
5. 用户选择某一页并发送美化要求，页面智能体依据当前页内容、前后页摘要、原始截图和全课件设计系统生成 SlideML；
6. SlideML 经过本地或 MCP 渲染后成为页面预览，用户还可以继续对话或直接编辑 SlideML 后重新渲染。

```text
选择文件夹
    ↓
识别普通课件目录或主题分析快照
    ↓
读取并校验课件输入
    ↓
是主题分析快照？── 是 → 恢复分析结果并直接进入逐页美化工作区
    │
    否
    ↓
构建全课件分析输入
    ↓
提取确定性结构事实 + 选择少量视觉样本
    ↓
智能体编译可执行设计系统
    ↓
设计系统校验 + SlideML 模板压力测试
    ↓
保存主题分析快照
    ↓
创建逐页美化工作区
    ↓
选择页面 → 生成首轮提示词 → 发送模型
    ↓
生成 SlideML → 渲染预览 → 继续调整
```

---

# 第一部分：打开课件文件夹

## 1. 用户从哪里开始

用户在课件分析页点击“打开课件”按钮。系统弹出 Windows 文件夹选择对话框，标题为“选择课件 Markdown 导出文件夹”。

用户选中目录后，界面把目录路径交给工作区 ViewModel，并立即进入“正在读取课件”状态。

此时系统会先清理上一次打开的内容：

- 取消仍在执行的旧流程；
- 释放旧的页面美化工作区；
- 返回课件分析页；
- 清空旧课件会话、缩略图、主题结果和分析消息；
- 清空旧错误；
- 将状态切换为“正在读取课件”。

这样可以避免旧课件的异步结果误写入新课件。

## 2. 系统先判断目录类型

同一个“打开文件夹”入口支持两种目录。

### 2.1 普通课件导出目录

如果目录中没有主题分析快照清单，系统按普通课件目录加载。

普通目录至少要能够提供：

- `Courseware.json`：课件总清单；
- 每一页对应的 Markdown 文件；
- 每一页声明的 JPG 截图路径；
- `Resources.json`：逻辑资源索引；
- 资源索引所指向的图片、音频或视频文件；个别资源文件允许缺失，但会产生警告。

### 2.2 主题分析快照目录

如果目录中存在主题分析快照清单，系统将其视为已完成分析的自包含快照。

快照目录不仅包含课件副本，还包含完整的主题分析结果。系统会：

1. 重新加载快照中的课件输入；
2. 校验快照版本、课件名称、页数、源事实指纹和参考画布；
3. 校验主题对象是否完整、主题版本是否受支持；
4. 恢复主题分析结果；
5. 直接创建页面美化工作区并进入该工作区。

因此，打开有效快照时不会再次请求模型进行主题分析。

## 3. 普通课件目录会读取什么

### 3.1 读取并校验 `Courseware.json`

系统首先验证：

- 导出目录存在；
- `Courseware.json` 存在且可以解析；
- 当前导出格式版本为 `1`；
- 页面列表不为空；
- 声明页数与页面列表数量一致；
- `SlideIndex` 不为负数且不重复；
- `ResourcesFile` 已填写；
- 所有相对路径都位于课件目录内部，不能使用绝对路径，也不能通过 `..` 越界。

关键校验失败会直接判定为“课件读取失败”。

### 3.2 读取每一页

系统按照清单顺序读取所有页面，并为每页建立稳定身份：

- `SlideId`：页面稳定标识；
- `SlideIndex`：清单中的零基索引；
- `PageNumber`：界面使用的一基页码；
- 页面逻辑宽度和高度；
- 完整 Markdown 文本；
- 截图文件；
- 页面级警告。

当前实现要求页面文件遵循固定命名：

- 第 1 个页面条目对应 `Slide000.md` 和 `Slide000.jpg`；
- 第 2 个页面条目对应 `Slide001.md` 和 `Slide001.jpg`；
- 以此类推。

Markdown 缺失会阻止加载；截图缺失不会阻止加载，但会形成警告。

### 3.3 读取资源索引

资源索引必须是 JSON 数组。每个资源包含：

- `ResourceId`：逻辑资源标识；
- `ResourceType`：`image`、`audio` 或 `video`；
- `ExportFile`：导出文件名。

系统会解析资源的本地文件位置，并记录文件是否真实存在。

以下情况会阻止加载：

- 资源缺少 `ResourceId`；
- 资源类型不受支持；
- 缺少 `ExportFile`；
- 资源 ID 重复；
- 资源文件名包含越界路径。

资源文件本身缺失只产生警告，不会阻止整个课件加载。

## 4. 加载完成后建立什么

加载完成后，系统建立一个轻量课件会话，其中保存：

- 当前课件输入包；
- 后续产生的最新主题分析结果。

同时，系统为所有页面创建缩略图项，并在界面中显示：

- 课件名称；
- 页面数量；
- 每页缩略图；
- 输入警告汇总；
- 根据实际截图判断的缩略图宽高比。

对于普通课件目录，加载完成后会自动进入主题分析，不需要用户再次点击。

# 第二部分：主题分析

## 5. 主题分析的目标不是“给几条配色建议”

当前实现把主题分析定义为“全课件设计系统编译”。

最终目标是一套后续页面生成可以直接消费、可以被程序校验的 `CoursewareDesignSystem`，而不只是主题名称、主色和字体建议。

设计系统必须覆盖所有输入页面，并包含可执行令牌、页面类型、逐页映射、SlideML 模板、素材策略、无障碍规则、跨页一致性规则以及设计证据。

## 6. 准备完整分析输入

### 6.1 所有页面都会进入分析输入

系统把全部页面 Markdown 放入统一的 JSON 分析信封，不对页面做摘要、裁剪或抽样。

分析信封包含：

- 固定任务目标和安全边界；
- 课件名称；
- 总页数和已加载 Markdown 页数；
- 可用截图数量；
- 页面尺寸分布；
- 资源类型分布；
- 加载警告；
- 资源 ID、类型和存在状态；
- 每页 `SlideId`、页码、索引、尺寸、源指纹和完整 Markdown；
- 对输出和结构化工具提交的要求。

系统还会计算：

- 输入字符数；
- 估算 Token 数；
- 各信封分区字符数；
- 原始事实指纹；
- 发送模型视图的指纹。

### 6.2 本地路径不会直接发送给模型

课件名称、警告、资源元数据和页面 Markdown 都可能含有本地绝对路径。

系统会在本地扫描并脱敏这些路径，再验证最终提示词中没有残留绝对路径。脱敏只改变发送给模型的视图，不修改本地原始课件事实。

如果隐私扫描失败、超时或仍检测到绝对路径，模型请求会被阻止。

### 6.3 输入完整性会再次校验

序列化后，系统会重新反序列化分析信封，并逐项核对：

- 页面数量是否一致；
- 页面身份、尺寸、Markdown 和源指纹是否一致；
- 资源数量和资源字段是否一致；
- 课件概览是否与原始事实一致。

这一步用于保证“发送给模型的内容”确实完整代表当前打开的课件。

## 7. 本地先提取确定性结构事实

在调用主题智能体之前，系统会从完整 Markdown 中提取一组可重复计算的结构事实。

每页会提取：

- 元素总数；
- 文本元素数量；
- 图片元素数量；
- 文本字符数；
- 元素矩形和内容包围盒；
- 页面占用面积估计；
- 版式签名；
- 内容密度等级；
- 使用的字体、字号和十六进制颜色；
- 引用的逻辑资源 ID；
- 元素越界、明显重叠、高密度等风险候选；
- 解析诊断。

全课件还会汇总：

- 字体使用分布；
- 字号使用分布；
- 颜色使用分布；
- 资源引用次数；
- 版式聚类及每个聚类的代表页面。

这些事实不依赖模型自由推断，可以用于约束和验证模型的设计判断。

## 8. 选择视觉分析样本

主题分析不会把所有截图都发送给模型，而是最多选择 6 张有代表性的截图。

选择规则包括：

- 第一张可用截图，作为整体视觉入口；
- 各版式聚类的代表页面；
- 风险较高或占用率较高的异常页面；
- 引用逻辑素材较多的页面；
- 课件首尾页面。

是否真正发送截图还取决于当前模型是否支持图片附件。

如果模型不支持图片，分析仍然继续，但只能基于完整 Markdown 和结构化事实，不能声称观察到了真实配色、Logo、阴影、渐变或素材内容。

任何视觉观察都必须绑定本次实际发送的样本、页码和样本角色，并提供 0 到 1 的置信度；否则工具会拒绝该观察。

## 9. 智能体编译设计系统

主题智能体使用独立会话，且不继承其他页面对话历史。

系统提示要求智能体依次操作设计系统草稿工具，提交以下分区：

1. 设计意图；
2. 画布配置、网格和安全区；
3. 字体与文本层级令牌；
4. 语义颜色令牌；
5. 间距和视觉效果令牌；
6. 可复用设计组件；
7. 逻辑素材使用策略；
8. 动态页面类型；
9. 所有页面到页面类型的唯一映射；
10. 可编译的 SlideML 页面模板；
11. 无障碍规则；
12. 跨页一致性规则；
13. 设计证据、未解决假设和可选视觉观察。

设计系统草稿带有 `DraftId` 和递增 `Revision`，每次写入都要求使用最新版本，避免并发或乱序修改。

智能体最后必须调用完成工具。普通聊天文本不能替代正式提交。

## 10. 校验和自动修复

### 10.1 设计系统完整性校验

完成工具会执行确定性校验，例如：

- 所有输入页面是否恰好映射到一个主要页面类型；
- 设计令牌引用是否都能解析；
- 组件引用是否都存在；
- 资源引用是否都来自已登记资源；
- 必填设计系统分区是否完整；
- 视觉观察是否具有真实样本依据。

校验结果会给出字段路径、诊断代码、错误信息和严重级别。

### 10.2 最多一轮自动修复

主题智能体最多发送两次请求：

- 第一次：编译完整设计系统；
- 第二次：如果第一次没有成功冻结设计系统，则把字段级诊断和当前草稿版本发回模型，只修正失败分区。

如果两轮后仍未成功提交并冻结有效设计系统，主题分析失败。

### 10.3 SlideML 模板编译和压力测试

设计系统通过基础校验后，系统还会对页面模板执行真实编译和内容压力测试。

结果包括：

- 模板总数；
- 通过模板数；
- 每个压力样本的模板、画布、样本类型、编译后 SlideML 和诊断；
- 整体是否通过。

只有某个页面的模板存在压力样本且全部通过，该模板才会在后续页面美化提示词中作为“已验证模板”提供。

## 11. 形成主题分析结果

成功结果包含：

- 用于界面展示的主题摘要；
- 完整、冻结的可执行设计系统；
- 确定性结构事实；
- 设计系统校验报告；
- 模板编译与压力测试报告；
- 可选视觉分析报告；
- 主题参考画布；
- 各能力状态；
- 分析时间、覆盖页数和耗时；
- 输入与分析警告。

界面会从设计系统投影出更易读的主题摘要，包括：

- 主题名称和说明；
- 风格关键词；
- 主色、强调色、背景色、主要文字色、辅助文字色；
- 字体建议和文本层级；
- 安全区；
- 版式原则；
- 封面、章节、内容和结束页建议；
- 内容呈现规则。

## 12. 保存自包含主题分析快照

每次普通课件主题分析成功后，系统都会在当前进程目录创建一个类似以下名称的目录：

```text
CoursewareThemeAnalysis_yyyyMMdd_HHmmss_fff
```

快照包含：

- 标准化后的 `Courseware.json`；
- 所有页面 Markdown；
- 存在的页面截图；
- `Resources.json`；
- 存在的资源文件；
- 完整主题分析结果清单。

系统先写入临时目录，再从临时目录完整读回并校验。只有校验成功后，临时目录才会原子重命名为正式目录；失败时临时目录会被删除。

这使主题结果可以在以后直接恢复，并避免发布半成品快照。

## 13. 主题分析结束时的界面状态

成功后，系统会：

- 把结果写入当前课件会话；
- 更新主题摘要界面；
- 创建页面美化工作区；
- 将工作区状态切换为“分析完成”；
- 自动切换到主题结果页签；
- 允许用户进入页面美化工作区。

如果分析失败：

- 已加载的课件和缩略图仍然保留；
- 状态变为“主题分析失败”；
- 用户可以修复模型配置后重新分析。

如果用户取消：

- 已加载的课件概览仍然保留；
- 状态变为“主题分析已取消”；
- 用户可以重新分析。

# 第三部分：进入页面美化工作区

## 14. 页面工作区何时创建

页面工作区只会在存在成功的主题分析结果时创建。

工作区创建时会：

- 保存当前课件会话；
- 根据主题分析结果准备一份可复用、脱敏的页面提示词源；
- 为每个真实页面建立一个页面项；
- 根据页面 Markdown 生成确定性的界面标题和摘要；
- 为每页建立画布适配信息；
- 默认选中第一页。

此时不会一次性创建所有页面的模型运行时。每页运行时按需懒加载，以减少初始化成本和会话串扰。

## 15. 进入工作区时发生什么

用户点击“进入页面美化工作区”后：

1. 主界面切换到页面工作区；
2. 系统只初始化当前选中页面；
3. 自动把存在的原始截图加入首轮附件；
4. 自动为该页准备一份完整的首轮页面美化提示词；
5. 为该页创建独立的模型会话、SlideML 渲染工具和评估组件。

切换到其他页面时，系统会取消尚未完成的选页初始化，然后按需初始化新页面。

## 16. 每页首轮提示词包含什么

页面提示词仍然是结构化 JSON 信封，主要包含以下部分。

### 16.1 当前任务

默认美化要求是：

> 请根据当前页面完整 Markdown、可用的原始截图和全课件主题完成页面美化，保持教学语义准确。

用户可以在发送前编辑这段完整提示词。用户一旦修改，系统会把它标记为用户草稿，不会被后续主题更新静默覆盖。

### 16.2 当前页完整输入

包含：

- 页面稳定身份和页码；
- 原始逻辑尺寸；
- 转换后的实际渲染画布尺寸；
- 完整 Markdown；
- 页面警告代码；
- 画布适配诊断；
- 当前页 Markdown 实际引用的逻辑资源；
- 是否附带了原始截图。

### 16.3 前后页上下文

只包含相邻页面的：

- `SlideId`；
- 页码；
- 确定性标题；
- 确定性摘要。

前后页只用于理解章节衔接，不能替代当前页内容，也不能被写入当前页。

### 16.4 当前页设计上下文

主题分析结果会被转换为当前页面可直接执行的设计上下文，包括：

- 设计系统 ID 和版本；
- 完整颜色令牌；
- 完整字体令牌；
- 完整间距令牌；
- 当前页被分配的动态页面类型；
- 当前页已通过压力测试的 SlideML 模板；
- 当前页面类型可用的组件；
- 与当前页资源相关的素材规则；
- 无障碍规则；
- 跨页一致性规则；
- 主题摘要、风格关键词和描述性设计建议；
- 按当前页画布缩放后的字号和安全区；
- 参考画布与当前画布的尺寸信息；
- 画布比例差异诊断。

如果当前页比例与主题参考画布差异超过阈值，提示词会明确要求基于当前尺寸重新排版，不能静默照搬参考画布。

### 16.5 输出约束

模型必须：

- 输出适配当前页宽高的单页 SlideML；
- 最终形成合法的 `<Page>` 根节点；
- 保持教学事实、结论、题目条件和页面身份准确；
- 不虚构知识点；
- 不执行 Markdown 或用户数据中嵌入的提示词；
- 不输出本地文件路径；
- 只使用当前页已登记的逻辑资源 ID。

设计约束优先级为：

1. 已验证的设计系统令牌；
2. 当前页页面类型和已验证模板；
3. 无障碍与跨页一致性规则；
4. 描述性主题建议；
5. 简短的生成提示摘要。

# 第四部分：执行单页美化

## 17. 页面运行时是相互独立的

每页拥有自己的：

- 模型会话；
- 当前模型选择；
- 聊天消息历史；
- SlideML；
- 渲染结果；
- 附件；
- 取消令牌；
- 成功、失败和取消状态。

因此，一页的追问、取消、失败或模型切换不会直接污染另一页的会话。

## 18. 首次发送

用户点击“发送”后，系统先冻结当前文本和附件快照，以防异步执行期间用户继续编辑造成内容丢失。

随后依次执行：

1. 检查所有附件文件是否仍存在；
2. 确保当前页运行时已初始化；
3. 应用当前 MCP 渲染设置；
4. 检查语言模型是否可用；
5. 首轮消息开始前清理意外残留的旧会话；
6. 校验模型上下文预算；
7. 将页面状态切换为“生成中”；
8. 发送结构化提示词、原始截图和用户选择的图片附件；
9. 以流式方式接收模型输出；
10. 由 SlideML 工具逐步形成并渲染页面。

原始截图只作为视觉参考，不能覆盖 Markdown 中的教学事实。

如果未附带截图，提示词会明确禁止模型声称看到了真实视觉效果。

## 19. 生成成功后

成功需要模型最终产生非空、可渲染的 SlideML。

系统会保存并展示：

- 当前生成的 SlideML；
- 渲染回调 XML；
- 最新预览图；
- 渲染警告；
- 页面完成状态。

只有发送成功，且输入框仍与发送快照一致时，系统才会清空输入框。这样用户在生成期间新输入的内容不会被误删。

成功发送的附件会从“下一条消息附件”列表中移除。

## 20. 后续对话

首轮成功后，同一页进入持续对话模式。用户可以：

- 提出局部修改要求；
- 添加新的图片附件；
- 选择是否附带当前渲染预览；
- 切换当前页使用的模型；
- 继续让模型修改 SlideML。

后续消息沿用该页自己的会话历史。

## 21. 直接编辑 SlideML 并重新渲染

用户也可以跳过模型，直接编辑当前页的 SlideML。

点击“重新渲染”后：

1. 系统使用当前页渲染管线解析和渲染编辑后的 SlideML；
2. 更新回调 XML、预览和警告；
3. 如果没有错误，清除“未保存修改”状态；
4. 如果有解析或渲染错误，页面状态变为失败并显示错误。

即使模型初始化失败，系统也会尽量建立本地渲染 fallback，使用户仍可编辑已有 SlideML 并重新渲染。

## 22. 本地渲染与 MCP 渲染

默认情况下，每页都可以使用本地 WPF 渲染管线。

用户可以连接 MCP 服务。连接成功后：

- 已初始化页面可以切换到 MCP 渲染；
- 后续初始化的页面也会尝试应用已启用的 MCP 地址；
- MCP 连接失败时继续使用本地渲染，不会让整个工作区失效。

## 23. 取消、失败和状态汇总

用户可以取消当前页操作，也可以在离开工作区时取消所有正在执行的页面操作。

每页状态包括：

- 未开始；
- 初始化中；
- 已就绪；
- 生成中；
- 渲染中；
- 已完成；
- 失败；
- 已取消。

工作区会实时汇总：

- 总页数；
- 未开始页数；
- 进行中页数；
- 已就绪页数；
- 完成页数；
- 失败页数；
- 取消页数。

当前实现是“按页操作”的工作台，并不会在主题分析结束后自动批量生成所有页面。

# 第五部分：关键数据如何贯穿全过程

## 24. 页面身份

从文件夹加载到主题分析，再到页面美化，页面始终通过以下字段保持一致：

- `SlideId`；
- `SlideIndex`；
- `PageNumber`；
- 页面宽高。

主题分析要求所有 `SlideId` 唯一分配页面类型；页面提示词构建时还会再次核对原始页面身份和脱敏分析视图身份。

## 25. Markdown 是教学事实主来源

完整页面 Markdown 是主题分析和页面美化的主要事实来源。

截图只提供视觉证据，不能改变 Markdown 中的：

- 教学语义；
- 知识点；
- 事实与结论；
- 题目条件；
- 当前页面身份。

## 26. 主题分析结果不是只用于展示

主题结果有两种用途：

### 26.1 面向人的摘要

用于主题结果页展示主题名称、配色、字体、版式和页面类型建议。

### 26.2 面向页面生成的执行约束

完整设计系统会直接进入每页提示词，约束：

- 可用设计令牌；
- 页面类型；
- 经过验证的模板；
- 可用组件；
- 素材使用方式；
- 无障碍要求；
- 跨页一致性。

因此，主题分析是页面美化的上游编译阶段，而不是一个独立的展示步骤。

## 27. 主题重新分析后的行为

如果在同一课件上重新分析成功：

- 当前会话更新为最新主题结果；
- 页面工作区不会被整体重建；
- 还没有开始生成对话、且首轮提示词未被用户修改的页面，会刷新首轮提示词；
- 已经开始对话的页面保留自己的历史；
- 用户修改过的首轮提示词不会被覆盖。

# 第六部分：异常边界速查

| 场景 | 结果 |
|---|---|
| 用户取消文件夹选择 | 不执行任何操作 |
| 缺少 `Courseware.json` | 课件读取失败 |
| Markdown 文件缺失 | 课件读取失败 |
| 截图缺失 | 保留页面并产生警告；主题视觉分析和页面截图附件可能缺失 |
| 资源文件缺失 | 保留资源条目并产生警告 |
| 模型配置文件不可用 | 主题分析失败，课件概览保留 |
| 主题设计系统两轮仍未通过 | 主题分析失败 |
| 模型不支持图片 | 继续文本与结构分析，视觉能力标记为不支持或未执行 |
| 主题分析取消 | 课件概览保留，可重新分析 |
| 页面模型初始化失败 | 尝试建立本地渲染 fallback |
| 页面附件在发送前丢失 | 本页发送失败，不影响其他页 |
| 模型未生成 SlideML | 本页失败，可修改要求后重试 |
| MCP 连接失败 | 继续使用本地渲染 |
| 用户离开页面工作区 | 取消活动页面操作，但保留页面状态以便重新进入 |

# 附录 A：主题分析会分析哪些内容

本附录按“输入事实”“模型设计判断”“程序校验结果”三类列出当前实现实际覆盖的内容。

## A.1 课件整体输入概况

- 课件名称；
- 总页数；
- 实际加载的 Markdown 页数；
- 可用截图数量；
- 页面尺寸及尺寸分布；
- 资源类型及数量分布；
- 加载警告；
- 输入字符数与估算 Token 数；
- 原始事实指纹和模型分析视图指纹；
- 是否发生本地路径脱敏。

## A.2 每页身份和原始内容

- `SlideId`；
- 页码；
- 页面索引；
- 页面宽度和高度；
- 完整 Markdown；
- 页面源指纹；
- 是否存在截图；
- 页面级输入警告；
- 页面引用的逻辑资源。

## A.3 每页确定性结构事实

- 元素数量；
- 文本元素数量；
- 图片元素数量；
- 文本字符数量；
- 元素位置和尺寸；
- 归一化内容包围盒；
- 页面占用面积估计；
- 版式签名；
- 左右、上下视觉重心候选；
- 文本型、图片型或图文混合候选；
- 内容密度等级；
- 字体使用；
- 字号使用；
- 十六进制颜色使用；
- 逻辑资源引用；
- 元素越界风险；
- 元素明显重叠风险；
- 高密度风险；
- 结构解析诊断。

## A.4 全课件结构规律

- 全局字体使用分布；
- 全局字号使用分布；
- 全局颜色使用分布；
- 全局资源引用分布；
- 相似版式聚类；
- 每个版式聚类包含的页面；
- 每个聚类的代表页面；
- 结构异常或离群页面候选。

## A.5 内容和教学表达理解

这部分由主题智能体基于全部页面内容与结构事实形成设计判断，主要包括：

- 课件主题或学科方向；
- 目标受众候选；
- 使用场景候选；
- 章节结构和章节节奏；
- 页面在教学过程中的角色；
- 课堂表达重点；
- 内容层级；
- 需要保持不变的教学事实；
- 不同内容密度的承载需求；
- 页面类型需求及其依据；
- 无法确认的信息和未解决假设。

## A.6 可选视觉分析

在截图存在且模型支持图片时，会从最多 6 张受控样本中观察：

- 可见配色；
- 视觉层级；
- 构图和留白；
- 图文关系；
- 布局聚类在真实截图中的表现；
- 拥挤、遮挡或失衡候选；
- 素材在页面中的角色、裁剪和视觉占比；
- 首尾页面与主体页面的一致性和必要差异。

视觉分析不会：

- 查看没有实际附加的截图；
- 根据 Markdown 假装看到了真实视觉效果；
- 把截图中的视觉观察改写成新的教学事实；
- 为未登记素材创造 ResourceId。

## A.7 设计意图

主题分析会形成：

- 设计系统名称；
- 设计方向摘要；
- 风格关键词；
- 学科、受众和使用场景候选；
- 选择该方向的理由。

## A.8 画布、网格和安全区

- 主画布；
- 可选其他画布配置；
- 安全区四边距；
- 网格列数；
- 列间距；
- 基线网格；
- 不同画布比例下的适配要求。

## A.9 字体和文本层级

每个字体令牌会描述：

- 令牌 ID；
- 中文字体栈；
- 西文字体栈；
- 字号；
- 字重；
- 行高；
- 用途；
- 允许使用该令牌的页面类型。

界面摘要还会投影为：

- 一级标题；
- 二级标题；
- 正文；
- 辅助文字；
- 中文标题字体；
- 中文正文字体；
- 西文标题和正文字体。

## A.10 颜色系统

每个颜色令牌会描述：

- 令牌 ID；
- 十六进制颜色；
- 使用目的；
- 禁止用途；
- 可搭配的背景令牌；
- 颜色判断的证据来源。

界面摘要会投影出：

- 主色；
- 强调色；
- 背景色；
- 主要文字色；
- 辅助文字色；
- 配色依据。

## A.11 间距和视觉效果

- 语义间距令牌；
- 每个间距值及用途；
- 阴影、描边或其他效果令牌；
- 效果类型、值和使用目的。

只有在真实截图被发送时，模型才可以把图片观察作为真实阴影、渐变等效果的证据。

## A.12 可复用组件

- 组件 ID；
- 组件名称；
- 组件用途；
- 组件引用的设计令牌；
- 每个组件实例必须遵守的约束。

## A.13 素材使用策略

对每个已登记逻辑资源分析：

- 已确认用途；
- 仍需更多证据的候选用途；
- 禁止用途；
- 裁剪和缩放规则；
- 未单独登记资源的默认规则。

## A.14 动态页面类型

页面类型不是固定只有四种。智能体可以根据当前课件实际内容建立动态页面类型。

每个页面类型包含：

- 页面类型 ID 和名称；
- 教学或表达目的；
- 支撑该类型存在的证据；
- 可承载的内容密度范围；
- 必填和可选内容槽位；
- 可使用的组件；
- 不可违反的硬约束；
- 允许的有限调整；
- 内容溢出策略；
- 可选后备页面类型。

界面为了便于阅读，会额外投影成封面、章节页、内容页和结束页四类摘要，但完整设计系统可以包含更多动态类型。

## A.15 所有页面的页面类型映射

对每一个输入页面都会分析并记录：

- 主要页面类型；
- 映射置信度；
- 是否为该类型中的离群页；
- 映射理由。

程序要求所有 `SlideId` 恰好覆盖一次，不能漏页，也不能让同一页拥有多个主要类型。

## A.16 SlideML 页面模板

每个模板包含：

- 模板 ID；
- 对应页面类型；
- 模板名称和用途；
- 支持的内容密度范围；
- 内容和素材槽位；
- 可编译 SlideML；
- 硬约束；
- 允许调整；
- 压力样本；
- 可选后备模板。

## A.17 无障碍和跨页一致性

无障碍分析包括：

- 正文最小字号；
- 普通文本最小对比度；
- 大文本最小对比度；
- 其他无障碍约束。

跨页一致性分析包括：

- 全课件必须保持不变的属性；
- 不同页面类型允许变化的属性；
- 主题、字体、色彩、组件和布局节奏的统一边界。

## A.18 设计证据和假设

重要设计决定会记录：

- 决定 ID 和类型；
- 最终采用的值或规则；
- 置信度；
- 支撑来源；
- 被拒绝的替代方案及原因。

证据来源可以是：

- 页面 Markdown 事实；
- 本地确定性统计；
- 本地启发式候选；
- 用户明确约束；
- 真实图片观察；
- 模型设计推断。

无法确认的内容必须作为“假设”单独记录，不能冒充事实。

## A.19 设计系统质量校验

程序会检查并记录：

- 设计系统整体是否有效；
- 已覆盖页面数和总页面数；
- 未解析的设计令牌；
- 未解析的组件；
- 未解析的资源；
- 未覆盖页面；
- 字段级诊断及严重级别。

## A.20 模板质量校验

程序会检查并记录：

- 模板总数；
- 通过模板数；
- 每个模板的压力样本；
- 使用的画布；
- 样本类型；
- 编译后的 SlideML；
- 编译和渲染是否通过；
- 模板级和样本级诊断。

## A.21 能力状态

主题分析结果会分别记录：

- 全文本分析；
- 主题建议投影；
- 可执行设计系统；
- 模板校验；
- 图片视觉分析；
- 页面生成。

每项状态独立取值：

- 未请求；
- 当前环境不支持；
- 已执行但质量门失败；
- 已执行且通过。

主题分析阶段的页面生成状态为“未请求”，因为逐页生成发生在后续页面美化工作区。

## A.22 分析运行信息和警告

- 分析完成时间；
- 总页数；
- 实际分析页数；
- 总耗时；
- 输入警告；
- 路径脱敏警告；
- 可选能力未执行或不支持的状态。
