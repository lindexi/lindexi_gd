## 当前实现复核与正式全局主题分析接入设计

> 本节基于当前仓库代码复核，优先级高于本文后续仍以“尚未开发”描述的旧方案内容。当前双页外壳、课件加载后自动分析、取消、重试和失败保留课件等基础流程已经开始实现；下一阶段不应重新搭建页面框架，而应把现有占位分析替换为正式的全局主题分析链路。

### 1. 当前代码已经具备的基础

当前实际入口是 `CoursewareWorkspaceViewModel`，不是旧方案中的 `MainWindowViewModel`。已经具备：

- `MainWindow` 同时承载 `CoursewareAnalysisView` 与 `SlideWorkspaceView`，通过 `PrototypePage` 切换。
- `CoursewareWorkspaceState` 已覆盖 Welcome、加载中、分析中、分析完成、加载失败、分析失败和取消。
- `CoursewareFolderLoader` 已加载 `Courseware.json`、全部页面 Markdown、页面尺寸、截图、资源索引和非阻塞警告。
- 加载成功后会创建轻量 `CoursewareWorkspaceSession` 和缩略图，并自动调用 `ICoursewareThemeAnalysisService.AnalyzeAsync`。
- 分析取消、分析失败、重新分析和进入工作台的基本状态转换已经存在。
- 测试已覆盖打开课件后自动调用分析、分析成功、输入加载失败和分析失败后保留课件。

因此，正式接入的主调用链可以保持为：

```text
CoursewareWorkspaceViewModel.OpenCoursewareFolderAsync
  -> CoursewareFolderLoader.LoadAsync
  -> 发布缩略图和 CoursewareWorkspaceSession
  -> ICoursewareThemeAnalysisService.AnalyzeAsync
  -> 发布结构化主题和分析诊断
  -> AnalysisReady
```

不需要为了正式分析重新设计导航状态机，也不应把 Agent 调用直接写入 ViewModel。

### 2. 当前占位实现与正式能力的差距

当前 `CoursewareThemeAnalysisService` 只根据课件页数、截图数量和加载警告同步构造固定标题与摘要，没有调用模型。`CoursewareThemeAnalysisResult` 也只包含标题、摘要、时间、页数和警告。

同时，第一页仍存在以下演示数据：

- `ThemeColors`、`TypographyLevels`、`LayoutPrinciples`、`PageTypeRecommendations` 和 `ContentRules` 为固定集合。
- `AnalysisSteps` 为固定的全部完成状态，不反映真实执行过程。
- `CoursewareThemeSummaryView` 中的风格关键词、字体、安全区和风险文案部分硬编码。
- `CoursewareAnalysisMessage.cs` 尚未形成实际消息模型。
- `SlideChatManagerFactory` 把个人本地配置路径、默认模型、页面渲染器、MCP 和评估器创建耦合在一起，不能作为主题分析工厂直接复用。

正式分析的目标不是“让当前固定 UI 看起来像完成了分析”，而是建立一份可校验、可持久化、可注入后续单页生成的课件级主题事实源，并让第一页完全由该事实源驱动。

### 3. 推荐的职责边界

建议保持现有 `ICoursewareThemeAnalysisService` 作为业务入口，但拆出以下职责：

```text
CoursewareWorkspaceViewModel
  -> ICoursewareThemeAnalysisService
       -> ICoursewareAnalysisInputBuilder
       -> IThemeAnalysisChatManagerFactory
       -> CoursewareThemeSubmissionTool / CoursewareThemeValidator
       -> CoursewareThemeAnalysisResult
       -> IProgress<CoursewareAnalysisEvent>
```

各组件职责如下：

#### `ICoursewareThemeAnalysisService`

- 编排输入构建、模型调用、结构校验和一次修复。
- 接受取消令牌。
- 通过进度事件发布面向用户的分析阶段，不引用 WPF Dispatcher 或 UI 类型。
- 只在得到有效 `CoursewareTheme` 后返回成功结果。
- 配置错误、模型错误、超时和结构校验失败均抛出明确的业务异常，由现有 ViewModel 转换为 `AnalysisFailed`。

#### `ICoursewareAnalysisInputBuilder`

- 从 `CoursewareInputPackage` 构建稳定、可测试、无本地绝对路径的文本快照。
- 提取课件名称、页数、尺寸分布、每页标题和正文摘要、资源类型统计、加载警告摘要。
- 保留封面、目录、章节页和结尾页的更多文本；普通内容页使用摘要。
- 负责输入长度预算，不能让 ViewModel 或 System Prompt 自行截断字符串。

#### `IThemeAnalysisChatManagerFactory`

- 创建只包含语言模型能力的独立 `CopilotChatManager`。
- 不创建 `SlideChatManager`、渲染器、评估器或 MCP 连接。
- 不设置 WorkspacePath，不开放默认文件系统工具。
- 与页面聊天共享模型配置加载基础设施，但不共享会话和消息集合。

#### `CoursewareThemeSubmissionTool`

- 向模型暴露唯一的结构化主题提交工具。
- 将工具参数映射为领域模型，并调用本地验证器。
- 验证失败时返回可读错误，要求模型修正后再次调用。
- 不把工具原始参数和本地路径直接暴露给第一页 UI。

#### `CoursewareThemeValidator`

- 执行不依赖模型的确定性校验。
- 可被主题分析服务、输出保存服务和测试共同复用。
- 返回字段级错误集合，而不是只返回 `bool`。

### 4. 模型配置必须先解耦

正式分析不能复用当前 `SlideChatManagerFactory.CreateAsync`，因为该方法会同时创建页面渲染管线、MCP、评估器和提示词优化器。建议先提取共享模型配置层：

```text
AgentModelOptions
  - ConfigurationFile
  - ThemeAnalysisModelName
  - SlideGenerationModelName
  - EvaluationModelName

ICopilotChatManagerFactory
  - CreateAsync(AgentWorkload workload, CancellationToken cancellationToken)

AgentWorkload
  - ThemeAnalysis
  - SlideGeneration
  - Evaluation
```

`SlideChatManagerFactory` 改为组合该基础工厂并追加页面渲染能力；`ThemeAnalysisChatManagerFactory` 只请求 `ThemeAnalysis` 工作负载。

当前开发阶段继续沿用项目原有的开发者本地配置文件路径和默认模型，不要求终端用户配置环境变量。配置值集中在 `AgentModelOptions.CreateDefault`，页面生成与主题分析共享同一基础工厂；待产品配置界面和持久化设置落地后，再由构造注入替换默认值。配置文件不可读或模型不存在时，应在课件已加载的前提下进入 `AnalysisFailed`，并显示可操作的错误说明。

### 5. 正式主题领域模型

不建议继续把所有结果压缩在 `ThemeTitle` 和 `ThemeDescription` 中。建议引入独立 `CoursewareTheme`，并让 `CoursewareThemeAnalysisResult` 包含主题和诊断信息。

```text
CoursewareThemeAnalysisResult
  - CoursewareTheme Theme
  - DateTimeOffset AnalyzedAt
  - int TotalSlideCount
  - int AnalyzedSlideCount
  - TimeSpan Duration
  - CoursewareAnalysisUsage? Usage
  - IReadOnlyList<CoursewareAnalysisWarning> Warnings

CoursewareTheme
  - string SchemaVersion
  - string Title
  - string Summary
  - IReadOnlyList<string> StyleKeywords
  - CoursewareColorScheme Colors
  - CoursewareTypography Typography
  - CoursewareFontRecommendation Fonts
  - CoursewareSafeArea SafeArea
  - IReadOnlyList<string> LayoutPrinciples
  - CoursewarePageTypeRecommendations PageTypes
  - IReadOnlyList<string> ContentPresentationRules
  - string GenerationPromptSummary
```

其中：

- `Colors` 至少区分主色、强调色、背景色、主文字色和次文字色，每项包含用途与十六进制值。
- `Typography` 至少包含一级标题、二级标题、正文和辅助文字的字号与字重。
- `Fonts` 区分东亚文字与拉丁文字、标题与正文，并明确这些是推荐值而非已安装保证。
- `SafeArea` 使用与页面尺寸一致的像素单位，分别保存上、下、左、右边距。
- `PageTypes` 包含封面、目录/章节、普通内容和结尾页建议。
- `GenerationPromptSummary` 是供单页提示词注入的机器可读摘要，不能依赖 UI 文案反向拼接。

领域模型和未来 `Theme.json` 应使用同一份契约，并使用 `System.Text.Json` 源生成上下文，避免再维护一套 UI 模型和一套输出模型。

### 6. 分析输入策略

首版正式分析建议以全部页面 Markdown 为主要事实来源，不附带全部截图。原因是：

- 全局主题主要依据学科属性、内容层级、页面类型和讲解节奏形成。
- 全量截图会显著增加多模态成本，并容易让模型过度模仿原课件中的局部视觉问题。
- 页面截图仍应保留给第二页逐页美化，以理解当前页原始布局和视觉元素。

输入快照建议包含：

```text
课件基本信息
  - 课件名称
  - 总页数
  - 页面尺寸分布
  - 有截图页数
  - 资源类型与数量
  - 输入警告统计

页面快照（按 SlideIndex 排序）
  - 页码与 SlideId
  - 页面尺寸
  - 从 Markdown 提取的标题
  - 内容摘要
  - 页面警告
```

输入中禁止包含课件根目录、Markdown 绝对路径、截图绝对路径和资源绝对路径。

为了从原型走向真实可用产品，输入构建器必须实现长度预算，而不是假定所有课件都能完整提交：

1. 优先保留课件元数据和每页标题。
2. 优先保留封面、目录、章节页、结尾页和存在警告页面的完整可读文本。
3. 普通页按统一上限生成摘要。
4. 仍超出预算时，按章节或固定页组先生成本地摘要输入，再进行最终主题分析。
5. 结果中记录 `AnalyzedSlideCount` 和覆盖策略，禁止 UI 无条件声称“已完整查看全部页面”。

首期可以先实现“普通规模一次提交 + 明确超限错误”，但接口和结果模型应预留分组摘要，不应在服务内部散落字符串截断。

### 7. 模型请求与结构化提交

AgentLib 当前已经支持正式分析所需 API。请求建议使用：

```text
WithHistory = false
CreateNewSession = true
AppendDefaultTools = false
Tools = [submit_courseware_theme]
SystemPrompt = 主题分析专用系统提示词
CancellationToken = 当前分析令牌
```

System Prompt 应约束：

- 只根据提供的课件快照分析，不推断不存在的具体页面问题。
- 输出适合教学投影和课堂阅读的主题，而不是网页或移动端 UI 主题。
- 配色、字号、字体、安全区和版式建议需要相互一致。
- 必须调用 `submit_courseware_theme`，聊天文本不能替代结构化提交。
- 不输出隐藏推理，不要求访问文件系统，不调用渲染工具。

提交工具执行以下验证：

- 必填文本不为空，关键词和原则数量在合理范围内。
- 颜色符合 `#RRGGBB` 或 `#AARRGGBB`。
- 字号为正数、层级单调合理，正文和辅助文字不低于产品允许值。
- 安全区非负，且结合课件主要页面尺寸后仍保留有效内容区域。
- 页面类型建议完整。
- `AnalyzedSlideCount` 与输入快照覆盖信息一致。

建议最多进行两轮模型发送：

1. 第一轮提交完整课件快照并要求调用工具。
2. 如果模型未调用工具或提交未通过校验，第二轮只发送校验错误和“重新调用提交工具”的要求。

第二轮仍失败时抛出 `CoursewareThemeValidationException`。不能使用固定默认主题掩盖失败，否则用户会误以为正式分析成功。

### 8. 分析过程事件，而不是直接绑定 Agent 消息

第一页的“分析过程”不应直接绑定 `CopilotChatManager.ChatMessages`。Agent 消息包含工具参数、内部消息项和未来可能变化的框架类型，不适合作为稳定 UI 契约。

建议定义：

```text
CoursewareAnalysisEvent
  - Guid Id
  - CoursewareAnalysisStage Stage
  - CoursewareAnalysisEventKind Kind
  - string Title
  - string Message
  - DateTimeOffset Timestamp
  - CoursewareAnalysisEventState State

CoursewareAnalysisStage
  - PreparingInput
  - ReadingStructure
  - AnalyzingContentHierarchy
  - DesigningTheme
  - ValidatingTheme
  - RepairingTheme
  - Completed

CoursewareAnalysisEventKind
  - Progress
  - ModelSummary
  - ToolSubmission
  - Warning
  - Error
```

服务层通过 `IProgress<CoursewareAnalysisEvent>` 或等效观察接口发布事件。ViewModel 将事件映射到可观察集合并通过现有 Dispatcher 更新 UI。

流式模型文本只选择面向用户的摘要文本映射为 `ModelSummary`；Reasoning 内容、完整工具参数和工具技术输出不进入 UI。即使模型没有产生适合展示的流式文本，服务阶段事件也能保证用户看到稳定、真实的进度。

取消后保留已发布事件；重新分析时为新一轮分析创建新的运行标识，避免旧请求晚到的事件写入新分析结果。

### 9. ViewModel 与 UI 的最小调整

`CoursewareWorkspaceViewModel` 继续负责工作流状态，但应移除主题与分析过程的 Demo 集合，改为从当前会话派生：

- `ThemeColors`、`TypographyLevels`、`LayoutPrinciples`、`PageTypeRecommendations` 和 `ContentRules` 映射自 `CoursewareTheme`。
- `AnalysisSteps` 替换为真实 `AnalysisEvents`。
- `ThemeTitle` 和 `ThemeDescription` 映射自 `CoursewareTheme.Title/Summary`。
- 输入风险从 `CoursewareInputPackage.Warnings` 与分析结果警告合并，不在 XAML 中写死具体页码。
- `CoursewareThemeSummaryView` 的风格关键词、字体、安全区、颜色和页面类型全部改为绑定。

`CoursewareWorkspaceSession` 继续作为两页共享边界，并增加当前分析运行信息。建议至少保存：

- `CoursewareThemeAnalysisResult? ThemeAnalysisResult`
- `Guid? ActiveAnalysisRunId`
- `DateTimeOffset? AnalysisStartedAt`
- 是否存在由旧主题生成的页面结果

重新分析开始时可以暂时隐藏旧主题，但不要先销毁已经进入工作台产生的页面结果；如果新主题成功，再替换主题并标记已有页面结果可能过期。新主题失败时应保留旧主题供用户查看，但禁止把失败状态误显示为最新主题有效。

### 10. 异常、取消与日志

建议区分以下异常类型或错误码：

- `ThemeAnalysisConfigurationError`：配置文件、模型名称或凭据不可用。
- `ThemeAnalysisRequestFailed`：网络、服务端或模型调用失败。
- `ThemeAnalysisTimedOut`：达到主题分析超时。
- `ThemeAnalysisNoSubmission`：模型未调用提交工具。
- `ThemeAnalysisValidationFailed`：结构化结果经修复后仍无效。
- `ThemeAnalysisInputTooLarge`：当前版本无法在预算内构建输入。

日志应记录课件页数、输入字符/Token 估算、模型名称、耗时、工具提交次数、验证错误码和 Token 用量；不得记录 API Key、完整本地路径或未经裁剪的整份课件正文。

取消令牌应贯穿输入构建、模型调用和重试。用户取消后不再进行结构修复请求，并保持 `Canceled`。重新打开课件时取消旧分析，旧分析完成回调必须通过运行标识检查后才能发布结果。

### 11. 测试设计

现有 ViewModel 测试可以继续保留，并补充以下层次：

#### 输入构建器测试

- 页面按 `SlideIndex` 排序。
- 标题和摘要从导出 Markdown 正确提取。
- 不包含任何绝对路径。
- 封面、章节、普通页的文本预算策略符合预期。
- 缺少截图和资源只形成警告，不阻止构建。
- 超出当前处理预算时返回明确结果或异常。

#### 验证器测试

- 有效主题通过。
- 非法颜色、倒置字号层级、负安全区、过大安全区和缺失页面类型分别失败。
- 多个字段错误一次性返回，便于模型修复。

#### 分析服务测试

- 请求禁用历史和默认工具，只注册主题提交工具。
- 成功工具调用返回结构化主题。
- 首次无提交或校验失败时执行一次修复。
- 第二次仍失败时返回明确错误。
- 取消后不再发送修复请求。
- 流式文本与工具状态正确映射为业务事件，不暴露 Reasoning 和工具参数。

#### ViewModel 测试

- 自动分析期间逐步追加事件。
- 成功后所有主题 UI 集合来自真实结果。
- 取消、失败和重试保留缩略图与事件。
- 旧分析的迟到结果不会覆盖新课件或新一轮分析。
- 没有有效主题时不能进入工作台。

Agent 调用测试应使用可注入的假 `IChatClient` 或聊天管理器适配器，不访问真实模型。可以另设显式或手工集成测试验证真实配置，但不能让默认测试套件依赖个人配置文件和网络。

### 12. 推荐实施顺序

正式接入建议按以下顺序一次形成可验证闭环：

1. 提取基础 `CopilotChatManager` 工厂，集中管理当前开发者本地配置路径与默认模型，并保留未来产品配置注入点。
2. 创建 `CoursewareTheme`、分析诊断、过程事件和 JSON 源生成上下文。
3. 实现输入快照构建器及其长度预算测试。
4. 实现主题验证器和结构化提交工具。
5. 用独立 `CopilotChatManager` 实现正式 `CoursewareThemeAnalysisService`，包括一次修复、取消和错误分类。
6. 将真实过程事件和主题结果接入 `CoursewareWorkspaceViewModel`。
7. 移除第一页 ViewModel 与 XAML 中的固定主题、固定步骤和固定风险文案。
8. 增加服务、ViewModel、会话隔离和迟到结果保护测试。
9. 最后再把 `CoursewareTheme.GenerationPromptSummary` 接入第二页单页生成提示词。

完成第 1 至 8 项后，第一页才算从交互原型升级为真实的全局主题分析产品；第 9 项完成后，全局分析结果才真正参与后续课件美化，而不只是展示结果。

## 一、改造目标

当前 Demo 采用固定三栏单页工作台：左侧页面列表、中间页面预览与 SlideML 编辑区、右侧 Copilot 聊天区。应用启动后立即创建空页面和页面级 `SlideChatManager`，用户在中间区域点击“打开课件文件夹”，加载完成后直接进入单页处理状态。

该交互不适合全课件美化，主要问题是：

- 打开课件、理解课件、分析全局主题和编辑单页被放在同一页面，没有清晰的任务阶段。
- 用户打开课件后立即看到单页工作台，无法直观看到程序正在从全课件视角进行分析。
- 全局主题分析产物缺少独立展示空间，容易退化为聊天文本，难以被用户确认和复用。
- 现有 Copilot 区包含输入框、附件、模型和渲染设置，不适合作为自动主题分析期间的只读过程界面。
- 应用启动时创建空页和页面会话，与“初始界面只提供打开课件入口”的产品目标冲突。

本次改造目标是将主流程拆成两个页面：

1. **课件导入与全局分析页**
   - 初始状态只突出“打开课件”操作。
   - 用户选择课件目录后自动加载课件并自动执行全局主题分析，不再要求用户额外点击“分析全局主题”。
   - 加载后展示全课件缩略图、只读分析过程和结构化主题分析产物。
   - 分析完成后由用户点击“进入单页美化工作台”。

2. **单页美化工作台页**
   - 复用当前三栏工作台的信息架构和主要能力。
   - 每页生成提示词和聊天上下文增加全局主题、课件摘要、相邻页面摘要和资源约束。
   - 提供返回全局分析概览的入口，但不因返回而丢失各页会话和生成结果。

## 二、现有实现约束

### 2.1 当前窗口结构

`MainWindow.xaml` 当前固定为三列：

```text
240 px 左侧页面列表
  +
自适应中间工作区
  +
420 px 右侧 Copilot
```

三个区域分别由以下视图承担：

- `Views/LeftSidebarPanel.xaml`
- `Views/MainContentPanel.xaml`
- `Views/CopilotPanel.xaml`

双页改造不能只在三个现有区域内部切换可见性，否则第一页仍会受到三栏尺寸和职责的限制。主窗口应提升为页面容器，由页面级视图决定自身布局。

### 2.2 当前启动行为

`App.xaml.cs` 在窗口显示前创建一个完整的 `SlideChatManager`；`MainWindowViewModel` 构造时继续创建一个空页面并立即选中。

这意味着当前应用在尚未打开课件时已经初始化：

- 页面聊天管理器。
- 渲染管线。
- 页面评估器。
- 一个空的 `CoursewareSlideItem`。
- 单页 Copilot 输入上下文。

新交互要求初始页面只显示打开入口，因此以上页面级对象应延迟到课件加载后，或进入第二页前按需创建。全局主题分析使用独立会话，不应借用启动时创建的页面会话。

### 2.3 当前打开课件行为

`MainContentPanel.xaml.cs` 负责文件夹选择事件桥接，随后调用 `MainWindowViewModel.OpenCoursewareFolderAsync`。加载完成后，ViewModel 为所有页面逐个创建 `SlideChatManager`，填充 `Slides`，并选中第一页。

新流程需要把打开动作改为一条连续编排链路：

```text
选择课件目录
  -> 加载并校验输入包
  -> 显示页面缩略图和课件摘要
  -> 自动创建独立主题分析会话
  -> 自动开始全局主题分析
  -> 流式显示分析过程
  -> 校验并展示结构化主题
  -> 用户进入单页工作台
```

### 2.4 当前聊天面板行为

`CopilotPanel.xaml` 固定包含：

- 模型选择。
- 聊天消息列表。
- 图片附件。
- MCP 设置。
- 用户输入框和发送按钮。

第一页不能直接复用完整 `CopilotPanel`。应抽取可复用的“消息流展示”部分，或创建主题分析专用只读视图，避免通过隐藏多个底部区域勉强复用一个承担不同职责的控件。

### 2.5 当前状态模型

当前只有全局 `IsBusy`，它同时控制发送消息、重新渲染、MCP 连接和打开课件按钮。双页流程至少需要区分：

- 未打开课件。
- 正在加载课件。
- 正在分析主题。
- 分析完成。
- 分析失败。
- 正在进入工作台。
- 工作台可交互。
- 正在执行当前页生成或渲染。

不能继续用多个布尔值随意组合表示页面阶段，应增加强类型页面状态和主题分析状态。

## 三、核心产品决策

### 3.1 打开课件后自动分析

主题分析不再是用户主动触发的独立步骤。

用户完成目录选择后，如果输入包加载成功，程序立即开始全局主题分析。用户不需要理解“为什么还要再点一次分析”，也不需要在加载和分析之间做额外决策。

自动分析的前置条件：

- `Courseware.json` 可读取。
- 至少存在一页有效课件页。
- 主题分析所需模型配置可用。
- 没有阻止分析的输入错误。

非阻塞警告，例如部分页面缺少截图或个别资源缺失，不阻止主题分析，但必须在第一页显示警告摘要。

### 3.2 分析完成后不自动跳转

分析完成后默认停留在第一页，由用户查看结构化主题产物并点击“进入单页美化工作台”。

原因：

- 用户需要看到自动分析的成果，而不是只看到加载动画后突然进入工作台。
- 用户可能需要重新分析或重新选择课件。
- 为后续增加主题编辑、主题确认和分析模型切换保留交互位置。

主按钮在不同阶段的行为：

| 阶段 | 主按钮 | 行为 |
| --- | --- | --- |
| 未打开 | 打开课件 | 选择目录 |
| 加载中 | 正在读取课件 | 禁用并显示进度 |
| 分析中 | 正在分析主题 | 禁用；旁边提供取消 |
| 分析完成 | 进入单页美化工作台 | 切换到第二页 |
| 分析失败 | 重试主题分析 | 使用已加载输入重新分析 |

### 3.3 第一页聊天区只读

第一页的聊天界面用于表达“系统正在如何理解课件”，不提供用户输入、图片附件、预览附件和 MCP 设置。

允许展示：

- 阶段性系统消息。
- 模型流式输出的可读分析内容。
- 结构化工具提交状态。
- 警告和错误。
- Token 或耗时摘要，可放在次要信息区域。

不建议默认展示完整内部推理内容。界面中的“分析过程”应是面向用户的工作过程和阶段结果，不应将模型隐藏推理直接作为产品内容。

### 3.4 第二页保持现有工作台心智

第二页不重新设计。当前左侧页面列表、中间预览与 SlideML、右侧 Copilot 的三栏结构本身适合单页美化，因此继续保留现状，只接入全局主题上下文和返回第一页的入口。

第二页主要新增：

- 顶部全局主题摘要入口。
- 当前课件名称与分析状态。
- 返回全局分析页。
- 页面生成提示词中的主题上下文。

## 四、双页导航与状态机

### 4.1 页面级状态

建议新增 `CoursewareWorkspacePage`：

- `CoursewareAnalysis`
- `SlideWorkspace`

第一页内部再使用 `CoursewareAnalysisStage`：

- `Welcome`
- `LoadingCourseware`
- `AnalyzingTheme`
- `AnalysisReady`
- `AnalysisFailed`
- `Canceled`

页面级状态和业务运行状态应分开。用户可以在主题已完成后多次往返两个页面，而不重新加载课件或重建所有页面状态。

### 4.2 主流程

```text
应用启动
  -> CoursewareAnalysis / Welcome
  -> 用户点击“打开课件”
  -> 打开目录选择器
  -> 用户取消选择：保持 Welcome
  -> 用户选择目录
  -> LoadingCourseware
  -> 加载失败：AnalysisFailed，显示输入错误，可重新选择
  -> 加载成功：立即进入 AnalyzingTheme
  -> 自动执行主题分析
  -> 分析成功：AnalysisReady
  -> 用户查看主题产物
  -> 用户点击“进入单页美化工作台”
  -> SlideWorkspace
```

### 4.3 返回与重新打开

从第二页点击“返回主题概览”时：

- 切回第一页的 `AnalysisReady`。
- 保留当前选中页。
- 保留所有页面聊天、SlideML、预览和日志。
- 不重新执行主题分析。

“打开其他课件”是破坏性操作。执行前如果当前课件已有生成结果，应提示用户尚未保存的结果可能丢失。确认后：

1. 取消当前主题分析或单页操作。
2. 释放当前课件的页面会话与渲染资源。
3. 清空旧输入、主题和页面状态。
4. 重新进入目录选择和自动分析流程。

### 4.4 取消

分析期间提供“取消分析”。取消后：

- 已加载的课件缩略图继续显示。
- 分析消息保留。
- 状态进入 `Canceled`。
- 提供“重新分析”和“打开其他课件”。
- 未获得有效主题前不能进入第二页。

工作台始终要求存在经过校验的全局主题，不设计无主题进入路径。

## 五、第一页信息架构

### 5.1 页面定位与应用背景

这是一个 Windows 桌面端课件美化应用。它读取已经导出的课件目录，目录中包含每一页的 Markdown 文本、原始页面截图和课件资源。应用先从整份课件的角度分析内容结构与视觉方向，形成统一的配色、字体层级、版式原则和安全区；随后用户进入现有的单页工作台，逐页生成和调整 SlideML。

第一页承担“打开课件并理解整份课件”的任务，不是单页编辑器。它需要覆盖以下连续状态：

1. 尚未打开课件时，提供唯一的“打开课件文件夹”入口。
2. 打开课件后，展示课件名称、页数和全部页面缩略图。
3. 自动开始全局主题分析，以只读聊天/时间线展示系统正在读取和理解什么。
4. 在同一页面展示逐步形成的结构化主题产物，例如主题摘要、颜色、字体层级、版式原则和安全区。
5. 分析完成后，用户查看结果并点击“进入单页美化工作台”。

设计时最重要的状态是第 3、4 项同时发生的界面：用户已经打开课件，能够看到课件页面概览、持续追加的分析过程，以及正在形成或已经形成的主题产物。该状态最能代表第一页的核心价值，也应作为视觉设计稿的目标画面。各区域的位置、占比和呈现方式由视觉设计师决定。

### 5.2 Welcome：初始空状态

初始界面只保留必要品牌信息、产品用途说明和打开入口，不能出现空白的三栏编辑器。首屏只有“打开课件文件夹”这一项主要操作；模型、MCP、添加空页面和聊天输入均不应出现。具体构图、文案和辅助视觉元素由视觉设计师决定。

### 5.3 LoadingCourseware：课件加载状态

目录选择完成后，不再返回 Welcome。页面进入加载状态，并让用户能够了解正在读取的课件和当前进度：

- 显示课件目录名称，不显示绝对路径。
- 显示当前阶段，例如读取 `Courseware.json`、读取页面、检查资源；具体文案不固定。
- 内容出现时避免闪烁或明显布局跳变，但不能为了展示动画而延长实际流程。

### 5.4 AnalyzingTheme：自动主题分析状态

课件加载成功后，页面需要同时容纳课件页面缩略图概览、全局主题分析过程和逐步形成的主题结果。具体区域关系由视觉设计师确定，但必须保证缩略图按行自动换行，并在不同窗口尺寸下维持清晰的信息层级和可操作性。

### 5.5 AnalysisReady：分析完成状态

分析完成后，缩略图概览和分析过程继续保留，结构化主题结果变为可确认状态，并提供“进入单页美化工作台”主要操作。按钮位置和结果出现方式不作限定。

用户进入第二页前，应能在一个屏幕内回答以下问题：

- 程序认为这是一份什么课件？
- 推荐采用什么视觉风格？
- 核心颜色是什么？
- 标题和正文如何分级？
- 页面布局应遵循哪些原则？
- 是否完整分析了所有页面？
- 是否存在输入警告？

## 六、第一页缩略图设计

### 6.1 内容与规格

缩略图图片区域保持原页面宽高比，并在常见窗口尺寸下兼顾可辨识度与全课件覆盖感。具体卡片尺寸和间距由视觉设计确定，工程实现不写死单一视觉尺寸。

每项显示：

- 原始页面截图。
- 可辨识的页码。
- 页面标题；空间不足时允许省略。
- 页面警告标识，例如缺少截图。

分析期间可以表达以下状态：

- 尚未扫描。
- 已完成摘要读取。
- 当前正在处理。
- 存在输入警告。

各状态的视觉形式由设计师决定，但不能只依赖颜色表达，也不能使用强烈闪烁。

### 6.2 布局

首期按普通课件规模设计。缩略图区域使用 WPF `WrapPanel`：从左到右按行排列，当前行空间不足时自动换到下一行，不使用横向滚动容器，也不显示横向滚动条。缩略图区域或页面整体允许纵向滚动，以容纳多行缩略图和下方分析内容。大课件的虚拟化、分页和缩略图内存优化不在本计划范围内。

### 6.3 缩略图交互

第一页缩略图的主要用途是展示全课件范围，不承担单页编辑。

建议交互：

- 单击：查看大图、标题和 Markdown 摘要；具体承载方式由视觉设计师决定。
- 双击：分析完成后进入第二页并定位该页；分析中不允许进入工作台。
- 键盘：支持方向键移动焦点和 Enter 查看。
- Tooltip：显示页码、页面 ID、标题和警告摘要。

首期也可以只实现单击预览，不实现双击跳转，以降低交互复杂度。

## 七、只读分析过程设计

### 7.1 分析过程的定位

第一页面中的聊天区不是传统问答，而是只读的 AI 工作过程。标题和辅助说明应让用户明确这是全局主题分析，并能与第二页的单页聊天区分；具体文案不固定，不应只使用含义过于宽泛的“Copilot”。

### 7.2 推荐消息阶段

系统应输出面向用户的阶段性消息，而不是只有模型的一段长文本：

1. 已读取课件基本信息。
2. 正在识别封面、目录、章节页和内容页。
3. 正在归纳学科属性与内容层级。
4. 正在分析页面尺寸和资源分布。
5. 正在形成配色、字体和版式建议。
6. 正在校验主题结构。
7. 主题分析完成。

消息可来自服务层进度事件和模型流式文本。系统阶段、模型摘要、工具提交、警告和错误需要能够被用户区分；工具技术详情默认不应干扰主要内容，错误状态需要提供重试入口。具体组件样式、颜色和展开方式由视觉设计师决定，状态不能只依赖颜色表达。

### 7.3 不显示的内容

默认不显示：

- 用户消息气泡，因为没有用户输入。
- 图片附件编辑区。
- “附带当前预览截图”。
- MCP 服务配置。
- 单页模型生成工具。
- 完整工具参数和本地文件路径。
- 模型内部隐藏推理。

### 7.4 完成后的处理

分析完成后保留分析过程，并明确表达完成状态、耗时和课件页数。用户位于消息末尾时可以继续自动跟随，否则不应强制改变滚动位置。

可以提供查看分析详情的入口，但结构化主题结果才是主要产物，不能要求用户从聊天文本中自行提炼主题。详情入口的具体形式不作限定。

---

## 八、主题分析产物展示

### 8.1 必须展示的内容

第一版主题产物至少包含以下信息。具体分组、顺序和组件形式由视觉设计师决定。

#### 1. 主题摘要

展示：

- 简洁的主题概述。
- 能帮助用户快速理解方向的风格关键词，例如“现代、清晰、理性、轻量科技”。
- 适用的整体视觉语气。

该信息应具有足够的视觉优先级，让用户能够快速理解分析结果。

#### 2. 色彩系统

展示：

- 主色。
- 强调色。
- 背景色。
- 主文字色和次文字色。

需要同时表达颜色、用途名称和十六进制值。颜色较多时优先展示核心色，避免次要信息干扰主要结果；具体展示方式由视觉设计师决定。

#### 3. 字体与字号层级

展示：

- 推荐标题字体。
- 推荐正文字体。
- 一级标题、二级标题、正文、辅助文字的字号或相对层级。
- 字重建议。

字体不可用时要说明后续将使用替代字体，不能让分析结果看起来已经保证字体存在。

#### 4. 版式原则

展示精炼的版式原则，例如：

- 标题区保持固定高度。
- 主要信息优先放在左上视觉起点。
- 每页只保留一个主视觉焦点。
- 图文内容遵守统一对齐线。
- 章节页与内容页形成明确节奏变化。

原则内容应便于快速浏览，避免在主界面形成长文档。

#### 5. 安全区

展示上、下、左、右安全边距及其数值，并让用户能够直观理解这些数值在幻灯片中的含义。具体可视化方式由视觉设计师决定。

#### 6. 页面类型建议

展示：

- 封面页建议。
- 目录或章节页建议。
- 普通内容页建议。
- 结尾页建议。

首期不需要为每种页面类型生成完整预览稿，但应让用户能够查看各类型建议。具体组织方式不作限定。

#### 7. 输入概览与风险

展示：

- 总页数。
- 缺少截图页数。
- 缺失资源数量。
- 其他加载警告数量。

该区域必须清晰但视觉优先级低于主题本身。

### 8.2 可后置内容

以下内容不必在首期主界面完整展示：

- 原始模型 JSON。
- 完整提示词摘要。
- 每个页面的分类结果。
- 全部资源列表。
- 主题版本历史。
- 用户编辑主题。

可以保留“查看原始分析数据”开发入口，但不作为面向普通用户的主要操作。

### 8.3 主题确认

首期不增加复杂的字段级编辑器。分析完成后提供：

- `进入单页美化工作台`：主操作。
- `重新分析`：次操作。
- `打开其他课件`：次操作。

如果重新分析会覆盖当前主题，应提示用户。尚未进入工作台时不需要保存旧主题版本；进入工作台并已产生页面结果后重新分析，则应提示新主题可能与既有页面结果不一致。

## 九、第二页设计

### 9.1 基础布局

第二页继续采用现有三栏布局：

```text
左侧：课件页面列表
中间：当前页预览、SlideML 和日志
右侧：当前页 Copilot 对话与生成控制
```

现有控件、尺寸、布局关系和主要交互尽可能保持不动。仅为了主窗口承载双页导航，可将现有三栏整体包入 `SlideWorkspaceView`，不借此重做工作台内部结构。

### 9.2 顶部全局信息条

在不挤压现有工作区的前提下，只增加必要的课件级入口：

- 返回主题概览。
- 当前课件名称。
- 主题状态，例如“已应用全局主题”。
- 打开主题详情的入口。

主题详情不建议再次占据一个永久列，具体展示方式由视觉设计师决定。

### 9.3 左侧页面列表

现有页面缩略图宽度、选择方式和列表行为保持不动。第一页缩略图用于全课件概览，第二页继续使用现有列表选择和处理当前页，不新增批量状态或批次进度交互。

### 9.4 当前页 Copilot 上下文

进入第二页后，每页仍使用独立 `SlideChatManager`。首次生成提示词除当前页 Markdown 和截图外，还应包含：

- 全局主题分析结果。
- 全课件摘要。
- 前后三页摘要。
- 页面尺寸。
- 当前页资源白名单。
- 安全区与版式约束。

右侧聊天界面仍允许用户输入、附加图片、切换模型和继续追问。主题分析会话不能与当前页聊天会话混用。

### 9.5 进入工作台时的初始化策略

不建议在第一页刚加载课件时同步创建所有完整页面会话后才开始主题分析，否则会延迟自动分析的启动。

建议顺序：

1. 加载 `CoursewareInputPackage`。
2. 立即构建轻量缩略图 ViewModel。
3. 启动主题分析。
4. 在分析期间后台受控创建页面级会话，或进入第二页时按需创建。
5. 进入第二页前保证当前选中页所需页面会话已就绪；其余页面可继续受控初始化。

最终选择“全部预创建”还是“按需创建”需要结合 `SlideChatManager` 初始化耗时和模型配置加载成本测量。无论采用哪种方式，都不能阻塞第一页缩略图和分析过程的首屏显示。

## 十、推荐 View 与 ViewModel 拆分

### 10.1 View 层

建议新增：

```text
Views/
  CoursewareAnalysisView.xaml
  CoursewareAnalysisChatView.xaml
  CoursewareThemeSummaryView.xaml
  CoursewareThumbnailOverview.xaml
  SlideWorkspaceView.xaml
```

职责：

- `CoursewareAnalysisView`：第一页页面级布局和状态切换。
- `CoursewareAnalysisChatView`：只读分析消息与阶段时间线。
- `CoursewareThemeSummaryView`：结构化主题结果。
- `CoursewareThumbnailOverview`：全课件缩略图概览。
- `SlideWorkspaceView`：组合现有三栏控件形成第二页。

现有 `CopilotPanel` 继续服务第二页。聊天消息 ItemTemplate 中可复用的部分应抽为资源或专用控件，避免复制整套消息模板。

### 10.2 ViewModel 层

建议将当前职责过重的 `MainWindowViewModel` 拆为：

```text
MainWindowViewModel
  ├── CoursewareAnalysisViewModel
  └── SlideWorkspaceViewModel
```

`MainWindowViewModel` 负责：

- 当前页面。
- 页面切换。
- 当前课件生命周期。
- 重新打开课件时的清理与确认。

`CoursewareAnalysisViewModel` 负责：

- 打开课件命令。
- 输入包加载状态。
- 自动主题分析。
- 只读分析消息。
- 主题结果和警告。
- 取消、重试和进入工作台命令。

`SlideWorkspaceViewModel` 负责：

- 页面集合和选择。
- 页面级聊天。
- SlideML 编辑与渲染。
- MCP 设置。
- 主题上下文展示。

避免让两个子 ViewModel 各自持有不同的输入包或主题副本。课件输入和主题结果应通过课件会话对象或明确的只读上下文共享。

### 10.3 课件会话边界

建议新增 `CoursewareWorkspaceSession`，表示当前已打开课件在应用中的一次会话，至少包含：

- `CoursewareInputPackage`。
- `CoursewareThemeAnalysisResult?`。
- 页面 ViewModel 集合。
- 当前选中页面索引。
- 是否存在未保存结果。

它只作为两个页面共享的课件级状态边界。主题分析服务、页面生成服务和输出服务仍保持独立。

## 十一、服务与自动分析编排

### 11.1 自动分析编排服务

“打开后自动分析”不应写成 ViewModel 中一段不断增长的 `try/finally`。建议由课件打开编排服务或明确的 ViewModel 私有流程协调：

```text
OpenCoursewareAndAnalyzeAsync
  -> 选择目录由 View 进行平台桥接
  -> LoadAsync
  -> 创建 CoursewareWorkspaceSession
  -> 发布缩略图和输入摘要
  -> AnalyzeThemeAsync
  -> 本地校验主题
  -> 发布 CoursewareThemeAnalysisResult
```

文件夹选择仍可保留在 code-behind 作为平台事件桥接，但“选择成功后加载并自动分析”的业务链路属于 ViewModel/服务层。

### 11.2 主题分析会话

主题分析必须使用专用 `CopilotChatManager`：

- 不复用任何页面会话。
- 不注册 SlideML 渲染工具。
- 关闭默认文件系统工具。
- 不显示用户输入区。
- 接受取消令牌。
- 将流式消息映射到第一页只读消息集合。

主题分析会话创建失败时，不应影响已加载输入包和缩略图展示。界面进入 `AnalysisFailed`，用户可以修复配置后重试。

### 11.3 页面会话初始化

当前 `SlideChatManagerFactory` 同时包含个人本地配置路径、默认模型、渲染器、评估器和 MCP 尝试连接。实施双页模式前，应按原全课件方案将配置外置，并区分：

- 主题分析聊天管理器工厂。
- 页面 SlideChatManager 工厂。
- 评估器聊天管理器工厂。

不能通过给主题分析创建一个“没有使用渲染功能的完整 SlideChatManager”来假装职责分离。

### 11.4 取消令牌生命周期

建议至少区分：

- 当前打开/分析流程的 `CancellationTokenSource`。
- 当前单页操作的取消令牌。

重新打开课件时必须取消旧课件的所有未完成操作。页面切换不应误取消全局主题结果或其他页面已完成状态。

## 十二、动画与转场设计

### 12.1 原则

动画用于解释状态变化，不用于延长流程或制造炫技效果。具体动效形式、方向和时长由视觉设计师确定，整体应克制、连贯且适合桌面生产力软件。

需尊重系统减少动画设置；无法读取时至少提供关闭复杂动画的应用设置扩展点。动画不能阻塞业务流程，也不能成为用户判断状态的唯一依据。

### 12.2 Welcome 到加载状态

应在同一窗口中清楚表达从欢迎状态进入加载状态，不要让整个窗口闪白或重新创建窗口。具体过渡方式由视觉设计师决定。

### 12.3 课件内容出现

课件加载完成后，应平稳呈现缩略图、第一条分析阶段信息和主题结果区域，避免明显布局跳变。具体内容占位和出现方式由视觉设计师决定。

### 12.4 分析完成

分析完成时，应让用户清楚感知状态已经完成、主题结果已经可用，并能够发现进入工作台的主要操作。具体转场方式由视觉设计师决定，不使用循环闪烁或干扰工作的娱乐化动效。

### 12.5 两页切换

两页在同一窗口内切换，保持窗口尺寸、位置和标题栏不变。具体转场方向和形式由视觉设计师决定。动画期间应阻止重复导航命令，返回操作应与进入操作保持一致的交互逻辑。

## 十三、异常与边界状态

### 13.1 输入加载失败

展示：

- 简洁错误标题。
- 可理解的错误原因。
- “重新选择课件”主操作。
- 可查看和复制的技术详情。

加载失败时不显示容易被误解为有效内容的空缩略图区域和空主题结果区域。

### 13.2 部分输入警告

如果缺少部分截图或资源：

- 继续自动分析。
- 显示警告数量。
- 对应缩略图显示警告标识。
- 主题结果的“分析覆盖与风险”中汇总。

### 13.3 主题分析失败

缩略图和课件基本信息继续保留。分析区域明确表达失败状态，并提供：

- 重试主题分析。
- 打开其他课件。
- 查看错误详情。

首期不允许在没有有效主题时进入工作台，避免第二页产生缺少全局约束的页面结果。

### 13.4 主题结构校验失败

如果模型输出可读文本但结构化主题无效：

- 分析状态保持进行中或进入“正在修复主题结构”。
- 最多按服务策略执行一次修复请求。
- 修复仍失败后进入 `AnalysisFailed`。
- 聊天区保留已产生的可读文本，但明确标记“未形成可用主题”。

### 13.5 超出当前处理规模的课件

当前方案只覆盖能够一次完整提交给模型并正常展示缩略图的普通课件。涉及长度限制、摘要、采样、分段归并、缩略图虚拟化或页面会话规模优化的课件不在本计划范围内，后续另行设计。

## 十四、无障碍与可用性

- 所有主要操作支持键盘焦点和 Enter/Space 激活。
- 页面切换后将焦点移动到新页面标题，不留在已隐藏按钮上。
- 状态不能只依赖颜色表达，需要图标和文字。
- 色块旁显示十六进制值，满足色觉差异用户读取需求。
- 分析消息流新增内容时不强制抢走用户滚动位置；用户位于底部时才自动跟随。
- 动画不使用快速闪烁。
- 错误详情可复制。
- 缩略图提供可访问名称，例如“第 3 页，二次函数，摘要已读取”。

## 十五、视觉设计师任务书

以下任务书只描述产品背景、核心信息和必要的布局关系。除明确说明的信息层级与缩略图位置外，组件形态、色彩、间距和文案呈现由视觉设计师根据专业判断完成。

以下代码块可直接复制给外部视觉设计师：

```text
请为 Windows 桌面端课件美化应用“课件智绘”设计“全课件分析页”的高保真界面。

页面用途：
用户打开一整份课件后，应用会自动分析课件内容与已有页面视觉，形成一套统一主题，再由用户进入单页工作台逐页美化。本页面用于让用户确认系统已经分析整份课件，并查看后续美化将采用的主题方案。

目标画面：
示例课件为“初中数学八年级上册”，共 24 页。全课件分析已经完成，输入完整且主题方案可用，用户正在查看结果并准备进入单页美化工作台。

页面需要承载以下信息和操作：

1. 应用与当前课件
   - 产品名称。
   - 当前课件名称和页数。
   - 分析已完成的状态。
   - 打开其他课件的入口。
   - 这些信息用于建立上下文，保持简洁，不应占据页面主体。

2. 顶部缩略图概览
   - 缩略图区域必须位于页面内容顶部，形成一条横向的全课件概览带，不能设计成左侧栏、页面导航栏或占据主要空间的大区域。
   - 每项只展示原始页面缩略图，不显示页面名称、页码、页面 ID 或其他逐页文字。
   - 缩略图从左到右排列，当前行空间不足时自动换到下一行，不使用横向滚动条。
   - 此区域只用于让用户感受到系统查看了整份课件，不承担单页选择、编辑、排序或批量操作，视觉优先级应明显低于主题方案。

3. 主题方案
   - 主题方案是页面的视觉主体。首先突出一段简洁的主题摘要和风格关键词，让用户先理解推荐的整体视觉方向，再阅读具体规范。
   - 核心规范包括配色及用途、标题与正文字体层级，以及少量关键版式原则。颜色需要同时表达用途和十六进制值。
   - 安全区、封面页与普通内容页的布局建议属于补充信息，应弱化或收纳，不能与主题摘要等权排列。
   - 不要把所有字段设计成视觉权重相同的指标卡片；应形成一个主次明确、能够整体阅读的主题结果区域。
   - 这是已经形成的一套统一主题，不设计成需要用户从多个候选主题中选择的方案列表。
   - 示例主题应适合初中数学教学，强调清晰推导、例题讲解、理性和现代感；具体视觉语言与配色由你设计。

4. 分析过程
   - 以紧凑的只读过程或阶段摘要，说明系统完成了课件读取、内容结构识别、视觉方向分析和主题生成。
   - 分析过程用于帮助用户建立信任，是辅助信息，不应成为页面主体。
   - 各阶段描述同一份 24 页课件的处理过程，不要为阶段编造不同的页数、百分比、评分或其他统计指标。
   - 不提供聊天输入，不展示模型内部推理、工具参数、本地文件路径或其他技术细节。

5. 后续操作
   - “进入单页美化工作台”是清晰可发现的主要操作。
   - “重新分析”和“打开其他课件”是次要操作。

整体应呈现专业、清晰、可信并适合教学内容的桌面生产力软件体验。虽然缩略图位于内容顶部，但视觉层级应依次为主题方案、分析完成状态与主要操作、分析过程、缩略图概览。除上述层级关系外，其他区域的布局与组件形式可由你决定。

本次只设计分析完成这一种状态下的全课件分析页，不需要呈现单页编辑器、聊天输入框、代码编辑器、模型配置或技术配置。示例输入完整，不要为了展示功能而添加缺少截图、字体过小、具体问题页码等警告。
```

## 十六、实施阶段建议

### 阶段一：主窗口页面化

目标：建立双页容器，不改变现有工作台内部业务。

步骤：

1. 将 `MainWindow` 从固定三列改为页面内容宿主。
2. 新增 `CoursewareAnalysisView`。
3. 将现有三栏组合迁入 `SlideWorkspaceView`。
4. 新增强类型页面导航状态。
5. 实现无数据时只显示第一页 Welcome。
6. 实现两个页面之间的导航和状态保留。

验收标准：

- 应用启动时不显示三栏工作台。
- 初始页面只有打开课件主入口。
- 可以进入和返回工作台。
- 页面切换不丢失现有单页状态。

### 阶段二：加载流程与缩略图概览

目标：打开课件后先展示完整课件概览。

步骤：

1. 将打开课件入口迁到第一页。
2. 增加 `CoursewareWorkspaceSession` 或等效课件级上下文。
3. 加载成功后创建轻量页面缩略图 ViewModel。
4. 实现缩略图概览和警告状态。
5. 增加加载中、加载失败和重新选择状态。

验收标准：

- 选择目录后在第一页看到课件名称、页数和缩略图。
- 缺少截图页面有明确占位和警告。
- 不因创建所有页面聊天会话而延迟缩略图首屏。

### 阶段三：打开后自动主题分析

目标：课件加载成功后无需额外点击即可完成主题分析。

步骤：

1. 新增主题分析专用聊天管理器工厂。
2. 实现主题分析服务和结构化主题校验。
3. 将加载成功事件直接衔接主题分析。
4. 增加只读分析消息模型和阶段进度。
5. 实现取消、重试和错误详情。
6. 确保分析会话不注册页面渲染或默认文件系统工具。

验收标准：

- 打开课件后自动开始主题分析。
- 用户无需点击第二个“分析”按钮。
- 第一页没有输入框。
- 分析失败不清空已加载课件。
- 取消后可以重新分析。

### 阶段四：主题产物与工作台承接

目标：展示可确认的结构化主题，并将其注入第二页。

步骤：

1. 实现主题摘要、配色、字体、版式、安全区和覆盖信息组件。
2. 分析完成后启用“进入单页美化工作台”。
3. 将主题结果保存到课件会话上下文。
4. 第二页顶部增加主题摘要入口。
5. 将主题结果注入页面提示词构建服务。
6. 保持页面会话与主题分析会话隔离。

验收标准：

- 用户无需阅读聊天全文即可理解主题。
- 第二页能查看当前应用的主题。
- 页面生成提示词包含全局主题。
- 返回第一页不会重新分析或丢失页面结果。

### 阶段五：动效、设计时数据与质量收尾

目标：完成体验细节和可测试性。

步骤：

1. 增加状态切换动效和减少动画降级。

## 十八、测试建议

### 18.1 导航状态测试

覆盖：

- 启动时处于 `CoursewareAnalysis / Welcome`。
- 未加载课件时不能进入第二页。
- 分析完成后可以进入第二页。
- 从第二页返回后主题和页面结果保留。
- 重复导航不会重复创建主题分析会话。
- 导航动画期间不能重复执行切换命令。

### 18.2 自动分析流程测试

覆盖：

- 选择有效目录后自动调用主题分析服务。
- 用户不需要额外执行分析命令。
- 加载失败时不调用主题分析。
- 非阻塞加载警告不阻止主题分析。
- 分析取消后不进入工作台。
- 分析失败后可复用已加载输入重试。
- 重新打开课件会取消旧分析。

### 18.3 第一页 ViewModel 测试

覆盖：

- Welcome 只允许打开课件。
- Loading 状态禁用重复打开。
- Analyzing 状态允许取消。
- AnalysisReady 启用进入工作台。
- AnalysisFailed 启用重试。
- 缩略图按页面索引排序。
- 警告统计正确。

### 18.4 上下文隔离测试

覆盖：

- 主题分析使用独立 `CopilotChatManager`。
- 主题分析消息不出现在任何页面聊天中。
- 页面聊天历史不进入主题分析请求。
- 第二页各页仍使用独立 `SlideChatManager`。
- 返回第一页不替换当前页聊天管理器。

### 18.5 UI 与可用性测试

覆盖：

- `1720 × 1000` 和 `1360 × 820` 布局。
- 分析消息自动滚动规则。
- 页面切换后的键盘焦点。
- 减少动画模式。
- 错误和警告不只依赖颜色表达。

## 十九、风险与待验证项

### 19.1 页面会话初始化成本

当前每页创建完整 `SlideChatManager`，并可能尝试连接 MCP。需要测量创建 20、50、100 页会话的耗时，决定后台受控预创建还是按需创建。

### 19.2 分析消息来源

主题分析过程既有服务阶段事件，又有模型流式输出。需要定义统一的只读消息模型，避免 UI 直接绑定具体 Agent 类型后难以插入加载、校验和错误阶段。

### 19.3 主题重分析后的既有页面结果

进入第二页并生成部分页面后，如果用户返回并重新分析主题，旧页面结果可能与新主题不一致。首期应明确警告并标记页面需要重新生成，不应静默替换主题。

### 19.4 当前配置耦合

`SlideChatManagerFactory` 当前包含个人本地配置文件路径。双页自动分析会使配置错误更早暴露，因此配置外置是实现自动分析前的必要前置工作。

### 19.5 窗口最小尺寸

当前最小宽度为 `1360 px`。第一页可以更灵活，但第二页三栏在更窄窗口下可用性有限。首期保持当前最小尺寸，后续再设计右侧 Copilot 可折叠能力。

## 二十、推荐实施决策

1. 将主窗口改为页面宿主，第一、第二页各自管理布局，不通过隐藏三栏模拟双页。
2. 应用启动时不创建空页面，不提前进入单页会话。
3. 选择课件目录并加载成功后立即自动执行全局主题分析。
4. 分析完成后停留第一页，由用户查看主题并主动进入第二页。
5. 第一页使用只读分析时间线，不显示聊天输入、附件和 MCP 设置。
6. 主题产物必须结构化展示，至少覆盖摘要、颜色、字体层级、版式、安全区和分析覆盖。
7. 第一页缩略图使用 WPF `WrapPanel` 按行自动换行，不使用横向滚动容器；内容超出可用高度时仅纵向滚动。
8. 第二页复用现有三栏工作台，并增加全局主题入口，不新增长期占用空间的第四栏。
9. 主题分析会话与各页会话严格隔离。
10. 使用强类型页面与分析状态，替代单一 `IsBusy`。
11. 课件加载、主题分析和当前页生成分别管理取消令牌。
12. 第一版不提供主题字段级编辑，只提供进入工作台、重新分析和打开其他课件。

按此方案实施后，用户的操作路径将从“打开课件后面对复杂单页编辑器”变为“打开课件—自动理解全局—查看统一主题—进入单页工作台”，既能承接全课件美化能力，也能保留现有单页 SlideML 生成和编辑体验。
