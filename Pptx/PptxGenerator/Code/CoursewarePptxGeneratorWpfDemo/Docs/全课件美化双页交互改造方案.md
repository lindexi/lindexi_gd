# CoursewarePptxGeneratorWpfDemo 全课件美化双页交互改造方案

> 状态：交互与实施方案设计阶段，尚未开发。
>
> 本文是独立于 `计划/全课件美化能力设计方案.md` 的交互改造方案，聚焦“双页模式”、打开课件后自动执行全局主题分析、第一页视觉设计要求，以及第二页如何承接现有单页工作台。
>
> 本文只描述方案与实施计划，不包含代码改动。

---

## 一、改造目标

当前 Demo 采用固定三栏单页工作台：左侧页面列表、中间页面预览与 SlideML 编辑区、右侧 Copilot 聊天区。应用启动后立即创建空页面和页面级 `SlideChatManager`，用户在中间区域点击“打开课件文件夹”，加载完成后直接进入单页处理状态。

该交互不适合全课件美化，主要问题是：

- 打开课件、理解课件、分析全局主题和编辑单页被放在同一页面，没有清晰的任务阶段。
- 用户打开课件后立即看到单页工作台，无法直观看到程序正在从全课件视角进行分析。
- 全局主题分析产物缺少独立展示空间，容易退化为聊天文本，难以被用户确认和复用。
- 现有 Copilot 区包含输入框、附件、模型和渲染设置，不适合作为自动主题分析期间的只读过程界面。
- 应用启动时创建空页和页面会话，与“初始界面只提供打开课件入口”的产品目标冲突。

本次改造目标是将主流程拆成两个页面：

1. **课件导入与全局分析页**
   - 初始状态只突出“打开课件”操作。
   - 用户选择课件目录后自动加载课件并自动执行全局主题分析，不再要求用户额外点击“分析全局主题”。
   - 加载后展示全课件缩略图、只读分析过程和结构化主题分析产物。
   - 分析完成后由用户点击“进入单页美化工作台”。

2. **单页美化工作台页**
   - 复用当前三栏工作台的信息架构和主要能力。
   - 每页生成提示词和聊天上下文增加全局主题、课件摘要、相邻页面摘要和资源约束。
   - 提供返回全局分析概览的入口，但不因返回而丢失各页会话和生成结果。

## 二、现有实现约束

### 2.1 当前窗口结构

`MainWindow.xaml` 当前固定为三列：

```text
240 px 左侧页面列表
  +
自适应中间工作区
  +
420 px 右侧 Copilot
```

三个区域分别由以下视图承担：

- `Views/LeftSidebarPanel.xaml`
- `Views/MainContentPanel.xaml`
- `Views/CopilotPanel.xaml`

双页改造不能只在三个现有区域内部切换可见性，否则第一页仍会受到三栏尺寸和职责的限制。主窗口应提升为页面容器，由页面级视图决定自身布局。

### 2.2 当前启动行为

`App.xaml.cs` 在窗口显示前创建一个完整的 `SlideChatManager`；`MainWindowViewModel` 构造时继续创建一个空页面并立即选中。

这意味着当前应用在尚未打开课件时已经初始化：

- 页面聊天管理器。
- 渲染管线。
- 页面评估器。
- 一个空的 `CoursewareSlideItem`。
- 单页 Copilot 输入上下文。

新交互要求初始页面只显示打开入口，因此以上页面级对象应延迟到课件加载后，或进入第二页前按需创建。全局主题分析使用独立会话，不应借用启动时创建的页面会话。

### 2.3 当前打开课件行为

`MainContentPanel.xaml.cs` 负责文件夹选择事件桥接，随后调用 `MainWindowViewModel.OpenCoursewareFolderAsync`。加载完成后，ViewModel 为所有页面逐个创建 `SlideChatManager`，填充 `Slides`，并选中第一页。

新流程需要把打开动作改为一条连续编排链路：

```text
选择课件目录
  -> 加载并校验输入包
  -> 显示页面缩略图和课件摘要
  -> 自动创建独立主题分析会话
  -> 自动开始全局主题分析
  -> 流式显示分析过程
  -> 校验并展示结构化主题
  -> 用户进入单页工作台
```

### 2.4 当前聊天面板行为

`CopilotPanel.xaml` 固定包含：

- 模型选择。
- 聊天消息列表。
- 图片附件。
- MCP 设置。
- 用户输入框和发送按钮。

第一页不能直接复用完整 `CopilotPanel`。应抽取可复用的“消息流展示”部分，或创建主题分析专用只读视图，避免通过隐藏多个底部区域勉强复用一个承担不同职责的控件。

### 2.5 当前状态模型

当前只有全局 `IsBusy`，它同时控制发送消息、重新渲染、MCP 连接和打开课件按钮。双页流程至少需要区分：

- 未打开课件。
- 正在加载课件。
- 正在分析主题。
- 分析完成。
- 分析失败。
- 正在进入工作台。
- 工作台可交互。
- 正在执行当前页生成或渲染。

不能继续用多个布尔值随意组合表示页面阶段，应增加强类型页面状态和主题分析状态。

## 三、核心产品决策

### 3.1 打开课件后自动分析

主题分析不再是用户主动触发的独立步骤。

用户完成目录选择后，如果输入包加载成功，程序立即开始全局主题分析。用户不需要理解“为什么还要再点一次分析”，也不需要在加载和分析之间做额外决策。

自动分析的前置条件：

- `Courseware.json` 可读取。
- 至少存在一页有效课件页。
- 主题分析所需模型配置可用。
- 没有阻止分析的输入错误。

非阻塞警告，例如部分页面缺少截图或个别资源缺失，不阻止主题分析，但必须在第一页显示警告摘要。

### 3.2 分析完成后不自动跳转

分析完成后默认停留在第一页，由用户查看结构化主题产物并点击“进入单页美化工作台”。

原因：

- 用户需要看到自动分析的成果，而不是只看到加载动画后突然进入工作台。
- 用户可能需要重新分析或重新选择课件。
- 为后续增加主题编辑、主题确认和分析模型切换保留交互位置。

主按钮在不同阶段的行为：

| 阶段 | 主按钮 | 行为 |
| --- | --- | --- |
| 未打开 | 打开课件 | 选择目录 |
| 加载中 | 正在读取课件 | 禁用并显示进度 |
| 分析中 | 正在分析主题 | 禁用；旁边提供取消 |
| 分析完成 | 进入单页美化工作台 | 切换到第二页 |
| 分析失败 | 重试主题分析 | 使用已加载输入重新分析 |

### 3.3 第一页聊天区只读

第一页的聊天界面用于表达“系统正在如何理解课件”，不提供用户输入、图片附件、预览附件和 MCP 设置。

允许展示：

- 阶段性系统消息。
- 模型流式输出的可读分析内容。
- 结构化工具提交状态。
- 警告和错误。
- Token 或耗时摘要，可放在次要信息区域。

不建议默认展示完整内部推理内容。界面中的“分析过程”应是面向用户的工作过程和阶段结果，不应将模型隐藏推理直接作为产品内容。

### 3.4 第二页保持现有工作台心智

第二页不重新设计。当前左侧页面列表、中间预览与 SlideML、右侧 Copilot 的三栏结构本身适合单页美化，因此继续保留现状，只接入全局主题上下文和返回第一页的入口。

第二页主要新增：

- 顶部全局主题摘要入口。
- 当前课件名称与分析状态。
- 返回全局分析页。
- 页面生成提示词中的主题上下文。

## 四、双页导航与状态机

### 4.1 页面级状态

建议新增 `CoursewareWorkspacePage`：

- `CoursewareAnalysis`
- `SlideWorkspace`

第一页内部再使用 `CoursewareAnalysisStage`：

- `Welcome`
- `LoadingCourseware`
- `AnalyzingTheme`
- `AnalysisReady`
- `AnalysisFailed`
- `Canceled`

页面级状态和业务运行状态应分开。用户可以在主题已完成后多次往返两个页面，而不重新加载课件或重建所有页面状态。

### 4.2 主流程

```text
应用启动
  -> CoursewareAnalysis / Welcome
  -> 用户点击“打开课件”
  -> 打开目录选择器
  -> 用户取消选择：保持 Welcome
  -> 用户选择目录
  -> LoadingCourseware
  -> 加载失败：AnalysisFailed，显示输入错误，可重新选择
  -> 加载成功：立即进入 AnalyzingTheme
  -> 自动执行主题分析
  -> 分析成功：AnalysisReady
  -> 用户查看主题产物
  -> 用户点击“进入单页美化工作台”
  -> SlideWorkspace
```

### 4.3 返回与重新打开

从第二页点击“返回主题概览”时：

- 切回第一页的 `AnalysisReady`。
- 保留当前选中页。
- 保留所有页面聊天、SlideML、预览和日志。
- 不重新执行主题分析。

“打开其他课件”是破坏性操作。执行前如果当前课件已有生成结果，应提示用户尚未保存的结果可能丢失。确认后：

1. 取消当前主题分析或单页操作。
2. 释放当前课件的页面会话与渲染资源。
3. 清空旧输入、主题和页面状态。
4. 重新进入目录选择和自动分析流程。

### 4.4 取消

分析期间提供“取消分析”。取消后：

- 已加载的课件缩略图继续显示。
- 分析消息保留。
- 状态进入 `Canceled`。
- 提供“重新分析”和“打开其他课件”。
- 未获得有效主题前不能进入第二页。

工作台始终要求存在经过校验的全局主题，不设计无主题进入路径。

## 五、第一页信息架构

### 5.1 页面定位与应用背景

这是一个 Windows 桌面端课件美化应用。它读取已经导出的课件目录，目录中包含每一页的 Markdown 文本、原始页面截图和课件资源。应用先从整份课件的角度分析内容结构与视觉方向，形成统一的配色、字体层级、版式原则和安全区；随后用户进入现有的单页工作台，逐页生成和调整 SlideML。

第一页承担“打开课件并理解整份课件”的任务，不是单页编辑器。它需要覆盖以下连续状态：

1. 尚未打开课件时，提供唯一的“打开课件文件夹”入口。
2. 打开课件后，展示课件名称、页数和全部页面缩略图。
3. 自动开始全局主题分析，以只读聊天/时间线展示系统正在读取和理解什么。
4. 在同一页面展示逐步形成的结构化主题产物，例如主题摘要、颜色、字体层级、版式原则和安全区。
5. 分析完成后，用户查看结果并点击“进入单页美化工作台”。

设计时最重要的状态是第 3、4 项同时发生的界面：用户已经打开课件，左侧或上方能看到课件页面概览，中间能看到持续追加的分析聊天内容，右侧能看到正在形成或已经形成的主题产物。该状态最能代表第一页的核心价值，也应作为视觉设计稿的目标画面。

### 5.2 Welcome：初始空状态

初始界面只保留必要品牌信息和打开入口，不能出现空白的三栏编辑器。

建议结构：

```text
┌────────────────────────────────────────────────────────────┐
│ 产品标识 / Courseware SlideML Studio                       │
│                                                            │
│                  [ 简洁插图或图形 ]                         │
│            打开课件，生成统一风格的 SlideML                 │
│   将读取已导出的 Markdown、截图和资源，并自动分析全局主题。 │
│                                                            │
│                    [ 打开课件文件夹 ]                       │
│                                                            │
│     支持说明 / 输入格式入口 / 最近打开（后续能力）           │
└────────────────────────────────────────────────────────────┘
```

首屏主操作只能有一个高优先级按钮。模型、MCP、添加空页面和聊天输入均不应出现。

### 5.3 LoadingCourseware：课件加载状态

目录选择完成后，不再返回 Welcome。页面主体原地过渡为加载状态：

- 显示课件目录名称，不显示绝对路径。
- 显示“正在读取课件索引”。
- 显示骨架缩略图。
- 显示当前阶段，例如读取 `Courseware.json`、读取页面、检查资源。
- 如果加载时间很短，动画至少保持一个可感知但不过度拖延的最小时长，避免闪烁。

### 5.4 AnalyzingTheme：自动主题分析状态

课件加载成功后，主内容分为上下或左右两个主要区域：

1. 课件页面缩略图概览。
2. 全局主题分析过程。

推荐在宽屏下使用“上方缩略图带 + 下方双栏内容”：

```text
┌ 顶部栏：课件名称 / 24 页 / 正在分析 / 打开其他课件 ┐
├────────────────────────────────────────────────────┤
│ 横向或自动换行缩略图区，单项约 160 px 宽              │
│ [1] [2] [3] [4] [5] [6] ...                         │
├───────────────────────┬────────────────────────────┤
│ 只读分析过程           │ 主题产物占位 / 分析进度      │
│ 流式消息、阶段、警告    │ 骨架卡片、当前识别结果       │
│                       │                            │
└───────────────────────┴────────────────────────────┘
```

当窗口宽度不足时，下方双栏改为上下堆叠。

### 5.5 AnalysisReady：分析完成状态

分析完成后，缩略图区保留；下方右侧从骨架状态切换为结构化主题结果，底部或右上角出现“进入单页美化工作台”主按钮。

用户进入第二页前，应能在一个屏幕内回答以下问题：

- 程序认为这是一份什么课件？
- 推荐采用什么视觉风格？
- 核心颜色是什么？
- 标题和正文如何分级？
- 页面布局应遵循哪些原则？
- 是否完整分析了所有页面？
- 是否存在输入警告？

## 六、第一页缩略图设计

### 6.1 内容与规格

建议缩略图卡片宽度基准为 `160 px`，图片区域保持原页面宽高比。对于常见 16:9 页面，可使用约 `160 × 90 px` 的图片区域，卡片总高度约 `122–138 px`。

每项显示：

- 原始页面截图。
- 左上或左下角页码。
- 一行页面标题，过长时省略。
- 页面警告标识，例如缺少截图。

分析期间可以按当前分析进度给缩略图增加轻量状态：

- 未扫描：普通边框。
- 已完成摘要读取：主题色细边框或勾选标识。
- 有警告：琥珀色标识。
- 当前提取摘要：轻量呼吸边框，不使用强烈闪烁。

### 6.2 布局

首期按普通课件规模设计。缩略图区域采用横向滚动带或有限高度的网格区域，避免无限向下延伸。大课件的虚拟化、分页和缩略图内存优化不在本计划范围内。

### 6.3 缩略图交互

第一页缩略图的主要用途是展示全课件范围，不承担单页编辑。

建议交互：

- 单击：在轻量浮层或侧栏查看大图、标题和 Markdown 摘要。
- 双击：分析完成后进入第二页并定位该页；分析中不允许进入工作台。
- 键盘：支持方向键移动焦点和 Enter 查看。
- Tooltip：显示页码、页面 ID、标题和警告摘要。

首期也可以只实现单击预览，不实现双击跳转，以降低交互复杂度。

## 七、只读分析过程设计

### 7.1 分析过程的定位

第一页面中的聊天区不是传统问答，而是“AI 工作过程时间线”。建议标题使用：

- `全局主题分析`
- 次级状态：`正在理解课件结构与视觉方向`

不要只显示“Copilot”，否则用户无法理解该区域与第二页单页聊天的区别。

### 7.2 推荐消息阶段

系统应输出面向用户的阶段性消息，而不是只有模型的一段长文本：

1. 已读取课件基本信息。
2. 正在识别封面、目录、章节页和内容页。
3. 正在归纳学科属性与内容层级。
4. 正在分析页面尺寸和资源分布。
5. 正在形成配色、字体和版式建议。
6. 正在校验主题结构。
7. 主题分析完成。

消息可来自服务层进度事件和模型流式文本，两者视觉上应有区分：

- 系统阶段：紧凑时间线项。
- 模型摘要：普通内容卡片。
- 工具提交：可折叠技术项，默认收起。
- 警告：琥珀色提示卡。
- 错误：红色错误卡，提供重试入口。

### 7.3 不显示的内容

默认不显示：

- 用户消息气泡，因为没有用户输入。
- 图片附件编辑区。
- “附带当前预览截图”。
- MCP 服务配置。
- 单页模型生成工具。
- 完整工具参数和本地文件路径。
- 模型内部隐藏推理。

### 7.4 完成后的处理

分析完成后，消息区保留并停留在底部。标题状态变为“分析完成”，显示耗时和课件页数。

可以提供“查看分析详情”折叠项，但结构化主题卡片才是主要产物，不能要求用户从聊天文本中自行提炼主题。

## 八、主题分析产物展示

### 8.1 必须展示的内容

第一版建议将主题产物组织为以下卡片。

#### 1. 主题摘要

展示：

- 一段 2–4 行的主题概述。
- 3–6 个风格关键词，例如“现代、清晰、理性、轻量科技”。
- 适用的整体视觉语气。

这是用户最快理解分析结果的区域，应放在主题产物顶部。

#### 2. 色彩系统

展示：

- 主色。
- 强调色。
- 背景色。
- 主文字色和次文字色。

每个颜色以色块、用途名称和十六进制值显示。颜色过多时只展示核心色，完整列表放在展开区。

#### 3. 字体与字号层级

展示：

- 推荐标题字体。
- 推荐正文字体。
- 一级标题、二级标题、正文、辅助文字的字号或相对层级。
- 字重建议。

字体不可用时要说明后续将使用替代字体，不能让分析结果看起来已经保证字体存在。

#### 4. 版式原则

展示 3–6 条短原则，例如：

- 标题区保持固定高度。
- 主要信息优先放在左上视觉起点。
- 每页只保留一个主视觉焦点。
- 图文内容遵守统一对齐线。
- 章节页与内容页形成明确节奏变化。

每条原则控制在两行以内，避免形成长文档。

#### 5. 安全区

以迷你幻灯片示意图展示上、下、左、右边距；同时显示数值。相比只显示四个数字，示意图更适合设计确认。

#### 6. 页面类型建议

展示：

- 封面页建议。
- 目录或章节页建议。
- 普通内容页建议。
- 结尾页建议。

首期可以是四个紧凑标签页或折叠项，不需要为每种类型生成完整预览稿。

#### 7. 输入概览与风险

展示：

- 总页数。
- 缺少截图页数。
- 缺失资源数量。
- 其他加载警告数量。

该区域必须清晰但视觉优先级低于主题本身。

### 8.2 可后置内容

以下内容不必在首期主界面完整展示：

- 原始模型 JSON。
- 完整提示词摘要。
- 每个页面的分类结果。
- 全部资源列表。
- 主题版本历史。
- 用户编辑主题。

可以保留“查看原始分析数据”开发入口，但不作为面向普通用户的主要操作。

### 8.3 主题确认

首期不增加复杂的字段级编辑器。分析完成后提供：

- `进入单页美化工作台`：主操作。
- `重新分析`：次操作。
- `打开其他课件`：次操作。

如果重新分析会覆盖当前主题，应提示用户。尚未进入工作台时不需要保存旧主题版本；进入工作台并已产生页面结果后重新分析，则应提示新主题可能与既有页面结果不一致。

## 九、第二页设计

### 9.1 基础布局

第二页继续采用现有三栏布局：

```text
左侧：课件页面列表
中间：当前页预览、SlideML 和日志
右侧：当前页 Copilot 对话与生成控制
```

现有控件、尺寸、布局关系和主要交互尽可能保持不动。仅为了主窗口承载双页导航，可将现有三栏整体包入 `SlideWorkspaceView`，不借此重做工作台内部结构。

### 9.2 顶部全局信息条

在不挤压现有工作区的前提下，只增加必要的课件级入口：

- 返回主题概览。
- 当前课件名称。
- 主题状态，例如“已应用全局主题”。
- 打开主题详情的入口。

主题详情可以使用抽屉或浮层展示，不建议再次占据一个永久列。

### 9.3 左侧页面列表

现有页面缩略图宽度、选择方式和列表行为保持不动。第一页缩略图用于全课件概览，第二页继续使用现有列表选择和处理当前页，不新增批量状态或批次进度交互。

### 9.4 当前页 Copilot 上下文

进入第二页后，每页仍使用独立 `SlideChatManager`。首次生成提示词除当前页 Markdown 和截图外，还应包含：

- 全局主题分析结果。
- 全课件摘要。
- 前后三页摘要。
- 页面尺寸。
- 当前页资源白名单。
- 安全区与版式约束。

右侧聊天界面仍允许用户输入、附加图片、切换模型和继续追问。主题分析会话不能与当前页聊天会话混用。

### 9.5 进入工作台时的初始化策略

不建议在第一页刚加载课件时同步创建所有完整页面会话后才开始主题分析，否则会延迟自动分析的启动。

建议顺序：

1. 加载 `CoursewareInputPackage`。
2. 立即构建轻量缩略图 ViewModel。
3. 启动主题分析。
4. 在分析期间后台受控创建页面级会话，或进入第二页时按需创建。
5. 进入第二页前保证当前选中页所需页面会话已就绪；其余页面可继续受控初始化。

最终选择“全部预创建”还是“按需创建”需要结合 `SlideChatManager` 初始化耗时和模型配置加载成本测量。无论采用哪种方式，都不能阻塞第一页缩略图和分析过程的首屏显示。

## 十、推荐 View 与 ViewModel 拆分

### 10.1 View 层

建议新增：

```text
Views/
  CoursewareAnalysisView.xaml
  CoursewareAnalysisChatView.xaml
  CoursewareThemeSummaryView.xaml
  CoursewareThumbnailOverview.xaml
  SlideWorkspaceView.xaml
```

职责：

- `CoursewareAnalysisView`：第一页页面级布局和状态切换。
- `CoursewareAnalysisChatView`：只读分析消息与阶段时间线。
- `CoursewareThemeSummaryView`：结构化主题结果。
- `CoursewareThumbnailOverview`：全课件缩略图概览。
- `SlideWorkspaceView`：组合现有三栏控件形成第二页。

现有 `CopilotPanel` 继续服务第二页。聊天消息 ItemTemplate 中可复用的部分应抽为资源或专用控件，避免复制整套消息模板。

### 10.2 ViewModel 层

建议将当前职责过重的 `MainWindowViewModel` 拆为：

```text
MainWindowViewModel
  ├── CoursewareAnalysisViewModel
  └── SlideWorkspaceViewModel
```

`MainWindowViewModel` 负责：

- 当前页面。
- 页面切换。
- 当前课件生命周期。
- 重新打开课件时的清理与确认。

`CoursewareAnalysisViewModel` 负责：

- 打开课件命令。
- 输入包加载状态。
- 自动主题分析。
- 只读分析消息。
- 主题结果和警告。
- 取消、重试和进入工作台命令。

`SlideWorkspaceViewModel` 负责：

- 页面集合和选择。
- 页面级聊天。
- SlideML 编辑与渲染。
- MCP 设置。
- 主题上下文展示。

避免让两个子 ViewModel 各自持有不同的输入包或主题副本。课件输入和主题结果应通过课件会话对象或明确的只读上下文共享。

### 10.3 课件会话边界

建议新增 `CoursewareWorkspaceSession`，表示当前已打开课件在应用中的一次会话，至少包含：

- `CoursewareInputPackage`。
- `CoursewareThemeAnalysisResult?`。
- 页面 ViewModel 集合。
- 当前选中页面索引。
- 是否存在未保存结果。

它只作为两个页面共享的课件级状态边界。主题分析服务、页面生成服务和输出服务仍保持独立。

## 十一、服务与自动分析编排

### 11.1 自动分析编排服务

“打开后自动分析”不应写成 ViewModel 中一段不断增长的 `try/finally`。建议由课件打开编排服务或明确的 ViewModel 私有流程协调：

```text
OpenCoursewareAndAnalyzeAsync
  -> 选择目录由 View 进行平台桥接
  -> LoadAsync
  -> 创建 CoursewareWorkspaceSession
  -> 发布缩略图和输入摘要
  -> AnalyzeThemeAsync
  -> 本地校验主题
  -> 发布 CoursewareThemeAnalysisResult
```

文件夹选择仍可保留在 code-behind 作为平台事件桥接，但“选择成功后加载并自动分析”的业务链路属于 ViewModel/服务层。

### 11.2 主题分析会话

主题分析必须使用专用 `CopilotChatManager`：

- 不复用任何页面会话。
- 不注册 SlideML 渲染工具。
- 关闭默认文件系统工具。
- 不显示用户输入区。
- 接受取消令牌。
- 将流式消息映射到第一页只读消息集合。

主题分析会话创建失败时，不应影响已加载输入包和缩略图展示。界面进入 `AnalysisFailed`，用户可以修复配置后重试。

### 11.3 页面会话初始化

当前 `SlideChatManagerFactory` 同时包含个人本地配置路径、默认模型、渲染器、评估器和 MCP 尝试连接。实施双页模式前，应按原全课件方案将配置外置，并区分：

- 主题分析聊天管理器工厂。
- 页面 SlideChatManager 工厂。
- 评估器聊天管理器工厂。

不能通过给主题分析创建一个“没有使用渲染功能的完整 SlideChatManager”来假装职责分离。

### 11.4 取消令牌生命周期

建议至少区分：

- 当前打开/分析流程的 `CancellationTokenSource`。
- 当前单页操作的取消令牌。

重新打开课件时必须取消旧课件的所有未完成操作。页面切换不应误取消全局主题结果或其他页面已完成状态。

## 十二、动画与转场设计

### 12.1 原则

动画用于解释状态变化，不用于延长流程或制造炫技效果。

建议统一时长：

- 微交互：`120–180 ms`。
- 内容出现：`200–280 ms`。
- 页面级切换：`260–360 ms`。
- 骨架呼吸：约 `1200–1600 ms` 循环，低对比度。

需尊重系统减少动画设置；无法读取时至少提供关闭复杂动画的应用设置扩展点。

### 12.2 Welcome 到加载状态

建议：

- 打开按钮轻微缩小并淡出。
- 欢迎卡片向上移动少量距离。
- 缩略图骨架和状态区淡入。

不要让整个窗口闪白或重新创建窗口。

### 12.3 课件内容出现

课件加载完成后：

- 缩略图按小范围错峰淡入，但总时长应受控。
- 分析区从骨架过渡到第一条阶段消息。
- 主题结果区保持骨架，避免布局突然塌陷。

### 12.4 分析完成

建议：

- 状态图标从进行中平滑切换为完成。
- 主题卡片整体淡入并轻微上移。
- 色块可短暂依次出现，但不做旋转、弹跳等娱乐化动效。
- “进入单页美化工作台”按钮最后出现并获得一次轻量高亮，不循环闪烁。

### 12.5 两页切换

建议采用同一窗口内的内容切换：

- 第一页向左轻移并淡出。
- 第二页从右侧轻移并淡入。
- 保持窗口尺寸、位置和标题栏不变。

返回时反向播放。动画期间应阻止重复导航命令。

## 十三、异常与边界状态

### 13.1 输入加载失败

展示：

- 简洁错误标题。
- 可理解的错误原因。
- “重新选择课件”主操作。
- 可折叠技术详情。

不显示空缩略图网格和空主题卡片。

### 13.2 部分输入警告

如果缺少部分截图或资源：

- 继续自动分析。
- 顶部显示警告数量。
- 对应缩略图显示警告标识。
- 主题结果的“分析覆盖与风险”中汇总。

### 13.3 主题分析失败

缩略图和课件基本信息继续保留。分析区显示失败卡片，提供：

- 重试主题分析。
- 打开其他课件。
- 查看错误详情。

首期不允许在没有有效主题时进入工作台，避免第二页产生缺少全局约束的页面结果。

### 13.4 主题结构校验失败

如果模型输出可读文本但结构化主题无效：

- 分析状态保持进行中或进入“正在修复主题结构”。
- 最多按服务策略执行一次修复请求。
- 修复仍失败后进入 `AnalysisFailed`。
- 聊天区保留已产生的可读文本，但明确标记“未形成可用主题”。

### 13.5 超出当前处理规模的课件

当前方案只覆盖能够一次完整提交给模型并正常展示缩略图的普通课件。涉及长度限制、摘要、采样、分段归并、缩略图虚拟化或页面会话规模优化的课件不在本计划范围内，后续另行设计。

## 十四、无障碍与可用性

- 所有主要操作支持键盘焦点和 Enter/Space 激活。
- 页面切换后将焦点移动到新页面标题，不留在已隐藏按钮上。
- 状态不能只依赖颜色表达，需要图标和文字。
- 色块旁显示十六进制值，满足色觉差异用户读取需求。
- 分析消息流新增内容时不强制抢走用户滚动位置；用户位于底部时才自动跟随。
- 动画不使用快速闪烁。
- 错误详情可复制。
- 缩略图提供可访问名称，例如“第 3 页，二次函数，摘要已读取”。

## 十五、视觉设计师任务书

以下文字可直接复制给外部视觉设计师：

> 请为一个 Windows 桌面端课件美化应用设计 **一张高保真界面图**。只需要输出一张图，不需要初始页、加载页、失败页、第二页或多套状态稿。
>
> 这个应用会读取一个已经导出的课件目录。目录中包含每一页的文本内容、原始页面截图和图片等课件资源。应用打开课件后，会自动从整份课件的角度分析内容结构和视觉方向，生成统一的配色、字体层级、版式原则与页面安全区；之后用户才会进入另一个现有工作台逐页美化课件。你要设计的是前面的“全课件分析页”，不是后面的单页编辑器。
>
> **请表现的唯一时刻：** 用户已经点击“打开课件”并成功加载课件，全局主题分析接近完成。结构化主题方案已经生成并可用于后续美化，主按钮“进入单页美化工作台”已经可用；与此同时，只读分析聊天区的最后一条消息仍显示“正在完成最终校验”，用来保留 AI 工作过程感。顶部状态统一写“主题分析即将完成”。界面中要同时看到“分析聊天内容”和“分析产物”。这张图应让第一次接触项目的人一眼明白：左侧/上方是整份课件的页面概览，中间是 AI 正在如何理解课件的只读聊天过程，右侧是 AI 输出的可直接用于后续美化的结构化主题方案。
>
> **画布与风格：** Windows 桌面应用，建议按 `1720 × 1000 px` 设计浅色界面。整体风格专业、现代、轻量、可信，偏生产力工具；可以使用蓝紫主色、浅灰背景、白色卡片、细边框和适度圆角。不要做成网页落地页，也不要使用娱乐化 AI 视觉或大面积毛玻璃。
>
> **界面必须包含以下区域：**
>
> 1. **顶部栏**：产品标识；当前课件名称，例如“初中数学八年级上册”；页数，例如“24 页”；固定状态“主题分析即将完成”；次要操作“打开其他课件”。
> 2. **课件缩略图概览**：展示一排或两排课件页面缩略图，每张约为 16:9，带页码和简短标题，例如“封面”“目录”“勾股定理”“例题讲解”。缩略图放在有限高度的可滚动区域中，用轻量标识表现当前正在读取的页面或有警告的页面。该区域只用于表达“系统正在查看整份课件”，不承担编辑功能。
> 3. **只读分析聊天区**：标题使用“全局主题分析”，不要只写 Copilot。这里没有输入框、发送按钮、附件、模型选择或设置。内容可包含系统阶段时间线和 AI 可读摘要，例如“已读取 24 页课件”“已识别封面、章节页和内容页”“课件以几何证明与例题讲解为主，建议采用清晰、理性的教学视觉”“已形成配色、字体和版式建议”。系统阶段与 AI 内容要有清楚但克制的视觉层级；最后一条消息固定表现为“正在完成最终校验”，并带低干扰的生成中状态。
> 4. **结构化主题产物区**：这是 AI 的输出结果，不要只做成聊天气泡。请用紧凑、可扫描的卡片展示：
>    - 主题摘要：2–4 行，并带“清晰 / 理性 / 现代教学”等风格关键词；
>    - 色彩系统：主色、强调色、背景色、文字色，显示色块、用途和十六进制值；
>    - 字体与层级：标题字体、正文字体，以及一级标题、二级标题、正文、辅助文字的层级；
>    - 版式原则：3–5 条短规则；
>    - 安全区：一个迷你幻灯片示意图和四边边距；
>    - 输入概览与风险：总页数、缺少截图页数、缺失资源数量、警告数量，视觉优先级低于主题本身。
> 5. **主要操作**：在主题产物已经可用的位置放置主按钮“进入单页美化工作台”；可同时放置低优先级操作“重新分析”。主按钮应明确但不要遮挡主题内容。
>
> **布局建议：** 可采用“顶部信息栏 + 上方缩略图带 + 下方左右双栏”。下方左侧为只读分析聊天区，右侧为结构化主题产物区；主题产物区可略宽于聊天区。整张图要有真实生产力软件的信息密度，不要做成只有几个巨大卡片的概念页。
>
> **明确不要出现：** 用户聊天输入框、发送按钮、图片附件区、模型选择、MCP 设置、单页 SlideML 编辑器、代码编辑器、批量任务或批次进度、多个页面状态拼图、本地绝对路径、密钥或内部技术配置。
>
> 最终交付物只有一张完整界面图：即“打开课件后，全局主题分析接近完成，最后一条分析消息正在进行最终校验，同时结构化主题输出已经可见且可以进入单页工作台”的状态。请把主要间距、字号层级、色彩和滚动区域表达清楚，确保该视觉能够在 WPF 桌面应用中合理实现。

## 十六、实施阶段建议

### 阶段一：主窗口页面化

目标：建立双页容器，不改变现有工作台内部业务。

步骤：

1. 将 `MainWindow` 从固定三列改为页面内容宿主。
2. 新增 `CoursewareAnalysisView`。

## 十九、风险与待验证项

### 19.1 页面会话初始化成本

当前每页创建完整 `SlideChatManager`，并可能尝试连接 MCP。需要测量创建 20、50、100 页会话的耗时，决定后台受控预创建还是按需创建。

### 19.2 分析消息来源

主题分析过程既有服务阶段事件，又有模型流式输出。需要定义统一的只读消息模型，避免 UI 直接绑定具体 Agent 类型后难以插入加载、校验和错误阶段。

### 19.3 主题重分析后的既有页面结果

进入第二页并生成部分页面后，如果用户返回并重新分析主题，旧页面结果可能与新主题不一致。首期应明确警告并标记页面需要重新生成，不应静默替换主题。

### 19.4 当前配置耦合

`SlideChatManagerFactory` 当前包含个人本地配置文件路径。双页自动分析会使配置错误更早暴露，因此配置外置是实现自动分析前的必要前置工作。

### 19.5 窗口最小尺寸

当前最小宽度为 `1360 px`。第一页可以更灵活，但第二页三栏在更窄窗口下可用性有限。首期保持当前最小尺寸，后续再设计右侧 Copilot 可折叠能力。

## 二十、推荐实施决策

1. 将主窗口改为页面宿主，第一、第二页各自管理布局，不通过隐藏三栏模拟双页。
2. 应用启动时不创建空页面，不提前进入单页会话。
3. 选择课件目录并加载成功后立即自动执行全局主题分析。
4. 分析完成后停留第一页，由用户查看主题并主动进入第二页。
5. 第一页使用只读分析时间线，不显示聊天输入、附件和 MCP 设置。
6. 主题产物必须结构化展示，至少覆盖摘要、颜色、字体层级、版式、安全区和分析覆盖。
7. 第一页缩略图视觉宽度按约 `160 px` 设计，并放入有限高度的可滚动区域。
8. 第二页复用现有三栏工作台，并增加全局主题入口，不新增长期占用空间的第四栏。
9. 主题分析会话与各页会话严格隔离。
10. 使用强类型页面与分析状态，替代单一 `IsBusy`。
11. 课件加载、主题分析和当前页生成分别管理取消令牌。
12. 第一版不提供主题字段级编辑，只提供进入工作台、重新分析和打开其他课件。

按此方案实施后，用户的操作路径将从“打开课件后面对复杂单页编辑器”变为“打开课件—自动理解全局—查看统一主题—进入单页工作台”，既能承接全课件美化能力，也能保留现有单页 SlideML 生成和编辑体验。
