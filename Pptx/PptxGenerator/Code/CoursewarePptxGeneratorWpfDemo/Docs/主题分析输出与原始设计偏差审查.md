# CoursewarePptxGeneratorWpfDemo 主题分析输出与原始设计偏差审查

> 状态：当前实现审查结论。
>
> 本文对照外部《全课件美化方案设计》的产品化最终架构，复核 `CoursewarePptxGeneratorWpfDemo` 当前主题分析的输入、输出、Agent、工具、验证、UI、下游消费、持久化和测试。
>
> 本文只记录分析结论和后续修复建议，不代表相关代码已经完成改造。

---

## 一、结论

用户观察到的偏差成立，而且实际缺口明显大于“少了封面页和内容页的 SlideML 描述”。

当前实现已经完成了一条可验证页面覆盖、不会静默截断的**完整 Markdown 输入链路**，但这条链路还不等于稳定身份、严格隐私和强边界的事实协议；主题分析的最终产物也仍是一份较小的视觉建议对象 `CoursewareTheme`。它可以表达五个颜色、四级字号、字体、安全区、若干自然语言原则和固定四类页面建议，但不能表达、编译或验证真正的课件设计系统。

综合代码、测试和文档，结论如下：

1. **封面页和内容页的 SlideML 骨架确实已经从当前输出契约中消失。**
   - 外部原始设计后半保留的早期草案明确要求 `CoverLayoutSkeleton` 和 `ContentLayoutSkeleton` 同时包含自然语言说明与 `SlideMlSkeleton`。
   - 当前 `CoursewarePageTypeRecommendation` 只有 `Name` 和 `Description`，没有任何 SlideML 字段。
   - Agent 提示词、提交工具、验证器、UI 和测试也都没有接收、校验或展示 SlideML 骨架的能力。

2. **但最终修复目标不应只是恢复两个固定骨架。**
   - 外部原始设计文档顶部已经将自身定义为“产品化最终架构”，并明确否定固定输出“封面页 + 内容页”两种骨架。
   - 最终要求是根据课件实际内容动态形成页面类型，并为页面类型提供可执行的槽位、约束、组件、设计令牌和可编译 SlideML 模板目录。
   - 因此，只补回两个字符串字段只能作为临时兼容措施，不能视为完成原始设计。

3. **当前实现不是“设计系统编译器”，而是“文本输入驱动的小型主题建议生成器”。**
   - `CoursewareTheme` 缺少网格、间距、效果、组件、素材策略、动态页面类型、页面覆盖映射、证据链、假设、无障碍规则、一致性规则和模板。
   - `submit_courseware_theme` 单工具协议和 `CoursewareThemeValidator` 只能证明小型对象字段基本合法，不能证明主题可执行、模板可编译或整份课件被页面类型完整覆盖。

4. **当前主题结果没有真正进入课件美化执行链路。**
   - 应用启动后使用的是 `CoursewareWorkspaceViewModel` 原型壳。
   - 第二页仍使用 60 页演示数据、硬编码主题名、不可由当前解析器直接接受的伪 SlideML、静态预览和演示聊天。
   - 打开真实课件后只更新第一页的缩略图集合；第二页绑定的 `Slides` 仍是构造函数创建的演示集合，`EnterWorkspaceCommand` 只切换视图，没有把已加载页面迁移到工作台。
   - 项目中存在的真实 `MainWindowViewModel` 单页生成链路没有接入当前主窗口，而且它构造的生成提示词也没有包含全局主题。

5. **截图和资源已经加载，但没有作为主题分析的视觉证据使用。**
   - 当前模型只收到完整 Markdown 和逻辑资源清单。
   - 外部最终架构要求全页面缩略图总览、高清代表页、关键素材视觉样本、视觉统计和品牌/延续策略。
   - 当前实现无法观察文字前景色、形状样式、图片内容、裁剪、视觉重心、重复品牌组件和真实视觉质量，因此不能达到外部最终架构定义的多模态设计系统编译。

6. **现有输入完整性改造是正确成果，后续不应回退。**
   - 全部页面完整 Markdown、规范 JSON 信封、不截断、逻辑资源清单、全字段路径隐私、双指纹、Token 估算、预算预检和带原始 JSON 上下文的修订均已落地。
   - 后续应在此基础上增加事实、规划和设计系统能力，而不是恢复 500/1500 字摘要、随机抽样或尾页截断。

7. **主要问题是产品契约主动收缩，不是模型偶发跑偏。**
   - System Prompt 明确只要求小型主题。
   - 工具 Schema 只能提交小型主题。
   - 验证器只接受小型主题。
   - UI 和测试继续固化这一结构。
   - 即使模型完全遵守当前指令，也不可能输出外部设计要求的完整结果。

8. **完整 Markdown 的页面身份、路径和结构边界已升级为明确协议。**
   - `courseware-analysis-envelope/v2` 为每页独立携带 `SlideId`、页码、索引、尺寸和源指纹；构建器校验 Markdown 首部页面信息中的 `Id`、页码和尺寸与清单一致。
   - 本地不可变快照保存原始 Markdown，模型只接收经过绝对路径脱敏或阻断后的分析视图；诊断仅保留哈希和定位信息，不保留敏感原值。
   - Markdown 作为 JSON 字符串值传输，正文中的 `</slides>`、旧页分隔符或嵌套 JSON 只能作为页面数据，不能伪造外层协议字段。

总体判断：**当前“页面原文覆盖、稳定事实协议、路径隐私与预算诊断”已经具备较好基础；“主题输出”仍处于原型级；“设计系统编译、模板验证、真实页面消费和全课件美化闭环”尚未实现。**

## 二、设计基线的层级与冲突

### 2.1 外部原始设计的最高优先级结论

外部原始设计文档顶部的产品化最终架构要求：

```text
不可变事实快照
  -> 全量语义、布局、视觉与素材分析
  -> 全局内容结构规划
  -> 多模态设计系统编译
  -> 模板编译、渲染和视觉评审
  -> 逐页计划与受约束 SlideML 生成
  -> 页面级与课件级质量门
  -> 事务式提交
```

其主题阶段的主要输入包括：

- 无损内容账本；
- 每页结构化语义、布局和视觉事实；
- 全局统计、聚类、重复模式和质量问题；
- 全页面缩略图总览；
- 高清代表页；
- 关键图片与重复素材视觉样本；
- 全局内容结构规划；
- 用户与品牌约束；
- SlideML 的真实能力清单和模板约束。

其主要输出包括：

- 可执行的课件设计系统；
- 全局结构计划；
- 逐页页面计划；
- 动态页面类型和模板目录；
- 设计令牌、组件、素材策略；
- 无障碍、一致性和证据链；
- 可编译、可渲染、可压力测试的 SlideML 模板。

### 2.2 外部文档后半仍保留早期草案

同一份外部文档后半仍保留了一套较早的实施草案，其中包括：

- 纯文本发送全部 Markdown；
- 不发送截图；
- 固定生成封面页和内容页两份版式骨架；
- 每份骨架包含 `Description + SlideMlSkeleton`；
- 将两份骨架直接注入单页美化提示词。

这套草案与文档顶部“产品化最终架构”存在明显冲突。顶部结论已经明确指出：

- 不能只把全部 Markdown 无差别拼到一次请求中；
- 不能完全不提供视觉样本却要求判断品牌和原视觉；
- 不能固定输出封面页和内容页两种骨架。

因此，本审查采用以下判定规则：

1. 外部文档顶部的产品化最终架构为最高基线；
2. 后半两骨架草案只用于识别历史字段回退；
3. “补回两个固定骨架”不等于完成最终架构；
4. 最终应升级为动态页面类型和可编译模板目录。

### 2.3 仓库内部文档改变了基线优先级

仓库内 `Pptx/PptxGenerator/Docs/设计文档/全课件主题分析输入与设计系统产品化方案.md` 明确写明：

- 外部原始设计只作为业务背景参考；
- 结论以当前仓库已有数据、Agent 调用方式和 SlideML 能力为准；
- 多模态视觉分析、结构计划和 SlideML 模板被拆成可选或后续阶段。

这份内部文档准确识别了当前小型主题的很多问题，也给出了较合理的 `CoursewareDesignSystem` 演进模型；但如果外部原始设计才是用户要求的权威基线，那么“把外部设计降为背景参考”本身就是本次偏差的主要来源之一。

换言之，当前代码并非在不知情的情况下遗漏原设计，而是在后续方案中主动把目标收敛为“先完成完整 Markdown 输入，再逐步升级输出”。这可以解释当前状态，但不能证明当前状态已经符合外部最终设计。

## 四、当前已实现且应保留的能力

本次审查不建议推翻当前全部实现。以下能力与外部最终架构并不冲突，应作为后续升级的基础保留。

### 4.1 全页完整 Markdown

`CoursewarePptxGenerator.Core/Analysis/CoursewareAnalysisInputBuilder.cs` 按 `SlideIndex` 顺序把每一页完整 Markdown 写入 `slides[].markdown` JSON 字符串值，不添加或依赖页分隔符。

`ValidateCompleteInput` 逐页比较原始 Markdown，能够发现页面缺失、顺序错误或内容被改写。

### 4.2 不静默截断

当前成功输入中：

- `AnalyzedSlideCount == TotalSlideCount`；
- `WasTruncated == false`；
- 不再使用旧 60000 字符截断线；
- 超预算时在模型配置完整的情况下显式失败。

对应测试：

- `CoursewareAnalysisInputBuilderTests.BuildShouldKeepAllSlidesWhenInputExceedsLegacyCharacterLimit`
- `CopilotCoursewareThemeAgentTests.AnalyzeAsyncShouldFailBeforeSendingWhenCompleteInputExceedsBudget`

### 4.3 元数据路径和工具边界

当前实现已经做到：

- 不把 `RootDirectory`、Markdown 文件绝对路径、截图绝对路径和资源解析路径放进构建器生成的元数据区；
- 默认工具关闭；
- 主题分析只暴露 `submit_courseware_theme`；
- System Prompt 明确把页面 Markdown 视为不可信数据；
- 不允许模型声称已经看见截图或图片内容。

路径安全现在覆盖完整模型 Prompt，包括页面 Markdown 分析视图。Windows 路径、UNC 路径、`file://` URI 和 Unix 风格绝对路径默认在发送视图中脱敏，原始事实不修改；诊断只保存路径哈希、页面身份、分区和处理动作，不保存敏感原值。产品也可选择阻断模式，在检测到路径时拒绝模型请求。

如果产品要求模型请求中绝不出现本地路径，需要在“不改写教学原文”和“发送前隐私扫描”之间建立显式策略：建议保留本地原文，构建一份可追溯的分析视图，对命中的路径执行阻断或可逆脱敏，并将差异写入诊断；不应静默改写后仍声称输入与原文完全一致。

此外，JSON 序列化提供不可与 Markdown 正文混淆的结构边界；正文中的旧 XML 标签、页分隔符和嵌套 JSON 均按字符串转义并逐字往返。语义提示注入仍由 System Prompt 明确的不可信数据边界、工具最小化和本地输出验证共同防护。

### 4.4 输入指纹与 Token 诊断

当前 `CoursewareAnalysisInput` 已包含：

- 字符数；
- Token 估算；
- 分区字符数；
- 原始逻辑事实 `SourceFingerprint` 与发送视图 `AnalysisViewFingerprint` 两个 SHA-256 指纹；
- 截断状态；
- 页面覆盖数。

这些字段可继续用于缓存、诊断和回归比较。

### 4.5 修订轮次保留原始上下文

`CopilotCoursewareThemeAgent.BuildRepairPrompt` 使用 `courseware-theme-repair/v1` JSON 信封，把规范化错误和完整原始分析信封作为嵌套对象一起发送，已经修复了“只发送错误列表、模型无法重新理解课件”的问题。

### 4.6 独立会话与本地确定性校验

主题分析使用独立工作负载和独立请求，不共享页面聊天历史；工具提交后由本地验证器判断基本字段合法性。这个边界可以直接扩展为后续有状态设计系统草稿和多工具协议。

## 五、总体差异矩阵

状态含义：

- **已实现**：当前代码已经具备，可继续保留；
- **部分实现**：存在基础数据或入口，但没有达到外部设计要求；
- **缺失**：当前项目内没有对应实现；
- **回退**：早期设计曾明确要求，但当前契约已删除；
- **口径偏移**：仓库后续文档主动降低或改变了外部设计要求。

| 能力 | 外部最终设计要求 | 当前实现 | 状态 | 影响 |
| --- | --- | --- | --- | --- |
| 全页原始内容 | 全量、无损、不可用摘要替代 | 全部 Markdown 原样进入 Prompt | 已实现 | 正确基础，应保留 |
| 稳定页面身份 | 事实层和后续计划使用统一 SlideId | 显式外层 SlideId，并校验 Markdown Id、页码和尺寸 | 已实现 | 可为页面类型覆盖和后续计划提供稳定引用 |
| Prompt 路径隐私 | 模型输入不包含本地绝对路径 | 原始事实与分析视图分离；全 Prompt 扫描并脱敏或阻断 | 已实现 | 隐私差异可审计，模型请求不携带绝对路径 |
| Prompt 分区边界 | 数据与指令具有可验证、不可混淆的边界 | 使用规范 JSON 信封；Markdown 作为字符串值；System Prompt 声明全部来源数据不可信 | 已实现 | 结构边界不可由 Markdown 内标签或 JSON 文本伪造；语义注入仍由最小工具和本地验证控制 |
| 内容块账本 | 稳定 ContentBlockId、来源、覆盖关系 | 只有 Markdown 字符串 | 缺失 | 无法确定性验证拆分、合并、遗漏和重复 |

| 完整设计令牌 | 颜色、字体、字号、间距、圆角、描边、阴影等 | 固定五色和四级字号 | 部分实现 | 无法统一复杂页面和组件 |

## 六、关键问题详解

### 6.1 SlideML 缺失是完整链路上的系统性删除

当前 SlideML 缺失不是 UI 忘记展示一个字段，而是从整条链路中被删除：

1. `Models/CoursewareTheme.cs`
   - 没有 `LayoutSkeleton` 类型；
   - 没有 `CoverLayoutSkeleton`；
   - 没有 `ContentLayoutSkeleton`；
   - 没有 `SlideMlSkeleton`；
   - 没有通用 `PageTemplate`。

2. `Services/CopilotCoursewareThemeAgent.cs:14-30`
   - System Prompt 只要求固定五色、四级字号、字体、安全区、四类页面建议和摘要；
   - 没有要求模型输出任何 SlideML。

3. `Services/CoursewareThemeSubmissionTool.cs:42-64`
   - 工具参数只有一个 `CoursewareTheme`；
   - 当前 Schema 无法承载模板。

4. `Services/CoursewareThemeValidator.cs:17-136`
   - 不解析 XML；
   - 不校验标签和属性；
   - 不调用真实 SlideML 解析器；
   - 不做模板压力测试。

5. `Views/CoursewareThemeSummaryView.xaml`
   - 没有模板预览、XML 或编译状态区域。

6. 测试
   - 没有任何测试搜索或断言 `SlideMlSkeleton`、`SlideMlTemplate` 或模板编译；
   - 当前有效主题测试只构造自然语言页面说明。

因此，模型不可能通过当前接口提交 SlideML，即使模型在普通文本中输出 XML，也不会进入结构化结果或后续执行。

### 6.2 当前页面类型是固定枚举式结构，不是动态目录

`CoursewarePageTypeRecommendations` 将页面类型固定为：

- Cover；
- Section；
- Content；
- Ending。

这会把课件中的不同内容形态压缩为四类。例如：

- 图文页；
- 双栏对话；
- 选择题；
- 公式推导；
- 步骤流程；
- 表格；
- 古诗词；
- 时间线；
- 总结复习；
- 媒体页。

这些页面只能落入一个宽泛的 `Content.Description`，不能获得独立槽位、约束或模板。

`CoursewareWorkspaceViewModelTests` 还显式断言页面类型数量为 4，这意味着当前测试会阻止动态页面类型演进，除非同步修改测试契约。

### 6.3 当前主题不是设计系统

当前 `CoursewareTheme` 主要是 UI 可读建议，不具备设计系统所需的引用关系：

- 颜色不是可被组件和模板引用的稳定 Token；
- 字号没有语义 Token、行高和 fallback 字体栈；
- 页面类型无法引用组件；
- 组件无法引用颜色、字体、间距和效果；
- 模板无法引用页面类型、槽位和设计令牌；
- 没有 ID 唯一性和跨对象引用验证。

`GenerationPromptSummary` 试图把主题压缩成一段自然语言供下游使用，但当前项目没有任何运行时代码消费它。即使未来接入，也会重新引入自由文本歧义，不能替代强类型设计系统。

### 6.4 当前没有外部最终架构要求的视觉证据

课件输入包已经加载页面截图，第一页 UI 也会展示全部缩略图；但主题 Agent 只收到文本。

当前代码明确告诉模型：

- 截图未发送；
- 不得声称看见图片内容、Logo、主色、阴影、渐变或视觉质量。

这项限制本身是正确的诚实边界，但也说明当前结果只能基于 Markdown 推断，不能承担外部最终架构中的以下职责：

- 判断真实品牌信号；
- 识别固定页眉、Logo 和重复视觉组件；
- 判断页面视觉质量；
- 分析图片风格和裁剪；
- 比较全课件视觉节奏；
- 识别视觉离群页；
- 基于真实视觉决定 Preserve/Evolve/Redesign。

此外，现有 `Docs/CoursewareMarkdownExportFormat.md` 只保证导出背景、元素坐标尺寸、主要字体字号和素材 ID 等信息，并不包含：

- 文本前景色及字符覆盖率；
- 形状填充、描边、圆角、阴影和透明度；
- 图片视觉语义、主色和裁剪；
- Z 序、旋转和完整样式；
- 留白、对齐线、视觉重心；
- 跨页重复组件和品牌事实。

因此，不能仅依靠“完整 Markdown 已发送”推导出“完整视觉分析已完成”。

### 6.5 `StatisticsVersion` 存在，但实际统计没有落地

`CoursewareAnalysisInputBuilder` 设置了：

```text
courseware-analysis-statistics/v1
```

但当前 Prompt 中没有 `<derived-statistics>` 分区，也没有对应的 `CoursewareMarkdownStatisticsBuilder`。

当前所谓统计只有：

- 页面尺寸分布；
- 资源类型数量；
- 截图存在页数；
- 警告数量。

没有实现内部产品化文档计划的：

- 字体按字符数加权分布；
- 字号分布；
- 背景颜色分布；
- 元素数量与面积占比；
- 内容密度；
- 越界、重叠和过密候选；
- 布局分组；
- 素材引用次数、页码和位置类型。

因此，当前 `StatisticsVersion` 更像为后续统计预留的版本标识，不能被理解为统计能力已经完成。

### 6.6 没有“先规划内容，再编译设计系统”

外部最终架构要求在模板设计之前完成：

- 章节和叙事结构；
- 页面拆分和合并；
- 最终输出页面清单；
- 内容块覆盖计划；
- 动态页面类型需求。

当前主题 Agent 直接从原始页面集合生成固定四类主题建议。它不知道：

- 哪些页面将被拆成多页；
- 哪些页面将被合并；
- 最终会存在多少页面；
- 每个输出页面必须承载哪些内容块；
- 页面类型在最终结构中的数量和密度范围。

这会导致模板需求建立在错误的页面集合上，也是当前无法形成真正动态页面类型的根本原因之一。

### 6.7 本地验证只能证明字段格式，不证明可执行

当前验证器检查：

- 必填字符串；
- 集合数量；
- Hex 颜色格式；
- 字号 10-96；
- 四级字号递减；
- 安全区非负且不超过页面 60%；
- 四类页面名称和说明非空。

它没有检查：

- 颜色对比度；
- 字体可用性或 fallback；
- 页面类型是否来自真实内容；
- 每个 SlideId 是否被覆盖且只覆盖一次；
- 组件和 Token 引用；
- 素材引用；
- 证据来源和置信度；
- 本地路径是否出现在模型输出；
- 无截图时是否伪造视觉观察；
- SlideML XML、标签、属性和模板槽位；
- 模板是否能被真实解析器编译；
- 真实内容是否溢出、裁切或越界。

因此，UI 中“主题结构校验通过”当前只表示小型对象通过了基础格式检查，不能解释为设计系统或模板质量已经通过。

### 6.8 UI 进一步裁剪了已有结果

`CoursewareWorkspaceViewModel.ApplyThemePresentation` 已经把以下内容映射到集合：

- `PageTypeRecommendations`；
- `ContentRules`。

但 `Views/CoursewareThemeSummaryView.xaml` 没有绑定这两个集合。

当前主题结果页实际展示：

- 主题说明；
- 风格关键词；
- 配色；
- 字号和字体；
- 版式原则；
- 安全区；
- Warning。

当前没有展示：

- 模型生成的 `ThemeTitle`；
- 四类页面建议；
- 内容呈现规则；
- `GenerationPromptSummary`；
- 输入覆盖率和字符/Token 诊断；
- 设计证据和未知项；
- 模板数量和编译状态；
- 页面类型覆盖映射。

因此，即使当前小型主题中的部分字段已经生成，用户也不能在正式结果页查看。

### 6.9 主题和真实页面都没有进入活动单页工作台

当前应用启动绑定：

```text
CoursewareWorkspaceViewModel
```

而不是项目中已有的真实：

```text
MainWindowViewModel
```

`CoursewareWorkspaceViewModel` 的单页工作台包含明显的原型逻辑：

- `DemoSlideCount = 60`；
- 页面标题来自内置数组；
- `SelectedSlideSummary` 硬编码“数学推导与理性空间”主题；
- `SelectedSlideXml` 硬编码示例 XML，根标签为 `<Slide>`，子标签为 `<Title>` 和 `<Content>`；当前 `SlideMlParser` 要求根标签必须是 `<Page>`，且页面元素只识别 `Panel`、`Rect`、`TextElement` 和 `Image`；
- 预览图为静态 XAML；
- 聊天回复明确写着“真实生成流程尚未接入”。

真实课件加载完成时，代码只填充 `CoursewareThumbnails` 并保存 `CoursewareSession.InputPackage`；构造时创建的 `Slides = CreateSlides()` 从未被替换。`EnterWorkspaceCommand` 只执行 `CurrentPage = PrototypePage.SlideWorkspace`。因此第二页不仅没有消费主题，也没有消费刚刚加载的真实页面、真实截图或真实 SlideId。

真实 `MainWindowViewModel.CreateBeautifyPrompt` 虽然会传入当前页 Markdown 和截图，但没有读取 `CoursewareThemeAnalysisResult`，也没有注入 `GenerationPromptSummary`、页面类型、Token、模板或其他主题字段。

因此，当前全局主题分析对后续页面生成没有实际约束效果。

### 6.10 页面类型模板与真实 SlideML 之间缺少编译层

仓库已经有可复用的单页 SlideML 解析和渲染能力：

- `SlideMlParser` 要求根元素为 `<Page>`；
- 可渲染元素为 `Panel`、`Rect`、`TextElement` 和 `Image`；
- `FontSize`、坐标和尺寸通过 `double.TryParse` 读取；
- `SlideMlRenderPipeline` 会执行 XML 规范化、解析、布局、测量和渲染。

但这不等于仓库已经具备“设计系统模板编译器”。当前缺少把以下业务层表达转换为运行时 SlideML 的代码：

- `L1`-`L5` 等语义字号 Token 到数值字号；
- 颜色、间距、组件和效果 Token 到具体属性或结构化子元素；
- 槽位占位符到合法 XML 和实际内容；
- 字体栈到当前平台字体选择、可用性和 fallback 诊断；
- `ResourceId` 到渲染资源；
- 模板约束到编译和压力测试诊断。

这也是仓库内 `课件视觉规范制定指引.md` 不能被当前运行时直接执行的原因。该文档要求 `FontSize="L1"` 至 `FontSize="L5"`，而当前解析器只接受数值；示例还把 `<!-- 此处填入 ... -->` 放在 `Text` 或 `Source` 属性值内，这在 XML 语法上本身无效。与此同时，当前流式单页提示词又要求模型“优先使用字号等级 L2/L3/L4”，但代码中没有对应 Token 解析器，提示词与运行时契约也存在漂移。

因此，后续不能把规范文档中的模板字符串直接交给 `SlideMlRenderPipeline`。必须先生成真实能力清单和模板 DSL/占位符规范，再由确定性编译器解析 Token、槽位和资源引用。

现有单页运行时本身还是宽松解析，不能直接等同于严格模板编译器：

- 未知标签和未知属性会记录 Warning 后被忽略，而不是阻断；
- `FontSize="L1"` 等非法数值会进入 `SlideMlRenderResult.Errors`，但元素仍可能以默认值继续构造；
- 部分非法颜色会在 WPF 转换时退化为透明、黑色、白色或无画刷，没有统一错误诊断；
- 非流式 `render_slide` 工具只把 `Warnings` 写回文本和 `LatestWarnings`，没有把 `SlideMlRenderResult.Errors` 返回给模型，解析失败也可能得到“渲染完成”的外层文案。

因此，模板质量门必须直接检查 `SlideMlRenderResult.Errors`，建立哪些 Warning 必须阻断的策略，并额外执行颜色、标签、属性、槽位和资源的严格白名单校验。`Errors` 为空只能作为必要条件，不能单独证明模板有效。

### 6.11 ResourceId 尚未接入实际 WPF 渲染

`CoursewareFolderLoader` 已经把资源加载为 `CoursewareResourceEntry`，并保存 `ResourceId`、解析后的文件路径和存在状态；但页面生成和渲染链路没有消费这份映射：

- `MainWindowViewModel.OpenCoursewareFolderAsync` 只根据 `package.Slides` 创建页面，没有把 `package.Resources` 注册到每页渲染上下文；
- `SlideChatManagerFactory` 创建默认 `SlideMlPipelineContext`，没有注入课件资源；
- 虽然 `SlideMlPipelineContext` 暴露了 `MaterialResourceManager`，当前生产渲染代码没有读取它；
- `WpfSlideMlRenderEngine.TryLoadBitmap` 只把 `Image.Source` 当作文件路径、当前目录文件名或 `Images/Assets` 下的候选文件查找。

因此，模型或模板输出 `<Image Source="img_1" />` 时，当前 Demo 通常无法根据 `Resources.json` 找到真实文件，只会使用占位图。主题阶段即使新增素材策略和模板，也必须同时补齐 `ResourceId -> ImageSlideMlMaterial/文件` 的注册与解析链路，否则模板仍不可执行。

### 6.12 结果保存只有设计文档，没有实现闭环

项目中已经存在：

- `Docs/CoursewareOutputExportFormat.md`；
- `CoursewareExportJsonSerializerContext` 中的 `CoursewareTheme` 元数据。

但当前没有：

- 主题结果保存服务；
- `Theme.json` 写入；
- `ThemeSummary.md` 写入；
- 页面 SlideML 和预览结果保存；
- 重新打开结果的恢复逻辑；
- Schema 迁移器。

同时，输出格式文档和当前模型也不完全一致：

- 文档写 `SchemaVersion = 1.0.0`，模型常量为 `1.0`；
- 文档使用 `PrimaryColors`、`AccentColors` 等数组式字段，模型使用嵌套 `Colors` 单值结构；
- 文档未完整反映当前 `Typography`、`PageTypes` 和 `ContentPresentationRules`；
- 代码虽注册序列化元数据，但没有实际导出调用。

### 6.13 测试保护了输入改造，也固化了弱输出契约

当前测试的强项：

- 全部页面完整输入；
- 超长输入不截断；
- 页面分隔符和原文一致；
- 元数据路径脱敏；
- 页面原文中的绝对路径保持不变；
- 修订请求重新携带原始输入；
- 模型预算不足时发送前失败；
- 小型主题颜色、字号和安全区校验；
- UI 状态、消息流和缩略图。

当前测试没有覆盖：

- 动态页面类型；
- SlideId 到页面类型的覆盖和唯一性；
- 设计令牌和引用；
- 组件和素材策略；
- 设计证据与未知项；
- SlideML 模板 Schema；
- 标签属性白名单；
- 真实解析器编译；
- Warning 阻断策略和非法颜色诊断；
- `render_slide` 工具完整回传 `Errors`；
- 模板压力测试；
- 视觉证据输入；
- 结构计划和页面计划；
- 主题注入真实页面生成；
- 清单 SlideId 与 Markdown 内 Id 一致；
- 活动工作台使用真实加载页面；
- 规范模板 Token 和占位符经过编译后可被真实解析器接受；
- ResourceId 能解析到真实课件素材；
- 结果持久化。

此外，`CoursewareWorkspaceViewModelTests` 断言固定存在 4 个页面类型，测试本身已经成为动态类型改造时必须调整的契约。

## 七、代码与仓库文档的一致性问题

### 7.1 与项目历史总体方案不一致

`Docs/计划/全课件美化能力设计方案.md` 的 `CoursewareTheme` 建议字段包括：

- `CoverLayout`；
- `ContentLayout`。

当前 `CoursewareTheme` 已不包含这两个字段。即使不考虑外部设计，项目内部历史方案与代码也存在直接差异。

### 7.2 与课件视觉规范制定指引不一致

`Pptx/PptxGenerator/Docs/课件视觉规范制定指引.md` 要求：

- 先根据内容形成动态页面类型清单；
- 为每一种页面类型输出完整、内容留空的 SlideML 模板；
- 模板使用稳定槽位 ID；
- 执行者替换内容后即可直接渲染；
- 自查必须确认每种页面类型都有 SlideML 骨架。

当前主题输出既没有动态类型，也没有任何模板，因此与这份仓库级指引也不一致。

还需要注意，这份指引描述的是偏业务层的模板契约，并不等于当前运行时语法规范：

- `L1`-`L5` 需要先解析为数值，当前 `SlideMlParser` 不直接识别；
- 属性值内的 XML 注释占位符不是合法 XML；
- 字体栈的可用性与替换诊断、设计 Token 和逻辑资源 ID 都需要编译或解析步骤；
- 当前真正支持的标签和属性应以 `SlideMlParser` 为准，而不是仅凭示例文档。

因此，正确修复不是降低指引要求，而是补齐“页面类型契约/模板 DSL -> 当前 SlideML”的确定性编译层，并同步修正文档示例，使其明确区分模板源码与编译后的可渲染 XML。

### 7.3 内部产品化文档准确识别问题，但降低了最终要求

`Pptx/PptxGenerator/Docs/设计文档/全课件主题分析输入与设计系统产品化方案.md` 已经准确指出：

- 当前五色、四级字号、固定四类页面和 `GenerationPromptSummary` 不足；
- 需要设计令牌、组件、素材规则、动态页面类型、页面覆盖、证据和多工具草稿；
- 当前验证器不能证明主题可执行；
- 第二页尚未消费主题；
- 保存流程尚未闭环。

但该文档同时将：

- 多模态视觉分析；
- SlideML 模板；
- 内容块账本；
- 结构计划；
- 页面计划；

拆成可选或后续阶段。

这与外部原始设计顶部“所有分析、验证、事务、缓存和质量闭环均属于产品能力，不以可选实现省略”的定义不一致。若外部原始设计是最终验收标准，内部产品化文档需要重新明确哪些是阶段交付，哪些是最终不可省略能力。

## 九、根因判断

### 9.1 输出 Schema 在实现时被主动收缩

当前 `CoursewareTheme` 不是从外部最终设计的 `SlideMlCoursewareDesignSystem` 直接演进而来，而是围绕第一页 UI 可展示内容设计的小型对象。

一旦 Schema 被收缩，后续所有层自然同步收缩：

```text
小型 CoursewareTheme
  -> 小型 System Prompt
  -> 单个 submit_courseware_theme
  -> 浅层 Validator
  -> 摘要型 UI
  -> 固定四类测试
```

### 9.2 输入完整性成为阶段性主目标，输出升级被延后

仓库内部产品化文档优先解决了旧实现的摘要和截断问题。这项优先级本身合理，但后续代码停留在“输入已升级、输出仍旧”的中间状态。

当前 UI 的“全课件分析已完成”没有区分：

- 完整 Markdown 已读取；
- 小型主题已生成；
- 设计系统是否形成；
- 模板是否编译；
- 视觉分析是否执行；
- 下游是否实际消费。

### 9.3 原型 UI 成为正式启动入口

双页交互原型为了验证第一页体验，创建了 `CoursewareWorkspaceViewModel`、演示页面和静态工作台。当前 `App.OnStartup` 已把它作为主入口，但真实单页生成链路没有完成迁移。

这使得主题分析看似已经接入完整产品流程，实际只接入了第一页。

### 9.4 测试以当前实现为契约，而不是以外部最终设计为契约

现有测试对当前输入链路提供了良好保护，但主题输出测试以“小型主题合法”为成功标准，没有建立外部设计要求的产品验收测试。

### 9.5 业务模板、运行时 SlideML 和素材解析是三套未打通的契约

仓库目前同时存在：

1. 视觉规范中的业务模板表达：动态页面类型、槽位、`L1`-`L5`、字体栈和内容占位符；
2. `SlideMlParser` 的运行时语法：`Page`、`Panel`、`Rect`、`TextElement`、`Image` 和数值属性；
3. `CoursewareFolderLoader` 的素材目录：逻辑 `ResourceId` 到本地导出文件。

三者之间没有确定性编译器和资源解析器。结果是设计文档可以写出模板、单页运行时可以渲染具体 XML、输入加载器可以找到素材文件，但三项能力无法组合成“模板引用设计令牌和 ResourceId 后可直接编译渲染”的产品链路。

### 9.6 输入无损、身份一致和隐私安全已拆分为独立协议

当前实现已经分别定义：

- 清单 SlideId 与 Markdown 首部 Id、页码和尺寸的一致性；
- 原始本地事实与模型分析视图的分离；
- Redact/Block 路径隐私策略和无原值审计诊断；
- JSON 结构边界与全部来源数据的不可信语义边界；
- `SourceFingerprint` 与 `AnalysisViewFingerprint` 的独立语义。

这些协议由构建器和 Agent 发送前共享校验器双重验证。后续仍不应使用一个“完整输入”标签替代各项独立诊断。

## 十、修复建议

### 10.1 不建议的修复方式

后续改造不应采用以下方式：

1. **只在 `CoursewareTheme` 增加两个字符串字段后宣布完成。**
   - 这只能恢复早期草案的一部分；
   - 仍然缺少动态页面类型、槽位、约束、组件和模板验证。

2. **让模型自由输出一大段 SlideML，但不提供能力清单和编译验证。**
   - 模型可能使用当前渲染器不支持的标签、属性或 Token；
   - 结果无法作为可执行契约。

3. **继续把 `GenerationPromptSummary` 当作主要执行接口。**
   - 自由文本会丢失引用关系和硬约束；
   - 下游会重新解释主题并产生跨页漂移。

4. **在未发送图像时输出品牌、Logo、真实主色或视觉质量结论。**
   - 必须保留当前诚实的证据边界。

5. **为了支持超大课件恢复摘要、采样或尾页截断。**
   - 应显式失败或实现带覆盖校验的分批归并。

6. **把视觉规范文档中的模板示例直接作为可渲染 SlideML 保存。**
   - `L1`-`L5`、模板槽位和占位符属于业务模板层，必须经过编译；
   - 属性值中的 XML 注释不是合法 XML；
   - 编译结果必须使用当前解析器真实支持的标签和数值格式。

### 10.2 真实页面工作台接入时暂不处理的已知问题

[`计划/主题分析接入真实页面美化实施计划.md`](计划/主题分析接入真实页面美化实施计划.md) 当前只负责把真实课件页面、`CoursewareTheme` 主题建议、实际画布和原始截图接入既有单页美化链路。以下问题已经确认存在，但不属于该期改造；实现真实工作台时不得把它们重新变成前置条件，也不得只做半套修复。

1. **WPF 本地渲染尚不能按逻辑 `ResourceId` 加载课件素材。**
   - `CoursewareInputPackage.Resources` 已有素材名与文件映射，但 `WpfSlideMlRenderEngine` 仍按路径和候选目录查找图片。
   - 本期保持现有本地渲染行为，不新增 `MaterialResourceManager` 注册、严格资源模式、资源白名单验证或路径 fallback 改造。
   - 后续应作为独立任务完整打通素材注册、逻辑 ID 解析、缺失诊断和相应测试。

2. **通用 `SlideMlPromptProvider` 存在与运行时或课件主题冲突的提示。**
   - 固定“尽量使用浅色主题”可能覆盖当前课件主题。
   - `L2/L3/L4` 字号 Token 没有当前运行时解析层。
   - `image_001` 一类占位资源建议会鼓励模型创造未登记素材名。
   - 本期不修改通用提示词；后续应在通用单页生成链路中统一修正，不能只为课件工作台复制一套分叉实现。

3. **`SlideMlRenderTool` 的完整错误和 UI 回填时序不可可靠观察。**
   - 当前只保存 Preview、Warnings、RenderedXml 和 SlideXml，没有保存完整 `SlideMlRenderResult.Errors`。
   - `ApplyRenderResult` 以 fire-and-forget 派发 UI 更新，调用方无法等待所有属性完成回填。
   - 本期每页继续复用现有单页行为，不修改此机制；后续应在通用渲染链路中统一设计完整结果传播和可等待状态应用。

4. **主题重新分析后的既有页面会话没有迁移协议。**
   - 本期不增加主题指纹、页面过期标识、SessionId、AnalysisRunId 或 GenerationRunId。
   - 新分析成功后，尚未开始的页面生成可以读取当前主题结果；已创建会话如何切换主题应另行设计。

5. **流式生成重试预算仍沿用现有实现。**
   - 本期不调整 `StreamingSlideGenerator` 的默认重试次数或成本策略。
   - 若实际使用暴露取消不及时或重试成本过高，应在通用单页链路中独立修复。

这些已知问题不影响“真实页面已接入现有单页工作流”的阶段验收，但会限制本地素材渲染、严格质量判断、主题重分析和长期运行可靠性；对外说明时必须保留这些边界。

### 10.3 建议的完整修复顺序

#### 阶段 0：重新确认权威基线

先明确产品验收以外部最终设计为准，并更新仓库内部文档：

- 区分“当前阶段交付”和“最终不可省略能力”；
- 不再把外部最终设计仅标记为背景参考；
- 明确固定两骨架只是历史草案；
- 明确最终目标是动态、可编译模板目录。

#### 阶段 1：建立不可变事实快照和低成本确定性分析

在现有 `CoursewareInputPackage` 基础上增加：

- 结构化内容块和稳定 ID；
- 元素几何、Z 序、组合、旋转、裁剪和透明度；
- 文字 Run 的字体、字号、字重、颜色和字符覆盖；
- 形状填充、描边、圆角、阴影和渐变；
- 图片尺寸、宽高比、哈希、裁剪和候选角色；
- 字体、字号、颜色、背景、密度、布局和素材引用统计；
- 页面质量问题和布局聚类。

输入协议已完成以下修正：

- 在结构化页面清单中显式携带清单 `SlideId`；

## 十四、最终判断

当前项目已经解决了“模型没有读到整份课件”的重要问题，但尚未完整解决“每页以哪个稳定身份进入分析、原文中的敏感路径如何处理、模型读完整份课件后应该产出什么、模板如何编译、素材如何解析、如何证明可执行、如何真正约束后续页面生成”的问题。

因此，本次偏差不能归结为少了一个 UI 区块或模型回答不够详细。它是以下多个层面的共同偏差：

```text
设计基线优先级改变
  + 输入身份、隐私与分区契约未闭合
  + 输出领域模型收缩
  + Agent 工具协议收缩
  + 验证范围收缩
  + UI 展示裁剪
  + 业务模板与运行时 SlideML 未编译打通
  + ResourceId 与渲染素材未打通
  + 真实页面和下游未接入
  + 持久化未实现
  + 测试固化当前弱契约
```

用户指出的“封面页和内容页 SlideML 消失”是最直观的症状。按早期草案，它们确实属于明确回退；按外部文档顶部的最终架构，更准确的目标则是：

> 不再返回固定四类页面的自然语言建议，而是基于具有稳定身份、明确隐私边界的全量事实、视觉证据和结构规划，形成动态页面类型、完整设计令牌、组件和素材策略；再由确定性编译器把模板 Token、槽位、字体和 ResourceId 转换为当前运行时真正接受的 SlideML，经过编译、渲染和压力测试后，最终由真实单页工作台和保存流程消费。

在达到上述目标前，当前主题分析应被定位为“完整 Markdown 驱动的主题建议原型”，不应被视为外部原始设计所定义的产品化全课件设计系统编译能力。
