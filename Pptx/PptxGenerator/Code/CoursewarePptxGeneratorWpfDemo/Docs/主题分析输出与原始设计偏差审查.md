# CoursewarePptxGeneratorWpfDemo 主题分析输出与原始设计偏差审查

> 状态：基于当前工作区实现重新审查。
>
> 上一版审查形成时，项目主要输出小型 `CoursewareTheme`。当前仓库已完成 `CoursewareDesignSystem` v2、多工具草稿、动态页面类型、页面覆盖映射、SlideML 模板编译与压力测试、受控截图附件、真实页面工作台和主题分析快照。因此，上一版关于这些能力“完全缺失”的结论已过时。本文只保留当前仍可由代码证明的偏差。

---

## 一、当前结论

当前主链路为：

```text
CoursewareFolderLoader
  -> CoursewareAnalysisInputBuilder / Validator
  -> CoursewareStructuredFactBuilder
  -> CoursewareVisualSampleSelector
  -> CopilotCoursewareThemeAgent
  -> CoursewareDesignSystemToolSet
  -> CoursewareDesignSystemDraftBuilder
  -> CoursewareDesignSystemValidator
  -> CoursewareTemplateCompiler
  -> CoursewareTemplateStressTester
  -> CoursewareThemeAnalysisResult
  -> CoursewareThemeAnalysisSnapshotStore
  -> CoursewareSlideWorkspaceViewModel
  -> CoursewareSlidePromptBuilder
```

综合当前代码和测试：

1. **SlideML 输出缺失已完成实质性修复。**
   - 当前不是只补回固定封面页和内容页两个字符串骨架。
   - `CoursewareDesignSystem.PageTemplates` 保存动态页面类型对应的 `SlideMlTemplate`。
   - 模板使用 `{{token:id}}` 和 `{{slot:id}}`，由 `CoursewareTemplateCompiler` 转换为当前运行时接受的具体 SlideML。
   - `CoursewareTemplateStressTester` 对每个模板、每个画布执行最小、短、中、长、特殊字符和缺图样本，并调用真实 `SlideMlRenderPipeline`。

2. **固定四类页面已不再是核心执行契约。**
   - `CoursewarePageTypeContract` 具有稳定 ID、证据、密度范围、槽位、组件引用、硬约束、可调整范围、过载策略和 fallback。
   - `CoursewarePageTypeAssignments` 要求每个输入 `SlideId` 恰好映射一次。
   - 旧 `CoursewareTheme` 和固定 Cover/Section/Content/Ending 仍作为 UI 兼容投影保留，但页面生成已优先消费 `CoursewareDesignSystem`。

3. **当前已具备版本化、可交叉验证的设计系统主体。**
   - Schema 版本为 `2.0`。
   - 包含设计意图、画布、网格、间距、字体、颜色、效果、组件、素材策略、动态页面类型、页面分配、模板、无障碍、一致性、证据和假设。
   - Agent 通过带 `DraftId + Revision` 的多工具草稿逐段写入并冻结，而不是单个大对象工具。

4. **确定性统计和受控视觉输入已落地，但仍不是完整事实层。**
   - `CoursewareStructuredFactBuilder` 从完整 Markdown 提取页面几何、元素数量、文本量、占用面积、字体、字号、颜色、ResourceId、风险代码和布局簇。
   - 模型支持图片时，Agent 会附加有限代表页截图，并约束视觉观察必须绑定实际发送样本。
   - 但当前仍没有字符 Run、Z 序、组合、旋转、裁剪、透明度、完整形状样式、图像语义和内容块覆盖账本。

5. **主题结果已进入真实页面美化链路。**
   - `CoursewareSlideWorkspaceViewModel` 使用真实 `CoursewareInputPackage.Slides`，不再依赖 60 页演示集合。
   - `CoursewareSlidePromptBuilder` 为当前页注入页面类型、已验证模板、组件、素材规则、结构化事实、完整 Markdown、相邻页摘要和截图证据边界。

6. **主题分析结果已可事务式保存和恢复。**
   - `CoursewareThemeAnalysisSnapshotStore` 在临时目录写入自包含课件副本和完整分析结果，回读验证后原子发布。
   - 工作台可识别快照目录并在不重新调用模型的情况下恢复分析结果。

7. **当前仍未达到外部最终架构定义的完整全课件产品闭环。**
   - 没有独立内容块账本、全局结构计划和拆分/合并后的最终逐页计划。
   - 视觉分析只覆盖有界代表页，没有全页联系表、素材视觉样本和独立视觉评审。
   - 本地 WPF 渲染仍不能按 `ResourceId` 从课件资源清单解析图片。
   - 没有课件级生成结果暂存、全局质量门、最终原子提交、回滚和 Undo/Redo。
   - 页面生成结果尚未纳入快照。

总体判断：

> 当前主题分析已经具备“可执行设计系统编译”的核心形态。主要差距已从输出 Schema 缺失，转移到无损事实与结构规划、全量视觉证据、ResourceId 实际渲染、严格质量门、页面结果持久化和事务式全课件提交。

## 二、设计基线和当前阶段边界

### 2.1 外部最终架构

外部最终架构要求：

```text
不可变事实快照
  -> 全量语义、布局、视觉与素材分析
  -> 全局内容结构规划
  -> 多模态设计系统编译
  -> 模板编译、渲染和视觉评审
  -> 逐页计划与受约束 SlideML 生成
  -> 页面级与课件级质量门
  -> 事务式提交
```

该目标仍是判断最终完成度的最高基线。

### 2.2 固定两骨架只属于历史草案

早期草案要求 `CoverLayoutSkeleton` 和 `ContentLayoutSkeleton` 各提供一个 `SlideMlSkeleton`。当前实现没有恢复这两个固定字段，而是实现了更高层级的动态 `PageTypes + PageTemplates`。

因此：

- “没有两个固定骨架字段”不再构成当前缺陷；
- 应检查每个动态页面类型是否具有有效模板、是否覆盖真实内容形态、是否通过编译和压力测试；
- 只有在兼容旧消费者时，才需要额外提供固定两骨架适配层。

### 2.3 仓库内部产品化方案顶部状态已过时

`Pptx/PptxGenerator/Docs/设计文档/全课件主题分析输入与设计系统产品化方案.md` 顶部仍写着“当前仅完成完整 Markdown 文本分析和小型主题建议；可执行设计系统、模板编译、视觉分析、真实页面生成和持久化均不得标记为完成”。

该状态与当前代码不一致：

- 可执行设计系统已实现；
- 模板编译与压力测试已实现；
- 受控截图视觉分析入口已实现；
- 真实页面生成工作台已实现；
- 主题分析快照持久化已实现。

该文档应另行同步为“已实现能力、阶段限制和最终架构缺口”三层口径。

## 三、输入与确定性事实

### 3.1 已实现能力

当前输入协议已经具备：

- `courseware-analysis-envelope/v2` 规范 JSON；
- 每页显式 `SlideId`、页码、索引、尺寸、源指纹和完整 Markdown；
- 清单身份与 Markdown 首部 Id、页码、尺寸一致性校验；
- 原始事实快照与模型分析视图分离；
- Windows、UNC、`file://` 和 Unix 绝对路径的脱敏或阻断；
- `SourceFingerprint` 与 `AnalysisViewFingerprint` 双指纹；
- 页面覆盖、分区字符数、Token 估算和预算预检；
- 超预算显式失败，不静默截断或丢弃页面。

`CoursewareStructuredFactBuilder` 当前提取：

- 每页稳定身份和尺寸；
- 元素矩形及文本/图片数量；
- 文本字符数；
- 内容边界与占用面积比例；
- 字体、字号、颜色分布；
- ResourceId 引用；
- 越界、重叠候选和高密度候选；
- 布局签名、布局簇和代表页；
- 全局字体、字号、颜色和资源引用分布。

上一版关于“统计没有落地”的结论已过时。

### 3.2 仍缺失的事实

当前事实层依赖 Markdown 和正则解析，不等于无损内容账本或完整视觉事实。仍缺少：

- 稳定 `ContentBlockId` 及来源、拆分、合并和覆盖关系；
- 文本 Run 级字体、字号、字重、前景色和字符覆盖率；
- Z 序、组合关系、旋转、裁剪、透明度；
- 形状填充、描边、圆角、阴影、渐变和效果参数；
- 图片像素尺寸、宽高比、哈希、裁剪、候选角色和视觉语义；
- 确定性品牌组件识别和跨页重复对象事实；
- 原始页面元素到最终输出元素的可追溯映射。

因此，当前 `StructuredFacts` 应解释为“Markdown 可提取的低成本确定性事实”，不能解释为无损事实账本。

2. `set_canvas_and_grid`；

草稿具有稳定 `DraftId`、单调递增 `Revision`、旧 Revision 冲突拒绝、分区级 upsert、局部诊断和冻结后禁止修改。

当前风险：

- 推荐顺序通过 `AllowedNextTools` 返回，但草稿层没有严格状态机阻止模型跳过中间分区；
- 自动修订仍只有一轮，复杂课件可能无法在两次请求内完成；
- 测试主要覆盖失败和预算边界，缺少模拟完整多工具成功链路的 Agent 端到端测试；
- 动态类型是否真正由内容驱动仍是语义质量问题，确定性验证只能证明结构和引用合法。

### 4.3 确定性验证

`CoursewareDesignSystemValidator` 已验证：

- Schema、必填文本和稳定 ID；
- 唯一主画布、画布尺寸和网格；
- Token 唯一性、格式和值域；
- 组件、页面类型、fallback、模板和资源引用；
- 每个输入 SlideId 恰好映射一次；
- 页面类型证据和槽位；
- 视觉证据必须来自实际图片分析；
- 无障碍基础阈值；
- 输出不得包含本地绝对路径。

仍建议补充：

- 每个页面类型至少一个已通过模板；
- 页面类型与模板槽位、密度和约束一致性；
- 颜色对比度实际计算，而不是只验证阈值为正数；
- 页面生成后硬约束执行；
- 关键渲染 Warning 阻断。

## 五、动态页面类型与结构规划

### 5.1 已实现能力

`CoursewarePageTypeContract` 已替代固定四类页面，包含稳定 ID、用途、页面证据、密度范围、槽位、组件、硬约束、可调整范围、过载策略和 fallback。

`CoursewarePageTypeAssignments` 当前证明：

```text
Input SlideId -> Primary PageTypeId
```

并由验证器检查全部输入页唯一覆盖。

### 5.2 当前边界

当前覆盖映射不包含：

- 页面拆分和合并；
- 最终输出页 ID；
- 内容块到输出页的覆盖；
- 页面顺序重排；
- 每页具体素材选择；
- 每页槽位填充计划；
- 页间叙事和章节结构计划。

因此当前 `PageTypeAssignments` 是设计系统覆盖映射，不是外部最终架构中的 `StructurePlan` 或 `PagePlan`。

## 六、SlideML 模板编译与压力测试

### 6.1 已实现能力

`CoursewareTemplateCompiler` 已实现：

- 将 `{{token:id}}` 解析为具体设计值；
- 将字体 Token 解析为字号和字体名；
- 将 `{{slot:id}}` 安全转义为 XML 内容；
- 检查槽位和值；
- 检查模板中的 ResourceId 白名单；
- 输出可交给当前 SlideML 运行时的具体 XML。

`CoursewareTemplateStressTester` 已实现：

- 遍历全部模板和画布；
- 生成最小、短、中、长和特殊字符样本；
- 对图片槽位增加缺图样本；
- 调用真实 `SlideMlRenderPipeline`；
- 收集渲染错误和警告；
- 按模板汇总通过数量和完整样本结果。

这已经解决上一版指出的 `L1`-`L5`、槽位占位符与运行时语法之间缺少确定性编译层的问题。

### 6.2 当前风险

1. 模板验证不是独立视觉评审，只能证明编译和渲染管线没有 Error。
2. Warning 不会自动导致样本失败，缺失图片等问题可能仍被标记为通过。
3. 默认 WPF 管线没有注册课件资源映射，图片样本可能使用占位图。
4. 分析服务测试使用 `FakeSlideMlRenderPipeline`，缺少生产 WPF 模板压力测试回归。
5. 设计系统验证只要求模板非空，没有直接证明每个页面类型都具有合适的已通过模板。

## 七、视觉分析

### 7.1 已实现能力

当前视觉链路已经不是“截图只在 UI 展示、模型完全不接收”：

- `CoursewareVisualSampleSelector` 根据结构化事实选择有限代表页；
- Agent 检查模型是否支持图片附件；
- 只附加实际存在且被选择的截图；
- Prompt 提供无路径的 `visualSamples` 清单；
- 视觉观察必须绑定本轮实际发送的 SlideId、页码和样本角色；
- 未执行图像分析时，验证器拒绝 `VisualObservation`；
- UI 分开展示视觉分析能力状态。

### 7.2 当前偏差

仍缺少：

- 全页面缩略图联系表及分页总览；
- 对所有布局簇、页面类型、密度极值和视觉离群页的可证明覆盖；
- 高清代表页和低清联系表的分层策略；
- 关键图片、重复素材、Logo 候选和品牌资产联系表；
- 图片裁剪、主色、主体和风格的结构化视觉事实；
- 独立视觉评审角色；
- 模板渲染联系表和全课件视觉节奏评审；
- Preserve/Evolve/Redesign 的强类型策略和证据门槛。

模型支持图片且附加了样本，但没有提交视觉观察时，当前能力状态可能仍显示“未请求”，不能准确表达“已请求但无有效观察”。

## 八、真实页面工作台和下游消费

### 8.1 已实现能力

当前应用已经使用真实页面工作台：

- 加载真实课件页面和缩略图；
- 分析成功后创建 `CoursewareSlideWorkspaceViewModel`；
- `Slides` 直接来自 `session.InputPackage.Slides`；
- 每页按需创建独立 `SlideChatManager`；
- 页面 Prompt 使用完整当前页 Markdown 和有界相邻页摘要；
- 当前页可附加原始截图；
- 注入画布、安全区、颜色、字体、版式和内容规则；
- 注入当前 SlideId 对应的动态页面类型；
- 注入已验证模板、相关组件和素材规则；
- 约束图片只能使用当前页登记的 ResourceId；
- 支持生成、追问、重渲染、取消和页面状态汇总。

上一版关于“第二页仍使用 60 页演示数据、静态预览和演示聊天”“主题未进入真实页面生成”的结论已过时。

### 8.2 当前偏差

- 页面生成仍由模型根据设计上下文重新生成 SlideML，不是确定性地先实例化模板再只填槽位。
- 页面类型硬约束、可调整范围和过载策略主要作为 Prompt 数据传递，尚无生成后规则执行器。
- 没有逐页 `PagePlan` 固定内容块、素材、模板和槽位绑定。
- 没有课件级批量调度、跨页一致性检查和返修闭环。
- 重新分析只更新未来初始页面 Prompt；已有会话没有主题指纹、过期标识或迁移协议。
- 页面生成结果、聊天记录、SlideML、预览和诊断尚未随快照保存。

## 九、ResourceId 与本地渲染

### 9.1 已建立的逻辑契约

ResourceId 已用于：

- 输入资源清单；
- 结构化事实中的页面资源引用；
- 设计系统素材策略；
- 模板编译白名单；
- 页面 Prompt 允许资源列表；
- 快照资源复制和 `Resources.json`。

### 9.2 尚未打通的执行链路

本地 WPF 渲染仍未消费课件资源映射：

- `SlideChatManagerFactory` 创建默认 `SlideMlPipelineContext`，没有注册 `CoursewareInputPackage.Resources`；
- `WpfSlideMlRenderEngine.TryLoadBitmap` 仍把 `Image.Source` 当作路径、当前目录文件名或 `Images/Assets` 候选查找；
- `Image.Source="img_1"` 无法稳定解析到课件导出的真实文件；
- 找不到图片时只产生 Warning 并显示占位图。

因此，当前模板“可编译、可渲染”对纯文本模板成立；对包含课件素材的模板，只能证明 ResourceId 合法和运行时可退化渲染，不能证明真实素材已经呈现。

建议优先闭合：

```text
CoursewareInputPackage.Resources
  -> page-scoped ResourceResolver
  -> SlideMlPipelineContext
  -> WpfSlideMlRenderEngine
  -> logical ResourceId bitmap
```

## 十、UI 与结果解释

### 10.1 已实现能力

`CoursewareThemeSummaryView` 已展示独立能力状态：

## 十八、最终判断

上一版审查所指出的以下核心问题已经完成修复：

```text
小型主题输出
  -> CoursewareDesignSystem v2
固定四类页面
  -> 动态页面类型与 SlideId 覆盖
无 SlideML 模板

  -> 自包含主题分析快照保存与恢复
```

当前项目已经跨过“主题建议原型”阶段，进入“可执行设计系统和真实单页消费”阶段。

最准确的定位是：

> 基于完整 Markdown、确定性派生事实和可选代表页截图，编译动态页面类型、设计令牌、组件、素材策略和 SlideML 模板；通过本地验证与渲染压力测试后，将设计系统用于真实单页美化，并支持分析快照恢复。尚未完成无损内容账本、全局结构规划、全量视觉与素材评审、ResourceId 实际渲染、逐页计划、课件级质量门、页面成果持久化和最终事务提交。

后续不应再把主要精力放在恢复固定封面/内容两份骨架，而应优先闭合：

```text
内容块账本与结构计划
  + 全页/素材视觉证据
  + ResourceId 真实渲染
  + 页面计划与生成后约束验证
  + 页面成果持久化
  + 课件级质量门和事务提交
```
