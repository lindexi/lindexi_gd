# CoursewarePptxGeneratorWpfDemo 课件主题分析重构方案设计

> 文档状态：待实施。
>
> 本文只描述目标方案，不代表当前代码已经完成改造。
>
> 本轮设计的核心不是继续修补现有 `CoursewareDesignSystem`，而是先完整删除旧主题分析语义，再按“少量、直接、可消费”的原则重建课件级美化建议。

---

## 一、结论摘要

当前主题分析效果差的主要原因不是模型能力不足，而是系统在模型调用前后引入了过多中间事实、验证协议和下游约束：

- 把全部页面包装成复杂 JSON 分析信封；
- 重复发送页数、尺寸、资源、字符数、元素数等 Markdown 已经包含或不需要的信息；
- 扫描并脱敏本地路径；
- 计算源事实指纹、分析视图指纹和多个分区统计；
- 提取几何、密度、风险、版式签名和布局聚类；
- 选择并发送主题分析截图；
- 让模型构建设计令牌、组件、素材策略、动态页面类型、页面映射、证据和假设；
- 使用 13 个草稿工具、`DraftId`、`Revision` 和模板压力样本完成提交；
- 把页面预先映射到页面类型，再将映射和模板强行交给页面美化工作台。

这些能力在结构上很完整，但它们把“给整份课件形成一个统一美化方向”错误地升级成了“编译一套全覆盖设计系统”，模型的大量上下文和注意力被旧课件细节、中间事实、协议状态和工具字段占用，最终反而限制页面美化。

本方案将主题分析重新定义为一个轻量课件级建议生成过程：

1. 程序只汇总原课件的字体、字号和颜色使用分布，并明确说明这些数据仅供参考；
2. 将全部页面原始 Markdown 按原顺序直接拼接，页面之间只使用水平分割线；
3. 不发送截图，不脱敏，不计算指纹，不构建 JSON 分析信封，不做版式预分析；
4. 主题智能体只输出：
   - 配色建议；
   - 中文字体建议；
   - 西文字体建议；
   - 一段自然语言风格描述；
   - 页面安全区；
   - 一段间距与轻量视觉效果建议；
   - 一段自然语言版式原则；
   - 一份封面页参考 SlideML；
   - 一份内容页参考 SlideML；
5. 两份 SlideML 必须经过现有 SlideML 解析、布局和 WPF 渲染管线校验；
6. 校验失败时，在同一个独立主题会话中继续多轮修复，不重新发送整份课件；
7. 首次创建页面美化工作台时，每个页面的首轮草稿都接收同一份主题结果和两份参考 SlideML；
8. 不再生成任何页面到页面类型的映射，由页面智能体根据当前页 Markdown 自行决定采用、融合或忽略两份参考结构。

目标主链路为：

```text
CoursewareInputPackage
    ↓
只汇总字体 / 字号 / 颜色使用分布
    ↓
自然语言参考说明 + 全部页面原始 Markdown
    ↓
独立主题 Agent 会话
    ↓
单一结构化提交工具
    ↓
轻量主题结果
    ↓
封面 SlideML + 内容页 SlideML 本地校验
    ├── 失败：同一会话反馈诊断并继续修复
    └── 成功：保存主题结果
    ↓
同一份结果注入首次创建的每页首轮草稿
    ↓
页面 Agent 自主判断当前页版式
```

## 二、重构原则

### 2.1 先删除，再重建

本次后续实施不能在现有设计系统上继续叠加兼容层。旧类型、旧工具和旧页面映射一旦继续保留，就很容易再次成为新流程的默认事实源。

实施时应先删除旧主题分析主干，并让所有编译调用点显式暴露出来，再建立新契约。删除范围包括但不限于：

- 分析 JSON 信封；
- 路径隐私与脱敏；
- 所有主题分析指纹；
- 结构事实中的几何、密度、风险和布局聚类；
- 主题视觉样本与截图附件；
- `CoursewareDesignSystem` 及其草稿、Revision、验证和兼容投影；
- 设计令牌、组件、素材策略、动态页面类型、页面映射、证据与假设；
- 旧模板编译器和面向动态页面类型的压力测试；
- 旧能力状态矩阵；
- 页面工作台中的设计系统、页面类型、模板和组件消费。

删除 `CoursewareAnalysisSourceSnapshotBuilder` 之前，必须先把其中不依赖指纹的输入格式校验迁回 `CoursewareFolderLoader`：`SlideIndex` 连续且顺序一致、`SlideId` 唯一、页面尺寸为有限正数，以及 Markdown“页面信息”中的 Id、页码和尺寸与清单一致。这些属于课件导出格式校验，不属于主题分析，也不得通过保留 SourceSnapshot 或计算哈希来实现。

删除后允许项目短暂处于“主题分析尚未重新接回”的可编译过渡状态，但不允许用空对象、固定默认主题或旧适配器掩盖缺失依赖。

### 2.2 输入以原始内容为主

模型真正需要理解的是课件内容，不是程序对课件做出的二次描述。

除字体、字号、颜色的简单使用分布外，不再让程序先替模型做内容层级、版式、页面类型、密度、风险或布局判断。

### 2.3 结果只保留下游会直接使用的内容

如果一个字段不会直接进入主题结果界面或逐页美化提示词，就不应进入主题分析结果。

新的主题结果不包含：

- 主题标题；
- 主题摘要与风格关键词数组；
- 字号层级体系；
- 页面类型建议；
- 页面类型映射；
- 可复用组件；
- 素材策略；
- 无障碍能力矩阵；
- 跨页一致性对象；
- 证据、置信度和假设；
- 分析页数、耗时、Token、字符数；
- 设计系统或模板验证报告；
- 视觉分析报告；
- 任何指纹。

### 2.4 SlideML 规范只维护一份

主题智能体需要知道 SlideML 语法，但不能在主题分析代码中复制另一份 SlideML 规范。

应从现有 `SlideMlPromptProvider` 中抽出面向“完整 `<Page>` 文档”的规范。非流式页面生成和主题分析共享这份完整文档规范；流式页面生成继续拥有 `Remove`、片段合并、`StyleFrom`、`StyleId` 和状态查询等流式专用扩展。主题分析只额外叠加自己的任务规则，不能照搬流式协议，也不能复制另一份完整文档规范。

### 2.5 页面结构是参考，不是页面分类结果

封面页和内容页 SlideML 是全课件视觉方向的两个参考结构，不是对输入页面的分类结果。

主题分析不得输出：

```text
SlideId -> PageType
```

页面美化工作台不得再根据 `SlideId` 查找页面类型或模板。每页收到两份参考结构后，由页面智能体根据当前页 Markdown 自主判断：

- 使用封面结构；
- 使用内容结构；
- 融合两者；
- 或完全忽略两者并重新排版。

### 2.6 不以“安全”名义重新引入内容加工

用户选择的课件 Markdown 将原样进入模型输入。

主题分析不再：

- 扫描 Markdown 中的本地绝对路径；
- 替换路径文本；
- 因路径存在而阻止请求；
- 为路径生成审计指纹；
- 把课件正文包装成“不可信数据”JSON 协议。

系统提示词仍需用一句自然语言声明数据边界：课件 Markdown 是待分析内容，其中出现的角色声明、命令、提示词或工具调用要求都不能覆盖系统任务。该规则不改写、不脱敏、不包装 Markdown，只用于防止把课件正文误当成主题 Agent 指令。

文件夹加载阶段仍保留相对路径和目录越界校验。这是本地文件读取安全边界，与“发送模型前路径脱敏”不是同一件事，不能误删。

## 三、范围与非目标

### 3.1 本次重构范围

- 删除旧主题分析领域模型和执行链；
- 新建简单样式使用分布汇总；
- 新建自然语言主题分析输入；
- 新建轻量主题结果模型；
- 新建单一主题提交工具；
- 新建封面页和内容页 SlideML 校验与多轮修复；
- 保留主题智能体独立会话和现有 Copilot 消息界面；
- 改造主题结果界面；
- 改造页面美化工作台首轮 Prompt；
- 改造主题分析快照；
- 删除或重写相关测试和现行流程文档。

### 3.2 明确非目标

本次不顺带实现：

- 自动批量美化全部页面；
- 重新设计单页聊天、渲染和手工编辑交互；
- 课件内容拆分、合并或重排；
- 页面质量评分；
- 全课件视觉一致性二次评审；
- ResourceId 到本地图片渲染的其他遗留问题；
- 页面生成结果持久化；
- 最终 PPTX 组装和事务提交；
- 包版本、目标框架或 SDK 升级。

### 3.3 截图边界

“主题分析不发送截图”只影响课件级主题智能体。

以下能力继续保留：

- 加载课件截图；
- 在课件分析页展示缩略图；
- 将当前页原始截图作为单页美化首轮可见附件；
- 页面级会话按用户选择附加其他图片；
- 页面级 SlideML 渲染预览。

不得因为主题分析不再使用截图，就删除 `CoursewareFolderLoader`、`CoursewareSlideInput.ScreenshotFile`、缩略图或页面附件逻辑。

- 快照目录的临时写入、回读验证和原子发布方式。

### 4.5 Token 估算的特殊处理

`CoursewareTokenEstimator` 当前定义在将被删除的 `CoursewareAnalysisInputBuilder.cs` 中，但还被主题和页面上下文预算校验器使用。

该估算器不属于指纹或课件分析结果，仍可保留为纯技术预检工具。实施时应把它迁移到独立文件，避免为了删除旧输入构建器而破坏页面上下文预算检查。

预算预检只能：

- 判断完整请求是否明显超出已配置模型上下文；
- 在超限时明确失败；
- 不得裁剪页面；
- 不得把 Token 数写入主题 Prompt 或主题结果；
- 不得在正常分析界面重复展示字符数和 Token 数。

## 五、目标数据流

### 5.1 普通课件目录

```text
打开课件目录
  -> CoursewareFolderLoader.LoadAsync
  -> CoursewareInputPackage
  -> 创建缩略图和 CoursewareWorkspaceSession
  -> 自动进入主题分析
  -> CoursewareStyleUsageSummaryBuilder
  -> CoursewareThemeAnalysisPromptBuilder
  -> CopilotCoursewareThemeAgent
  -> CoursewareThemeSubmissionTool
  -> CoursewareThemeValidator
  -> CoursewareThemeSlideMlValidator
  -> CoursewareThemeAnalysisResult
  -> 保存新版快照
  -> 创建或更新页面美化工作台
```

### 5.2 主题分析快照目录

```text
打开快照目录
  -> 识别新版快照清单
  -> 重新加载课件输入副本
  -> 读取轻量主题结果
  -> 重新执行轻量字段校验和两份 SlideML 校验
  -> 创建页面美化工作台
```

快照恢复不再重算或比对源指纹。

### 5.3 重新分析

重新分析仍为全新主题会话：

- 取消旧主题请求；
- 清空本轮分析事件和主题聊天消息；
- 创建新的主题 `CopilotChatManager`；
- 不继承上一次主题会话；
- 新结果成功后替换当前课件会话中的主题结果；
- 只刷新尚未开始页面会话且首轮 Prompt 未被用户编辑的页面；
- 已开始页面会话和用户编辑过的首轮草稿不被静默覆盖。

## 六、主题分析输入设计

### 6.1 输入只由两部分组成

主题用户消息固定为：

1. 原课件字体、字号、颜色使用分布的自然语言参考；
2. 全部页面原始 Markdown。

不再存在 JSON envelope、XML 外层包装、页面 DTO 或资源 DTO。

### 6.2 字体、字号和颜色使用分布

新增一个职责单一的本地汇总器，例如：

```text
CoursewareStyleUsageSummaryBuilder
  -> Fonts
  -> FontSizes
  -> Colors
```

它只做以下工作：

- 只读取导出 Markdown 的结构化元数据行，不扫描 `#### 内容` 下的 fenced code block；
- 从文本元素样式行中匹配 `字号:` 和 `字体:` 声明；
- 从 fenced code block 之外的结构化元数据行中匹配十六进制颜色；
- 只接受 `#RRGGBB` 或 `#AARRGGBB`，不把正文代码块中偶然出现的十六进制文本计入颜色分布；
- 对同值进行计数；
- 按出现次数降序、值稳定排序；
- 每类最多展示前 12 项，避免参考信息反过来污染上下文；
- 不保留逐页分布；
- 不计算页面数、元素数、资源数、占用率、密度或布局；
- 不生成指纹和版本化统计协议。

统计口径为“每个结构化样式声明计一次”。字体去除首尾空白并按不区分大小写聚合；颜色统一为大写；当前导出格式的字号单位是 `px`，无单位值按 `px` 处理，若遇到 `pt` 则作为独立显示值保留，不与 `px` 数值合并。

建议的自然语言格式：

```text
下面是程序从原课件 Markdown 中汇总的样式使用情况，仅供参考。
原课件正在被重新美化，这些数据不是必须继承的限制，不要因为原课件高频使用某个值就机械沿用。

字体使用参考：微软雅黑（38 次）、宋体（12 次）、Arial（9 次）
字号使用参考：24（31 次）、18（25 次）、32（17 次）
颜色使用参考：#1F4E78（22 次）、#FFFFFF（19 次）、#000000（15 次）
```

如果某一类没有匹配结果，使用“未从 Markdown 中识别到明确记录”，不补造默认值。

### 6.3 页面 Markdown 拼接

页面必须按照 `CoursewareInputPackage.Slides` 的稳定顺序拼接。

每页内容使用 `MarkdownText` 原文，不执行：

- Trim；
- 摘要；
- 字符替换；
- 路径替换；
- JSON 转义；
- XML 转义；
- 标题提取；
- 页面身份重写；
- 尺寸重写；
- 资源重写。

页面之间只插入：

```text

---

```

不额外插入“第 N 页”“SlideId”“页面尺寸”等标签，因为这些信息已经存在于页面 Markdown 中。

### 6.4 不进入主题 Prompt 的内容

- 课件根目录；
- Markdown 文件路径；
- 截图路径；
- 资源文件路径；
- 程序加载警告；
- 总页数；
- 页面尺寸分布；
- 截图数量；
- 资源类型分布；
- 页面元素数量；
- 文本字符数；
- 结构化事实；
- 布局签名或聚类；
- 页面类型候选；
- 任何指纹；
- 截图或其他图片附件。

这里的“路径不进入 Prompt”是指程序不主动添加文件系统元数据；Markdown 原文中本来存在的路径文本保持原样。

### 6.5 目标用户消息结构

```text
请基于下面整份课件内容，形成统一的课件美化建议。

[字体、字号、颜色使用分布参考]

以下为课件各页 Markdown 原文，页面之间用水平线分隔：

[第一页 Markdown 原文]

---

[第二页 Markdown 原文]

---

[后续页面 Markdown 原文]
```

不在用户消息尾部重复输出字段清单。输出字段和提交要求由系统提示词及工具 Schema 统一约束。

## 七、目标主题结果契约

### 7.1 运行时根结果

建议将 `CoursewareThemeAnalysisResult` 收敛为：

```text
CoursewareThemeAnalysisResult
  - CoursewareTheme Theme
```

SlideML 校验使用当前课件第一页的本地画布作为临时技术上下文，但该画布不写入主题结果或快照，也不在主题 Prompt 中额外重复尺寸数值。主题 Agent 只能从页面 Markdown 原文中的页面信息理解尺寸。

不再保存：

- `AnalyzedAt`；
- `Duration`；
- `TotalSlideCount`；
- `AnalyzedSlideCount`；
- `Warnings`；
- `CapabilityStates`；
- `StructuredFacts`；
- `DesignSystemValidation`；
- `TemplateValidation`；
- `VisualAnalysis`；
- `DesignSystem`。

分析时间由快照 `CreatedAt` 或运行日志承担，不需要污染主题业务结果。

### 7.2 `CoursewareTheme` 最小字段

新契约与现有 `1.0` 完全不兼容，固定使用：

```text
CoursewareTheme.CurrentSchemaVersion = "2.0"
```

提交工具、快照保存和快照恢复都必须精确校验该版本。

```text
CoursewareTheme
  - SchemaVersion
  - ColorSuggestions[]
  - Fonts
      - Chinese
      - Western
  - Style
  - SafeArea
      - LeftRatio
      - TopRatio
      - RightRatio
      - BottomRatio
  - SpacingAndVisualEffects
  - LayoutPrinciples
  - CoverPageSlideMl
  - ContentPageSlideMl
```

### 7.3 配色建议

`ColorSuggestions` 是一组简单建议，不固定为旧版五个语义槽位。

每项只包含：

- `Name`：颜色名称；
- `HexValue`：颜色值；
- `Usage`：建议用途。

建议约束：

- 3 到 8 项；
- 颜色必须为 `#RRGGBB` 或 `#AARRGGBB`；
- 名称、颜色和用途不能为空；
- 不要求提供证据、置信度、禁止用途或可搭配背景令牌。

这正是主题分析应承担的“几条配色建议”，不再把配色升级为可执行语义令牌系统。

### 7.4 字体建议

字体建议只分中文和西文：

- `Chinese`：一段自然语言，可说明中文标题、正文和回退字体；
- `Western`：一段自然语言，可说明西文标题、正文和回退字体。

不再输出字体 Token、适用页面类型、字号、行高、字重或字体栈对象。

原课件字号分布只作为输入参考。新主题结果不再强制生成一套字号层级，页面智能体可结合当前页内容、当前画布和两份参考 SlideML 决定字号。

### 7.5 风格

`Style` 是一个自然语言字符串，而不是关键词数组。

示例语义：

```text
整体采用清爽、克制的现代教学风格，以大面积浅色背景和少量高饱和强调色建立课堂投影视觉层级，避免装饰性堆叠。
```

不再生成主题标题、摘要、关键词、学科、受众或使用场景对象。

### 7.6 安全区

安全区使用相对比例，而不是绑定某个固定像素画布：

- `LeftRatio`、`RightRatio` 相对于页面宽度；
- `TopRatio`、`BottomRatio` 相对于页面高度；
- 值域建议为 `0` 到 `0.3`；
- 左右之和、上下之和必须小于 `0.6`。

页面美化 Prompt 构建时，可以按当前页实际画布换算为像素值后展示，不需要把参考画布尺寸和当前画布尺寸重复发送给页面模型。

### 7.7 间距与视觉效果

`SpacingAndVisualEffects` 是一段自然语言，允许描述：

- 页面边缘留白；
- 区块间距；
- 标题与正文间距；
- 卡片内边距；
- 分割线；
- 描边；
- 圆角；
- 色块层次；
- 留白节奏。

明确规则：

- 不输出阴影建议；
- 页面参考 SlideML 不允许出现 `Shadow` 属性或 `<Shadow>` 元素；
- 页面美化 Prompt 额外加入固定规则“不要使用阴影效果”；
- 不输出可复用组件、组件 ID、组件约束或 Style 组件库。

### 7.8 版式原则

`LayoutPrinciples` 是一段自然语言字符串，不是页面类型数组，也不是程序预计算的布局规则。

它可以描述：

- 信息层级；
- 对齐方式；
- 图文关系；
- 留白；
- 内容密度处理；
- 课堂投影阅读距离；
- 标题、正文和辅助信息的相对位置。

不要求拆成固定数量的条目，避免为了满足数组数量而生成空泛原则。

### 7.9 封面页和内容页 SlideML

`CoverPageSlideMl` 和 `ContentPageSlideMl` 必须是两份完整、独立、可直接交给当前 `SlideMlParser` 的 XML 文档：

- 包含 `<Page>` 根节点；
- 不使用 Markdown 代码块；
- 不包含 `{{token:*}}` 或 `{{slot:*}}`；
- 不依赖旧模板编译器；
- 不引用本地文件；
- 不使用阴影；
- 完全不允许出现 `<Image>` 元素，图片区域只能用 `Rect` 和占位文字表达结构；
- 使用具体的示例文字表达标题、正文和辅助信息位置；
- 两份结构应明显不同，分别表达封面与普通内容页面。

它们是“经过验证的参考页面”，不是可复用组件模板，也不与任何输入 `SlideId` 绑定。

## 八、SlideML 规范共享设计

### 8.1 当前问题

`SlideMlPromptProvider.BuildSystemPrompt()` 同时混合了：

- SlideML 标签与属性规范；
- 完整 XML 输出格式；
- 页面 Agent 必须调用 `render_slide` 的行为；
- 最多渲染次数；
- 页面生成场景的角色说明。

主题智能体只需要语法规范，不应被要求调用页面会话的 `render_slide` 工具，也不应获得页面级预览工具。

### 8.2 推荐拆分

扩展 `ISlideMlPromptProvider`，新增精确面向完整文档的规范 API：

```text
BuildCompleteDocumentSpecificationPrompt()
  - 当前 SlideMlParser 接受的完整文档标签、层级和属性
  - 坐标、颜色、数值和 XML 格式规则
  - <Page> 根节点及完整文档输出规则

BuildSystemPrompt()
  - 非流式页面 Agent 行为
  - BuildCompleteDocumentSpecificationPrompt()

BuildStreamingSystemPrompt()
  - 流式页面 Agent 行为
  - 只复用与完整文档真正相同的基础定义
  - 流式片段、Remove、Id、StyleFrom、StyleId、合并和状态查询规则
```

完整文档规范和流式规范不能被要求逐字相同。主题 Agent 只接收完整文档规范，绝不能看到流式专用属性后再把它们提交给普通 `SlideMlParser`。

建议同时建立一个窄范围的 `SlideMlCompleteDocumentSchema`，由完整文档 Prompt 生成器和严格验证器共同消费，避免“给模型的规范”和“程序实际校验的规范”再次分叉。该 Schema 只描述当前完整文档子集，不引入业务模板、Token 或槽位系统。

主题智能体系统提示词组合：

```text
主题分析角色与输出要求
+ 主题场景专用限制
+ SlideMlPromptProvider.BuildCompleteDocumentSpecificationPrompt()
```

这样 SlideML 标签或属性发生变化时，只需修改一个规范来源。

### 8.3 主题场景专用限制

主题系统提示词在共享规范之外增加：

- 只生成两份完整 SlideML；
- 不调用页面渲染工具；
- 不输出流式片段；
- 不使用 `Shadow`；
- 不使用图片资源；
- 不输出 token、slot 或组件占位符；
- 最终通过唯一提交工具提交结果。

## 九、SlideML 本地校验设计

### 9.1 校验目标

用户要求的是 SlideML 合法，而不只是 XML 能解析。

因此两份参考 SlideML 应复用完整本地管线：

```text
SlideMlXmlUtilities.ExtractXml
  -> NormalizeXml
  -> SlideMlParser.Parse
  -> SlideMlLayoutEngine.PreLayout
  -> WpfSlideMlRenderEngine.PreMeasure
  -> SlideMlLayoutEngine.FinalLayout
  -> WpfSlideMlRenderEngine.Render
```

不需要把渲染预览发送给主题模型。

### 9.2 新验证器

建议新增 `CoursewareThemeSlideMlValidator`，职责仅为：

- 分别校验封面和内容页 SlideML；
- 使用 `XDocument` 和共享 `SlideMlCompleteDocumentSchema` 执行严格标签、层级、属性和值域校验；
- 使用主题分析服务传入的临时本地校验画布创建 WPF 渲染管线；
- 收集解析错误、渲染错误和警告；
- 按 XML 元素和属性的 `LocalName` 拒绝阴影、图片和回填属性，而不是扫描原始字符串；
- 捕获 XML、解析、布局、测量和 WPF 渲染异常，并统一转换为字段级诊断；
- 返回字段级简明诊断；
- 在成功时返回规范化后的输入 XML，而不是带 `RenderSize`、`RenderLocation` 和 `ActualLineCount` 的回填 XML。

### 9.3 通过标准

每份 SlideML 必须同时满足：

- 非空；
- 可以提取为 XML；
- 根元素为 `<Page>`；
- 所有标签、父子层级和属性都存在于完整文档 Schema；
- 必填属性存在，显式 Id 不重复；
- 颜色严格符合 `#RRGGBB` 或 `#AARRGGBB`；
- 数值为有限值，并满足尺寸、字号、透明度、渐变 Offset 等各自值域；
- 枚举属性为当前运行时支持值；
- 解析和渲染 `Errors.Count == 0`；
- `Warnings.Count == 0`；
- 不包含 `Shadow` 属性或 `<Shadow>` 元素；
- 不包含任何 `<Image>` 元素；
- 不包含回填属性；
- 不包含 token、slot、StyleFrom、StyleId、Remove 或其他流式专用结构。

两份 SlideML 还应满足规范化 XML 不完全相同，避免模型把同一页面重复提交为封面和内容页。

主题参考页面数量只有两个，要求严格 Schema 通过且零 Warning 不会产生旧压力测试那样的大量误判，同时能够防止解析器“忽略未知标签或属性后仍返回页面”、无效颜色被 WPF 回退为默认色或异常值在渲染阶段被容错的情况被当成成功。

### 9.4 本地校验画布

程序使用 `CoursewareInputPackage.Slides[0]` 的宽高创建临时 `SlideDocumentContext`，只用于本地校验两份参考 SlideML。

完整文档 SlideML 规范不额外拼接课件画布数值。主题 Agent 从全部原始 Markdown 中读取页面尺寸，程序不再发送“主画布”“尺寸分布”或另一份页面尺寸说明。

如果课件存在多种页面尺寸：

- 主题智能体仍只提交两份参考结构；
- 页面工作台依据当前页 Markdown 和当前页 SlideML 系统画布自行适配；
- 不再为每种尺寸生成模板集合；
- 不再建立 CanvasProfile、fallback 或页面映射。

## 十、主题 Agent 与提交协议

### 10.1 独立会话

每次主题分析必须创建新的 `CopilotChatManager`：

- 工作负载为 `AgentWorkload.ThemeAnalysis`；
- 不复用任何页面的 `SlideChatManager`；
- 不继承其他页面聊天；
- 重新分析时也创建新会话；
- 主题分析结束后不把会话注入页面工作台。

### 10.2 单一提交工具

旧的 13 个设计系统工具全部删除，替换为一个工具：

```text
submit_courseware_theme_analysis
```

工具参数就是完整 `CoursewareTheme`。

工具职责：

1. 进行轻量字段校验；
2. 校验两份 SlideML；
3. 校验失败时返回字段路径和简明错误；
4. 校验成功时保存规范化后的主题结果；
5. 返回“主题结果已通过校验”。

不再存在：

- `DraftId`；
- `Revision`；
- begin/set/upsert/complete；
- 分区顺序；
- 草稿冻结；
- AllowedNextTools；
- 组件、页面类型、证据和视觉观察工具。

### 10.3 多轮修复必须是真正的同会话对话

当前实现每次修复都使用 `WithHistory = false` 和 `CreateNewSession = true`，实际并不是同一会话内的持续修复，而且会重复携带整份原始输入。

新流程应改为：

#### 首轮

- `CreateNewSession = true`；
- `WithHistory = true`；
- 发送完整自然语言课件输入；
- 注册唯一提交工具；
- 不注册默认工具；
- 不附加图片。

#### 修复轮

- 使用同一个 `CopilotChatManager`；
- `CreateNewSession = false`；
- `WithHistory = true`；
- 只发送校验诊断和“重新调用提交工具”的要求；
- 不重复发送整份课件 Markdown；
- 不创建新草稿或新会话。

默认最多允许 4 次模型请求：

- 1 次初始分析；
- 最多 3 次显式修复。

工具调用期间，模型如果根据工具返回错误在同一轮再次调用工具，可以继续处理，不额外消耗显式修复轮次。达到上限仍未得到有效结果时，主题分析明确失败。

### 10.4 修复消息

修复消息只包含当前错误，例如：

```text
上次提交未通过本地校验。请保持已有效的主题建议，只修正下面的问题，然后重新调用 submit_courseware_theme_analysis：

- coverPageSlideMl：Panel(Header) 存在未知属性 StyleId。
- contentPageSlideMl：元素超出页面并产生布局警告。
- contentPageSlideMl：不允许使用 Shadow。
```

不附带：

- 原始完整 Prompt；
- 结构化事实；
- 截图；
- 草稿 ID；
- Revision；
- 旧提交的完整 JSON 副本。

### 10.5 Copilot 消息界面

保留现有消息展示方式：

- 用户消息；
- 助手消息；
- 工具调用；
- 工具输出；
- 用量信息；
- 滚动到底部行为。

`CopilotCoursewareThemeAgent` 每轮仍通过 `IProgress<CopilotChatMessage>` 上报 `UserChatMessage` 和 `AssistantChatMessage`。

不要求修改 `CopilotMessageStream` 的交互模型，只需要让其展示新的单工具提交和多轮修复内容。

## 十一、主题系统提示词设计

主题系统提示词只承担以下职责：

1. 角色：全课件美化主题分析助手；
2. 读取全部课件 Markdown；
3. 明确原课件样式分布仅供参考，不应限制重新美化；

6. 字体分中文和西文；

## 十二、主题分析服务编排

新的 `CoursewareThemeAnalysisService.AnalyzeAsync` 建议按以下顺序执行：

1. 检查取消；
2. 发布“正在准备课件内容”事件；
3. 调用样式使用分布汇总器；
4. 调用自然语言 Prompt Builder；
5. 从第一页创建只供本地 SlideML 校验使用的画布上下文；
6. 创建或获取主题 Agent；
7. 调用主题 Agent；
8. Agent 内完成提交、SlideML 校验和多轮修复；
9. 构造最小 `CoursewareThemeAnalysisResult`；
10. 发布“主题分析完成”事件；
11. 返回结果。

服务不再：

- 构建分析快照事实；
- 构建 JSON 信封；
- 提取版式事实；
- 选择视觉样本；
- 检查图片能力；
- 加载截图附件；
- 编译设计系统；
- 执行动态模板压力测试；
- 生成能力状态。

### 12.1 进度阶段

建议将 `CoursewareAnalysisStage` 收敛为：

```text
PreparingInput
AnalyzingTheme
ValidatingResult
RepairingTheme
Completed
```

界面事件示例：

- 正在汇总原课件字体、字号和颜色使用情况；
- 正在读取全部页面 Markdown 并形成主题建议；
- 正在校验封面页和内容页 SlideML；
- 正在根据本地校验结果修复 SlideML；
- 主题分析完成。

不再显示“读取结构”“内容层级分析”“编译设计系统”“模板压力测试”“视觉能力”等与新流程不符的阶段。

## 十三、页面美化工作台分发设计

### 13.1 每页必须收到的主题内容

首次创建工作台时，每个页面的首轮 Prompt 都使用同一份当前主题结果。重新分析后，新的主题结果只用于尚未开始会话且首轮草稿未被用户编辑的页面；已编辑草稿或已有会话继续保留其原内容，不能宣称这些页面已经切换到最新主题。

首轮 Prompt 包含：

- 配色建议；
- 中文字体建议；
- 西文字体建议；
- 风格描述；
- 按当前页画布换算后的安全区；
- 间距与视觉效果建议；
- 固定“不要使用阴影效果”规则；
- 自然语言版式原则；
- 封面页参考 SlideML；
- 内容页参考 SlideML。

### 13.2 每页不再收到的内容

当前页 Markdown 已经包含页面身份、尺寸和内容，不再二次包装。

### 13.3 页面 Prompt 改为自然语言

`CoursewareSlidePromptBuilder` 不再生成 `CoursewareSlideGenerationEnvelope` JSON。

目标首轮文本：

### 13.4 页面类型由页面 Agent 自主判断

页面 Prompt 必须明确：

- 系统没有预先给当前页分类；

- 手工编辑 SlideML 和重新渲染继续工作。

### 13.6 页面 Prompt 类型简化

- `CoursewareSlidePromptSource` 只保存 `CoursewareInputPackage` 和 `CoursewareThemeAnalysisResult`；

## 十五、主题分析快照设计

### 15.1 保留快照能力

继续保留自包含快照，因为它可以让用户以后直接进入页面美化工作台。

快照仍复制：

- `Courseware.json`；
- 全部页面 Markdown；
- 存在的页面截图；
- `Resources.json`；
- 存在的资源文件；
- 轻量主题分析结果。

截图和资源的保留是为了页面工作台，不代表主题 Agent 使用它们。

### 15.2 新版 Manifest

快照版本提升为新的 Schema，例如：

```text
courseware-theme-analysis-snapshot/v2
```

Manifest 只需保存：

```text
SchemaVersion
CreatedAt
CoursewareName
SlideCount
AnalysisResult
```

明确删除：

- `SourceFingerprint`；
- 设计系统；
- 结构事实；
- 模板压力样本；
- 视觉分析；
- 能力状态；
- 分析 Token、字符数和覆盖统计。

### 15.3 恢复校验

恢复时执行：

1. 先用 `JsonDocument` 或轻量 Header 只读取 `SchemaVersion`；
2. 如果是旧 `v1`，在反序列化不兼容的新主题对象前立即返回明确的重新分析提示；
3. 版本为 `v2` 后，再反序列化完整 Manifest；
4. 正常加载快照中的课件目录；
5. 校验课件名称和页数；
6. 校验 `AnalysisResult.Theme.SchemaVersion == "2.0"` 和全部轻量主题字段；
7. 从已加载快照课件的第一页重新创建本地校验画布并校验两份 SlideML；
8. 成功后进入页面工作台。

不再：

- 重新计算源指纹；
- 检测 Markdown 内容是否与保存时逐字一致；
- 校验设计系统和页面映射。

移除指纹后，系统不再具备“快照 Markdown 是否被保存后修改”的篡改检测能力。修改后的 Markdown 只要仍通过普通课件格式校验就可能被加载，但产品不承诺保存的主题建议仍与修改后的内容匹配；用户修改快照内容后应重新分析。不得通过隐藏哈希、校验和或另一种指纹绕回旧逻辑。

### 15.4 旧快照策略

旧 `v1` 快照包含本次明确删除的 DesignSystem、页面映射和指纹语义，不建议转换后继续使用。

新版本应明确拒绝旧快照，并提示：

```text
该主题分析快照使用已移除的旧设计系统协议，请打开原始课件目录重新分析。
```

不编写兼容适配器，不从旧 `CoursewareDesignSystem` 投影新主题结果，避免把旧错误逻辑带入新系统。

### 15.5 主题聊天记录

保持当前边界：主题聊天消息和分析事件不写入快照。

快照恢复后：

- 主题结果可查看；
- 主题分析对话为空；
- 页面会话重新按需创建；
- 不恢复旧 Agent 会话。

## 十九、关键设计取舍

### 19.1 为什么保留简单样式分布

字体、字号、颜色是原课件中成本最低、最直接的可参考事实。它们可以帮助模型了解原有素材大致特征，但系统会明确说明“仅供参考，不是约束”，避免模型机械继承坏设计。

### 19.2 为什么不保留版式统计

版式正是本轮要重新设计的内容。程序提前生成布局签名、聚类和页面类型，会把旧课件的问题当作新设计边界，直接限制主题 Agent 和页面 Agent。

### 19.3 为什么只保留封面和内容页两份结构

两份结构足以提供全课件的视觉语言和基本空间节奏，同时不会形成动态页面类型、模板库和逐页映射的复杂协议。页面 Agent 仍有足够自由处理目录、章节、练习、总结等页面。

### 19.4 为什么采用单一提交工具

结果字段数量少，单工具提交最容易理解、最容易测试，也最容易让模型在同一会话中修复。继续使用分区草稿只会重新引入协议状态和工具顺序问题。

### 19.5 为什么仍进行完整 SlideML 渲染校验

SlideML 是本次结果中唯一真正需要程序保证合法性的部分。复用现有解析和渲染管线可以直接发现未知标签、未知属性、XML 错误和布局警告，不需要再发明模板编译器或压力样本协议。

### 19.6 为什么拒绝旧快照

旧快照以设计系统、页面映射、结构事实和指纹为核心。自动转换会把本轮明确删除的语义重新投影进新模型，与“先删除干净”冲突。重新打开原课件分析是更清晰、可验证的迁移路径。

## 二十、最终目标状态

改造完成后，主题分析应被准确描述为：

> 程序先从整份课件 Markdown 的结构化样式元数据中汇总少量字体、字号和颜色使用参考，再把全部页面原始 Markdown 交给一个独立主题智能体。主题智能体给出配色、中文与西文字体、自然语言风格、安全区、间距与视觉效果、自然语言版式原则，以及封面页和内容页两份 SlideML 参考结构。两份 SlideML 经严格 XML/语义校验和本地真实解析、布局、渲染校验，失败时在同一主题会话中多轮修复。首次创建工作台时，同一成功结果进入每个页面的首轮草稿；系统不预先给页面分类，也不生成页面映射、组件或设计系统，由每个页面智能体结合当前页 Markdown 自主完成美化。
