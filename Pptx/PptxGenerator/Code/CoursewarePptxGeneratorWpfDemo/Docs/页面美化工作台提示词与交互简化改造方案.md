# CoursewarePptxGeneratorWpfDemo 页面美化工作台提示词与交互简化改造方案

> 本文针对页面美化工作台当前存在的四项问题给出实现级改造方案：主题分析结果没有以用户可见的待发送内容进入每页、聊天区错误地区分“首次生成”和“发送追问”、页面预览上方存在重复操作信息条、页面缩略图信息过载。
>
> 本文基于当前代码实现进行分析，只描述后续改造方案，不在本次工作中修改产品代码。

---

## 一、结论摘要

当前工作台并非完全没有把主题分析结果传给页面生成模型。`CoursewareSlidePromptBuilder` 已经能够在首次发送时构建结构化页面生成信封，其中包含：

- 当前页完整 Markdown。
- 当前页身份、尺寸、资源和画布诊断。
- 前后相邻页的标题与摘要。
- 由主题分析结果适配出的 `CoursewarePageDesignContext`。
- 当前页页面类型、模板、组件和素材规则。
- 主题、设计令牌、可访问性和跨页一致性规则。
- 当前请求是否附带原始截图的视觉证据边界。

真正的问题是：这些内容直到用户点击“首次生成”后才在后台临时构建，右侧输入框初始为空，原始页面截图也没有作为可见附件预先放入附件区。界面展示的“待发送内容”和代码实际发送的内容不是同一个事实源，因此代码被迫引入“首次生成”和“发送追问”两套命令、两套按钮及大量状态分支。

推荐将工作台调整为以下统一模型：

```text
进入工作台或首次访问某页
  -> 为该页构建完整首轮页面美化 Prompt
  -> 将 Prompt 写入该页聊天输入框
  -> 将原始页面截图加入该页可见附件集合
  -> 用户可以检查、编辑 Prompt 和附件
  -> 点击唯一的“发送”按钮
  -> 发送当前输入框文本 + 当前可见附件
  -> 底层根据该页会话是否已经建立，内部选择首轮协议或续轮协议
```

核心原则是：

> **右侧输入框和附件区必须成为下一次发送请求的唯一业务事实源。**

发送时不再临时构建另一份隐藏业务 Prompt，不再隐式追加原始截图，也不再让 UI 决定“首次生成”还是“追问”。底层 `isFirstMessage` 可以继续存在，但只能作为 `SlideChatManager` 的内部协议参数，不能继续承担产品交互状态。输入框应与传给页面生成管线的用户消息正文一致；底层仍可按首轮协议追加系统提示、SlideML 语法约束和工具配置，这些协议内容不属于用户可编辑输入框。

## 二、当前实现与问题根因

### 2.1 当前主题到页面生成的数据流

当前工作台创建时，`CoursewareSlideWorkspaceViewModel` 会通过：

```text
CoursewareSlidePromptBuilder.PrepareSource
  -> CoursewareInputPackage
  -> CoursewareThemeAnalysisResult
  -> CoursewareAnalysisEnvelope
  -> CoursewareSlidePromptSource
```

准备全工作台共享的页面提示词来源。

首次生成时，调用链为：

```text
GenerateSelectedSlideCommand
  -> GenerateSlideAsync
  -> SendPageMessageAsync(requireInitialConversation: true)
  -> 判断 HasStartedGenerationConversation
  -> CoursewareSlidePromptBuilder.Build
  -> CoursewareSlideGenerationEnvelope
  -> 序列化为 JSON Prompt
  -> SlideChatManager.SendMessageAsync(isFirstMessage: true)
```

其中 `CoursewareSlideGenerationEnvelope` 已经包含 `DesignContext`。`CoursewareThemePageDesignAdapter` 会将主题分析结果转换为当前页面的执行上下文，包括：

- 主题标题、摘要和风格关键词。
- 配色与旧主题字号、字体、安全区、版式原则、页面类型建议。
- `CoursewareDesignSystem` 的颜色、字号和间距令牌。
- 当前页动态页面类型。
- 当前页对应模板及模板是否通过验证。
- 当前页面类型可用组件。
- 当前页资源对应的素材规则。
- 可访问性和跨页一致性规则。
- 主题参考画布和当前页面画布。
- 画布比例差异诊断。

所以当前第一项问题的准确表述应是：

> 主题分析结果已经进入首次请求的后台结构化 Prompt，但没有在进入页面时预先生成到聊天输入框，也没有成为用户可见、可编辑、可核对的待发送内容。

### 2.2 当前截图附件的数据流

`CoursewareSlideItemViewModel` 创建后：

- `AttachedImageFiles` 初始为空。
- 有原始截图时，`ScreenshotAttachmentState` 为 `NotPrepared`。
- 没有原始截图时为 `FileMissing`。

首次点击生成后，`SendPageMessageAsync` 才会：

1. 读取当前手工附件集合。
2. 如果是首条消息且原始截图存在，将原始截图路径隐式加入局部待发集合。
3. 根据局部集合计算 `sourceScreenshotAttached`。
4. 将该状态写入 Prompt。
5. 把截图作为 `requiredAttachedImageFiles` 传给底层。

这会造成两个问题：

- 用户在界面上看不到原始截图已经准备发送。
- 附件区并不是实际请求附件的完整表达，代码存在一条隐藏附件路径。

### 2.3 当前双命令和双按钮的根因

`CopilotPanel.xaml` 当前同时显示：

- “首次生成”，绑定 `GenerateSelectedSlideCommand`。
- “发送追问”，绑定 `SendMessageCommand`。

两条命令最终都进入 `SendPageMessageAsync`，差异来自：

- `requireInitialConversation` 参数。
- `HasStartedGenerationConversation` 状态。
- `isFirstMessage` 局部变量。

这些分支进一步决定：

- 是否调用 `CoursewareSlidePromptBuilder.Build`。
- 是否创建新会话。
- 是否隐式追加原始截图。
- 是否将输入框文字视为 `UserInstruction` 后嵌入结构化 Prompt。
- 是否附带当前渲染预览。
- 是否允许命令执行。

本质上，当前输入框在首轮时只保存一句“用户附加要求”，并不保存最终发送 Prompt。因此首轮发送必须由单独命令在点击时完成二次加工。

### 2.4 当前中间页面顶部信息条

`MainContentPanel.xaml` 的第 0 行完整展示：

- 当前页标题。
- 当前页摘要。
- 当前页状态。
- 当前页画布尺寸。
- “生成当前页”按钮。
- “取消”按钮。

其中“生成当前页”与右侧聊天区首次生成完全重复；标题、摘要、状态和尺寸也不是用户操作当前页面的必要主信息。整条区域增加了垂直占用和视觉复杂度。

### 2.5 当前缩略图模板

`LeftSidebarPanel.xaml` 当前每项包含：

- 页面缩略图。
- 警告徽标。
- 错误徽标。
- 页面标题。
- 页码。
- 页面状态。
- 页面 ID。

信息密度过高，而且状态、警告和错误通过多个位置重复表达。用户的主要任务是按页码快速定位页面，因此缩略图列表应回归导航职责。

## 三、整体目标交互

### 3.1 每页的目标状态

每一个 `CoursewareSlideItemViewModel` 都应独立保存：

- 当前可见聊天草稿。
- 当前可见图片附件。
- 当前会话是否已经成功建立。
- 当前首轮草稿所依据的主题版本或主题对象引用。
- 当前草稿是否经过用户编辑。
- 当前发送操作的文本和附件快照。
- 当前 SlideML、预览、日志和错误。

切换页面只切换绑定对象，不重新从全局临时拼装界面状态。

### 3.2 进入工作台

进入工作台时：

1. 创建全部轻量页面 ViewModel。
2. 为第一页准备完整首轮 Prompt。
3. 将第一页原始截图加入第一页可见附件集合。
4. 初始化第一页运行时，但 Prompt 和附件准备不依赖模型初始化成功。
5. 输入框直接显示将要发送给模型的完整文本。

其他页面可选择以下两种实现方式：

- 在工作台创建时为全部页面同步构建 Prompt。优点是切页立即可见；缺点是大课件会提前序列化大量文本。
- 首次访问页面时构建该页 Prompt。优点是保持懒加载；缺点是切页时需要一次确定性的本地初始化。

推荐使用第二种方式，但必须保证 Prompt 构建不依赖页面 AI 运行时，也不引入网络请求。第一页在工作台激活时准备，其他页在首次选中时准备。

### 3.3 首轮发送

用户看到并可编辑：

- 完整页面美化 Prompt。
- 原始页面截图附件。
- 用户手工添加的其他图片。

点击唯一“发送”按钮后：

1. 获取输入框文本快照。
2. 获取当前可见附件快照。
3. 校验文本非空、附件文件仍存在。
4. 根据页面会话阶段确定底层发送模式。
5. 将原样文本和附件快照发送给 `SlideChatManager`。
6. 不再调用 Prompt Builder。
7. 不再隐式追加任何截图。

### 3.4 后续追问

首轮成功并应用有效 SlideML 后：

- 页面会话进入已建立状态。
- 输入框清空，成为普通追问输入框。
- 原始截图从“下一次待发附件”集合移除，避免每次追问重复发送。
- 用户手工附件按发送成功策略清除。
- “附带当前渲染预览”仍可以保留，但它应被视为当前发送的显式附件策略，而不是另一个发送命令。

后续用户仍点击同一个“发送”按钮。内部会话协调层将请求映射为续轮协议。

### 3.5 首轮失败或取消

首轮失败、取消、模型未返回有效 SlideML 或必需截图发送失败时：

- 页面会话不能标记为已成功建立。
- 输入框原文本保留。
- 当前附件保留。
- 用户可以直接点击同一个“发送”按钮重试。
- 底层应清理或重新创建失败首轮留下的不完整会话，避免历史和流式状态污染重试。

## 四、问题一：主题分析结果拼接到各页面

### 4.1 目标

每页聊天输入框初始内容就是完整、最终、可发送的页面美化 Prompt。该 Prompt 在原单页美化内容基础上增加主题分析的页面级有效输出，并继续包含当前页完整 Markdown。

### 4.2 推荐 Prompt 结构

继续复用当前 `CoursewareSlideGenerationEnvelope`，不建议退回大段不可验证的自然语言拼接。结构可保持为：

```text
CoursewareSlideGenerationEnvelope
  - schemaVersion
  - task
  - currentSlide
  - neighbors
  - designContext
  - visualInput
  - outputRequirements
```

输入框显示其序列化文本。这样用户看到的内容与实际发送内容完全一致，同时保留当前已有的确定性、路径隐私和单元测试能力。

### 4.3 应拼接的主题分析内容

主题分析结果很大，不应整包复制到每页。应只注入经过验证且对当前页面有执行意义的内容。

#### 4.3.1 全局主题意图

保留：

- `Theme.Title`。
- `Theme.Summary`。
- `Theme.StyleKeywords`。
- `Theme.LayoutPrinciples`。
- `Theme.ContentPresentationRules`。
- `Theme.GenerationPromptSummary`，仅作为补充说明。

作用是帮助模型理解全课件的视觉方向和教学表达方向。

#### 4.3.2 当前页可执行设计令牌

保留：

- 颜色令牌。
- 字号令牌。
- 间距令牌。
- 适配当前画布后的字体层级。
- 适配当前画布后的安全区。
- 推荐字体。
- 可访问性规则。
- 跨页一致性规则。

这些是生成 SlideML 时真正需要执行的约束。

#### 4.3.3 当前页页面类型和模板

保留：

- 当前页被分配的页面类型。
- 当前页对应的模板。
- 模板是否通过真实编译和压力测试。
- 当前页面类型允许使用的组件。

模板处理必须遵守：

- 已验证模板可以作为强约束或优先骨架。
- 未通过验证的模板不能作为可执行模板发送；最多只能省略，不应把失败模板作为建议继续提供给模型。

当前实现会把模板对象和 `CurrentTemplateValidated` 一并放入上下文。后续改造时建议进一步确保：只有验证通过的模板才进入 `CurrentTemplate`，避免模型忽略布尔状态而使用未验证模板。

#### 4.3.4 当前页素材规则

只保留当前页实际引用资源对应的：

- 逻辑 `ResourceId`。
- 资源类型。
- 是否存在。
- 对应素材使用规则。

不能注入：

- 本地绝对路径。
- 截图真实路径。
- 全课件所有无关资源规则。

#### 4.3.5 当前页和邻页内容

保留：

- 当前页完整 Markdown。
- 当前页 `SlideId`、页码、尺寸和画布诊断。
- 前一页和后一页的标题与有限摘要。

不注入其他页面完整 Markdown，防止内容串页和上下文无谓膨胀。

### 4.4 不应拼接的主题分析输出

以下内容不应进入每页 Prompt：

- 主题分析聊天历史。
- 模型推理文本。
- 工具调用参数和原始模型响应。
- 分析耗时、时间戳、Token 用量等运行诊断。
- UI 展示状态和分析过程事件。
- 未经验证的候选主题或候选设计系统。
- 全量模板验证样本和压力测试日志。
- 全量结构化事实报告。
- 全量视觉分析原始证据和中间描述。
- 与当前页无关的组件、模板和素材规则。
- 其他页面截图。

如视觉分析已经形成经过验证的设计结论，应让结论进入设计系统或当前页设计上下文，而不是把原始视觉证据重复发送。

### 4.5 字段优先级

当前 `CoursewarePageDesignContext` 同时保留较旧的描述性主题字段和新设计系统字段。若两者存在冲突，应在 Prompt 中明确以下优先级：

```text
已验证的设计系统令牌
  > 当前页页面类型与已验证模板
  > 可访问性和跨页一致性规则
  > 描述性主题建议
  > GenerationPromptSummary
```

`GenerationPromptSummary` 只能补充说明，不能覆盖结构化令牌。

### 4.6 Prompt 初始化时机

建议为页面增加“首轮草稿是否已准备”的内部状态。准备逻辑在以下时点执行：

- 工作台激活时，为当前选中页准备。
- 首次切换到未准备页面时，为该页准备。
- 主题更新后，对尚未开始会话且未被用户编辑的首轮草稿重新准备。

Prompt 初始化时需要同时确定原始截图是否处于可见附件集合中，并据此设置：

- `CurrentSlide.ScreenshotAttached`。
- `VisualInput.WasAttached`。
- `VisualInput.EvidenceBoundary`。

不能先生成“已附截图”的 Prompt，再允许附件集合缺失截图而不更新 Prompt。Prompt 与附件必须来自同一份页面待发状态。

### 4.7 主题更新策略

主题重新分析成功后，应按页面状态处理：

#### 尚未建立会话，首轮草稿未编辑

- 自动使用新主题重建 Prompt。
- 保持原始截图附件。
- 更新该页草稿所依据的主题版本。

#### 尚未建立会话，但首轮草稿已被用户编辑

- 不静默覆盖用户内容。
- 保留当前草稿和附件。
- 标记“主题已更新，当前草稿基于旧主题”。
- 后续若要增加“按新主题重新生成草稿”，应作为显式操作，不应在普通发送时暗中替换。

#### 已建立会话

- 当前会话继续使用建立时的主题上下文。
- 普通追问不能在后台切换到新主题。
- 如需采用新主题，应未来提供“重新开始当前页会话”能力；这不属于本次四项界面简化的必要范围。

## 五、问题二：取消“首次生成”和“发送追问”区分

### 5.1 UI 改造

`CopilotPanel.xaml` 中：

- 删除“首次生成”按钮。
- 删除“发送追问”按钮。
- 增加唯一“发送”按钮。
- 绑定统一的 `SendMessageCommand` 或重命名后的 `SendSelectedSlideMessageCommand`。
- “取消”按钮保留。
- “追问时附带当前渲染预览”文案改为“发送时附带当前渲染预览”，避免继续暴露追问状态。

### 5.2 ViewModel 命令改造

`CoursewareSlideWorkspaceViewModel` 中：

- 删除 `_generateSelectedSlideCommand`。
- 删除 `GenerateSelectedSlideCommand`。
- 删除 `GenerateSlideAsync`。
- 删除 `requireInitialConversation` 参数。
- 统一保留一条发送命令。
- `CanExecute` 只判断：
  - 页面不忙。
  - 输入框非空。
  - 没有未重新渲染的手工 SlideML 修改。
  - AI 运行时可用或尚未创建。

发送流程只接收当前页面，不再接收“是否首次生成”的 UI 意图。

### 5.3 底层首轮协议保留方式

现有 `SlideChatManager.SendMessageAsync` 和生成管线仍需要 `isFirstMessage`，因为首轮和续轮可能在以下底层行为上不同：

- 首轮注入 SlideML 系统提示。
- 初始化流式页面状态。
- 建立或重建对话会话。
- 续轮对现有 SlideML 执行增量修改。

因此不建议为了界面统一而立即重写整个底层协议。正确做法是把该差异下沉：

```text
统一发送命令
  -> 页面会话阶段
       - NotStarted: InitializeConversation
       - Active: ContinueConversation
  -> SlideChatManager 适配
       - InitializeConversation => isFirstMessage: true
       - ContinueConversation => isFirstMessage: false
```

UI、按钮、输入框和附件准备都不再读取或设置 `isFirstMessage`。

### 5.4 页面会话阶段

可继续复用 `HasStartedGenerationConversation`，但建议将语义改为更清晰的内部阶段枚举：

```text
CoursewareSlideConversationState
  - NotStarted
  - SendingInitial
  - Active
```

如果希望保持最小改动，也可以暂时保留布尔属性，但必须满足：

- 只由统一发送流程内部读取。
- 只有首轮请求成功并应用有效 SlideML 后才设为 `true`。
- UI 不根据它显示不同按钮。
- Prompt 构建和截图准备不再依赖它在发送时临时执行。

### 5.5 发送请求快照

统一发送时应先创建不可变快照，至少包含：

- 当前页对象或 `SlideId`。
- 当前输入框文本。
- 当前附件集合。
- 是否附带当前渲染预览。
- 当前会话阶段。
- 草稿修订号。

之后异步发送只使用快照，防止发送期间用户继续编辑导致成功回调误清除新内容。

### 5.6 输入框清理规则

当前实现发送成功后直接执行：

- `slide.InputText = string.Empty`。
- `slide.AttachedImageFiles.Clear()`。

该方式存在并发问题：如果用户在请求发送期间继续编辑或添加附件，成功回调会把新内容一起删除。

建议增加草稿修订号或等价快照比较：

- 发送前记录文本及修订号。
- 成功后只有当前输入仍等于发送快照时才清空。
- 若用户已经编辑，则保留新草稿。
- 附件只移除本次成功发送的附件实例，不清空整个集合。

### 5.7 附件生命周期

建议将附件分成三类策略：

#### 原始页面截图

- 页面首轮草稿准备时加入可见附件集合。
- 只用于首轮请求。
- 首轮成功后移除。
- 首轮失败或取消后保留。
- 同一路径手工附件需要去重。

#### 用户手工图片

- 按页面独立保存。
- 下一次发送成功后移除本次已发送项。
- 失败或取消后保留。
- 发送期间新增的附件不能被旧请求完成回调移除。

#### 当前渲染预览

- 继续通过复选框显式选择。
- 只作用于下一次发送。
- 不进入持久附件缩略图集合，或以明确的临时附件项展示。

### 5.8 截图状态简化

当前 `CoursewareScreenshotAttachmentState.NotPrepared` 的含义来自“尚未点击首次生成”。改造后截图在页面准备时已经进入附件区，因此状态建议调整为：

- `Attached`：已在当前待发附件中。
- `FileMissing`：源截图不存在。
- `SendFailed`：最近一次发送失败。
- 可选 `Sent`：首轮已经成功发送，不再作为待发附件。

如果不需要继续显示截图状态文案，可以进一步删除该枚举的 UI 绑定，只保留附件模型和错误信息作为事实源。

### 5.9 重试与会话清理

首轮请求可能已经向聊天管理器写入部分消息后失败。重试前应确保：

- 不把失败首轮误当成有效续轮。
- 清理失败首轮的流式页面状态。
- 必要时创建新会话。
- 继续发送界面中保留的原 Prompt 和原附件。

首轮成功判定应至少满足：

- 请求正常结束。
- 得到非空 SlideML。
- SlideML 已经应用到当前页面状态。

只有满足这些条件，页面会话才进入 `Active`。

```text

```

```text

```

并调整：

- SlideML 编辑与日志区从原 `Grid.Row="2"` 移到 `Grid.Row="1"`。

- `Summary`：相邻页 Prompt 构建和内部摘要仍需要。
- `CanvasSizeText`：目前可能只用于被删除区域，可在全局引用检查后决定是否删除。
- `StatusText`：左侧状态将删除，但可能仍用于日志、可访问性或测试；应按引用结果最小清理。

本次产品目标是删除展示，不要求为了减少属性数量而扩大改造范围。

## 七、问题四：简化页面缩略图

### 7.1 目标布局

每个列表项改为水平布局：

```text
页码    页面缩略图
```

要求：

- 页码位于缩略图左边。
- 页码与缩略图顶部对齐。
- 列表项只显示页码和缩略图。
- 不显示标题。
- 不显示页面 ID。
- 不显示页面状态。
- 不显示“未开始”“等待生成”等文字。
- 不显示警告和错误状态徽标。

推荐 XAML 结构：

```text
Grid
  Column 0: Auto，页码
  Column 1: 缩略图
```

页码可使用当前 `PageNumber` 或 `NumberText`。如果希望保持紧凑，建议显示自然页码而不是固定两位补零；最终视觉可按现有产品风格决定。

### 7.2 选中状态

继续使用缩略图边框颜色表达选中状态：

- 未选中：透明边框。
- 选中：`PrimaryBlueBrush`。

页码可同步加粗或变为主色，但不是必须。选中状态属于导航反馈，不是用户要求删除的页面生成状态。

### 7.3 ToolTip 内容

整个列表项或缩略图设置统一 ToolTip，只显示：

```text
标题：{Title}
页面 Id：{SlideId}
```

不能包含：

- 页面状态。
- 运行时状态。
- 警告摘要。
- 错误信息。
- 是否已完成生成。

建议直接在 XAML 中使用 ToolTip 模板绑定 `Title` 和 `SlideId`，而不是在 ViewModel 中拼接包含其他状态的字符串。

### 7.4 警告和错误的去向

删除缩略图上的警告和错误徽标后，问题信息仍可从以下位置查看：

- 中间区域的错误覆盖提示。
- SlideML 下方“输入诊断”页签。
- 渲染日志。
- 右侧聊天区运行时错误。

因此无需在 ToolTip 中重新加入状态、警告或错误信息。

### 7.5 可访问性

视觉简化不应删除辅助技术所需信息。`AccessibleName` 可以继续包含页码和标题，但建议去掉状态或警告描述，使其与简化后的导航职责一致：

```text
第 N 页，{Title}
```

ToolTip 和自动化名称是不同用途；用户要求 ToolTip 不显示状态，不代表自动化名称必须与 ToolTip 完全相同，但也不建议继续重复过多状态。

## 十、测试方案

### 10.1 Prompt 构建测试

覆盖：

- 页面首次准备后输入框已经包含完整结构化 Prompt。
- Prompt 包含当前页完整 Markdown。
- Prompt 包含当前页设计上下文。
- Prompt 包含前后页有限摘要，不包含其他页完整 Markdown。
- Prompt 只包含当前页相关资源和素材规则。
- Prompt 只包含当前页页面类型、验证通过的模板及相关组件。

- Prompt 不包含本地绝对路径。

- 工作台激活后第一页输入框非空。
- 第一页原始截图出现在可见附件集合中。

覆盖：

- XAML 只有一个发送按钮。
- 统一命令在未建立会话页面上调用底层首轮协议。

## 十二、验收标准

### 12.1 主题与页面内容

- 进入某页时，右侧输入框已经显示该页最终待发 Prompt。
- Prompt 包含当前页完整 Markdown。
- Prompt 包含主题分析产生且与当前页相关的设计上下文。
- Prompt 不包含无关分析过程、其他页完整内容或本地路径。

- 有原始截图的页面，进入该页时截图已显示在附件区。
- 用户看到的附件集合与实际发送附件一致。
- 不存在发送阶段隐式追加截图的隐藏逻辑。
- 失败或取消后附件仍可直接用于重试。

### 12.3 发送交互

- 右侧只有一个“发送”按钮。
- 不再出现“首次生成”和“发送追问”。
- 用户点击发送时，仅发送输入框内容和当前附件。
- 首轮与续轮差异只存在于底层会话协议，不影响 UI 和内容准备。

### 12.4 中间预览区

- 页面预览上方不再显示标题、摘要、状态和画布尺寸条。
- 不再显示“生成当前页”按钮。
- 页面预览获得更多垂直空间。
- SlideML 的“重新渲染”功能保持可用。

### 12.5 缩略图

- 每项只显示左侧页码和右侧缩略图。
- 页码与缩略图顶部对齐。
- 不显示页面标题、页面 ID 和页面状态。
- 鼠标悬停时 ToolTip 只显示标题和页面 ID。
- ToolTip 不显示页面状态、警告或错误。

## 十三、风险与边界

### 13.1 完整 Prompt 可编辑风险

输入框显示结构化 JSON Prompt 后，用户可能修改或破坏其格式。本方案仍以输入框为权威事实源，因此发送时不应静默重建或覆盖用户修改。

可以执行：

- 非空检查。
- 上下文预算检查。
- 附件存在性检查。

不建议执行：

- 自动把用户编辑内容重新嵌入另一层 Prompt。
- 自动恢复成后台生成版本。
- 静默修复并发送与界面不同的文本。

如果后续认为结构化 Prompt 不适合直接编辑，可以增加“系统上下文只读区 + 用户要求编辑区”，但这会改变“输入框内容就是完整待发内容”的当前产品要求，不属于本次方案。

### 13.2 主题更新与活跃会话

主题更新后，已有页面会话是否重新开始是独立产品决策。本方案推荐：

- 未发送且未编辑草稿自动刷新。
- 已编辑草稿不覆盖。
- 已建立会话冻结原主题。

这样风险最小，也最符合用户对可见发送内容的预期。

### 13.3 大课件内存与性能

如果一次为全部页面生成完整 Prompt，会增加内存占用。推荐按页懒准备，并在页面 ViewModel 中缓存。Prompt 构建是本地确定性操作，不应依赖页面模型运行时。

### 13.4 底层协议不应被误删

取消 UI 上的首次/追问区分，不代表底层生成管线已经没有首轮/续轮差异。本次应下沉和封装差异，而不是直接把所有请求都强制设为同一种 `isFirstMessage` 值。

---

## 十四、最终建议

本次四项问题应作为一次统一的数据流修正，而不是四个彼此独立的 XAML 调整。

最终推荐架构为：

```text
主题分析结果 + 当前页事实
  -> CoursewareSlidePromptBuilder
  -> 页面首轮可见草稿

原始页面截图
  -> 页面可见附件

页面可见草稿 + 页面可见附件
  -> 唯一发送命令
  -> 页面会话协调层
  -> 首轮或续轮底层协议
  -> SlideChatManager
```

只要坚持“可见草稿和可见附件是发送请求的唯一业务事实源”，就可以同时解决：

- 主题结果看似没有进入页面的问题。
- 原始截图准备不可见的问题。
- 首次生成与追问双状态问题。
- 中间区域重复生成入口问题。
- 用户无法预先核对实际发送内容的问题。

中间信息条和缩略图简化则在此基础上进一步收敛界面职责：左侧只负责页面导航，中间只负责预览与 SlideML，右侧只负责统一的页面对话与发送。
