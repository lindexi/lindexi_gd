# 打开课件文件夹后的流程思考

本文记录 `CoursewarePptxGeneratorWpfDemo` 在点击“打开课件文件夹”之后，面向全课件美化流程应执行的工作拆分。

这里假设用户选择的文件夹已经符合 `Docs/CoursewareMarkdownExportFormat.md` 所定义的导出格式。也就是说，课件原始对象到 Markdown、截图、资源文件和资源索引的导出步骤已经完成；Demo 侧不再重新读取原始课件对象，也不再重新做 Markdown 提取和资源提取。

---

## 当前项目现状

当前 Demo 已经具备以下基础能力：

- 主界面中存在“打开课件文件夹”按钮，入口位于 `Views/MainContentPanel.xaml` 和 `Views/MainContentPanel.xaml.cs`。
- `MainWindowViewModel.OpenCoursewareFolder` 当前只保存文件夹路径，并更新状态文本。
- `MainWindowViewModel` 已经维护 `Slides` 集合、`SelectedSlide`、`SlideChatManager` 和当前页 SlideML 编辑区。
- 每个 `CoursewareSlideItem` 已经设计为持有独立的 `SlideChatManager`，这与全课件美化设计中“每页独立会话，避免跨页上下文污染”的原则一致。
- `SlideChatManagerFactory` 可以为每页创建独立的生成、渲染、评估上下文。
- 渲染链路已经存在本地渲染和 MCP 渲染切换能力。

因此，“打开课件文件夹”不应该只是记录路径，而应该成为全课件美化输入初始化的入口。

---

## 打开文件夹后应该做的事情

### 1. 验证课件导出目录

点击打开课件文件夹后，第一步应该验证目录是否为合法的课件 Markdown 导出目录。

建议校验内容：

- 根目录存在 `courseware.json`。
- `courseware.json` 能被解析，并且 `exportVersion` 是当前支持的版本。
- `slideCount` 与 `slides` 数量一致。
- 每个页面条目中的 Markdown 文件存在。
- 每个页面条目中的截图文件存在；如果不存在，可以允许降级，但 UI 应标记缺失截图。
- `resourcesFile` 指向的资源索引文件存在；如果不存在，可以允许无资源模式，但应提示资源不可用。
- 资源索引中的 `exportFile` 对应资源文件如果不存在，不应阻止加载，但应记录警告。

这一阶段只做本地 I/O 校验，不调用 LLM。

### 2. 加载课件输入包

校验通过后，应把导出目录加载成一个内存中的 `CoursewareInputPackage`，统一承载页面输入和资源索引。

建议包含信息：

- 课件根目录。
- 课件名称。
- 页面总数。
- 每页的页面索引、页面 ID、宽高、Markdown 文件路径、截图文件路径、Markdown 文本。
- 资源索引文件路径。
- 每个图片资源的 `imageId`、导出文件路径、原始文件名。

这里的关键点是：

- 不再重复执行 Markdown 提取。
- 不再执行截图。
- 不再重新提取资源。
- 使用导出目录中的 Markdown、截图和资源作为后续全课件美化的输入。

### 3. 初始化 UI 页面列表

加载输入包之后，应根据 `courseware.json` 初始化左侧页面列表和当前页状态。

建议行为：

- 清空当前默认空白页。
- 为每个导出页面创建一个 `CoursewareSlideItem`。
- 每个页面创建独立的 `SlideChatManager`。
- 页面标题可优先从 Markdown 中提取第一个标题；提取不到时使用“第 N 页”。
- 页面摘要可来自 Markdown 前几行正文的简短截断。
- 页面状态初始为“待美化”或“已加载”。
- 当前选中第一页。
- 中间预览区在尚未生成 SlideML 前，优先显示原始截图。
- SlideML 编辑区在尚未生成时保持为空，或者显示提示性占位文本，但不应伪造 SlideML。

当前 `CoursewareSlideItem` 只有生成后的 `SlideMl`、`RenderingLog`、`CallbackXml` 等字段。后续如果要完整承载打开文件夹后的状态，可能需要增加源 Markdown、源截图、页面 ID、页面尺寸、生成输出路径、错误信息等字段，或者新增更完整的页面输入模型，再由显示模型引用。

### 4. 建立资源可访问上下文

导出的资源文件位于 `resources` 目录，Markdown 中引用的是资源 ID 或导出资源相对路径。后续 LLM 生成 SlideML 时需要知道可用资源，也需要渲染器能够找到图片。

打开文件夹后应建立资源上下文：

- 读取 `resources/resources.json`。
- 解析 `imageId` 到实际资源文件路径的映射。
- 在提示词中向 LLM 描述可用图片资源，例如资源 ID、原始文件名、相对路径。
- 渲染 SlideML 时要能解析这些资源引用。

当前 `CoursewareWpfSlideMlRenderEngine` 的图片路径查找更偏向当前工作目录、`Images`、`Assets` 等固定位置。为了支持课件导出目录，后续可能需要为渲染上下文增加资源根目录，或者在生成 SlideML 前约定统一的图片引用方式，并在渲染前将引用转换为可解析路径。

这一点应该尽早确定，否则生成出来的 SlideML 可能引用了资源 ID，但本地预览无法显示图片。

### 5. 进入“可分析、可美化”的准备状态

打开文件夹完成后，系统应进入一个明确的准备状态：

- 输入课件已加载。
- 页面 Markdown 已在内存中可用。
- 页面截图已可用于单页美化视觉输入。
- 资源索引已可用于提示词和渲染。
- 尚未执行全局主题分析。
- 尚未执行逐页美化。
- 尚未执行一致性校验。

状态文本不应只显示“已选择课件文件夹”，而应更具体，例如“已加载 12 页，等待主题分析”。

## 全局主题分析应该放在哪里触发

全课件美化设计中，导出目录已经完成了 Markdown 提取和资源提取，因此打开文件夹之后，下一个真正的智能步骤是“全局主题分析”。

这里有两种触发方式：

### 方式一：打开文件夹后自动启动主题分析

优点：

- 用户操作少。
- 打开后很快进入可美化状态。
- 符合“全课件美化入口”的一体化体验。

缺点：

- 一选择文件夹就消耗 LLM 调用。
- 用户如果选错目录，会浪费调用。
- 不利于先检查页面加载结果。

### 方式二：打开文件夹只加载本地数据，由用户点击“开始分析/开始美化”再触发

优点：

- 避免误操作消耗 LLM。
- 用户可以先确认页面数量、截图和资源是否正确。
- 后续更容易加入“只分析主题”“批量美化”“重新分析主题”等按钮。

缺点：

- 多一步用户操作。

建议第一版采用方式二：打开课件文件夹只完成本地加载和 UI 初始化，不自动调用 LLM。全局主题分析由独立命令触发，可以命名为“分析全局主题”或“开始全课件美化”。如果用户点击“开始全课件美化”而主题尚未分析，则流程内部先分析主题，再逐页生成。

---

## 主题分析阶段的输入

主题分析不需要截图，输入应来自已经加载的全部 Markdown 文本。

建议组装内容：

- 课件名称。
- 页面数量。
- 每页索引、页面 ID、尺寸。
- 每页 Markdown 文本摘要或截断后的 Markdown。
- 可选：资源摘要，例如本课件包含多少张图片、图片原始文件名概览。

需要控制 Token：

- 每页 Markdown 可以限制最大字符数。
- 超过一定页数时，可以采用首页、目录页、章节页、末页和抽样内容来分析主题。
- 主题分析结果应结构化保存，作为后续逐页美化共享上下文。

主题分析产物可对应全课件设计文档中的 `CoursewareTheme`：配色、字号、版式骨架、安全区、字体建议等。

## 阶段二逐页美化在 Demo 中的落点

在当前 Demo 中，逐页美化直接从导出的 Markdown、截图和资源索引生成每页 SlideML，不再依赖原始运行时课件对象。

每一页的输入建议包括：

- 当前页 Markdown 全文。
- 当前页截图文件。
- 全局主题分析结果。
- 前后三页的 Markdown 摘要。
- 可用资源索引。
- 当前页尺寸。

每一页的输出建议包括：

- 生成后的 SlideML。
- 渲染预览图。
- 渲染日志。
- LLM 对话日志。
- 错误信息。
- 输出到目标目录的 `.slideml` 或 `.xml` 文件。

因为 `CoursewareSlideItem` 已经为每页持有独立 `SlideChatManager`，第一版可以串行处理页面：

1. 选中或定位当前页。
2. 构建包含全局主题的用户提示词。
3. 附带当前页截图。
4. 调用当前页自己的 `SlideChatManager.SendMessageAsync`。
5. 保存生成出的 SlideML 和日志。
6. 更新页面状态为成功或失败。

后续再考虑限制并发处理。并发前需要确认每个渲染管道、MCP 连接和 UI 更新是否可以安全并行。

## 阶段三一致性校验在 Demo 中的落点

一致性校验可以先作为可选步骤，不建议放在打开文件夹时执行。

推荐位置：

- 批量美化完成后提供“检查一致性”按钮。
- 或在“开始全课件美化”流程末尾提供可配置开关。

输入应是生成后的所有 SlideML 渲染截图，以及全局主题结果。输出可以是每页的一致性评分、问题说明和是否需要返工。

第一版可以先不实现自动返工，只在 UI 中标记问题页，允许用户人工选择重新生成。

---

## 建议的第一版用户流程

建议点击“打开课件文件夹”后的第一版完整流程如下：

1. 用户点击“打开课件文件夹”。
2. 选择符合导出格式的课件目录。
3. 程序验证 `courseware.json`、页面 Markdown、截图和资源索引。
4. 程序加载全部页面 Markdown、截图路径和资源索引。
5. 程序清空默认页，按导出页面创建页面列表。
6. 程序选中第一页，并显示原始截图预览。
7. 状态显示“已加载 N 页，等待主题分析”。
8. 用户点击“开始全课件美化”或“分析全局主题”。
9. 程序基于全部 Markdown 执行全局主题分析。
10. 程序保存主题结果，并在 UI 上展示主题摘要。
11. 用户确认后批量逐页美化，或直接进入串行逐页美化。
12. 每页生成 SlideML、渲染预览、保存输出和日志。
13. 批量完成后，可选执行一致性校验。

---

## 与全课件美化设计文档的衔接

当前 Demo 的课件输入来源是 Markdown 导出文件夹，Markdown、截图和资源索引已经准备完成。因此，全课件美化的预备阶段应直接执行：

- 读取并验证导出目录。
- 加载已有 Markdown 快照。
- 加载已有资源索引。
- 从已有 Markdown 执行全局主题分析。

“打开课件文件夹”完成后，`CoursewareInputPackage` 即作为后续全局主题分析和逐页美化的统一输入边界，不再重复提取页面内容和资源。

---

## 后续实现建议

建议后续按以下顺序实现：

1. 新增课件导出目录读取模型和读取服务。
2. 扩展 `CoursewareSlideItem` 或新增页面输入模型，承载 Markdown、截图、尺寸、页面 ID 和输出状态。
3. 修改 `OpenCoursewareFolder`，从只记录路径升级为加载课件输入包并初始化页面列表。
4. 增加原始截图预览能力，让未生成 SlideML 的页面也能展示导出截图。
5. 增加全局主题分析服务，基于全部 Markdown 生成结构化主题。
6. 增加“开始全课件美化”命令，第一版串行逐页生成 SlideML。
7. 增加输出目录和日志目录规则。
8. 增加资源引用解析能力，确保生成的 SlideML 能正确显示导出资源。
9. 后续再加入一致性校验和返工机制。

---

## 关键风险

- 资源路径如果没有统一约定，LLM 生成的 SlideML 可能无法被本地渲染器解析。
- 打开文件夹时如果自动触发 LLM，误选目录会浪费调用，因此建议本地加载和 LLM 分析分离。
- 大课件的 Markdown 总量可能超过模型上下文，需要摘要、截断或抽样策略。
- 当前页面模型更偏向生成后状态，承载导出输入信息可能不够，需要扩展。
- 批量并发美化需要确认渲染管道和 MCP 调用的线程安全性，第一版建议串行。
- 如果输出目录规则不明确，后续日志、SlideML 文件和渲染截图容易混在输入目录中，建议输出到独立目录，避免污染导出源文件。
