## 一、实施目标

本次实施需要完成一次“删除旧主题分析，再建立精简主题分析”的破坏性重构。

最终目标：

```text
课件输入
  -> 字体 / 字号 / 颜色简单分布参考
  -> 全部页面原始 Markdown 自然语言拼接
  -> 独立主题 Agent 单工具提交
  -> 配色 / 中西文字体 / 风格 / 安全区 / 间距视觉效果 / 版式原则
  -> 封面和内容页两份 SlideML
  -> 本地解析与 WPF 渲染校验
  -> 同一主题会话多轮修复
  -> 同一份轻量结果进入首次创建的每页首轮草稿
```

必须删除的旧目标：

```text
复杂 JSON 分析信封
路径脱敏
源事实和分析视图指纹
几何 / 密度 / 风险 / 布局聚类
主题分析截图和视觉样本
CoursewareDesignSystem
设计令牌 / 组件 / 素材策略
动态页面类型
SlideId 到页面类型映射
证据和假设
13 个草稿工具及 DraftId / Revision
动态模板编译与压力样本
能力状态矩阵
页面工作台页面类型和组件消费
```

## 二、实施约束

### 2.1 删除优先

不得在旧 `CoursewareDesignSystem` 上增加“简化模式”或继续保留兼容适配器。

正确顺序是：

1. 明确保留基础设施；
2. 迁出少量共享技术代码；
3. 删除旧主题分析生产链和类型；
4. 修复所有编译断点，建立无旧语义的过渡基线；
5. 新建最小主题分析；
6. 接回 UI、快照和页面工作台。

### 2.2 不误删截图和资源加载

“主题分析不发送截图”不等于课件不再加载截图。

必须保留：

- `CoursewareExportSlideEntry.ScreenshotFile`；
- `CoursewareSlideInput.ScreenshotFile`；
- `CoursewareFolderLoader` 截图读取和缺失警告；
- 分析页缩略图；
- 页面工作台首轮原始截图附件；
- 页面聊天中用户图片附件；
- 快照复制截图和资源，供页面工作台恢复使用。

只删除主题 Agent 的：

- 视觉样本选择；
- 图片能力判断；
- `DataContent.LoadFromAsync` 截图附件；
- 视觉观察和视觉能力状态。

### 2.3 不误删本地文件安全校验

必须保留 `CoursewareFolderLoader` 对相对路径、`..` 和目录越界的检查。

删除 `CoursewareAnalysisSourceSnapshotBuilder` 前，还必须把其中以下非指纹校验迁入 `CoursewareFolderLoader`：

- `SlideIndex` 连续并与清单顺序一致；
- `SlideId` 唯一；
- 页面宽高为有限正数；
- Markdown 以“## 页面信息”开始；
- Markdown 中的 Id、序号和尺寸与 `Courseware.json` 一致。

这些是导出格式校验，不属于主题分析。迁移后删除 SourceSnapshot 和哈希，不得为了保留校验而继续构建分析快照。

只删除模型发送前的：

- 绝对路径扫描；
- 路径替换；
- 路径隐私诊断；
- 路径审计指纹；
- 因 Markdown 路径文本而阻止模型请求。

### 2.4 不用默认主题掩盖过渡错误

在删除旧主题分析和接入新主题分析之间，不允许：

- 自动构造固定主题；
- 返回空 `CoursewareTheme`；
- 保留旧 `CoursewareDesignSystemThemeAdapter` 作为 fallback；
- 让页面工作台在没有有效新主题时继续发送旧 Prompt。

每个阶段都必须建立真实、可解释的检查点。只有标记为“可合并提交”的检查点要求解决方案完整构建；删除过程中的内部检查点可以只要求 Core 构建通过并给出明确的预期下游编译断点，但不得把这种中间状态提交或合并。

### 2.5 测试要求

新增或重写的 MSTest：

- 使用中文 `DisplayName`；
- 对异步测试设置硬超时；
- 默认测试不访问真实模型和网络；
- 通过可注入聊天管理器或假模型结果验证会话和工具协议；
- 不以保留旧测试数量为目标，应删除与新产品语义冲突的断言。

## 五、阶段 1：迁出共享技术能力

本阶段必须先完成，否则删除旧输入构建器会破坏页面和主题上下文预算检查。

### 5.1 迁出 Token 估算器

#### 当前位置

- `CoursewarePptxGenerator.Core/Analysis/CoursewareAnalysisInputBuilder.cs`

#### 新位置建议

- `CoursewarePptxGenerator.Core/Analysis/CoursewareTokenEstimator.cs`

或更中性的：

- `CoursewarePptxGenerator.Core/Text/CoursewareTokenEstimator.cs`

#### 修改调用方

- `Services/CoursewareModelContextBudgetValidator.cs`；
- `Services/CoursewareSlideContextBudgetValidator.cs`；
- 后续新的主题 Agent 预算预检；
- 后续新的页面 Prompt Builder，如仍保留估算结果。

#### 验收

- 删除 `CoursewareAnalysisInputBuilder.cs` 后仍可进行上下文预算预检；
- Token 数不进入业务结果或用户 Prompt；
- 估算器无指纹、隐私和课件分析依赖。

### 5.2 抽出完整文档 SlideML 共享规范

#### 当前文件

- `PptxGenerator.Core/Prompt/ISlideMlPromptProvider.cs`；
- `PptxGenerator.Core/Prompt/SlideMlPromptProvider.cs`。

#### 修改

新增精确面向完整 `<Page>` 文档的规范 API，例如：

```text
BuildCompleteDocumentSpecificationPrompt()
```

重构现有：

```text
BuildSystemPrompt()
BuildStreamingSystemPrompt()
```

`BuildSystemPrompt()` 复用完整文档规范。`BuildStreamingSystemPrompt()` 只复用真正相同的基础定义，并继续追加流式片段、`Remove`、Id、`StyleFrom`、`StyleId`、合并和状态查询规则。

同时在 `PptxGenerator.Core` 中建立一个窄范围的 `SlideMlCompleteDocumentSchema`，集中声明完整文档允许的标签、父子层级、属性、必填项、枚举和值域。完整文档 Prompt 生成器和后续严格验证器共同消费该 Schema，避免维护两份规范。

#### 注意事项

- 不能复制一份规范到 WPF Demo；
- 页面 Agent 继续保留自己的工具行为要求；
- 主题 Agent 只复用语法规范，不继承 `render_slide` 或 `get_slide_state` 行为；
- 主题 Agent 不能接收流式专用属性和合并规则；
- 现有 PromptProvider 测试需证明完整文档 Prompt 与 Schema 一致，并单独证明流式 Prompt 保留自己的扩展；不能要求三种 Prompt 逐字相同。

#### 验收

- SlideML 标签、属性、颜色、坐标和 XML 格式规则只有一个维护位置；
- 页面生成现有行为不回归。

### 5.3 迁移课件输入格式校验

#### 当前来源

- `CoursewarePptxGenerator.Core/Analysis/CoursewareAnalysisSourceSnapshotBuilder.cs`。

#### 目标位置

- `CoursewareFolderLoader` 的 Manifest 和页面加载校验；
- 如方法过长，可新增只服务加载格式的内部 `CoursewareMarkdownPageInfoValidator`，但不得保留 Snapshot、Fact 或 Fingerprint 模型。

#### 迁移内容

- 清单页顺序连续；
- `SlideId` 唯一；
- 有限正数尺寸；
- Markdown 页面信息章节；
- Id、序号和尺寸交叉核对。

#### 不迁移内容

- 课件名、资源、警告和 Markdown 的哈希；
- `CoursewareAnalysisSourceSnapshot`；
- `CoursewareAnalysisSlideFact`；
- `CoursewareAnalysisResourceFact`；
- 截图存在状态参与指纹；
- 任何算法版本或审计字段。

#### 测试

将对应一致性测试迁移到 `CoursewareFolderLoaderTests.cs`，并继续验证截图缺失只产生警告。

## 十一、阶段 7：实现 SlideML 共享规范与真实校验

### 11.1 新增主题 SlideML 验证器

建议文件：

- `Services/CoursewareThemeSlideMlValidator.cs`；
- 可选 `Models/CoursewareThemeSlideMlValidationResult.cs`。

依赖：

- `SlideMlRenderPipeline`；
- `SlideMlLayoutEngine`；
- `WpfSlideMlRenderEngine`；
- `WpfDispatcher`；
- `SlideMlPipelineContext`；
- 由主题分析服务从 `CoursewareInputPackage.Slides[0]` 创建的临时校验画布。

### 11.2 校验流程

对封面和内容页分别：

1. 非空检查；
2. `ExtractXml`；
3. `NormalizeXml`；
4. 使用 `XDocument` 和 `SlideMlCompleteDocumentSchema` 校验标签、父子层级、属性、必填项、枚举和值域；
5. 按元素和属性的 `LocalName` 拒绝：
   - `Shadow` 属性；
   - `<Shadow>` 元素；
   - 任意 `<Image>` 元素；
   - `RenderSize`、`RenderLocation`、`ActualLineCount`；
   - token、slot、`StyleFrom`、`StyleId`、`Remove` 和其他流式专用结构；
6. 严格校验颜色、有限数值、非负尺寸和字号、Opacity、渐变 Offset 等值域；
7. 调用真实本地 `RenderAsync`；
8. 捕获 XML、解析、布局、测量和 WPF 渲染异常并转换为字段诊断；
9. 将任何 Error 视为失败；
10. 将任何 Warning 视为失败；
11. 成功时保留规范化输入 XML；
12. 比较两份规范化 XML，完全相同则失败。

### 11.3 为什么 Warning 阻断

当前解析器对未知标签和属性常用 Warning 并忽略，而不是抛异常。若只检查 Error，非法 SlideML 可能被“容错渲染”后错误地视为合法。

WPF 颜色转换还可能对非法颜色使用透明色或默认色，因此严格 Schema/值域校验必须发生在渲染之前，零 Warning 不能单独作为合法性证明。

主题参考结构只有两份，要求零 Warning 是合理且可维护的门槛。

### 11.4 不发送预览图

验证器只在本地使用预览以完成 WPF 渲染，不将图片提供给主题 Agent，也不加入 Copilot 消息。

反馈给模型的只是文本诊断。

### 11.5 测试

使用真实或可替换渲染管线覆盖：

- 合法页面；
- XML 错误；
- 错误根节点；
- 未知标签；
- 未知属性；
- 阴影属性；
- Shadow 子元素；
- 任意 Image 元素；
- 非法颜色；
- 非有限值、负字号、负尺寸和非法透明度；
- token、slot、StyleFrom、StyleId、Remove 和回填属性；
- 布局越界 Warning；
- 两份相同页面；
- 输出不包含回填属性。

### 11.6 验收

只有两份参考 SlideML 都通过共享完整文档 Schema、严格 XML/语义校验和同一真实本地渲染管线，主题提交工具才成功。

7. 两份 SlideML 本地校验通过；
8. UI 展示新主题结果；
9. 进入页面工作台；
10. 首次创建工作台时，任一页面输入框显示同一主题结果和两份参考 SlideML；
11. 原始当前页截图仍在可见附件区；
12. 页面生成、追问和重新渲染正常。

### 21.2 SlideML 修复路径

1. 模型首次提交非法封面 SlideML；
2. 工具返回精简诊断；
3. UI 显示工具失败；
4. Agent 在同一会话发送修复要求；
5. 修复消息不重复课件 Markdown；
6. 模型再次调用同一工具；
7. 两份 SlideML 通过后才成功。

### 21.3 多轮失败路径

1. 连续四轮仍未提交有效结果；
2. 主题分析状态变为失败；
3. 已加载课件和缩略图保留；
4. 不创建新页面工作台或不替换已有有效主题；
5. 用户可重新分析；

1. 已有页面工作台；
2. 部分页面未开始且 Prompt 未编辑；

4. 部分页面已开始对话；
5. 重新分析成功；
6. 只刷新第一类页面；
7. 保留用户编辑和已有页面会话；
8. 后两类页面不宣称已经应用最新主题。
