# CoursewarePptxGeneratorWpfDemo 统一双比例缩略图显示方案

> 状态：方案设计阶段，尚未开发。
>
> 本文只描述截图缩略图在 WPF 界面中的显示适配，不包含代码改动。
>
> 本方案不会读取、修改或推导课件页面的逻辑尺寸，不影响课件加载、页面模型、生成、渲染或导出逻辑。

---

## 一、背景与目标

全课件概览当前把缩略图卡片固定为 16:9，并使用 `UniformToFill` 显示截图。该方式会为了填满固定外框而裁剪非 16:9 图片。

同一份课件导出的页面截图预期具有相同图片比例，因此没有必要为每一张截图单独执行比例判断。本次调整只解决界面显示问题：

1. 只取当前课件缩略图集合中按页面顺序第一张非空截图路径作为代表。
2. 由页面级纯 View 状态只解码该代表截图一次，读取 `PixelWidth/PixelHeight`。
3. 将代表截图比例映射到 16:9 或 4:3。
4. 当前课件的所有缩略图统一使用该外框比例。
5. 每张图片仍使用 `Uniform` 完整显示，实际比例存在偏差时允许黑边。

这里的“一次解码”是指整个 `CoursewareAnalysisView` 对当前课件只为比例探测解码一张代表截图。每张截图为了实际显示仍会由现有图片加载链路正常解码，但不再分别参与外框比例判断。

## 二、设计边界

### 2.1 本方案包含

- 在 `CoursewareAnalysisView` 的纯 View 范围内寻找第一张可用截图。
- 只解码该代表截图一次进行比例探测。
- 在 16:9 和 4:3 两种统一外框之间选择。
- 将探测结果应用到当前概览中的所有缩略图。
- 图片使用 `Uniform`，允许黑边，不裁剪内容。
- 第一张代表截图加载失败时的默认比例降级。
- XAML、界面行为和自动化测试方案。

### 2.2 本方案明确不影响

- `Courseware.json` 中的页面尺寸。
- `CoursewareSlideInput.Width/Height`。
- `CoursewareThumbnailItemViewModel.Width/Height`。
- `CoursewareInputPackage`。
- `CoursewareWorkspaceSession`。
- 课件加载校验。
- 页面生成、渲染和导出尺寸。
- 全局主题分析或单页美化逻辑。

代表截图只决定缩略图控件的统一视觉外框，不代表或修正课件页面的业务比例。

### 2.3 本方案不包含

- 不修改课件 Markdown 导出格式。
- 不建立业务层课件比例模型。
- 不逐张比较截图比例。
- 不为每个缩略图创建独立比例结果。
- 不新增逐图 `FilePathToThumbnailImageConverter`。
- 不重新生成、裁剪或写回截图文件。
- 不在本方案中实施代码。

## 三、现有实现探索结论

### 3.1 当前截图加载链路

`CoursewareThumbnailItemViewModel` 向界面提供 `ScreenshotFilePath`。

`Views/Converters/FilePathToImageSourceConverter.cs` 当前负责：

1. 检查文件路径。
2. 创建 `BitmapImage`。
3. 使用 `BitmapCacheOption.OnLoad` 完成图片加载。
4. 冻结并返回 `BitmapImage`。

当前单项显示链路为：

```text
ScreenshotFilePath
  -> FilePathToImageSourceConverter
  -> BitmapImage
  -> Image.Source
```

每张截图为了显示本身都会经过该链路。这是正常图片显示成本，本方案不改变这一点。

### 3.2 当前比例处理缺失

`CoursewareThumbnailOverview.xaml` 当前固定使用：

- 卡片宽度：`180`。
- 卡片高度：`101.25`。
- `Image.Stretch="UniformToFill"`。
- 横向 `WrapPanel`。

当前界面没有课件概览级的统一显示比例状态，也没有在第一张图片加载完成后调整全部缩略图外框的机制。

### 3.3 多个概览实例

`CoursewareAnalysisView.xaml` 在多个流程状态中声明了四个 `CoursewareThumbnailOverview`。如果让每个实例独立探测，同一课件最多会重复执行四次比例判断，不符合“一份课件只处理一次”的目标。

因此，统一外框高度应由共同的 `CoursewareAnalysisView` 持有。四个 Overview 只绑定同一个页面级纯 View 属性，不各自解码代表图，也不各自维护比例状态。

## 四、核心设计决策

### 4.1 第一张可用截图作为代表

比例探测按缩略图集合顺序进行：

1. 从第一项开始查找第一张 `ScreenshotFilePath` 非空的项。
2. 在集合本轮更新稳定后，由页面级探测逻辑只尝试解码该代表文件。
3. 如果图片解码成功，使用其 `BitmapSource.PixelWidth/PixelHeight` 决定统一外框。
4. 如果文件不存在、解码失败或像素尺寸无效，直接使用 16:9 作为纯界面默认外框。
5. 不继续扫描或解码后续图片。

这里的“第一张图片”准确含义是“按页面顺序第一张具有非空截图路径的页面截图”。若该代表文件不可用，方案接受默认 16:9 降级，以确保每份课件的比例探测最多只处理一个文件。

### 4.2 比例判断只解码第一张代表截图

比例探测由 `CoursewareAnalysisView` 统一执行：

```text
第一张可用 ScreenshotFilePath
  -> 使用 BitmapCacheOption.OnLoad 解码一次
  -> BitmapSource.PixelWidth / PixelHeight
  -> 计算统一外框比例
```

禁止对每个缩略图执行该流程，也禁止增加逐图比例 Converter。

比例探测解码与缩略图正常显示解码的职责不同：

- 页面级探测只解码一张代表截图，得到统一高度。
- 现有 `FilePathToImageSourceConverter` 仍负责每张图片的正常显示。
- 代表截图后续为显示可能再次经过现有 Converter；如果实测需要避免这一次重复，可让现有图片加载层复用冻结后的 `BitmapImage`，但不能演变为逐图比例处理。
- 其他截图只负责自身显示，不参与比例判断。

本方案所说的“只处理第一张图片”专指比例识别。所有缩略图要显示出来，仍然必须加载各自图片，这是界面显示的必要行为。

### 4.3 比例状态属于 CoursewareAnalysisView

统一外框比例是当前分析页面的共享视觉状态，建议由 `CoursewareAnalysisView.xaml.cs` 的依赖属性或控件内部状态承载。

建议属性：

- `double ThumbnailFrameHeight`

其值只可能为：

- `101.25`：16:9。
- `135`：4:3。

初始值使用 `101.25`。代表图片成功加载后，如果判断为 4:3，则更新为 `135`。四个 `CoursewareThumbnailOverview` 都绑定该页面级属性，再由每个 Overview 把统一高度传给内部模板。

该属性：

- 不进入业务 ViewModel。
- 不写入课件会话。
- 不作为课件数据公开。
- 不触发任何课件业务状态变化。

### 4.4 16:9 与 4:3 映射规则

代表截图比例为：

```text
ImageAspectRatio = PixelWidth / PixelHeight
```

目标比例：

```text
16:9 = 1.777...
4:3 = 1.333...
```

两个目标比例的中点为 `14:9`。距离相同时选择 4:3：

```text
ImageAspectRatio > 14 / 9
  -> 16:9
否则
  -> 4:3
```

由于像素宽高是正整数，可使用整数交叉乘法：

```text
9 × PixelWidth > 14 × PixelHeight
  -> 16:9
否则
  -> 4:3
```

实现时使用足够宽的整数类型保存中间乘积。

### 4.5 所有缩略图统一使用同一外框

探测完成后，当前 `CoursewareThumbnailOverview` 中的所有缩略图统一使用同一个 `ThumbnailFrameHeight`。

| 代表截图结果 | 所有缩略图内容外框 |
| --- | --- |
| 16:9 | `180 × 101.25` |
| 4:3 | `180 × 135` |
| 无可用截图 | 默认 `180 × 101.25` |

不再需要每项独立选择外框，也不需要为 16:9 与 4:3 混排设计不同高度槽位。

### 4.6 图片完整显示并允许黑边

所有图片统一使用：

```text
Stretch = Uniform
```

外框内使用主题资源定义的黑色或接近黑色背景。

如果某张截图实际比例与代表截图不同，或者与标准 16:9、4:3 略有偏差，该图片会在统一外框内产生黑边，但内容保持完整。

这是有意的降级策略：同一课件预期截图比例一致，因此不为异常单页改变整个布局，也不裁剪异常图片。

### 4.7 边框不改变内容比例

标准尺寸作用于图片内容区域。装饰边框不应挤占 `180 × 101.25` 或 `180 × 135` 的内容空间。

建议使用覆盖结构：

```text
180 × ThumbnailFrameHeight 的 Grid
  -> 黑边背景和 Image 铺满
  -> 装饰 Border 作为覆盖层，不参与内容测量
```

圆角需要使用实际可裁切图片的几何裁切、遮罩或项目中已有的圆角图片容器验证，不能只依赖 `Border.CornerRadius` 和 `ClipToBounds`。

## 五、建议 View 实现

### 5.1 CoursewareAnalysisView 页面级状态

建议在 `CoursewareAnalysisView.xaml.cs` 中维护：

- `ThumbnailFrameHeight` 依赖属性或只读依赖属性。
- 是否已经完成当前课件集合的代表图片探测。
- 当前探测代次，用于丢弃旧集合的迟到任务。
- 当前代表图片的文件版本标识。

该状态只服务当前页面实例，由四个 Overview 共享，不进入业务 ViewModel。

### 5.2 代表图片探测方式

当前 `FilePathToImageSourceConverter` 使用 `BitmapCacheOption.OnLoad`，在 `Convert` 的 `EndInit()` 中同步完成文件读取。`Image.Loaded` 只表示控件进入可视树，不表示解码成功，因此比例探测不依赖模板图片的 `Loaded` 顺序。

推荐由页面级代码按以下方式探测：

1. 监听当前 `CoursewareThumbnails` 集合的 `INotifyCollectionChanged.CollectionChanged`。
2. 集合发生 `Clear/Add` 时只安排一次 Dispatcher 合并任务，不逐次立即探测。
3. Dispatcher 任务执行时，按 ItemsControl 实际 Items 视图的索引顺序遍历 `ScreenshotFilePath`。
4. 只对第一张非空路径使用 `BitmapCacheOption.OnLoad` 解码。
5. 解码成功且像素尺寸有效时，只执行一次比例计算。
6. 路径不存在、解码异常或像素尺寸无效时直接保持默认 16:9，不尝试后续项。
7. 集合中没有非空截图路径时保持默认 16:9。
8. 使用探测代次确认结果仍属于当前集合，再更新共享高度。

“按页面顺序”在当前实现中指 ItemsControl 实际 Items 视图的索引顺序，不按图片加载速度、文件名、路径或容器生成顺序选择。

### 5.3 ItemsSource 变化

`CoursewareThumbnails` 是同一个 `ObservableCollection`，加载另一份课件时通过 `Clear()` 后连续 `Add()` 更新，并非简单替换 ItemsSource。

因此需要：

- 高度先回到默认 16:9。
- 合并同一批 `Clear/Add` 通知，集合稳定后只开始一轮探测。
- 增加探测代次，旧任务不得覆盖新课件结果。
- DataContext 或 ItemsSource 改变时解除旧集合订阅，避免控件泄漏。
- 一轮比例确定后，集合尾部普通追加或图片控件布局变化不重复计算；只有明确的新课件加载轮次才重置。

该重置只影响控件显示，不通知或修改业务 ViewModel。

### 5.4 多个概览实例

四个 `CoursewareThumbnailOverview` 不执行比例探测，只消费 `CoursewareAnalysisView.ThumbnailFrameHeight`。因此当前页面对一份课件只运行一轮比例识别。

四个 Overview 当前可能各自为显示创建图片实例。这属于现有多视图结构带来的正常显示解码，页面级比例状态不能解决该问题。若产品未来要求同一截图在整个页面只正常解码一次，应改为单个 Overview 在不同状态间复用或延迟创建不可见状态内容；该优化独立于本方案。

### 5.5 不新增逐图显示结果类型

本方案不需要新增：

- `ThumbnailImageSource`。
- `ThumbnailImageLoadState`。
- `FilePathToThumbnailImageConverter`。
- 每个缩略图的 `FrameHeight` 状态。

现有 `FilePathToImageSourceConverter` 继续只负责把路径转换为 `ImageSource`。

无截图状态继续依据现有 ViewModel 属性显示。代表图片路径非空但实际解码失败时直接使用默认 16:9，不尝试其他图片；是否为该单项增加“加载失败”占位属于独立的图片错误体验改进，不是统一比例方案的必要前置。

### 5.6 不修改加载和业务模型

本方案不需要修改：

- `CoursewareFolderLoader`。
- `CoursewareSlideInput`。
- `CoursewareInputPackage`。
- `CoursewareWorkspaceSession`。
- `CoursewareThumbnailItemViewModel` 的尺寸逻辑。
- `CoursewareWorkspaceViewModel`。
- 页面生成、渲染和导出服务。

## 六、界面数据流

建议数据流如下：

```text
CoursewareThumbnails ItemsSource
  -> 合并本轮 Clear/Add 集合通知
  -> 按 Items 视图顺序取得第一张非空 ScreenshotFilePath
  -> 页面级探测只解码该代表文件一次
  -> 读取 BitmapSource.PixelWidth / PixelHeight
  -> 计算统一 ThumbnailFrameHeight
  -> 四个 Overview 中所有模板绑定同一页面级高度
```

其他截图的数据流保持不变：

```text
ScreenshotFilePath
  -> 现有 FilePathToImageSourceConverter
  -> Image.Source
  -> 在统一外框内使用 Uniform 显示
```

比例探测不会为其他图片执行任何额外处理。

## 七、XAML 设计

### 7.1 统一高度绑定

四个 `CoursewareThumbnailOverview` 先绑定页面级统一属性，再由 Overview 内部所有缩略图绑定控件属性：

```text
CoursewareAnalysisView.ThumbnailFrameHeight
  -> CoursewareThumbnailOverview.ThumbnailFrameHeight
  -> DataTemplate 外框 Height
```

不绑定单项 ViewModel 的 `Width/Height`，也不为每项调用比例 Converter。

### 7.2 统一外框

每项使用：

- 内容宽度：`180`。
- 内容高度：绑定统一 `ThumbnailFrameHeight`。
- 项间距：`0,0,8,8`。

由于当前概览内所有项高度相同，不需要固定 4:3 槽位再垂直居中 16:9 外框。`WrapPanel` 会自然保持整齐行高。

### 7.3 图片区域

图片区域使用：

- 现有 `FilePathToImageSourceConverter`。
- `Image.Stretch="Uniform"`。
- 主题化黑边背景。
- 覆盖式装饰边框。
- 有效的圆角裁切。

视觉资源集中定义：

- 黑边背景 Brush。
- 边框 Brush。
- 圆角。
- 边框厚度。
- 项间距。

### 7.4 无截图和代表图失败

单项无截图继续显示现有“无截图”占位内容。

代表图探测失败时：

1. 不尝试后续截图。
2. 统一使用默认 16:9 外框。
3. 不修改任何业务警告或页面状态。

### 7.5 宿主高度

统一外框可能为：

- 16:9：`101.25` 高。
- 4:3：`135` 高。

“正在分析主题”状态的内层 `ScrollViewer.MaxHeight` 需要能够容纳最坏情况下的一行 4:3 缩略图：

```text
135 + 8 底部间距 = 143
```

`143` 是理论最小值，最终值结合 ScrollViewer 模板和 DPI 实测确定。

## 九、测试方案

### 9.1 比例判断单元测试

将像素尺寸到外框高度的纯函数提取为 `Views` 范围内的内部辅助方法，覆盖：

1. 16:9 返回 `101.25`。
2. 4:3 返回 `135`。
3. 16:10 映射到 16:9。
4. 3:2 映射到 4:3。
5. 21:9 映射到 16:9。
6. 1:1 和纵向图片映射到 4:3。
7. 14:9 映射到 4:3。
8. 略宽于 14:9 映射到 16:9。
9. 等比例缩放后的像素尺寸得到相同结果。

### 9.2 代表图探测测试

建议通过 STA 界面测试覆盖：

1. 第一项截图有效时只使用第一项作为代表。
2. 第一项路径为空时使用下一张非空路径作为代表。
3. 第一张非空路径对应文件不存在或解码失败时直接使用默认 16:9，不解码下一张。
4. 第一张非空路径截图成功解码且为 4:3 时，所有项统一绑定 `135`。
5. 第一张非空路径截图成功解码且为 16:9 时，所有项统一绑定 `101.25`。
6. 后续截图比例不同也不会改变已经确定的统一高度。
7. 后续图片控件 Loaded 或布局事件不会执行比例计算。
8. 集合没有非空截图路径时保持默认 16:9。
9. 同一个 `ObservableCollection` 执行 `Clear` 后连续 `Add` 只产生一轮最终比例判断。
10. 旧集合探测任务晚于新集合时，不能覆盖新课件比例。
11. DataContext 或 ItemsSource 变化后解除旧集合订阅。
12. 四个 Overview 同时存在时，页面级比例只判断一次。
13. 集合尾部普通追加、后续图片加载和模板重建不重复计算。
14. 路径不存在、解码异常和像素尺寸无效时均直接降级，不推进到下一候选。
15. 页面级比例探测最多创建一张代表图片的 `BitmapImage`。

### 9.3 XAML 与界面自动化测试

建议验证：

- `Image.Stretch` 为 `Uniform`。
- 所有外框高度最终绑定同一个 `CoursewareAnalysisView` 页面级属性。
- 模板不使用单项 `Width/Height` 决定外框。
- 模板不包含逐图比例 Converter。
- 统一 16:9 和统一 4:3 两种状态下 `WrapPanel` 行布局整齐。
- 警告标记位于实际外框内部。
- 无截图占位保持可见。

### 9.4 人工验证

建议验证：

- 第一张代表截图为 16:9 时，全部缩略图统一为 16:9。
- 第一张代表截图为 4:3 时，全部缩略图统一为 4:3。
- 后续存在异常比例截图时只产生黑边，不改变整体布局。
- 图片内容没有被裁剪。
- 125%、150% 和 175% DPI 下首行不裁切。
- 打开另一份不同比例课件后正确重新探测。

## 十二、风险与取舍

### 12.1 假设同一课件截图比例一致

方案明确接受该产品假设。后续截图比例异常时，不重新计算外框，只使用 `Uniform` 和黑边完整显示。

### 12.2 第一张代表截图可能不可用

“第一张”采用页面顺序中第一张非空截图路径。若该文件缺失或损坏，直接回退到默认 16:9，不扫描后续截图。这是为了严格保证比例探测最多只处理一张图片。

### 12.3 必须保持页面顺序语义

实现必须按 ItemsControl 实际 Items 视图顺序选择候选，不能按文件名、路径或容器生成顺序选择。

### 12.4 多个概览实例仍可能重复正常显示解码

页面级共享状态确保比例只判断一次，但四个静态 Overview 仍可能各自通过现有 Converter 加载图片。这是当前多视图结构的正常显示成本，不是逐图比例处理。若要消除，应另行改造为单个 Overview 复用或延迟创建不可见状态内容。

### 12.5 当前方案依赖非虚拟化布局

当前 ItemsControl 使用普通 `WrapPanel`，全部项目会生成绑定。页面级探测直接基于 ItemsSource 文件路径，不依赖视觉容器，因此即使后续改为虚拟化，也仍能按页面顺序找到代表截图；但探测仍只允许解码第一张成功候选，不能扩展为逐图比例分析。

### 12.6 比例变化会触发布局更新

控件初始使用 16:9，若代表截图最终判断为 4:3，所有缩略图会统一增高并重新布局一次。这是预期的纯界面变化。

## 十三、最终建议

推荐采用以下落地方案：

1. 只使用当前课件第一张非空截图路径作为比例代表。
2. 由 `CoursewareAnalysisView` 页面级逻辑只解码该代表截图一次，不逐张判断。
3. 不逐张判断截图比例，不新增逐图 Converter 或结果模型。
4. 代表截图像素比例大于 `14:9` 时统一选择 16:9，否则统一选择 4:3。
5. 所有缩略图统一使用 `180 × 101.25` 或 `180 × 135` 外框。
6. 每张图片使用 `Uniform` 和主题化黑边背景，异常比例只显示黑边，不改变整体布局。
7. 比例状态只保存在 `CoursewareAnalysisView` 页面内部，并由四个 Overview 共享。
8. 打开另一份课件后重置并重新探测第一张可用截图。
9. 不修改加载器、课件模型、会话、ViewModel 业务状态以及生成导出逻辑。

该方案对当前课件只运行一轮比例探测，并且只成功解码一张代表截图用于判断。其他截图仅执行原本就需要的图片显示加载，并统一复用代表图确定的外框比例。
