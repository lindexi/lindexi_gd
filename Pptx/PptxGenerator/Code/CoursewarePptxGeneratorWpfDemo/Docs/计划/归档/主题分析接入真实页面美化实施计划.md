# CoursewarePptxGeneratorWpfDemo 主题分析接入真实页面美化实施计划

> 状态：Gate A 已完成并归档。真实页面、懒运行时、实际画布、结构化主题、截图附件和单页生成编辑闭环均已接入；Gate B、Gate C 等后续能力已整理到 [`../CoursewarePptxGeneratorWpfDemo未完成事项.md`](../CoursewarePptxGeneratorWpfDemo未完成事项.md)。
>
> 本文承接 [`../../主题分析输出与原始设计偏差审查.md`](../../主题分析输出与原始设计偏差审查.md) 的结论，聚焦一个明确目标：让已经加载并完成主题分析的真实课件进入真实单页美化、SlideML 生成和渲染链路。
>
> 本文不把“接通旧工作台”误写成“完整设计系统已经落地”。当前活动分析结果仍是 `CoursewareTheme` 主题建议；`CoursewareDesignSystem` 可执行设计系统目前只有 Core DTO 和 JSON 元数据。两者的阶段边界、能力状态和最终验收口径必须分别表达。

## 一、结论与推荐路径

当前主题分析完成后的真实调用链停在：

```text
CoursewareWorkspaceViewModel.PublishAnalysisResult
  -> CoursewareWorkspaceSession.ThemeAnalysisResult
  -> ApplyThemePresentation
  -> AnalysisReady
```

之后执行 `EnterWorkspaceCommand` 时，代码只切换 `PrototypePage`。第二页仍读取构造期创建的 60 页演示集合、静态预览、伪 SlideML 和演示聊天，没有使用刚刚加载的 `CoursewareInputPackage`，也没有创建真实页面会话。

项目中另有一条未启用的真实单页链路：

```text
MainWindowViewModel
  -> CoursewareFolderLoader
  -> 为每页创建独立 SlideChatManager
  -> 准备当前页 Markdown 和原始截图
  -> SlideGenerationPipeline
  -> SlideML 流式生成
  -> WPF / MCP 渲染
```

该链路可以复用，但不能原样恢复为应用根 ViewModel。推荐采用以下路径：

1. **保留 `CoursewareWorkspaceViewModel` 作为唯一应用壳和导航状态源。**
   - 不把应用启动入口切回 `MainWindowViewModel`。
   - 不让第二页重新加载一次课件，也不维护第二份 `CoursewareInputPackage`。

2. **新增独立的真实工作台子 ViewModel。**
   - 推荐命名为 `CoursewareSlideWorkspaceViewModel`。
   - 由它管理真实页面列表、当前页、页面会话、生成命令、渲染命令和工作台状态。
   - `CoursewareWorkspaceViewModel` 只负责课件级会话、主题分析、页面导航和工作台生命周期。

3. **用可观察页面 ViewModel 替换 `CoursewareSlideItem` 和演示页模型。**
   - 推荐命名为 `CoursewareSlideItemViewModel`。
   - 页面输入保持只读，生成状态、SlideML、预览、错误、警告和附件均为页面级可观察状态。

4. **首个里程碑先完成“真实选中页美化闭环”。**
   - 用户选择真实页面后手工触发生成。
   - 使用真实 Markdown、真实页面尺寸和当前主题结果。
   - 原始截图文件存在时默认随首次生成请求发送，不为本期新增模型图片能力判断。
   - 继续复用现有单页生成、流式预览和重新渲染逻辑，不在本期重写其质量判断和结果传播机制。
   - 首期不直接启动全课件并行或自动批量生成。

5. **当前 `CoursewareTheme` 主题建议只作为过渡输入，不冒充可执行设计系统。**
   - 首期通过明确的适配边界把 `CoursewareTheme` 结构化字段提供给每页生成请求。
   - `DesignSystem` 和 `TemplateValidation` 能力继续保持 `NotSupported`。
   - 后续完整的 `CoursewareDesignSystem` 通过同一消费边界替换当前主题适配器，而不是再重写工作台。

6. **明确本期不改造的底层能力。**
   - `ResourceId` 到 WPF 本地渲染素材的解析仍是已知问题，转入后续改造，不作为本期真实工作台接入的前置条件。
   - MCP 远端服务按素材名解析已知素材；本计划不探索或修改其素材协议。
   - `SlideMlRenderTool` 的错误保存和 UI 回填时序保持现状；本期只复用现有单页行为。
   - 通用 `SlideMlPromptProvider` 中的默认配色、字号 Token 和占位资源建议不在本期修改。

### 1.1 核心实施原则

本计划实施时必须遵守以下原则：

- **禁止异步转同步。** 不得使用 `.Result`、`.Wait()`、`GetAwaiter().GetResult()` 或其他阻塞等待把异步流程转换为同步流程。优先使用可等待的异步命令；现有入口暂时无法直接等待时，宁可保留受控的 fire-and-forget，并在异步边界捕获异常、回填状态，也不能阻塞 UI 线程。
- **设计时与运行时使用同一类型。** 不得为设计器另建一套页面或聊天 ViewModel。可以删除设计时演示数据，也可以用真实运行时类型构造设计样例；如果无法兼顾，宁可放弃设计时预览体验。
- **复用现有单页逻辑。** 每页美化继续使用当前单页生成、聊天、流式预览和重新渲染链路。本期只完成真实课件页面和主题结果的接入，不顺带重写通用生成器、渲染器或质量门。
- **默认发送现有截图。** 页面原始截图存在时直接作为首次请求附件发送；除非复用代码已经具有能力判断，否则不新增模型能力探测和文本降级分支。
- **不引入无必要的运行标识。** 本期不新增主题指纹、Prompt 指纹、页面输入指纹、SessionId、AnalysisRunId 或 GenerationRunId。页面与当前课件的归属由对象生命周期、取消和替换关系管理。
- **不改流式工具机制。** 保持现有流式页面工具白名单及其注册方式，不追加默认工具，也不重构这部分机制。
- **名称表达完整。** 文档使用 `CoursewareTheme` 主题建议和 `CoursewareDesignSystem` 可执行设计系统等完整名称，不使用脱离对象语境的版本编号或方案代号。

## 二、“真实页面美化”的阶段定义

### 2.1 首个真实闭环必须具备

只有同时满足以下条件，才可以把第二页从“工作台原型”改称“单页美化工作台”：

- 页面列表来自当前 `CoursewareWorkspaceSession.InputPackage.Slides`。
- 页面数量、顺序、`SlideId`、页码、尺寸和截图均来自真实输入。
- 每页拥有独立的生成会话和渲染状态，切页不会混用聊天历史。
- 初次生成请求消费当前有效主题的结构化字段，而不是只拼接 UI 文案。
- 当前页完整 Markdown 进入页面生成输入，不静默截断。
- 构建请求时不主动拼接资源和截图的本地绝对路径；若原始 Markdown 中存在路径，只做不增加结构复杂度的简单处理。
- 页面使用真实画布尺寸创建 `SlideDocumentContext` 和渲染上下文。
- 原始截图文件存在时以二进制附件发送到首次流式请求。
- 生成结果能够形成合法 `<Page>` 根节点的 SlideML。
- 当前页生成结果、编辑后的 SlideML、现有渲染预览和聊天在切换页面后仍保留。
- 当前页失败、取消或模型初始化失败不会清空其他页面已有结果。

### 2.2 首个真实闭环不等于完整产品化

完成上述闭环后，仍然不能宣称以下能力已完成：

- 可执行 `CoursewareDesignSystem` 设计系统。
- 动态页面类型与 100% SlideId 类型覆盖。
- SlideML 模板编译和压力测试。
- 全课件视觉分析。
- 页面拆分、合并或重排。
- 全课件批量生成全部通过。
- 跨页一致性自动评审和返工。
- 结果持久化、恢复和事务式提交。
- 最终 `.pptx` 组装与导出。

### 2.3 最终产品化验收

最终产品仍应以 [`../../../../../Docs/设计文档/全课件主题分析输入与设计系统产品化方案.md`](../../../../../Docs/设计文档/全课件主题分析输入与设计系统产品化方案.md) 为目标：冻结设计系统、动态页面类型、模板、资源策略、逐页执行契约、质量门、持久化和恢复均真实执行后，才能把全课件页面生成能力标记为 `Passed`。

## 三、当前代码断点

### 3.1 活动工作台仍是演示数据

`ViewModels/CoursewareWorkspaceViewModel.cs` 当前具有以下原型逻辑：

- `DemoSlideCount = 60`。
- 构造函数执行 `Slides = new ObservableCollection<DemoSlideViewModel>(CreateSlides())`。
- `SelectedSlideXml` 生成 `<Slide><Title/><Content/></Slide>`，不符合当前解析器要求的 `<Page>` 根元素和元素白名单。
- `RenderingLog`、`ChatMessages` 和页面预览均为固定演示内容。
- `EnterWorkspaceCommand` 只执行 `CurrentPage = PrototypePage.SlideWorkspace`。

因此主题分析结果和真实课件页面都没有进入第二页。

### 3.2 未启用的真实单页链路可复用但存在缺陷

`ViewModels/MainWindowViewModel.cs` 已包含以下可复用能力：

- 从 `CoursewareInputPackage.Slides` 创建页面列表。
- 为每页创建独立 `SlideChatManager`。
- 选择页面时准备 Markdown、截图附件和生成提示词。
- 支持流式生成、继续聊天、编辑 SlideML 和重新渲染。
- 支持本地渲染和 MCP 切换。

但它不能直接成为新根 ViewModel，原因包括：

- 自己再次调用 `CoursewareFolderLoader`，会产生第二份课件根状态。
- 构造时先创建一个空页面，与当前双页入口冲突。
- 不接收 `CoursewareWorkspaceSession` 和主题结果。
- `CreateBeautifyPrompt` 只有当前页 Markdown 和尺寸，没有全局主题、相邻摘要或资源约束。
- 使用单一 `IsBusy` 表达加载、生成、渲染和 MCP 等不同操作。
- `CoursewareSlideItem.SlideMl`、`RenderingLog` 和 `CallbackXml` 是只读初始化值；生成后没有回写页面对象。
- 切换页面时 `SyncSelectedSlideContext` 会重新读取初始 `SlideMl`，已生成内容可能被重置为空。
- 页面会话创建失败时用 fallback 继续显示，但 UI 无法准确区分“本地可渲染”和“模型可生成”。
- 一次为所有页面创建完整生成模型、评估模型和渲染管线，且每页尝试后台连接 MCP，规模较大时初始化成本和失败面过大。

推荐迁移其有效逻辑，随后删除或归档重复实现，不保留两条可运行的页面工作流。

### 3.3 页面尺寸仍固定为默认值

`Services/SlideChatManagerFactory.cs` 当前创建：

```text
SlideMlRenderPipeline(
  SlideMlLayoutEngine,
  WpfSlideMlRenderEngine,
  dispatcher)
```

没有传入页面级 `SlideDocumentContext`，因此默认画布始终为 `1280 × 720`。同时：

- `SlideMlPromptProvider` 使用默认画布。
- `WpfSlideMlRenderEngine` 按默认上下文创建预览图。
- `Views/MainContentPanel.xaml` 的预览容器也固定为 `1280 × 720`。

真实工作台必须让提示词、布局、渲染和 UI 预览使用同一份页面级画布描述。

### 3.4 原始截图没有进入当前流式请求

`MainWindowViewModel.SendMessageAsync` 会把 `AttachedImageFiles` 传给：

```text
SlideChatManager.SendMessageAsync(... useStreaming: true)
```

但 `SlideGenerationPipeline.SendMessageAsync` 的流式分支只把 `attachPreview` 传给 `StreamingSlideGenerator.GenerateAsync`，没有传递 `attachedImageFiles`。因此当前“已准备原始截图附件”只体现在 UI 集合，实际首次流式请求并未发送原始截图。

真实工作台应修复这段参数传递：截图文件存在时默认加入首次流式用户消息，并把实际状态显示为“已附带原始截图”“未找到原始截图”或“截图发送失败”。本期不新增 `ModelDefinition.Capabilities` 判断；如果复用链路中已经存在能力判断，则保持原有行为，不另建第二套分支。

### 3.5 ResourceId 没有进入 WPF 渲染

当前输入包已经提供：

- `CoursewareResourceEntry.ResourceId`。
- `ResourceType`。
- `ResolvedFilePath`。
- `Exists`。

`SlideMlPipelineContext` 也具有 `MaterialResourceManager`，但 `WpfSlideMlRenderEngine.PreMeasureImage` 只调用 `TryLoadBitmap(image.Source)`，后者把 `Image.Source` 当成：

- 直接文件路径；
- 当前目录相对路径；
- `Images` 或 `Assets` 目录下的候选文件名。

它不会读取 `MaterialResourceManager`。所以仅仅在页面工厂中注册素材仍然不能解决逻辑资源渲染问题。

该问题本期不改造，已作为后续待办记录在 [`../../主题分析输出与原始设计偏差审查.md`](../../主题分析输出与原始设计偏差审查.md)。真实工作台接入时保持现有单页渲染行为，不新增严格资源模式，也不把 `ResourceId` 本地渲染闭环列为本期验收条件。

### 3.6 MCP 素材解析采用现有远端约定

`McpSlideMlRenderPipeline.RenderAsync` 只发送：

```json
{ "slideXml": "..." }
```

本计划采用现有系统约定：远端服务知道课件素材，只要 SlideML 提供素材名即可解析。本期不再探索资源映射、文件上传或受控 URI 协议，也不因素材引用禁用 MCP。是否自动连接 MCP 只按现有单页工作流和页面运行时初始化成本决定，与素材协议无关。

### 3.7 通用提示词存在后续待办

`SlideMlPromptProvider.BuildDefaultStreamingUserPrompt` 当前包含：

- “尽量使用浅色主题”，可能覆盖课件主题。
- “标题 L2、正文 L3、辅助 L4”，但当前解析器只接受数值字号。
- “可用占位资源 ID，如 image_001”，会鼓励模型创造未登记资源。

这些问题已作为后续警告记录在 [`../../主题分析输出与原始设计偏差审查.md`](../../主题分析输出与原始设计偏差审查.md)。本期不修改通用 `SlideMlPromptProvider`，也不新增专用 Prompt Provider；真实工作台继续复用现有单页提示词构造方式，只在页面生成请求中补充当前主题的结构化内容。

### 3.8 渲染结果状态不完整

`SlideMlRenderTool` 当前只保存：

- `LatestPreviewImage`。
- `LatestWarnings`。
- `LatestRenderedXml`。
- `LatestSlideXml`。

没有保存 `SlideMlRenderResult.Errors`。同时 `ApplyRenderResult` 内部以 fire-and-forget 方式派发 UI 更新，调用方不能可靠等待状态回填完成。

该问题本期保持现状，已作为后续待办记录在 [`../../主题分析输出与原始设计偏差审查.md`](../../主题分析输出与原始设计偏差审查.md)。每个真实页面继续复用现有单页美化逻辑；本期页面状态只映射现有链路已经公开的结果、警告和失败信息，不新增完整结果快照、阻断 Warning 分级或新的质量门。

### 3.9 页面生成输入尚未复用路径隐私边界

主题分析通过 `CoursewareAnalysisInputBuilder` 对整个模型输入执行路径脱敏或阻断，但其 `PathPrivacyTransformer` 是私有嵌套实现。旧 `CreateBeautifyPrompt` 直接拼接原始 Markdown。

本期不提取新的 Core 路径隐私服务，也不增加路径哈希、审计模型、阻断策略或二次扫描协议。页面生成只做简单处理：不主动把资源文件路径和截图路径拼进 Prompt；如果现有文本构造中能以少量代码识别并移除明显绝对路径，则就地处理。任何需要新增抽象、状态模型或复杂解析的路径场景均不在本期解决。

### 3.10 当前主题缺少参考画布信息

`CoursewareThemeAnalysisResult` 当前没有保存主题分析所采用的参考画布。

但 `CoursewareThemeAnalysisService` 在生成主题前会选择主导页面尺寸，并据此校验字号和安全区。若课件存在混合尺寸，后续无法知道主题数值基于哪一个画布。

真实工作台如需按页面尺寸应用主题数值，只增加语义明确的 `ReferenceCanvasWidth` 和 `ReferenceCanvasHeight`。本期不增加输入指纹、主题指纹、Prompt 指纹或页面输入指纹，也不基于指纹实现缓存、迟到结果保护或过期判断。

### 3.11 当前异步命令仍以 fire-and-forget 启动

`RelayCommand` 只接受同步 `Action<object?>`。当前分析重试、发送消息、重新渲染、连接 MCP 和添加页面等操作普遍使用 `_ = SomeAsync()` 启动，命令本身无法：

- 等待任务完成；
- 统一捕获异常；
- 阻止重复执行；
- 暴露命令级取消和执行状态；
- 让测试可靠等待命令完成。

真实工作台不应继续扩大这种模式。页面初始化、生成、取消、重新渲染和批处理需要使用可等待的 `AsyncRelayCommand` 或等效异步命令抽象；最后一道服务边界仍需捕获异常并转换为页面状态，不能让未观察任务异常进入 Dispatcher。

无论是否新增异步命令，都禁止为了适配同步命令而调用 `.Result`、`.Wait()` 或 `GetAwaiter().GetResult()`。如果某个现有命令入口暂时无法改造成可等待命令，应保留受控的 fire-and-forget，并在异步方法内部处理异常和状态；不得用阻塞等待制造表面同步。

### 3.12 流式会话必须继续保持工具白名单

`StreamingSlideGenerator` 通过 `ManualSendMessageContext` 创建 Agent。该 Context 自身持有默认工具，但当前代码通过替换 `ChatClientAgentOptions.ChatOptions`，实际只给流式页面生成注册 `get_slide_state` 和 `get_slide_preview`。

本期不修改这套工具注册机制和相关逻辑。截图附件透传、主题内容补充和工作台迁移不得顺带调整 `ManualSendMessageContext`、`ChatClientAgentOptions.ChatOptions`、默认工具集合或 `WorkspacePath`。

## 四、可复用与应替换的现有能力

| 能力 | 处理方式 | 说明 |
| --- | --- | --- |
| `CoursewareFolderLoader` | 直接复用 | 继续作为唯一课件加载入口 |
| `CoursewareInputPackage` | 直接复用 | 作为工作台输入事实边界 |
| `CoursewareWorkspaceSession` | 直接复用 | 继续保存当前输入包和当前主题分析结果，不新增会话 ID 或指纹 |
| `CoursewareSlideSummaryService` | 复用 | 用于标题和有限相邻页摘要 |
| `SlideChatManager` | 直接复用 | 保留每页独立会话，仅通过工厂提供页面实际画布和现有生成参数 |
| `SlideGenerationPipeline` | 最小修改 | 只补齐流式分支对现有图片附件参数的透传 |
| `SlideMlRenderPipeline` | 复用并注入上下文 | 使用页面实际画布，其他行为保持现状 |
| `WpfSlideMlRenderEngine` | 本期不修改 | `ResourceId` 解析作为后续已知问题 |
| `CopilotMessageStream` | 直接复用 | 作为页面真实聊天消息视图 |

## 五、目标架构

```text
App.OnStartup
  -> CoursewareWorkspaceViewModel                         应用壳/唯一根状态
       ├── CoursewareWorkspaceSession                    当前课件输入和主题分析结果
       ├── CoursewareAnalysisView                        全课件分析页
       └── CoursewareSlideWorkspaceViewModel             真实单页工作台
            ├── ObservableCollection<CoursewareSlideItemViewModel>
            ├── SelectedSlide
            ├── ICoursewareSlidePromptBuilder
            ├── ICoursewarePageDesignContextFactory
            └── ISlideChatManagerFactory
                 └── 每页懒创建 CoursewareSlideRuntime
                      ├── SlideChatManager
                      ├── SlideDocumentContext
                      ├── SlideMlPipelineContext
                      └── 页面级取消
```

### 5.1 `CoursewareWorkspaceViewModel`

继续负责：

- 打开课件。
- 自动主题分析。
- 分析取消、失败和重试。
- 全课件分析页与单页工作台导航。
- 创建和销毁当前工作台子 ViewModel。
- 打开新课件时取消旧分析和全部页面操作。
- 主题重新分析成功后把新的当前主题结果交给工作台，供后续生成使用。

不再负责：

- 页面演示集合。
- 页面 SlideML 字符串拼接。
- 页面聊天消息。
- 当前页生成和渲染。
- 页面级附件和模型选择。

### 5.2 `CoursewareWorkspaceSession`

继续保持为简单的当前课件状态对象：

- `CoursewareInputPackage? InputPackage`：当前已加载课件。
- `CoursewareThemeAnalysisResult? ThemeAnalysisResult`：当前主题分析结果。
- 现有加载、分析状态和错误信息。

本期不增加 `SessionId`、`ActiveAnalysisRunId`、`ThemeFingerprint` 或单独的 `ActiveThemeAnalysisResult`。打开新课件时，由根 ViewModel 先取消并释放旧工作台，再替换整个 `CoursewareWorkspaceSession` 和工作台对象；旧异步操作即使完成，也只会更新已经脱离当前 UI 的旧页面对象。

主题参考画布属于分析结果本身，而不是课件会话身份。若页面主题适配需要该信息，只在 `CoursewareThemeAnalysisResult` 中增加名称明确的 `ReferenceCanvasWidth` 和 `ReferenceCanvasHeight`。

### 5.3 `CoursewareSlideWorkspaceViewModel`

建议职责：

- 从现有 `CoursewareInputPackage` 一次创建轻量页面 ViewModel。
- 暴露真实 `Slides`、`SelectedSlide` 和当前课件标题。
- 选择页面时懒初始化该页运行时。
- 公开生成、取消、重新渲染、继续追问和返回主题概览所需状态。
- 聚合页面进度：未开始、生成中、已完成、失败和已取消。
- 接收当前主题分析结果，供尚未开始的新生成请求使用。
- 关闭或切换课件时取消页面操作并解除事件订阅。

不建议在进入工作台前为全部页面创建模型和评估器。页面列表、标题、源截图和 Markdown 可以立即可用；`SlideChatManager` 只在选中或开始生成时创建。

### 5.4 `CoursewareSlideItemViewModel`

建议包含以下分区。

#### 只读输入事实

- `CoursewareSlideInput Input`。
- `SlideIndex`、`PageNumber`、`SlideId`。
- 原始逻辑宽高和统一转换后的栅格画布宽高。
- `Title`、`Summary`。
- `SourceMarkdownText`。
- `SourceScreenshotFilePath`。
- 页面输入警告。

#### 页面运行状态

建议至少分开两个枚举，避免把所有状态压缩为字符串：

```text
CoursewareSlideRuntimeState
  - NotCreated
  - Creating
  - Ready
  - ModelUnavailable
  - Failed

CoursewareSlideGenerationState
  - NotStarted
  - Generating
  - Completed
  - Failed
  - Canceled
```

另外保留：

- `string? ErrorMessage`。
- `IReadOnlyList<string> Warnings`。
- `CoursewareScreenshotAttachmentState ScreenshotAttachmentState`，区分未准备、已附带、文件缺失和发送失败。

#### 页面结果

- `string EditableSlideMl`。
- `string RenderedXml`。
- `IPreviewImage? PreviewImage`。
- 页面级聊天管理器和真实聊天消息。
- 页面级输入文本、手工附件和是否附带当前预览。

这些字段只映射现有单页链路已经公开的状态，不要求本期新增完整 `SlideMlRenderResult` 快照或 `LatestErrors`。

页面切换只改变工作台的 `SelectedSlide`，不能重新从初始值覆盖这些字段。

### 5.5 页面运行时工厂

扩展 `ISlideChatManagerFactory`，推荐使用创建选项：

```text
CoursewareSlideRuntimeOptions
  - SlideDocumentContext
  - CancellationToken
```

接口应：

- 接受 `CancellationToken`。
- 使用页面实际画布构造同一份 `SlideDocumentContext`。
- 让现有 Prompt Provider、布局和渲染共享该上下文。
- 保持现有单页模型、评估和 MCP 行为；由于运行时改为按页懒创建，不再在打开课件时一次初始化全部页面。
- 模型初始化失败时返回可区分的结果：页面仍可查看源截图和使用本地手工渲染，但 AI 生成命令不可用。

并发选择同一页时，`EnsureRuntimeAsync` 必须复用同一个初始化任务，不能重复创建多个页面会话。

### 5.6 页面画布适配器

`CoursewareSlideInput.Width/Height` 是 `double`，而 `SlideDocumentContext` 和 WPF `RenderTargetBitmap` 使用整数像素。不能在 Prompt Provider、渲染器和 UI 中分别执行隐式强制转换。

建议新增单一 `CoursewareSlideCanvas` 或等效适配器：

- 保留输入中的原始逻辑宽高；
- 校验数值有限、为正且可转换到 `int`；
- 使用固定舍入规则一次生成渲染宽高；
- 对非整数输入记录精度转换诊断；
- Prompt、`SlideDocumentContext`、预览容器和输出元数据全部消费同一转换结果；
- 禁止不同层各自 `Convert.ToInt32`、强制截断或重新计算比例。

### 5.7 异步命令和异常边界

新增 `AsyncRelayCommand` 或等效抽象，至少支持：

- `Task ExecuteAsync`；
- 执行期间自动刷新 `CanExecute`；
- 统一异常回调；
- 可选取消入口；
- 测试可等待完成。

工作台的页面初始化、生成、重新渲染、模型重试和批处理都使用异步命令。普通 `RelayCommand` 只保留给同步选择、导航和纯状态切换。

`AsyncRelayCommand` 的实现和调用链必须全程异步，不得在命令内部或调用端阻塞等待异步任务。若个别现有入口暂时只能 fire-and-forget，应由入口显式丢弃任务，并由被调用异步方法自行捕获异常和更新状态。

## 六、主题到页面执行契约

### 6.1 不直接消费 `GenerationPromptSummary`

`GenerationPromptSummary` 可以继续作为用户可读摘要或兼容字段，但不能作为唯一执行接口。页面生成应消费强类型 `CoursewarePageDesignContext`，至少包含：

- `CoursewareTheme` 主题建议中的结构化颜色、字号、字体和安全区。
- 主题分析使用的参考画布。
- 当前页解析后的实际画布。
- 版式原则。
- 内容呈现规则。
- 当前固定四类页面建议，仅作为自然语言参考，不声称已经完成模板绑定。
- 当前能力明确为 `ThemeSuggestion`，不能写成 `DesignSystem`。

### 6.2 `CoursewareTheme` 页面适配器

推荐新增 `ICoursewarePageDesignContextFactory`，首个实现为 `CoursewareThemePageDesignContextFactory`。

它将当前 `CoursewareTheme` 主题建议转换为当前页执行配置：

- 颜色直接使用结构化 Hex 值。
- 字号按主题参考画布与当前页画布确定性缩放。
- 左右安全区按 X 轴比例缩放，上下安全区按 Y 轴比例缩放。
- 字体继续使用当前单个推荐字体，不声称已实现字体栈 fallback。
- 四类页面建议可以作为候选指导，但由于没有稳定页面类型分配，不得标记为模板绑定。
- `GenerationPromptSummary` 只附加为补充说明，不覆盖结构化字段。

对于与参考画布等比例的页面，字号建议使用统一比例缩放；对于长宽比显著不同的页面，应记录 `CanvasProfileFallback` 诊断，并要求模型基于当前页尺寸重排，而不是静默套用主导页面版式。

### 6.3 主题分析结果补充字段

建议扩展 `CoursewareThemeAnalysisResult`：

- `double ReferenceCanvasWidth`。
- `double ReferenceCanvasHeight`。

这两个字段只表达主题分析数值所依据的画布，不承担版本、缓存或并发保护职责。

### 6.4 `CoursewareDesignSystem` 后续接入点

未来增加 `CoursewareDesignSystemPageDesignContextFactory`：

- 根据 `CanvasProfiles` 选择画布。
- 根据 `PageTypeAssignments` 找到当前 SlideId 的页面类型。
- 解析设计 Token、组件、素材策略和模板。
- 生成编译后的数值 SlideML 约束。
- 把模板和硬约束交给同一个页面 Prompt Builder。

工作台、页面 ViewModel 和生成服务不应因后续从 `CoursewareTheme` 主题建议切换到完整 `CoursewareDesignSystem` 而再次重写。

## 七、页面生成输入协议

### 7.1 使用结构化信封

不建议继续在 ViewModel 中散落拼接自由文本。新增 `ICoursewareSlidePromptBuilder`，输出结构明确的 JSON 信封，例如：

```text
courseware-slide-generation-envelope
  - task
  - designContext
  - currentSlide
  - neighborSlides
  - visualInput
  - outputRequirements
```

其中：

- `task` 和 `outputRequirements` 由本地固定生成，属于可信指令。
- 主题文本、页面 Markdown、相邻摘要、资源字段均标记为不可信数据。
- 当前页 Markdown 必须完整保留，不静默摘要或截断。
- 相邻页只发送有限摘要，不发送完整正文。
- Prompt 不包含资源本地路径和截图本地路径。

### 7.2 当前页字段

至少包含：

- `SlideId`。
- `SlideIndex`。
- `PageNumber`。
- 实际宽高。
- 完整 Markdown。
- 页面输入警告摘要。

### 7.3 相邻页上下文

首期默认只包含前一页和后一页：

- SlideId。
- 页码。
- 标题。
- `CoursewareSlideSummaryService` 生成的摘要。

必须明确相邻页仅用于理解章节连续性，不得把其内容写入当前页。

后续扩大窗口前应先测量上下文成本，不直接固定发送前后三页完整内容。

### 7.4 素材名称

本期沿用当前 Markdown 和 `Resources.json` 中已有的逻辑素材名：

- 页面 Prompt 可以携带与当前页相关的素材名和资源类型，但不携带本地解析路径。
- 不新增资源白名单提取器、`MaterialResourceManager` 注册或严格路径策略。
- MCP 按现有约定仅凭素材名解析远端已知素材。
- WPF 本地渲染不能按 `ResourceId` 找到素材的问题作为已知问题保留。

### 7.5 路径隐私

路径处理保持简单：

- Prompt Builder 不主动加入截图路径、资源解析路径或课件根目录。
- 对容易识别的绝对路径可以在当前构造方法中做简单移除或替换。
- 不为页面生成新增 Core 路径隐私抽象、哈希诊断、可配置阻断模式或二次完整扫描。
- 需要复杂解析才能处理的原始 Markdown 路径本期忽略，不因此扩大实现复杂度。

### 7.6 上下文预算

页面生成也要执行模型预算预检：

- System Prompt。
- 页面 JSON 信封。
- 流式查询工具 Schema。
- 输出预留。
- 安全余量。

当前页完整 Markdown 超出上下文时应明确失败，不允许静默删正文。相邻摘要和资源目录必须在构建时保持有界。

## 八、截图和视觉输入

### 8.1 截图发送规则

本期不新增模型图片能力判定。当前页原始截图文件存在时默认发送；文件不存在时继续使用 Markdown 和结构化主题。若复用链路内部已经存在能力检查，则保持其现有行为，不在工作台再判断一次。

### 8.2 修复流式附件传递

需要修改：

- `SlideGenerationPipeline.SendMessageAsync`。
- `StreamingSlideGenerator.GenerateAsync`。
- `StreamingSlideGenerator.RunStreamingLoopAsync`。

让 `attachedImageFiles` 在首次流式请求中转换为 `DataContent` 并加入用户消息。附件只在首次用户请求加入，后续流式修复轮次沿用现有行为，不重复读取同一文件。

### 8.3 视觉证据边界

页面 Prompt 必须明确：

- `visualInput.wasAttached = true` 时，模型可以参考当前页截图布局和视觉元素。
- 没有截图或文件缺失时，声明本轮只使用 Markdown 和结构化主题。
- 页面截图参与单页生成不等于全课件主题阶段已经完成视觉分析。
- `CoursewareAnalysisCapabilityStates.VisualAnalysis` 不因此改为 `Passed`。

### 8.4 手工附件

用户手工选择的图片首期仅作为视觉参考：

- 不把文件路径写入 Prompt。
- 不自动成为可被 SlideML 引用的资源。
- 如果未来允许生成结果使用该图片，应先执行显式导入，分配稳定 ResourceId 并进入受控资源目录。

## 九、本期素材与渲染边界

### 9.1 WPF 本地 `ResourceId` 解析

`WpfSlideMlRenderEngine` 尚未从 `MaterialResourceManager` 按逻辑素材 ID 加载课件图片。这是已知问题，本期不修改渲染器、不新增严格资源模式，也不把真实图片本地渲染作为工作台接入的验收条件。

### 9.2 MCP 素材约定

MCP 远端服务按素材名解析其已知素材。本期继续沿用当前 `{ "slideXml": "..." }` 请求，不扩展素材上传或映射协议，也不因页面包含逻辑素材名而禁用 MCP。

### 9.3 通用提示词待办

`SlideMlPromptProvider` 中固定浅色主题、`L2/L3/L4` 字号 Token 和 `image_001` 占位资源建议均为后续待办。本期不修改通用提示词实现，只通过页面请求补充当前 `CoursewareTheme` 的结构化内容。

### 9.4 渲染结果待办

`SlideMlRenderTool` 未保存完整 `Errors`，且结果应用时序不可等待。本期保持现有逻辑，页面 ViewModel 仅消费当前单页链路能够获得的 SlideML、预览、Warning、回填 XML 和失败信息。

## 十、复用现有单页生成与渲染

### 10.1 主题内容注入

不新增 `CoursewareSlideMlPromptProvider`，也不修改通用 `SlideMlPromptProvider`。`ICoursewareSlidePromptBuilder` 在现有单页用户请求外层补充：

- 当前页实际画布。
- `CoursewarePageDesignContext` 中的结构化主题字段。
- 当前页完整 Markdown 和有限相邻页摘要。
- 截图是否实际附带。

现有标签、属性、流式合并、工具和渲染提示保持不变。

### 10.2 页面操作状态

本期不创建 `GenerationRunId`、Prompt 指纹或页面输入指纹。每个页面 ViewModel 自己持有当前生成任务、取消令牌和运行时：

- 同一页面生成期间禁用重复生成命令。
- 切换选中页不会改变正在生成任务所引用的页面对象。
- 打开新课件时取消并释放整个旧工作台。
- 任务完成后只更新启动该任务的页面 ViewModel。

这种对象归属和生命周期管理足以满足本期需求，不额外引入身份校验层。

### 10.3 结果传播

页面 ViewModel 在现有单页生成调用完成或现有状态通知到达后更新：

- `EditableSlideMl`。
- `RenderedXml`。
- `PreviewImage`。
- 现有 Warnings 和错误文本。
- 生成状态。

切换页面时不得重新从初始模型覆盖这些字段。重新生成是否清空旧预览、失败时如何保留结果，继续遵循现有单页逻辑，本期不额外改变。

### 10.4 完成与失败口径

页面生成状态按现有单页链路能够观察到的结果设置：

- 调用正常结束且获得当前链路认可的 SlideML/预览时标记为 `Completed`。
- 初始化、生成或重新渲染抛出异常时标记为 `Failed` 并显示错误文本。
- 用户取消时标记为 `Canceled`。
- Warning 按现有逻辑展示，不在本期新增阻断分级。

完整 `SlideMlRenderResult.Errors` 传播、未知标签阻断、裁切与溢出质量门、资源白名单校验均属于后续改造。

### 10.5 重试行为

继续使用现有 `StreamingSlideGenerator` 重试逻辑，本期不调整默认次数或新增重试预算模型。页面取消必须继续传递到现有取消入口；若后续发现重试成本或取消响应问题，再作为通用单页链路的独立已知问题处理。

## 十一、工作流与生命周期

### 11.1 打开课件

```text
打开新课件
  -> 取消旧主题分析
  -> 取消旧工作台全部页面操作
  -> 释放旧工作台事件订阅和页面运行时
  -> 加载 CoursewareInputPackage
  -> 创建分析页缩略图
  -> 自动分析主题
```

根 ViewModel 在替换当前会话前先让旧工作台退出当前 UI 并执行取消、释放。页面异步操作始终捕获其所属页面对象，不通过全局 `SelectedSlide` 回写，因此不需要额外的 SessionId 或回调身份校验。

### 11.2 主题分析成功

```text
PublishAnalysisResult
  -> 保存当前 ThemeAnalysisResult 和参考画布
  -> 从当前 InputPackage 创建轻量 CoursewareSlideWorkspaceViewModel
  -> 让 EnterWorkspaceCommand 可执行
```

工作台页面列表可以在此时创建，但不创建所有页面模型会话。

### 11.3 进入工作台

```text
EnterWorkspaceCommand
  -> 切换 SlideWorkspace
  -> 保持真实页面列表
  -> 选择第一页或上次选择页
  -> 异步 EnsureRuntimeAsync(SelectedSlide)
```

源截图应立即可见，不必等待模型初始化。

### 11.4 选择页面

- 取消上一页的“选择触发初始化等待”不等于取消上一页正在生成的任务。
- 同一页重复选择复用既有运行时和聊天历史。
- 当前页初始化失败时显示错误，并允许重试初始化或使用本地手工渲染。
- 页面级附件、输入框和编辑内容随页面保存，不能使用一个全局集合在切页时丢失。

### 11.5 初次生成

```text
GenerateCurrentSlide
  -> 读取当前 CoursewareTheme 结构化结果
  -> 构建页面生成输入
  -> 执行上下文预算预检
  -> 原始截图文件存在时默认附带
  -> 流式生成和实时渲染
  -> 按现有单页逻辑回写当前页面结果
```

### 11.6 继续追问

同一页后续消息继续使用该页既有会话和聊天历史，与当前单页工作流保持一致。本期不增加主题指纹、过期检测或自动重建会话；主题重新分析后的既有页面会话如何迁移作为后续问题单独设计。

### 11.7 编辑与重新渲染

- `EditableSlideMl` 属于页面 ViewModel。
- 手工编辑后标记页面为“有未保存修改”。
- 重新渲染调用现有单页重新渲染逻辑，并同步现有预览、Warning、回填 XML 和错误文本。
- 重新渲染不调用模型，也不改变聊天历史。

### 11.8 重新分析主题

本期不改造重新分析机制，不增加分析 RunId、主题指纹、页面过期状态或自动迁移聊天会话。重新分析仍按现有根 ViewModel 逻辑执行；新分析成功后，后续尚未开始的页面生成读取新的当前主题结果，已经创建的页面会话和结果保持不变。

若实际接入时发现现有 `ReanalyzeAsync` 会破坏工作台基本可用性，只允许做最小状态修正，不扩展为主题版本管理系统。

### 11.9 关闭应用或打开其他课件

工作台需要统一 `CancelAll` / `Dispose` 生命周期：

- 取消所有页面 CTS。
- 调用页面 `CopilotChatManager.CancelCurrentChat()`。
- 解除 `PropertyChanged`、渲染事件和集合事件订阅。
- 不等待无限期模型请求。
- MCP 客户端后续若实现可释放接口，应由创建它的运行时负责释放。

## 十三、能力状态口径

### 13.1 分析结果能力

主题分析成功后：

- `TextAnalysis = Passed`。
- `ThemeSuggestion = Passed`。
- `DesignSystem = NotSupported`。
- `TemplateValidation = NotSupported`。
- `VisualAnalysis = NotRequested`。
- 页面生成运行时已经接入后，新的主题分析结果可把 `PageGeneration` 写为 `NotRequested`，表示本轮分析没有执行页面生成，而不是运行时不支持。
- `CoursewareThemeAnalysisResult.CapabilityStates` 是分析运行的事实快照，后续页面生成不能原地回写该对象；动态页面状态由工作台汇总和最终输出结果单独记录。

### 13.2 页面级能力

每页单独记录：

- 是否初始化模型运行时。
- 是否发送视觉附件。
- 是否生成 SlideML。
- 是否获得现有单页链路的渲染预览。
- 当前状态是未开始、生成中、已完成、失败还是已取消。

### 13.3 课件级页面生成状态

不要把“生成过一页”标记为整课件 `Passed`。建议新增独立汇总：

```text
CoursewarePageGenerationSummary
  - TotalCount
  - NotStartedCount
  - GeneratingCount
  - CompletedCount
  - FailedCount
  - CanceledCount
```

首期没有自动全课件生成，因此“至少生成一页”不能把整课件页面生成标记为 `Passed`。只有后续明确要求处理的全部页面都完成相应生成与渲染后，才可以单独定义整课件完成口径。

## 十四、分阶段实施计划

### 阶段 A：最小底层接入

目标：只补齐真实工作台必须依赖的画布、异步命令和截图附件传递，不扩展通用单页生成与渲染职责。

1. 仅在需要主题尺寸换算时，为 `CoursewareThemeAnalysisResult` 增加 `ReferenceCanvasWidth` 和 `ReferenceCanvasHeight`。
2. 新增唯一的页面画布适配结果，让 Prompt、`SlideDocumentContext`、渲染和 UI 使用同一宽高。
3. 新增可等待的 `AsyncRelayCommand` 或等效抽象，保证调用链全程异步，禁止任何异步转同步。
4. 扩展 `ISlideChatManagerFactory.CreateAsync`，接收页面实际 `SlideDocumentContext` 和 `CancellationToken`。
5. 让 `attachedImageFiles` 真正进入首次流式用户消息；截图存在时默认发送，不新增模型能力判断。
6. 为现有流式工具注册行为增加回归测试，但不修改工具白名单机制。

本阶段明确不修改 `SlideMlRenderTool`、`WpfSlideMlRenderEngine`、`SlideMlPromptProvider`、MCP 素材协议和流式重试策略。

验收：

- 单元测试证明原始截图内容真正进入首次流式用户消息。
- 页面 Prompt、`SlideDocumentContext`、渲染预览和 UI 使用同一实际画布。
- 非整数页面尺寸只经过一次确定性转换。
- 新增异步调用链中不存在 `.Result`、`.Wait()` 或 `GetAwaiter().GetResult()`。
- 流式工具集合和修改前保持一致。

### 阶段 B：真实工作台状态模型

目标：用真实页面替换第二页演示集合，但暂不要求所有页面自动生成。

1. 新增 `CoursewareSlideItemViewModel`。
2. 新增 `CoursewareSlideWorkspaceViewModel`。
3. 从当前 `CoursewareWorkspaceSession.InputPackage` 创建轻量页面列表。
4. 页面运行时改为选中时懒初始化。
5. 页面级保存输入框、附件、SlideML、预览、现有日志、聊天和错误文本。
6. 页面运行时初始化失败时保留源截图和本地手工渲染能力。
7. 根 ViewModel 持有工作台并在新课件打开时取消和替换。
8. `EnterWorkspaceCommand` 只进入绑定当前课件对象的真实工作台。
9. 删除 `DemoSlideViewModel` 和演示聊天；设计时数据如需保留，必须使用正式运行时 ViewModel 类型。

验收：

- 工作台页数和 SlideId 与输入包完全一致。
- 切换页面不会丢失任何一页的编辑内容和聊天历史。
- 快速重复选择同一页只创建一个运行时。

- 设计时和运行时不存在两套页面或聊天 ViewModel 类型。

### 阶段 C：主题约束下的真实单页生成

目标：完成首个可对外演示的真实页面美化闭环。

1. 新增 `CoursewareTheme` 到 `CoursewarePageDesignContext` 的页面适配器。
2. 新增 `ICoursewareSlidePromptBuilder`。
3. 注入结构化主题、当前页、有限相邻摘要、素材名和截图发送状态。
4. 页面生成前执行上下文预算预检。
5. 截图存在时默认发送到首次流式请求。

8. 支持页面级取消，且取消或失败不影响其他页面。

验收：

- 页面尺寸、主题颜色、字号和安全区可在构建的页面请求中验证。

## 十六、测试矩阵

### 16.1 页面 Prompt Builder

- 当前页完整 Markdown 被保留。
- 主题颜色、字号、字体和安全区来自结构化主题。
- 不只依赖 `GenerationPromptSummary`。
- 参考画布与当前页尺寸缩放正确。
- 不同比例页面产生明确 fallback 诊断。
- 只包含前一页和后一页摘要。
- 相邻页正文不会混入当前页。
- 可包含当前页相关素材名，但不包含素材解析路径和截图路径。
- 明显绝对路径的简单处理不会引入额外服务或审计模型。
- JSON 信封规范序列化后可往返。
- 预算预检覆盖最终发送内容。

### 16.2 流式图片附件

- 截图文件存在时，首次流式用户消息收到原始截图 `DataContent`。
- 截图缺失时仍可生成文本页面。
- 取消期间停止读取和发送附件。
- 后续流式修复轮次不会重复加载首次截图。
- 用户手工附件不会自动成为 SlideML 资源。
- 工作台不会新增模型图片能力判断。

### 16.3 页面运行时工厂

- 每页使用实际画布尺寸。
- Prompt Provider 和渲染器共享同一画布。
- 原始 `double` 尺寸只通过统一适配器转换一次，舍入和诊断稳定。
- 同一页并发初始化只创建一个运行时。
- 未选中页面不创建模型会话。
- 模型初始化失败时本地渲染仍可用。
- MCP 行为与现有单页逻辑一致。
- 流式会话只注册 `get_slide_state`、`get_slide_preview` 等允许工具，不包含默认文件工具。

### 16.4 异步边界

- `AsyncRelayCommand.ExecuteAsync` 可在测试中等待完成。
- 页面初始化、生成和重新渲染异常会转换为页面状态。
- 取消入口能够传递到页面操作。
- 新增代码不调用 `.Result`、`.Wait()` 或 `GetAwaiter().GetResult()`。
- 暂时保留的 fire-and-forget 入口不会产生未观察任务异常。

### 16.5 现有单页结果映射

- 现有 SlideML、预览、Warning、回填 XML 和错误文本映射到所属页面。
- 手工编辑后重新渲染更新同一页结果。
- 页面完成、失败和取消状态与现有单页调用结果一致。
- 测试不要求新增 `LatestErrors`、阻断 Warning 或资源质量门。

### 16.6 页面 ViewModel

- 页面列表与输入包页数、顺序和 SlideId 一致。
- 选择页时显示真实源截图。
- 页面切换后 SlideML、聊天、附件和预览保留。
- 两页聊天历史完全隔离。
- 取消当前页不影响其他页。
- 模型不可用时生成命令禁用但手工渲染可用。
- 生成任务始终更新其启动时所属的页面对象，而不是当前全局选择页。
- 设计数据构造使用 `CoursewareSlideItemViewModel` 正式类型，不存在 `DemoSlideViewModel` 设计时替代类型。

### 16.7 根 ViewModel 与导航

- 主题分析完成前不能进入真实工作台。
- 分析成功后工作台绑定当前 `CoursewareWorkspaceSession.InputPackage` 和 `ThemeAnalysisResult`。
- 返回分析页再进入时页面选择和结果保留。
- 打开新课件会取消旧页面操作。
- 打开新课件后旧工作台对象不再是当前 UI 数据源。
- 异步命令异常被转换为可观察状态，测试中不存在未观察任务异常。

### 16.8 UI 自动化和 XAML 契约

- `SlideWorkspaceView` 不再包含静态演示预览和伪 SlideML。
- 左侧绑定真实页面类型。
- 中间预览尺寸绑定当前页实际画布。
- 右侧绑定真实 `CopilotChatManager.ChatMessages`。
- 运行时文案不再出现“演示”“原型”“尚未接入真实页面生成”。
- `DesignSystem` 和 `TemplateValidation` 未完成警告仍存在。
- 焦点在双页往返后落到正确导航或当前页操作。

### 16.9 最小集成测试

构建一个 3 页课件：

- 第 1 页：有截图和图片资源。
- 第 2 页：无截图。
- 第 3 页：与主导画布尺寸或比例不同。

使用 Fake 主题服务和 Fake 页面模型：

1. 打开课件并完成主题分析。
2. 进入真实工作台。
3. 验证真实页面列表。
4. 第 1 页首次生成请求包含源截图 `DataContent`。
5. 第 2 页因无截图而只发送文本，并仍可进入现有生成链路。
6. 第 3 页使用自己的画布并记录主题缩放诊断。
7. 切页后结果和聊天保留。
8. 页面失败或取消不会破坏其他页。
9. 打开另一份课件会取消并替换旧工作台。

默认测试不得访问真实网络、个人模型配置或真实 MCP 服务。

## 十七、风险与约束

### 17.1 `CoursewareTheme` 主题约束仍然偏弱

即使接入真实工作台，当前 `CoursewareTheme` 仍没有动态页面类型、模板、组件和素材策略。处理方式：

- 首期如实标记 `ThemeSuggestion`。
- 使用结构化字段而不是自由文本摘要。
- 不把一页生成成功解释为设计系统已落地。
- 保持完整 `CoursewareDesignSystem` 的后续接入边界稳定。

### 17.2 大课件页面会话成本

如果恢复旧逻辑并一次为所有页面创建生成和评估会话，加载可能很慢且失败面大。缓解方式：

- 页面 ViewModel 轻量创建。
- 运行时按页懒加载。
- 不在打开课件时一次创建所有页面模型、评估器和渲染管线。
- 本期不增加批处理。
- 单个页面运行时内部保持现有 MCP 行为。

### 17.3 混合页面尺寸

当前 `CoursewareTheme` 主题建议只基于主导画布校验。缓解方式：

- 保存参考画布。
- 确定性缩放。
- 不同比例页面记录 fallback。
- 后续完整 `CoursewareDesignSystem` 使用 `CanvasProfiles` 解决多画布配置。

### 17.4 模型忽略主题

自然语言约束仍可能被模型忽略。缓解方式：

- 发送结构化执行上下文。
- 在页面请求中明确当前主题颜色、字号、安全区和画布。
- 本期接受现有单页自由生成的限制，不新增专用系统提示词或后置质量验证器。
- 后续完整 `CoursewareDesignSystem` 使用模板和硬约束替代自由生成。

### 17.5 渲染结果与 UI 状态竞态

当前结果应用是 fire-and-forget。缓解方式：

- 本期保持现有 `SlideMlRenderTool` 行为，不把它改造成可等待结果应用。
- 页面 ViewModel 只订阅和映射现有公开状态，并确保回写到发起操作的页面对象。
- 该竞态及完整 Errors 缺失继续作为后续已知问题。

### 17.6 MCP 与本地素材行为不同

本计划按既有约定认为 MCP 远端服务知道素材名；本期不修改或禁用 MCP。本地 WPF 不能按 `ResourceId` 解析素材则是另一个已知问题，不在本期补齐。

### 17.7 旧实现长期并存

如果 `MainWindowViewModel`、演示 `Slides` 和新工作台同时存在，后续维护容易继续修错入口。缓解方式：迁移测试和功能后删除或明确归档旧运行时代码。

### 17.8 范围扩张和错误异步适配

真实工作台接入容易顺带引入指纹、运行 ID、严格资源模式、能力探测或异步转同步。缓解方式：

- 代码审查按本文“核心实施原则”和“后续独立阶段”逐项核对。
- 任何新增 `.Result`、`.Wait()`、`GetAwaiter().GetResult()` 都视为阻断问题。
- 任何设计时专用页面或聊天类型都视为阻断问题。
- 非本期底层缺陷只记录待办，不以顺手修复扩大改动面。

## 十八、禁止的实现方式

实施时不要采用以下捷径：

1. **只把 `CoursewareThumbnails` 复制到第二页。**
   - 缩略图不是页面生成状态，不能替代真实页面会话。

2. **只把 `GenerationPromptSummary` 拼到旧 Prompt。**
   - 应同时传递当前 `CoursewareTheme` 中的结构化颜色、字号、字体和安全区。

3. **顺带实现本期之外的资源严格模式。**
   - `MaterialResourceManager` 和 WPF `ResourceId` 解析是后续独立工作，本期不得只做半套注册，也不得扩大渲染器改动。

4. **继续使用固定 1280 × 720。**
   - Prompt、布局、预览和输出会与真实页面尺寸不一致。

5. **把截图放进 UI 集合后就宣称模型看到了截图。**
   - 必须验证流式请求实际包含图片 `DataContent`。

6. **为了获得完整结果而改写现有 `SlideMlRenderTool`。**
   - 完整 Errors 传播和可等待结果应用是已知问题，本期保持现有单页逻辑。

7. **恢复一次性为全部页面创建模型和评估器。**
   - 应使用轻量页面 + 懒运行时。

8. **为主题或页面新增指纹、SessionId 或 RunId。**
   - 本期使用对象生命周期、取消和替换关系管理归属，不增加防御性标识。

9. **新增模型图片能力判断。**
   - 截图存在时默认发送；除非现有复用链路已经判断，否则不新建能力分支。

10. **修改现有流式工具注册机制。**
    - 截图和主题接入不得顺带改变工具白名单、默认工具或 WorkspacePath。

11. **把生成一页解释为全课件页面生成 `Passed`。**
    - 本期只交付真实选中页工作流，不交付全课件自动生成。

12. **在完整 `CoursewareDesignSystem` 未实现时把其 DTO 的存在当作能力完成。**
    - DTO、Agent、验证、模板和下游消费必须分别验收。

13. **把异步流程阻塞成同步。**
    - 禁止 `.Result`、`.Wait()`、`GetAwaiter().GetResult()` 及任何等价阻塞等待；宁可保留受控 fire-and-forget，也不得阻塞 UI 线程。

14. **为设计时另建页面或聊天类型。**
    - 设计时和运行时必须共用正式 ViewModel；无法提供设计样例时宁可放弃设计器预览。

15. **使用含义不清的版本简称。**
    - 不写脱离对象语境的版本编号或方案代号；必须写完整对象名称或先定义清晰简称。

## 十九、完成定义

### Gate A：真实选中页美化可用

- 第二页显示当前输入包的真实页面。
- 至少一页能够在当前主题约束下生成和渲染合法 SlideML。
- Prompt、渲染和 UI 使用页面实际画布。
- 原始截图存在时真正进入首次流式请求；缺失和发送失败状态可见。
- 页面状态切换后保留。
- 原型文案和演示数据退出运行时。
- 设计时和运行时使用同一页面及聊天类型。
- 新增异步调用链不存在异步转同步。
- `ResourceId` 本地渲染、完整 Errors 传播和通用提示词缺陷仍明确标记为后续已知问题。

通过 Gate A 后，可以对外描述为：

> 已接入真实单页美化工作台，可基于当前文本主题建议逐页生成、渲染和调整 SlideML；可执行设计系统、模板和全课件自动生成仍在后续阶段。

### Gate B：后续全课件逐页工作流可靠

- 所有页面均可独立生成、取消、重试和保留结果。
- 串行批处理、整体进度和部分失败恢复可用。
- 结果可以增量保存和恢复。
- 主题重新分析后的既有页面会话迁移已有明确策略。

### Gate C：完整 `CoursewareDesignSystem` 产品化闭环

- 完整 `CoursewareDesignSystem` 真实生成并通过验证。
- 动态页面类型覆盖全部 SlideId。
- 模板编译、资源策略和压力测试通过。
- 全课件视觉分析、跨页质量门和事务提交完成。
- 全部目标页使用当前冻结设计系统并通过页面级和课件级验收。

只有 Gate C 才能对外描述为完整的全课件美化产品闭环。

## 二十、推荐执行顺序

建议按以下顺序开发，避免 UI 先行后再次返工：

1. 建立唯一画布适配器和全异步命令边界。
2. 扩展页面运行时工厂，支持实际画布、取消和懒创建。
3. 修复流式截图附件透传，并保持现有工具机制不变。
4. 新增 `CoursewareTheme` 页面适配器和页面 Prompt Builder。
5. 新增真实工作台与页面 ViewModel。
6. 把旧三栏控件接入新工作台并删除演示内容。
7. 完成单页生成、取消、继续追问、编辑、重新渲染和状态保留。
8. 删除设计时专用演示类型，或改用正式运行时类型构造设计样例。
9. 执行单元测试、XAML 契约测试和最小集成测试。
10. 复查没有混入指纹、SessionId、RunId、能力探测、异步转同步、资源严格模式或通用渲染器改造。

这个顺序优先形成真实、可验证的页面美化闭环，同时严格限制改动面，避免把当前 `CoursewareTheme` 主题建议、现有单页链路或静态工作台误包装为完整产品能力。
