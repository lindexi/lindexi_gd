# CoursewarePptxGeneratorWpfDemo 全课件美化能力设计方案

> 状态：方案设计阶段，尚未开发。
>
> 本文只描述 `CoursewarePptxGeneratorWpfDemo` 的全课件美化落地方案，不包含代码改动。
>
> 双页交互、第一页视觉设计和打开课件后自动分析的详细方案见 `../全课件美化双页交互改造方案.md`。

---

## 一、背景与目标

`CoursewarePptxGeneratorWpfDemo` 当前已经具备以下基础能力：

- 从课件 Markdown 导出目录加载 `Courseware.json`、逐页 Markdown、原始截图和资源索引。
- 在左侧页面列表中展示课件页面，并在中间区域预览原始截图或 SlideML 渲染结果。
- 为每一页创建独立的 `SlideChatManager`、`SlideGenerationPipeline` 和渲染管线。
- 将当前页 Markdown 和原始截图预填到 Copilot 输入区，由用户手工触发单页 SlideML 生成。
- 支持流式生成、SlideML 编辑、重新渲染、本地渲染和 MCP 渲染切换。

当前能力仍然是“用户逐页手工生成”。不同页面之间没有共享主题约束，也没有课件级主题分析、统一提示词和结果保存流程。因此，即使每一页单独生成成功，也难以保证整个课件在配色、字号、字体、版式、安全区和视觉节奏上保持一致。

本方案的目标是在现有输入格式和单页生成能力之上，增加完整的全课件美化流程：

1. 对整个课件进行一次全局主题分析。
2. 将统一主题、相邻页面摘要和资源约束注入每一页的生成提示词。
3. 在单页美化工作台中复用统一主题，逐页生成并渲染 SlideML。
4. 按现有输出格式保存主题、逐页 SlideML、预览图、日志和索引。
5. 为后续页面拆分、一致性校验和最终 PPTX 导出预留扩展点。

## 二、范围定义

### 2.1 首期目标

首期建议实现以下闭环：

1. 用户打开课件 Markdown 导出目录。
2. 程序在课件加载成功后自动执行全局主题分析。
3. 程序生成并展示结构化的全局主题。
4. 用户查看主题产物并进入单页美化工作台。
5. 用户在单页工作台中逐页生成、渲染和调整 SlideML，各页共用同一主题约束。
6. 用户将生成结果保存到独立输出目录。

### 2.2 首期非目标

首期不实现以下能力：

- 不重新截图原始页面，直接使用导出目录中的截图。
- 不自动拆分内容过载页面，只预留拆分决策模型和接入位置。
- 不自动执行跨页一致性校验和返工。
- 不实现高并发逐页生成，第一版默认串行处理。
- 不承诺生成最终 `.pptx` 文件；首期产物仍是 SlideML、预览图和结构化结果目录。
- 不要求用户在打开课件后再次点击“分析主题”；输入包加载成功后自动启动主题分析。
- 不在用户尚未选择课件目录时调用 LLM，也不在输入加载失败时启动主题分析。
- 不覆盖或修改原始课件 Markdown 导出目录。
- 不允许模型通过默认文件系统工具访问任意本地文件。

## 三、现有实现探索结论

### 3.1 项目基础

目标项目：

```text
Pptx/PptxGenerator/Code/CoursewarePptxGeneratorWpfDemo
```

项目当前实际配置：

- 目标框架：`net6.0-windows`
- WPF：已启用
- Nullable reference types：已启用
- C# 语言版本：`latest`
- 项目引用：`PptxGenerator.Wpf`

设计与实现必须以项目文件的实际配置为准，不应沿用旧文档中已经过时的目标框架描述。

### 3.2 课件加载链路

`Services/CoursewareFolderLoader.cs` 已实现课件输入加载：

```text
用户选择目录
  -> MainWindowViewModel.OpenCoursewareFolderAsync
  -> CoursewareFolderLoader.LoadAsync
  -> 读取 Courseware.json
  -> 读取每页 Markdown 和原始截图
  -> 读取 Resources.json
  -> CoursewareInputPackage
  -> 为每页创建 CoursewareSlideItem + 独立 SlideChatManager
  -> Slides 集合
```

当前输入模型已经包含全课件美化所需的大部分静态数据：

- `CoursewareInputPackage`
  - 课件名称
  - 根目录
  - 页面集合
  - 资源集合
  - 加载警告
- `CoursewareSlideInput`
  - 页面索引和页码
  - 页面 ID
  - 页面尺寸
  - Markdown 文件和文本
  - 原始截图文件
  - 页面加载警告
- `CoursewareResourceEntry`
  - 资源 ID
  - 资源类型
  - 导出文件
  - 解析后的本地路径
  - 文件存在状态

因此，全课件美化直接使用当前导出目录作为稳定、可测试的课件输入快照，不重复实现 Markdown、截图和资源提取。

### 3.3 单页生成链路

当前选中页面的生成链路如下：

```text
SelectedSlide
  -> PrepareSelectedSlideCopilotInput
  -> 将原始截图加入 AttachedImageFiles
  -> CreateBeautifyPrompt 组合页面 Markdown 和尺寸
  -> 用户执行 SendMessageCommand
  -> SlideChatManager.SendMessageAsync
  -> SlideGenerationPipeline
  -> LLM 生成 SlideML
  -> SlideMlRenderTool / RenderPipeline
  -> WPF 或 MCP 渲染预览
  -> SlideChatManager 属性通知
  -> MainWindowViewModel 刷新预览、XML 和日志
```

当前 `MainWindowViewModel` 只支持用户对选中页手工发送消息，没有全课件主题和课件级提示词上下文。

### 3.4 页面会话隔离

`SlideChatManagerFactory.CreateAsync` 为每页创建独立对象，包括：

- `CopilotChatManager`
- `SlideGenerationPipeline`
- `SlideMlRenderTool`
- `SwitchableSlideMlRenderPipeline`
- 本地 `SlideMlRenderPipeline`
- `WpfSlideMlRenderEngine`
- 页面评估器和提示词评估器

这已经满足逐页生成最重要的状态隔离要求。后续仍需保持每页独立实例，禁止为了节省对象而共享单个 `SlideChatManager` 或 `SlideGenerationPipeline`。

### 3.5 当前输出契约

`Docs/CoursewareOutputExportFormat.md` 已经定义全课件输出目录，包括：

```text
CoursewareOutput_.../
├── CoursewareOutput.json
├── Theme/
│   ├── Theme.json
│   └── ThemeSummary.md
├── Slides/
│   ├── Slide_000/
│   │   ├── Slide.slideml
│   │   ├── Preview.png
│   │   ├── Generation.md
│   │   ├── RenderLog.txt
│   │   ├── SlideResult.json
│   │   └── SourceMarkdown.md
│   └── ...
└── Summary.md
```

全课件美化实现应直接遵循该格式，不再设计第二套输出结构。

### 3.6 当前关键缺口

现有代码仍缺少：

- 全局主题模型和主题分析服务。
- 统一的课件级提示词构建服务。
- 资源 ID 到 SlideML 渲染资源的完整映射。
- 结果保存服务。
- 一致性校验和页面拆分实现。

## 五、核心数据模型

### 5.1 CoursewareTheme

`CoursewareTheme` 表示全局主题分析的机器可读结果。字段应与 `CoursewareOutputExportFormat.md` 中 `Theme.json` 的定义保持一致，避免主题分析模型和输出模型重复。

建议包含：

- `SchemaVersion`
- `Summary`
- `StyleKeywords`
- `PrimaryColors`
- `AccentColors`
- `BackgroundColors`
- `TextColors`
- `FontSuggestions`
- `FontHierarchy`
- `LayoutPrinciples`
- `CoverLayout`
- `ContentLayout`
- `SafeArea`
- `GenerationPromptSummary`

主题模型需要满足：

- 颜色统一使用 `#RRGGBB` 或 `#AARRGGBB`。
- 字号必须在合理范围内。
- 安全区边距不能为负数，且不能占满页面。
- 所有集合属性默认非 null。
- 使用 `System.Text.Json` 源生成上下文进行序列化。

### 5.2 CoursewareThemeAnalysisResult

除了结构化主题，还应保存本次分析的诊断信息：

- `CoursewareTheme Theme`
- `DateTimeOffset AnalyzedAt`
- `string RawResponse`
- `IReadOnlyList<string> Warnings`
- `int TotalSlideCount`

### 5.3 页面 ViewModel 与输入模型分离

推荐新增 `CoursewareSlideItemViewModel`，内部引用只读的 `CoursewareSlideInput`，并持有当前页的 `SlideChatManager` 和可观察运行状态。

不建议继续扩大 `CoursewareSlideItem`，原因是：

- 输入数据与运行状态的生命周期不同。
- 输出序列化不应直接依赖 UI 模型。
- 页面生成和渲染状态需要通过绑定属性更新。
- 重新生成页面时不应重新构造整个输入模型。

### 5.4 页面拆分预留模型

首期只创建模型和空策略接口，不执行实际拆分：

- `SplitDecision`
  - `NeedsSplit`
  - `Segments`
- `SplitSegment`
  - `Description`
  - `SourceElementNames`

逐页编排中保留拆分评估接入点，但首期始终返回不拆分。

## 六、全局主题分析设计

### 6.1 独立会话

主题分析必须使用专用 `CopilotChatManager`，不能复用任意页面的 `SlideChatManager.Pipeline.ChatManager`。

原因：

- 页面聊天历史不能污染全局主题。
- 主题分析内部消息不应显示在当前页面 Copilot 面板。
- 取消主题分析不应取消当前页聊天。
- 主题分析不需要 SlideML 渲染工具。

主题分析请求建议配置：

- `WithHistory = false`
- `CreateNewSession = true`
- `AppendDefaultTools = false`
- 不注册 `render_slide`、`get_slide_preview` 等页面工具
- 传入当前主题分析操作的 `CancellationToken`
- 使用主题分析专用 System Prompt

### 6.2 结构化结果获取

首期推荐使用工具回调提交结构化主题，而不是只要求模型输出一段自由文本。

主题提交工具负责：

1. 接收配色、字号、字体、安全区、版式和主题摘要。
2. 校验颜色格式、字号范围、安全区和必填字段。
3. 校验失败时返回明确错误，要求模型修正后再次提交。
4. 校验成功后构造 `CoursewareTheme`。

如果当前 Agent API 对复杂工具参数支持不足，可以降级为：

1. 要求模型返回严格 JSON。
2. 使用 `System.Text.Json` 解析。
3. 执行本地模型校验。
4. 最多进行一次 JSON 修复重试。

不应把模型聊天文本作为主题的唯一持久化数据源。

### 6.3 主题分析输入

输入内容建议包括：

- 课件名称。
- 总页数。
- 页面尺寸分布。
- 每页索引、标题和 Markdown 摘要。
- 封面、目录、章节页和结尾页的较完整内容。
- 资源类型和数量统计。
- 课件加载警告摘要。

主题分析不附带全部页面截图。主题分析主要关注学科属性、内容层级和全局设计方向；逐页截图应留到逐页美化时使用。

当前计划以能够一次完整提交给模型的普通课件为边界。大课件的长度限制、摘要、采样和分段归并不在本方案范围内，后续另行设计。

## 七、逐页提示词设计

### 7.1 提示词构建职责

当前 `MainWindowViewModel.CreateBeautifyPrompt` 只包含当前页 Markdown 和尺寸。全课件模式不应继续在 ViewModel 中扩展大段字符串拼接。

建议新增 `ICoursewareSlidePromptBuilder`，统一构建逐页提示词。

输入：

- 当前 `CoursewareSlideInput`
- `CoursewareTheme`
- 全课件页面摘要
- 资源索引
- 页面拆分提示或紧凑排版提示

输出：

- 当前页生成提示词
- 当前页允许附加的截图和资源
- 用于输出日志的提示词摘要

### 7.2 提示词内容

逐页提示词至少包含：

1. **当前任务**
   - 美化第几页。
   - 保持教学语义准确。
   - 输出完整、可渲染的 SlideML。

2. **全局主题**
   - 主色、辅助色、背景色、文字色和强调色。
   - 一级至辅助文本的字号层级。
   - 标题和正文字体建议。
   - 封面页与内容页版式原则。
   - 安全区。

3. **当前页信息**
   - 页面 ID。
   - 页面尺寸。
   - 当前页 Markdown。
   - 原始截图附件。

4. **上下文页面**
   - 前后三页的标题和摘要。
   - 明确这些内容只用于理解章节连续性，不得混入当前页正文。

5. **资源白名单**
   - 当前课件中允许引用的资源 ID。
   - 资源类型和简短描述。
   - 禁止引用白名单之外的本地路径。

6. **生成约束**
   - 内容必须位于安全区内。
   - 不得随意改变主题颜色和字号层级。
   - 不得删除关键教学信息。
   - 图片缺失时应使用布局占位或纯文本方案，不能编造文件路径。

### 7.3 上下文摘要

相邻页面摘要建议使用现有 `CoursewareSlideSummaryService.CreateSummary` 作为第一版基础，后续再根据 Markdown 结构增加章节标题、元素类型和资源引用摘要。

只注入摘要，不注入前后三页完整 Markdown，以控制 Token 并减少相邻页面内容干扰。

## 八、资源处理设计

### 8.1 当前断点

`CoursewareFolderLoader` 已将 `Resources.json` 解析为 `CoursewareResourceEntry`，但这些资源尚未完整接入 SlideML 渲染上下文。

当前 WPF 图片加载仍可能按路径或工作目录查找资源。如果模型只输出资源 ID，而渲染器不知道如何解析，就会出现“提示词允许引用、渲染阶段无法找到”的断链。

### 8.2 资源解析服务

建议新增 `ICoursewareResourceResolver`：

- 根据 `ResourceId` 查找 `CoursewareResourceEntry`。
- 再次校验文件仍位于课件导出目录内。
- 将图片资源注册为 `ImageSlideMlMaterial`。
- 将资源注册到每页独立的 `SlideMlPipelineContext.MaterialResourceManager`。
- 为提示词生成受控资源摘要。
- 缺失资源返回结构化警告，而不是生成任意绝对路径。

### 8.3 安全原则

- 模型只看到资源 ID、类型和必要描述，不看到不必要的本地绝对路径。
- `AppendDefaultTools` 保持关闭。
- 不允许模型读取任意文件。
- 渲染前再次验证资源路径没有越过课件根目录。
- 输出 SlideML 和 JSON 中只写资源 ID或相对路径。

### 8.4 首期资源范围

首期优先完整支持图片资源。音频和视频可以继续保留在资源索引中，但不承诺在 WPF SlideML 预览里完整渲染。

## 九、取消与线程模型

### 9.1 取消令牌生命周期

主题分析和当前页生成分别持有独立的 `CancellationTokenSource`。取消主题分析只终止当前分析，不影响已经加载的课件；取消单页生成只影响当前页，不影响其他页面已有结果。

取消令牌应贯穿对应操作中的模型请求、SlideML 渲染和文件异步写入。

### 9.2 ConfigureAwait 使用原则

与 UI 无关的服务层 I/O 可以使用 `ConfigureAwait(false)`，但必须确认后续代码不访问 WPF 对象或绑定集合。

需要回到 UI 的操作必须显式通过 `IViewModelDispatcher` 或 WPF Dispatcher：

- 修改 `ObservableCollection`。
- 更新页面 ViewModel 的绑定属性。
- 修改 `SelectedSlide`。
- 访问需要 UI 线程的 WPF 图像对象。
- 执行 WPF 测量和渲染。

实施时应对完整异步调用链进行线程审查，不能机械地给所有 `await` 添加 `ConfigureAwait(false)`。

### 9.3 WPF 渲染约束

现有 `SlideMlRenderPipeline` 已将 WPF 测量和渲染分派到主线程，应继续复用该机制。

需要额外确认：

- `WpfPreviewImage.Save` 的调用线程。
- 渲染结果应用到 `SlideMlRenderTool` 的完成时序。
- 保存预览图时是否需要 Dispatcher。
- 本地渲染和 MCP 渲染返回的预览对象是否具有相同线程语义。

保存结果时应优先使用明确返回的渲染结果，不应依赖尚未完成的异步属性回填。

## 十、ViewModel 设计

### 10.1 MainWindowViewModel 新增状态

建议新增：

- `CoursewareInputPackage? LoadedPackage`
- `CoursewareThemeAnalysisResult? ThemeAnalysisResult`
- `CoursewareTheme? Theme`
- `CoursewareWorkspacePage CurrentPage`
- `CoursewareAnalysisStage AnalysisStage`
- `bool IsThemeAnalyzing`
- `bool CanRetryThemeAnalysis`
- `bool CanEnterSlideWorkspace`
- `int TotalSlideCount`
- `DirectoryInfo? OutputDirectory`

不建议继续只依赖一个全局 `IsBusy`。课件加载、主题分析、当前页生成、结果保存和 MCP 工具连接需要不同状态，否则命令启用条件和 UI 提示会互相覆盖。

### 10.2 新增命令

建议新增：

- `RetryThemeAnalysisCommand`
- `EnterSlideWorkspaceCommand`
- `ReturnToThemeOverviewCommand`
- `SaveResultsCommand`
- `OpenOutputFolderCommand`

后续可增加：

- `CheckConsistencyCommand`
- `RegenerateInconsistentSlidesCommand`

### 10.3 手工单页聊天兼容

现有单页 Copilot 能力应继续保留：

- 用户可选择某页后进行手工追问。
- 可以编辑 SlideML 并重新渲染。
- 可以附加额外图片。
- 可以切换模型。

进入第二页后，当前页仍使用该页独立 `SlideChatManager`，允许用户继续迭代。

## 十一、界面设计

### 11.1 MVVM 原则

界面改造继续遵循：

- View 只负责布局、样式、数据模板和绑定。
- code-behind 只保留文件夹选择、文件选择、滚动等平台事件桥接。
- 全课件美化逻辑放在服务层。
- 命令和状态放在 ViewModel。
- 新增控件样式放入独立资源文件，不在页面中复制大量内联样式。

### 11.2 建议界面区域

#### 顶部操作区

双页交互下，打开课件和主题分析过程放在独立的第一页；第二页工作台顶部增加：

- 返回主题概览。
- 查看全局主题详情。
- 保存结果。

按钮启用状态由 ViewModel 控制。

#### 全局主题区

可以在主内容区增加可折叠主题卡片，展示：

- 主题摘要。
- 主要配色。
- 字体建议。
- 字号层级。
- 版式原则。
- 安全区。

首期只读展示，后续可以支持用户修改后确认。

第一页的课件缩略图、只读分析过程、主题产物和页面转场要求见 `../全课件美化双页交互改造方案.md`。

### 11.3 设计时支持

新增 ViewModel 和控件应提供设计时数据或可预览状态，至少覆盖：

- 已加载但未分析主题。
- 主题分析完成。
- 当前页正在生成。
- 当前页生成失败。
- 当前页生成完成。

## 十二、结果保存设计

### 12.1 输出服务

建议新增 `ICoursewareOutputService`，严格遵循 `CoursewareOutputExportFormat.md`。

职责：

- 创建输出目录。
- 保存 `Theme.json` 和 `ThemeSummary.md`。
- 为每页创建稳定的 `Slides/Slide_XXX` 目录。
- 保存 SlideML、预览图、渲染日志和页面结果 JSON。
- 保存可选的源 Markdown 副本。
- 原子更新 `CoursewareOutput.json`。
- 生成 `Summary.md`。

### 12.2 保存时机

建议采用增量保存：

1. 主题分析完成后即可保存主题结果。
2. 每页完成后保存该页结果。
3. 每页保存完成后更新总索引。
4. 应用中断时，已经保存的页面结果仍可读取。

### 12.3 路径和敏感信息

- 输出 JSON 只记录相对路径。
- 不保存模型访问令牌。
- 不保存本地模型配置文件路径。
- 不保存不必要的绝对输入路径。
- 不将课件输入目录中的文件原地修改。

### 12.4 JSON 序列化

所有新增输出模型使用 `System.Text.Json`，并创建对应 `JsonSerializerContext` 源生成上下文。

禁止引入 `Newtonsoft.Json` 或兼容层。

## 十三、页面拆分预留设计

首期不执行页面拆分，只预留：

- `ISlideSplitEvaluator`
- `SplitDecision`
- `SplitSegment`
- 逐页生成前的评估接入点

空实现始终返回不拆分。

后续可以采用两阶段漏斗：

1. CPU 快筛
   - 基于 Markdown 字符数、行数、字号和安全区估算内容密度。
2. LLM 精判
   - 只对疑似过载页面判断是否拆分，并给出分段方案。

未来拆分时应在输入页面与输出页面之间建立稳定映射，而不是修改原始输入集合。建议在输出模型中允许一个输入页对应多个生成页。

## 十四、一致性校验预留设计

一致性校验放在后续阶段。

建议输入：

- 全局主题。
- 全部生成预览图。
- 每页结构化渲染指标。

建议输出：

- 配色一致性评分。
- 字号层级一致性评分。
- 版式一致性评分。
- 问题页面索引。
- 返工建议。

第一版一致性校验只标记问题页，不自动返工。自动返工会引入循环次数、成本上限和结果回退问题，应单独设计。

## 十五、安全与配置

### 15.1 模型配置

当前聊天管理器工厂仍存在本地环境耦合。全课件美化能力上线前，应把以下配置外置：

- Agent 配置来源。
- 默认模型。
- MCP 服务地址。
- 主题分析模型。
- 单页生成模型。

配置来源可以是应用设置、环境变量或用户选择，不能将个人配置路径写入方案、输出文件或新增代码常量。

### 15.2 工具白名单

主题分析：

- 不注册渲染工具。
- `AppendDefaultTools = false`。

逐页生成：

- 只注册 SlideML 渲染和预览所需工具。
- 不启用默认文件系统工具。
- 资源通过受控资源解析器提供。

### 15.3 输入校验

- 继续保留输入目录路径越界防护。
- 资源文件必须位于课件根目录或规定资源目录内。
- 颜色、字号、安全区和布局模型必须本地校验。
- LLM 返回的 SlideML 仍需经过解析和渲染验证。

## 十七、分阶段实施计划

### 阶段一：模型和资源闭环

目标：建立全课件主题模型和资源解析基础。

步骤：

1. 新增主题和拆分预留模型。
2. 为新增 JSON 模型创建源生成上下文。
3. 新增课件资源解析接口。
4. 将 `CoursewareResourceEntry` 注册到每页 SlideML 渲染上下文。
5. 验证图片资源可以通过受控 ID 或相对路径渲染。

验收标准：

- 主题模型可序列化和反序列化。
- 已加载图片资源可被 SlideML 稳定引用。
- 不需要暴露本地绝对路径给模型。

### 阶段二：全局主题分析

目标：完成课件级主题分析并展示结果。

步骤：

1. 新增主题分析专用聊天管理器工厂或配置入口。
2. 实现课件文本快照。
3. 实现主题提交工具或严格 JSON 解析。
4. 实现主题结果本地校验。
5. 在 ViewModel 增加课件加载完成后自动启动主题分析的编排状态。
6. 在独立的课件分析页增加只读分析过程和主题摘要展示区。
7. 增加取消、重试分析和进入单页美化工作台的命令。

验收标准：

- 用户选择有效课件目录后自动开始主题分析，无需再次点击分析按钮。
- 分析完成后停留在课件分析页展示主题产物，由用户进入单页美化工作台。
- 主题分析不污染任意页面聊天历史。
- 主题结果结构化且通过本地校验。

### 阶段三：单页工作台接入全局主题

目标：让现有单页工作台在全局主题约束下逐页生成。

步骤：

1. 实现课件级提示词构建器。
2. 将全局主题、相邻页摘要和资源白名单注入当前页提示词。
3. 保持每页独立 `SlideChatManager` 和渲染管线。
4. 为当前页生成和渲染传递取消令牌。
5. 保留现有单页编辑、重新渲染和继续追问能力。

验收标准：

- 所有页面使用同一全局主题。
- 每页使用独立会话和管线。
- 当前页失败不会破坏其他页面已有状态。
- 第二页整体布局和现有交互尽可能保持不变。

### 阶段四：结果保存

目标：按既定格式保存可继续编辑的完整中间结果。

步骤：

1. 实现输出模型和 JSON 源生成上下文。
2. 实现主题结果保存。
3. 实现逐页 SlideML、预览和日志保存。
4. 实现总索引和摘要保存。
5. 支持逐页增量更新。
6. 增加保存结果和打开输出目录命令。

验收标准：

- 输出目录符合 `CoursewareOutputExportFormat.md`。
- 所有索引使用相对路径。
- 已保存页面可以在应用中断后继续使用。
- 输出不包含敏感配置。

### 阶段五：质量增强

目标：在已有闭环稳定后增加质量能力。

步骤：

1. 测量 LLM 生成和 WPF 渲染耗时。
2. 增加一致性检查。
3. 增加页面内容密度快筛。
4. 增加拆分精判和多输出页支持。

验收标准：

- UI 线程保持可响应。
- 结果顺序稳定。
- 一致性问题可定位到具体页面。

## 十九、风险与开放问题

### 19.1 资源注册尚未形成闭环

输入资源已经加载，但渲染器未必能直接根据资源 ID 定位图片。这是逐页生成前必须优先验证和解决的问题。

### 19.2 全局主题与单页模型偏离

单页模型可能忽略主题约束。后续可在页面评估中增加“主题一致性”维度，但首期先通过明确提示词和结构化主题约束解决。

### 19.3 页面状态模型迁移

当前页面项属性多为初始化属性。引入可观察状态时需要决定是改造现有类型，还是新增页面 ViewModel。推荐新增 ViewModel，避免输入模型、UI 状态和输出模型混合。

### 19.4 主题结果是否允许用户编辑

首期建议只读展示并提供“重新分析”。后续如允许用户修改主题，需要增加主题确认、验证和版本管理。

### 19.5 MCP 工具可用性

MCP 工具不是主题分析的前置条件。单页工作台继续使用现有的 MCP 工具切换与本地回退能力，不在本方案中展开 MCP 的具体实现。

### 19.6 最终 PPTX 产物

当前已确认的是 SlideML 生成和预览渲染链路。最终多页 `.pptx` 合并、母版、字体嵌入和资源打包不属于本方案首期范围，需要在现有 SlideML 结果稳定后单独探索。

## 二十、推荐决策

综合当前代码和风险，建议采用以下落地决策：

1. 以 `CoursewareInputPackage` 作为全课件美化输入边界。
2. 保持每页独立 `SlideChatManager` 和渲染管线。
3. 使用独立 `CopilotChatManager` 执行全局主题分析。
4. 第二页继续沿用现有单页工作台，由用户逐页生成和调整。
5. 在逐页生成之前先完成资源 ID 到渲染上下文的闭环。
6. 将课件级提示词构建从 `MainWindowViewModel` 移到独立服务。
7. 使用强类型页面和分析状态，避免继续使用自由字符串承载流程状态。
8. 各项异步操作传递对应的 `CancellationToken`，UI 更新显式经过 Dispatcher。
9. 输出严格遵循现有 `CoursewareOutputExportFormat.md`。
10. 大课件、页面拆分、一致性自动返工和最终 PPTX 导出后置。

按此方案实施，可以复用当前 Demo 已有的输入加载、单页会话、SlideML 生成以及 WPF/MCP 工具渲染能力，形成可测试、可取消、可保存、可逐步扩展的全课件美化闭环。
