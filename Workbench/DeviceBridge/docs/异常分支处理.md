# DeviceBridge 异常分支处理

## 一、扫码 & 解析阶段

| # | 异常场景 | 影响 | 处理策略 |
|---|---|---|---|
| 1.1 | 二维码过期（code 超过 90s TTL） | Agent 拿到过期 code，连接请求被拒 | 服务器返回 `code_expired`，Agent 提示"二维码已过期，请重新扫描" |
| 1.2 | 二维码被遮挡、损坏、无法识别 | 用户扫不出任何内容 | 二维码旁加文字提示"如无法扫描，请联系管理员" |
| 1.3 | Agent 解析二维码文本失败 | 提取不到 deviceId 或 code | 二维码采用固定格式，首行标记 `【设备连接码】`。Agent 判读是否同时含 `设备ID`、`连接码`、`服务器` 三个关键词，缺一则提示"无法识别此二维码，请确认扫描的是设备连接码" |
| 1.4 | 用户误扫其他二维码 | Agent 拿到不相关内容 | 同上，缺关键词则提示"这不是设备连接码" |
| 1.5 | 二维码中 server 地址与 Agent 已知不一致 | 可能是旧版设备或测试设备 | Agent 优先使用二维码中自描述的 server 地址，不硬编码 |

## 二、连接请求阶段

| # | 异常场景 | 影响 | 处理策略 |
|---|---|---|---|
| 2.1 | code 已过期 | 连接请求被拒绝 | 服务器返回 `code_expired`，Agent 引导重新扫码 |
| 2.2 | deviceId 不存在 | 设备从未注册或已注销 | 服务器返回 `device_not_found`，Agent 提示"未找到该设备" |
| 2.3 | 设备离线（超过 N 个 refresh 周期未上报） | 设备不可达 | 服务器标记设备离线（如 3 个周期 = 180s 无心跳），返回 `device_offline` + 最后在线时间。Agent 提示"设备当前离线，最后在线于 xx 分钟前" |
| 2.4 | 同一设备同时收到多个连接请求 | 多个用户先后或同时扫码 | 设备端维护请求队列，按先后顺序处理。当前确认框未关闭时，后续请求排队。Agent 收到 `queued` 状态，提示"前面还有 N 个请求，请稍候" |
| 2.5 | 同一用户对同一设备重复扫码 | 重复提交连接请求 | 服务器按 (userId, deviceId) 去重：若已有 pending 请求，返回现有请求状态，不创建新请求 |
| 2.6 | 服务器过载 | 无法处理新请求 | 返回 HTTP 503 + Retry-After 头，Agent 自动重试或提示稍后再试 |

## 三、设备确认阶段

| # | 异常场景 | 影响 | 处理策略 |
|---|---|---|---|
| 3.1 | 确认框超时（用户 120s 内不点击） | 连接请求作废 | 设备端确认框 120s 倒计时，超时自动关闭并通知服务器。服务器返回 `request_timeout` 给 Agent |
| 3.2 | 用户点"拒绝" | 连接被拒绝 | 服务器返回 `rejected`，Agent 提示"设备拒绝了连接请求" |
| 3.3 | 确认过程中 code 刷新为新值 | 请求与 code 生命周期解耦 | requestId 独立于 code 生命周期。只要 request 在有效期内（120s），就继续等待确认。code 刷新不影响已创建的 request |
| 3.4 | 设备确认过程中离线或重启 | 确认框消失，请求悬空 | 设备离线时服务器检测到，自动将该设备所有 pending request 置为 `cancelled`。Agent 收到后提示"设备已断开，请稍后重试" |
| 3.5 | 恶意频繁扫码骚扰 | 设备确认框不断弹出 | 频率限制：同一 deviceId 每 30s 最多创建 3 个 request。超出返回 `too_many_requests`。设备端也可设置"屏蔽连接请求 5 分钟"选项 |

## 四、会话阶段

| # | 异常场景 | 影响 | 处理策略 |
|---|---|---|---|
| 4.1 | WebSocket 断开（设备断网） | 指令无法下达 | 服务器检测断开后通知 Agent `session_disconnected`。Agent 提示"设备连接中断"。设备恢复后自动回到空闲态（HTTP refresh），需重新扫码连接 |
| 4.2 | WebSocket 断开（服务器侧） | 同上 | 同上。设备端检测断开后回退到 HTTP refresh 模式，不丢失状态 |
| 4.3 | 设备执行指令超时（如 30s 无响应） | Agent 无限等待 | 服务器设置指令超时 30s，超时返回 `command_timeout`。Agent 提示"指令执行超时，设备可能正忙"，并给用户重试选项 |
| 4.4 | 设备执行指令失败 | 操作未完成 | 设备返回失败原因（如"权限不足"、"应用未安装"、"设备正忙"）。Agent 将原因转述给用户 |
| 4.5 | 会话闲置（Agent 长时间不发指令） | 占用设备连接资源和确认框 | 服务器设置会话闲置超时（10 分钟），超时后自动断开 WebSocket。设备回到空闲态。Agent 收到 `session_idle_timeout` |
| 4.6 | WebSocket 升级失败 | 无法进入实时通信模式 | 回退到 HTTP 轮询：Agent 发指令用 HTTP POST，设备在 refresh 响应中带回结果。延迟增加但功能可用。用户感知不明显（交互结果在设备屏幕上呈现） |
| 4.7 | 同一设备被第二个用户尝试连接 | 操控冲突 | 同一时间只允许一个活跃会话。新请求返回 `device_busy` + 当前操控者名称 |

## 五、设备侧异常

| # | 异常场景 | 影响 | 处理策略 |
|---|---|---|---|
| 5.1 | 设备系统时间不准 | code 刷新周期错乱，code 在服务器侧提前或延迟过期 | 服务器以服务器时间为准校验 code 有效期。设备上报时携带设备时间戳，服务器计算偏差并记录告警，不依赖设备时间做判断 |
| 5.2 | 设备端程序崩溃 | 二维码消失，设备失联 | 守护进程或系统服务自动重启。崩溃时回到未连接态。重启后重新注册到服务器，生成新 code |
| 5.3 | 设备系统重启 | 短暂失联后恢复 | 重启后自动启动连接器，重新注册。服务器检测到同一 deviceId 重连，清理旧状态 |
| 5.4 | 设备网络切换（WiFi → 有线 → 4G） | 短暂断连 | refresh 请求自带重试机制（3 次，指数退避）。切换后 IP 变化对本架构无影响（设备以 deviceId 标识，不依赖 IP） |
| 5.5 | 设备存储空间不足 | 无法写日志或缓存 | 程序自身做最低存储保护，日志轮转，超阈值只写关键错误，不影响核心功能 |

## 六、服务器侧异常

| # | 异常场景 | 影响 | 处理策略 |
|---|---|---|---|
| 6.1 | 服务器重启或滚动更新 | 短时间内设备 refresh 失败 | 设备端重试机制（3 次，指数退避）。Kubernetes 滚动更新保证始终有实例在线。重启后 Code 映射表从 Redis 恢复 |
| 6.2 | 某服务器实例故障 | 部分设备连接断开 | 无状态设计：设备重连到其他健康实例。Code 映射存 Redis 集群，任意实例均可处理 |
| 6.3 | Redis 或缓存故障 | Code 映射丢失 | Redis 哨兵/集群保证高可用。降级方案：设备 refresh 响应中标记 `recovery_mode`，设备缩短 refresh 间隔到 10s 加速恢复 |
| 6.4 | Code 映射脏数据（过期 code 未清理） | 内存或存储浪费 | Code 使用 Redis key TTL（90s）自动过期。同时定时任务兜底清理 |
| 6.5 | 大规模断网恢复（千万设备同时重连） | 流量冲击（惊群效应） | 设备端加随机抖动（0-30s）。服务器端限流 + 排队。Kubernetes HPA 自动扩容 |

## 七、安全相关

| # | 异常场景 | 影响 | 处理策略 |
|---|---|---|---|
| 7.1 | 重放攻击（用旧 code 重放） | 冒用已过期授权 | Code 一次性使用：服务器校验后立即标记为已使用。Code 本身有 TTL（90s）。两者结合，重放必失败 |
| 7.2 | deviceId 枚举攻击 | 恶意扫描设备列表 | 无设备列表查询接口。`request_connection` 必须携带有效 code（GUID，不可猜测）。找不到设备时统一返回 `device_not_found`，不泄露设备是否存在 |
| 7.3 | 伪造设备向服务器上报 | 注入虚假设备 | 设备预置客户端证书（mTLS），服务器在 TLS 握手阶段验证 |
| 7.4 | 中间人攻击 | 窃听或篡改指令 | 全链路 TLS。mTLS 用于设备侧，标准 TLS 用于 Agent 侧 |
| 7.5 | 敏感指令（系统重置、卸载应用等） | 恶意指令破坏设备 | Agent.md 中标记敏感操作，执行前设备端二次弹窗确认。非敏感操作直接执行无需确认 |
