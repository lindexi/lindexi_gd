## 一、测试概述

### 1.1 测试目标

验证「图片查看器」是否符合需求 v1.1 中 14 条验收标准（AC-01 ~ AC-13）以及 UX/UI 设计的交互一致性，确保产品质量零缺陷交付。

### 1.2 测试范围

#### 1.2.1 包含范围（IN-SCOPE）

| 维度 | 具体内容 |
|---|---|
| 核心功能 | 打开、缩放、旋转、全屏、拖拽平移、目录翻看、图片信息显示 |
| 系统集成 | 文件关联、右键菜单、单实例、窗口管理 |
| 性能 | 冷启动、加载、切换、缩放帧率、内存占用 |
| 兼容性 | Windows 10/11 × Linux Ubuntu/Debian/Fedora 最小覆盖矩阵 |
| 安全性 | 离线运行、原文件保护、关闭行为 |
| 易用性 | 键盘快捷键、无二次确认、空/错状态文案 |
| UI 一致性 | 10 个 UI 状态全覆盖 + 视觉规范校验 + 装饰元素取色 |

#### 1.2.2 不包含范围（OUT-OF-SCOPE）

| 排除项 | 理由 |
|---|---|
| macOS 平台 | v1.0 明确不含，v2.0 候选（见 §8） |
| 命令行参数 | v1.0 不支持 |
| 编辑功能（裁剪/调色/滤镜） | v1.0 不在范围内 |
| 删除文件功能 | v1.0 不在范围内（§2.2） |
| 社交分享、云同步 | v1.0 不在范围内 |
| 幻灯片自动播放 | v1.0 不在范围内 |
| EXIF 信息查看 | v1.0 不在范围内 |
| 打印功能 | v1.0 不在范围内 |
| 任何二次确认对话框 | v1.0 不提供（§2.2 / AC-13） |

| 维度 | 策略 | 优先级 |

## 三、测试环境要求

### 3.1 硬件环境

| 平台 | 最低配置 | 推荐配置 |
|---|---|---|
| Windows | Intel i5 / 8GB RAM / 1080p | Intel i7 / 16GB RAM / 1440p |

### 3.2 软件环境

| 平台 | 操作系统 | 桌面环境 |
|---|---|---|
| Windows 测试机 A | Win10 21H2 x64 | 默认 |

### 3.3 网络环境

- 完全离线测试（验证"无网络请求"约束，DevTools 抓包验证）
- 如需下载样图，需在外网完成准备，测试时离线

### 3.4 测试数据集（响应 R6 风险）

| 类别 | 数量 | 说明 | 用途 |

## 五、专项测试要求

### 5.1 功能测试

按 §4.1 模块矩阵执行，每个模块的正常/异常/边界全覆盖。详细用例见 `02_功能模块用例.md`。

### 5.2 接口测试

| 接口 | 测试方法 | 关联 AC |
|---|---|---|
| ImageShape 解码 | 传入正常/损坏文件，验证返回与异常处理 | AC-02 |
| 平台右键菜单注册 | Windows 注册表 + Linux `.desktop` 文件 | AC-10 |

| 系统文件关联 | 双击 .jpg / .png 等触发本应用 | AC-10 |

### 5.3 性能测试（按 v1.1 §5.1）

| 指标 | 测试方法 | 阈值 |
|---|---|---|
| 冷启动速度 | 重启机器后冷启动 → 计时到第一帧图片渲染 | ≤ 1.5s |
| 20MB 图加载 | 计时从打开到完整渲染 | ≤ 500ms |
| 翻看切换延迟 | 同目录两张图间切换 | ≤ 300ms |
| 缩放帧率 | 录屏分析缩放过程帧率 | ≥ 50fps（无明显卡顿） |
| 内存占用（空闲） | 应用空闲时内存 | ≤ 80MB |
| 内存占用（开图后） | 打开 20MB 图片后内存 | ≤ 200MB |

> 每项指标取 5 次中位数，避免单次波动误判（R10 风险应对）。

### 5.4 兼容性测试

按 v1.1 §5.2.1 最小覆盖矩阵执行，重点关注：

| 关注点 | Win 测试点 | Linux 测试点 |

| 字体回退 | 无微软雅黑时回退 | 无 Noto Sans CJK 时回退 |
| HiDPI 缩放 | 100% / 150% / 200% | 100% / 150% / 200% |
| 快捷键差异 | `Alt+F4` 退出 | `Ctrl+Q` 退出 |
| 主题跟随 | 跟随系统亮/暗主题 | 跟随系统亮/暗主题 |
| 文件关联 | 默认关联生效 | MIME 类型注册生效 |
| 右键菜单 | 入口可用 | `.desktop` 文件可用 |

### 5.5 安全性测试

| 测试项 | 方法 | 关联 AC |

## 六、缺陷管理流程

### 6.1 严重等级定义

| 等级 | 定义 | 修复 SLA | 准入发布 |
|---|---|---|---|
| **P0-致命** | 主流程阻塞、数据丢失、安全漏洞、安装失败 | 立即 | **不准入** |
| **P1-严重** | 核心功能不可用，但有 workaround | 24h | **不准入** |
| **P2-一般** | 非核心功能异常、UI 错位、文本错误 | 3天 | ≤ 2 个准入 |
| **P3-轻微** | 优化建议、提示文案瑕疵 | 下版本 | ≤ 5 个准入 |
| **P4-建议** | 体验改进、长期优化 | 不限期 | 不计入 |

### 6.2 缺陷单规范

```
【缺陷标题】一句话描述问题
【所属模块】M1~M13
【严重等级】P0/P1/P2/P3/P4
【优先级】  P0=立即修；P1=24h内；P2=本轮修；P3=下轮修；P4=不限
【关联 AC】AC-01a 等（多个用逗号分隔）
【环境】    Win10 21H2 / Ubuntu 22.04 / ...
【复现步骤】1. ... 2. ... 3. ...
【预期结果】
【实际结果】
【影响范围】影响哪些功能/用户
【截图/录屏】占位（必须附图）
【日志】占位（如有）
【备注】其他补充信息
```

### 6.3 回归验证流程

1. **开发标记修复完成** → 通知测试
2. **测试执行"原缺陷复现步骤"** → 验证是否已修
3. **同时执行该模块全部 P0/P1 用例** → 验证无回归
4. **若仍存在或引入新缺陷** → 重新打开并升级严重等级
5. **验证通过后** → 关闭缺陷
